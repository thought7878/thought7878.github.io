>前端常见攻击与防护详细总结

前端安全是 Web 安全体系的第一道防线。虽然许多漏洞需要前后端协同防御，但前端在**数据渲染、用户交互、资源加载、状态管理**等环节承担着不可替代的安全责任。以下按攻击类型、原理、场景、防护方案进行系统化梳理，并附现代浏览器安全机制与工程化最佳实践。

---

## 一、核心攻击与防护详解

| 攻击类型                      | 原理简述                              | 典型场景                                                | 前端防护重点                                | 后端/全栈协同                                 |
| ------------------------- | --------------------------------- | --------------------------------------------------- | ------------------------------------- | --------------------------------------- |
| **XSS（跨站脚本）**             | 恶意脚本注入页面并在用户浏览器执行                 | *评论区*、搜索框、URL参数、富文本编辑器                              | 上下文转义、禁用危险API、CSP、DOMPurify           | HttpOnly Cookie、输入校验、输出编码、CSP头          |
| **CSRF（跨站请求伪造）**          | 诱导已登录用户发起非自愿的跨站请求                 | 转账、改密、删除数据、关注/点赞                                    | SameSite Cookie、自定义请求头、Token携带        | CSRF Token、验证 Origin/Referer、敏感操作二次确认   |
| **Clickjacking（点击劫持）**    | 透明 iframe 覆盖正常 UI，诱导误点击           | 授权按钮、支付确认、社交分享                                      | CSP `frame-ancestors`、X-Frame-Options | 服务端响应头控制、关键操作增加滑动/验证码                   |
| **开放重定向（Open Redirect）**  | 未校验跳转目标，被用于钓鱼或绕过认证                | `?redirect=evil.com`、OAuth回调、登录后跳转                  | 白名单校验、仅允许相对路径、外部跳转二次确认                | 服务端重定向参数严格校验、禁止开放回调                     |
| **敏感数据泄露/不安全存储**          | Token/密码/PII 存储在易被读取的位置           | JWT 存 localStorage、调试信息未清理、日志打印                     | 内存存储短期 Token、HttpOnly Cookie、清理敏感数据   | 短效 Access Token + 长效 Refresh Token、加密传输 |
| **供应链攻击**                 | 第三方包/CDN/插件被篡改或含恶意代码              | npm 依赖投毒、公共 CDN 脚本、浏览器扩展                            | SRI 校验、锁定依赖、定期审计、最小化第三方脚本             | 私有仓库、发布签名、CI/CD 安全扫描                    |
| **原型污染 / DOM Clobbering** | 恶意输入修改 `Object.prototype` 或覆盖全局变量 | 递归合并配置、`id`/`name` 与 JS 变量冲突                        | 安全合并逻辑、`Object.hasOwn`、避免全局依赖、严格模式    | 输入结构校验、JSON 解析沙箱、服务端同样防护                |
| **CORS 配置错误**             | 跨域策略过于宽松，导致数据越权读取                 | `Allow-Origin: *` + `credentials: true`、盲目反射 Origin | 不依赖前端绕过、严格携带凭证策略                      | 源白名单校验、禁止 `*` 与凭证共存、限制暴露头               |

## 二、重点攻击深度解析与代码级防护

### 🔴 1. XSS（跨站脚本攻击）
**分类**：存储型、反射型、DOM 型  
**前端高危 API**：`innerHTML`、`document.write()`、`eval()`、`setTimeout(string)`、`v-html`、`dangerouslySetInnerHTML`

✅ **防护方案**：
```js
// ❌ 危险
div.innerHTML = userInput;

// ✅ 安全替代
div.textContent = userInput; // 纯文本
// 或富文本场景使用 DOMPurify
import DOMPurify from 'dompurify';
div.innerHTML = DOMPurify.sanitize(userInput, { USE_PROFILES: { html: true } });
```
- **框架默认防护**：React/Vue/Angular 默认转义插值表达式，但显式使用危险 API 会绕过。
- **CSP 强化**：
  ```http
  Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-abc123'; object-src 'none';
  ```
  配合 `nonce` 或 `hash` 可彻底阻断内联脚本与外部注入。

---

### 🔴 2. CSRF（跨站请求伪造）
⚠️ **注意**：同源策略**不防御 CSRF**（请求会正常发出且携带 Cookie）。前端只能辅助，核心在服务端。

✅ **前端防护方案**：
1. **SameSite Cookie**（现代浏览器默认 `Lax`）：
   ```http
   Set-Cookie: session=xyz; SameSite=Strict; Secure; HttpOnly
   ```
2. **自定义请求头**（利用 CORS 预检拦截跨域）：
   ```js
   fetch('/api/transfer', {
     method: 'POST',
     headers: { 'X-CSRF-Token': csrfToken } // 跨域时会触发 OPTIONS 预检，恶意站无法伪造
   });
   ```
3. **双重提交 Cookie**：前端读取 Cookie 中的 CSRF Token 放入请求头/Body，服务端比对。

---

### 🔴 3. 敏感数据存储策略
| 存储方式 | 是否受 XSS 影响 | 是否自动携带 | 适用场景 |
|----------|----------------|--------------|----------|
| `localStorage` / `sessionStorage` | ✅ 极易被读取 | ❌ 需手动附加 | 非敏感缓存、UI 状态 |
| `Cookie`（无 HttpOnly） | ✅ 可被 JS 读取 | ✅ 自动携带 | 不推荐存 Token |
| `Cookie`（HttpOnly + Secure + SameSite） | ❌ JS 不可读 | ✅ 自动携带 | **Session / Refresh Token 首选** |
| 内存变量（闭包/Store） | ❌ 页面刷新丢失 | ❌ 需手动附加 | **Access Token 短期存储** |

✅ **推荐架构**：
```
前端内存存 Access Token（5-15分钟） 
→ 过期时用 HttpOnly Refresh Token 静默续期 
→ 登出时调用后端销毁 Refresh Token
```

---

### 🔴 4. 供应链与第三方资源安全
✅ **防护清单**：
- **锁定依赖**：提交 `package-lock.json` / `pnpm-lock.yaml`，禁止 `^` / `~` 浮动版本上生产。
- **定期审计**：`npm audit`、`pnpm audit`、GitHub Dependabot、Snyk。
- **SRI（子资源完整性）**：
  ```html
  <script src="https://cdn.example.com/lib.js"
          integrity="sha384-oqVuAfXRKap7fdgcCY5uykM6+R9GqQ8K/uxy9rx7HNQlGYl1kPzQho1wx4JwY8wC"
          crossorigin="anonymous"></script>
  ```
- **最小化第三方脚本**：异步加载、`sandbox` iframe 隔离、监控异常网络请求。

---

### 🔴 5. 原型污染 & DOM Clobbering
**原型污染示例**：
```js
// 危险：递归合并未过滤的输入
function merge(target, source) {
  for (let key in source) {
    if (typeof source[key] === 'object') merge(target[key], source[key]);
    else target[key] = source[key];
  }
}
merge({}, JSON.parse('{"__proto__":{"isAdmin":true}}'));
console.log({}.isAdmin); // true
```
✅ **防护**：
- 使用 `Object.hasOwn(obj, key)` 替代 `for...in`
- 合并前过滤 `__proto__`、`constructor`、`prototype`
- 使用安全库：`lodash.merge`（v4.17.21+）、`deepmerge`（配置 `isMergeableObject`）
- 冻结原型：`Object.freeze(Object.prototype)`（谨慎使用）

**DOM Clobbering**：HTML 中 `<a id="config">` 会覆盖 `window.config`。  
✅ 防护：避免全局变量依赖、使用 `document.getElementById()`、启用严格模式。

## 三、现代浏览器安全机制协同

| 机制 | 作用 | 配置示例 | 与前端关系 |
|------|------|----------|------------|
| **CSP**（内容安全策略） | 限制资源加载与脚本执行源 | `script-src 'self' 'nonce-xxx'; frame-ancestors 'none';` | 前端需配合 nonce/hash，避免内联脚本 |
| **SRI**（子资源完整性） | 校验 CDN/外部脚本哈希 | `integrity="sha384-..."` | 前端构建时自动生成 hash |
| **SameSite Cookie** | 控制 Cookie 跨站发送 | `SameSite=Strict/Lax/None` | 前端登录态管理需适配 |
| **Trusted Types** | 强制 DOM 操作使用安全类型 | `require-trusted-types-for 'script'` | 需改造 `innerHTML` 等调用为策略函数 |
| **COOP / COEP** | 跨源隔离，启用高性能 API | `Cross-Origin-Opener-Policy: same-origin`<br>`Cross-Origin-Embedder-Policy: require-corp` | 嵌入第三方资源需显式授权 |
| **Fetch Metadata Headers** | 浏览器自动携带请求意图 | `Sec-Fetch-Dest: document`<br>`Sec-Fetch-Mode: navigate` | 服务端可据此拦截异常请求 |

## 四、前端安全开发最佳实践

### ✅ 编码规范
1. **永远不信任用户输入**：URL 参数、表单、富文本、第三方回调均需校验/转义。
2. **禁用危险 API**：全局搜索 `eval`、`innerHTML`、`document.write`、`setTimeout(string)` 并替换。
3. **框架安全模式**：关闭 `v-html`/`dangerouslySetInnerHTML` 或强制走 sanitize 管道。
4. **错误处理不泄露信息**：生产环境隐藏堆栈、API 详情、内部路径。

### ✅ 架构与工程化
1. **安全头自动化**：通过 Nginx / Vercel / Cloudflare 注入 CSP、X-Frame-Options、HSTS 等。
2. **CI/CD 安全门禁**：集成 `npm audit`、ESLint 安全规则（`eslint-plugin-security`）、SAST 扫描。
3. **依赖治理**：定期更新、锁定版本、使用私有 Registry、审查高下载量包的维护状态。
4. **监控与告警**：捕获 `window.onerror`、CSP `report-uri`、异常 fetch 请求、可疑 localStorage 读写。

### ✅ 用户交互安全
1. 敏感操作（支付、改密、解绑）增加二次确认/验证码/生物识别。
2. 外部链接添加 `rel="noopener noreferrer"` 防 `window.opener` 钓鱼。
3. 文件上传限制类型/大小，前端预览不执行（如 PDF/HTML 需沙箱渲染）。

## 五、总结：纵深防御（Defense in Depth）

前端安全不是单点问题，而是**多层策略叠加**的结果：

```
用户输入 → 前端校验/转义 → 安全传输(HTTPS) → 服务端校验/鉴权 → 安全存储 → 安全渲染 → 浏览器策略拦截
          ↑框架默认防护     ↑HSTS/TLS1.3       ↑CSRF Token/CORS    ↑HttpOnly/SameSite ↑CSP/SRI/COEP
```

🔑 **核心原则**：
1. **最小权限**：只加载必要资源，只暴露必要接口，只存储必要数据。
2. **默认拒绝**：CSP 默认 `default-src 'none'`，CORS 默认不开放，Cookie 默认 `SameSite=Lax`。
3. **不依赖单一防线**：XSS 防护 = 转义 + CSP + HttpOnly；CSRF 防护 = SameSite + Token + 验证 Origin。
4. **安全左移**：在开发、构建、测试阶段拦截漏洞，而非依赖运行时修补。
