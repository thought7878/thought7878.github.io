`同源策略（Same-Origin Policy, SOP）`是 Web *浏览器最核心、最基础的安全机制之一*。它由 Netscape 于 1995 年首次引入，如今已被所有现代浏览器默认强制执行。理解同源策略，是掌握前端安全、跨域通信与 Web 架构设计的前提。

---

## 一、什么是同源策略？

**同源策略**规定：**不同源的客户端脚本，在未获得明确授权的情况下，不能读写对方的资源**。  
它由浏览器在客户端强制执行，服务器端无法直接关闭或绕过（除非主动配合放松限制）。

> 💡 核心思想：隔离不同来源的网页环境，防止恶意脚本窃取用户数据或越权操作。

---

## 二、“同源”的判定标准

两个 URL **同源**，当且仅当以下*三项完全一致：*

| 组成部分           | 说明                          | 示例                                                |
| -------------- | --------------------------- | ------------------------------------------------- |
| **协议（Scheme）** | `http` / `https` / `file` 等 | `http://` ≠ `https://`                            |
| **域名（Host）**   | 完整主机名，含子域名                  | `a.example.com` ≠ `b.example.com` ≠ `example.com` |
| **端口（Port）**   | 显式或默认端口                     | `:80`（http默认）≠ `:8080`；`:443`（https默认）可省略         |

✅ **同源示例**  
`https://example.com/page1` 与 `https://example.com:443/api` → 同源（端口默认一致）

❌ **不同源示例**
- `http://example.com` vs `https://example.com`（协议不同）
- `https://api.example.com` vs `https://example.com`（子域名不同）
- `https://example.com:3000` vs `https://example.com`（端口不同）
- `https://192.168.1.1` vs `https://example.com`（IP 与域名视为不同源）

> 📌 注意：**路径、查询参数、哈希值不影响同源判定**。  
> 例如 `https://example.com/a` 与 `https://example.com/b?x=1#top` 仍属同源。

## 三、同源策略限制的内容

同源策略*主要限制以下三类交互：*

| 限制类型 | 具体表现 |
|----------|----------|
| **DOM 访问** | 不同源的 `window`/`iframe` 无法直接读写对方 DOM（如 `iframe.contentWindow.document` 会抛 `SecurityError`） |
| **网络请求** | `XMLHttpRequest` / `fetch()` 默认禁止跨域读取响应（请求仍会发出，但浏览器拦截响应） |
| **客户端存储** | `Cookie`、`LocalStorage`、`SessionStorage`、`IndexedDB` 严格按源隔离，跨源不可见 |

> ⚠️ 常见误区：同源策略**不会阻止请求发送到服务器**，而是*阻止浏览器将响应数据暴露给前端脚本*。服务器*仍会收到请求并处理（这也是 CSRF 攻击能发生的原因）*。

## 四、为什么需要同源策略？（安全意义）

若无同源策略，恶意网站可轻易实施以下攻击：

1. **数据窃取**  
   `evil.com` 的 JS 执行 `fetch('https://bank.com/api/balance')`，直接读取用户账户余额。
2. **会话劫持**  
   读取其他站点的 `Cookie` 或 `LocalStorage` 中的 Token，冒充用户身份。
3. **DOM 篡改**  
   通过跨域 iframe 修改银行支付页面的表单内容，诱导用户转账。

*同源策略与 `SameSite Cookie`、`CSRF Token`、`CSP` 等机制共同构成现代 Web 的纵深防御体系*。

## 五、如何**合法跨域**？（标准解决方案）

| 方案                       | 原理                                 | 适用场景          | 注意事项                               |
| ------------------------ | ---------------------------------- | ------------- | ---------------------------------- |
| `CORS（跨域资源共享）`           | 服务端*通过响应头授权特定源访问*                  | 现代 API 跨域首选   | 需服务端配置；带凭证时 `Allow-Origin` 不能用 `*` |
| **postMessage**          | 窗口间安全通信 API，可指定 `targetOrigin`     | iframe/弹窗跨域通信 | 接收方需验证 `event.origin` 防伪造          |
| **反向代理/代理服务器**           | *开发服务器或 Nginx 将同源请求转为跨域请求*         | 前后端分离项目       | 浏览器始终认为请求同源，服务端需处理真实跨域             |
| **WebSocket**            | 协议层不受 SOP 限制                       | 实时双向通信        | 仍需服务端鉴权，防未授权连接                     |
| **JSONP**（历史方案）          | *利用 `<script>` 可跨域加载，回调函数参数保存返回数据* | 老旧系统兼容        | *仅支持 GET；易受 XSS 攻击；已淘汰*            |
| **document.domain**（已废弃） | 主域相同时可降级为同域                        | 子域名通信         | Chrome 106+/Firefox 109+ 已禁用，不推荐   |

### 🔍 CORS 关键机制补充
- **简单请求**：`GET/POST/HEAD` + 安全头（如 `Content-Type: application/x-www-form-urlencoded`），浏览器直接发送。
- **预检请求（Preflight）**：非简单请求（如 `PUT`、自定义头、`application/json`）会先发送 `OPTIONS` 请求，服务端需返回：
  ```http
  Access-Control-Allow-Origin: https://frontend.com
  Access-Control-Allow-Methods: POST, PUT
  Access-Control-Allow-Headers: Content-Type, Authorization
  Access-Control-Allow-Credentials: true  # 若前端 fetch(credentials: 'include')
  ```
- **凭证模式**：`fetch(url, { credentials: 'include' })` 会携带 Cookie，此时服务端 `Allow-Origin` 必须为具体源，且需返回 `Allow-Credentials: true`。

## 六、同源策略的 **“例外”** 与资源加载规则

浏览器允许以下标签**跨域加载资源**，但 JS **无法读取内容**（仅能执行或渲染）：

| 标签                            | 行为       | 安全限制                                                                   |
| ----------------------------- | -------- | ---------------------------------------------------------------------- |
| `<script src="...">`          | 可跨域加载并执行 | 受 CORS 或 SRI 限制；错误信息不暴露详情                                              |
| `<img>`, `<video>`, `<audio>` | 可跨域加载渲染  | JS 无法通过 Canvas `drawImage` 读取像素（除非服务端返回 `Access-Control-Allow-Origin`） |
| `<link rel="stylesheet">`     | 可跨域加载样式  | CSS 规则可生效，但 JS 无法读取 `CSSStyleSheet` 内容                                 |
| `<iframe>`                    | 可嵌入跨域页面  | DOM 完全隔离；仅能通过 `postMessage` 通信                                         |

> ✅ 总结：浏览器区分 **“加载/执行”** 与 **“读取/控制”**。跨域加载允许，但脚本级数据访问被 SOP 拦截。

## 七、现代浏览器的**多层安全策略**

同源策略是基石，但现代 Web 安全已发展为*多层策略协同：*

| 策略                         | 作用                                    | 与 SOP 关系                     |
| -------------------------- | ------------------------------------- | ---------------------------- |
| **CSP**（内容安全策略）            | 限制页面可加载/执行的资源来源                       | 补充 SOP，防 XSS 注入              |
| **COOP / COEP**（跨源隔离）      | 启用 `SharedArrayBuffer` 等高性能 API 需严格隔离 | 比 SOP 更严格，要求页面与所有嵌入资源同源或明确授权 |
| **SameSite Cookie**        | 控制 Cookie 是否在跨站请求中发送                  | 与 SOP 协同防御 CSRF              |
| **CORP / COEP**            | 控制资源是否可被跨源文档嵌入                        | 防止侧信道攻击（如 Spectre）           |
| **Fetch Metadata Headers** | 浏览器自动发送 `Sec-Fetch-*` 头，服务端可判断请求意图    | 增强服务端跨域决策能力                  |

> 🌐 趋势：浏览器默认策略日益严格（如第三方 Cookie 逐步禁用、跨域隔离要求提高），开发者需主动适配标准跨域方案。

## 八、最佳实践与常见陷阱

### ✅ 推荐做法
1. **优先使用 CORS**：服务端明确配置允许的源、方法、头，避免 `*` 滥用。
2. **跨域通信验证来源**：`postMessage` 接收方必须校验 `event.origin`。
3. **敏感接口加 CSRF 防护**：SOP 不防 CSRF，需配合 `SameSite=Strict/Lax` 或 Token。
4. **开发环境用代理**：Vite/Webpack `proxy` 或 Nginx 反向代理，避免修改浏览器安全设置。

### 🚫 危险操作
- 启动浏览器加 `--disable-web-security`（仅限本地调试，绝不可用于生产）
- 服务端配置 `Access-Control-Allow-Origin: *` 同时允许 `credentials: true`（浏览器会直接拒绝）
- 依赖 `document.domain` 或 JSONP 处理新业务

---

## 总结

同源策略不是“开发障碍”，而是保护用户隐私与 Web 生态安全的**第一道防线**。  
现代 Web 开发中，跨域需求应通过 **CORS、postMessage、代理** 等标准机制解决，而非试图绕过安全模型。理解 SOP 的边界、例外与演进方向，有助于构建更安全、可维护的前后端架构。
