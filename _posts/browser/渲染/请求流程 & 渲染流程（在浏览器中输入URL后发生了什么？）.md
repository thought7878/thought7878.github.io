在浏览器中输入 URL 到页面最终呈现，是一个涉及**网络协议、操作系统、浏览器内核、前端渲染**等多个层面的复杂过程。这也是计算机科学和前端开发中最经典的“全链路”问题。

以下是按时间顺序详细梳理的完整过程：

---

### 第一阶段：输入处理与缓存检查

#### 1. URL 解析与输入处理
*   **判断输入内容**：浏览器首先判断输入的是合法的 URL 还是搜索关键字。如果是关键字，会使用默认搜索引擎拼接成搜索 URL。
*   **解析 URL**：如果是 URL，浏览器会解析出协议（http/https）、域名（www.example.com）、端口（默认80/443）、路径（/path）和参数等。

#### 2. 缓存检查（关键性能优化点）
在发起网络请求前，浏览器会严格按照以下顺序检查缓存：
1.  **Service Worker 缓存**：如果注册了 SW，会拦截请求并返回缓存。
2.  **Memory Cache（内存缓存）**：读取当前页面已加载到内存中的资源（如预加载的资源）。
3.  **Disk Cache（磁盘缓存）**：
    *   **强缓存**：检查响应头中的 `Cache-Control` (max-age) 或 `Expires`。如果未过期，**直接使用本地缓存，不发起网络请求**（状态码 200 from disk cache）。
    *   **协商缓存**：如果强缓存失效，浏览器会携带 `If-Modified-Since` 或 `If-None-Match` 向服务器发起请求。如果资源未改变，服务器返回 **304 Not Modified**，浏览器继续使用本地缓存。
4.  **Push Cache（HTTP/2 推送缓存）**：如果以上都没命中，且使用了 HTTP/2，会检查推送缓存。

*如果强缓存命中，流程直接跳到“第七阶段：页面渲染”。否则继续向下执行。*

---

### 第二阶段：网络请求准备

#### 3. DNS 解析（域名转 IP）
浏览器需要知道目标服务器的 IP 地址，查找顺序如下：
1.  **浏览器缓存**：检查浏览器自身的 DNS 缓存。
2.  **操作系统缓存**：检查 OS 的缓存（如 Windows 的 DNS Client 服务，或 Mac/Linux 的 `/etc/hosts` 文件）。
3.  **本地 DNS 服务器**：向路由器或 ISP（网络服务提供商）的本地 DNS 服务器发起查询。
4.  **迭代查询**：如果本地 DNS 没有，它会向**根域名服务器** -> **顶级域名服务器 (.com)** -> **权威域名服务器 (example.com)** 依次查询，最终获取 IP 并层层返回，同时各级服务器都会进行缓存。

#### 4. 建立 TCP 连接（三次握手）
拿到 IP 和端口后，浏览器与服务器通过 TCP 协议建立可靠连接：
1.  **第一次握手（SYN）**：客户端发送 SYN 报文（seq=x），进入 `SYN-SENT` 状态。
2.  **第二次握手（SYN+ACK）**：服务端收到后，回复 SYN+ACK 报文（seq=y, ack=x+1），进入 `SYN-RCVD` 状态。
3.  **第三次握手（ACK）**：客户端收到后，回复 ACK 报文（ack=y+1），双方进入 `ESTABLISHED` 状态，连接建立成功。

#### 5. TLS/SSL 握手（仅限 HTTPS）
如果 URL 是 `https://`，在 TCP 连接之上还需要进行 TLS 握手，以加密通信：
1.  **Client Hello**：客户端发送支持的 TLS 版本、加密算法列表、随机数 Client Random。
2.  **Server Hello**：服务端确认 TLS 版本和加密算法，返回数字证书和随机数 Server Random。
3.  **证书验证**：客户端验证证书的合法性（是否过期、颁发机构是否可信、域名是否匹配）。
4.  **密钥交换**：客户端生成预主密钥（Pre-Master Secret），用证书中的公钥加密后发给服务端。
5.  **生成会话密钥**：双方利用 Client Random、Server Random 和 Pre-Master Secret 计算出相同的**对称加密会话密钥**，后续通信均使用该密钥加密。

---

### 第三阶段：HTTP 通信

#### 6. 发送 HTTP 请求
连接建立后，浏览器构建 HTTP 请求报文并发送：
*   **请求行**：方法（GET/POST）、URL路径、HTTP版本（HTTP/1.1 或 HTTP/2）。
*   **请求头（Headers）**：包含 `Host`、`User-Agent`、`Accept`、`Cookie`、`Connection: keep-alive` 等。
*   **请求体（Body）**：如果是 POST 请求，会携带表单数据或 JSON 数据。

#### 7. 服务器处理请求并返回响应
1.  **接收请求**：服务器（如 Nginx/Apache）接收到请求。
2.  **路由与处理**：根据 URL 路由到对应的后端应用（Node.js/Java/Python 等），应用可能会查询数据库、调用其他微服务。
3.  **构建响应**：生成 HTTP 响应报文。
    *   **状态行**：HTTP版本、状态码（200 OK, 404 Not Found, 500 等）。
    *   **响应头**：`Content-Type`、`Content-Length`、`Set-Cookie`、缓存控制头等。
    *   **响应体**：返回的实际数据（通常是 HTML 文档，也可能是 JSON、图片等）。
4.  **返回数据**：将响应报文通过 TCP 连接发送回浏览器。

---

### 第四阶段：浏览器解析与渲染（核心重点）

浏览器拿到 HTML 数据后，开始复杂的解析和渲染流水线：

#### 8. 解析 HTML，构建 DOM 树
*   浏览器将 HTML 字节流转换为字符，再解析成 Token，最后构建成 **DOM 树（Document Object Model）**。
*   **预解析（Preload Scanner）**：在解析 HTML 时，浏览器会启动一个副线程扫描文档，提前发现 `<link>`、`<script>`、`<img>` 等外部资源，并**异步发起网络请求**下载这些资源，以提升加载速度。

#### 9. 解析 CSS，构建 CSSOM 树
*   浏览器解析 CSS 文件（包括内联、内部、外部样式），构建 **CSSOM 树（CSS Object Model）**。
*   *注意：CSS 解析不会阻塞 DOM 解析，但会阻塞页面的渲染（Render）。*

#### 10. 处理 JavaScript
*   当解析器遇到 `<script>` 标签时，**DOM 解析会暂停**（因为 JS 可能会修改 DOM）。
*   等待 JS 下载并执行完毕后，再恢复 DOM 解析。
*   *优化手段*：使用 `<script defer>`（异步下载，DOM 解析完后执行）或 `<script async>`（异步下载，下载完立即执行）来避免阻塞。

#### 11. 构建渲染树（Render Tree）
*   将 **DOM 树** 和 **CSSOM 树** 合并，生成 **渲染树**。
*   渲染树只包含需要在页面上显示的节点（例如，`display: none` 的元素不会出现在渲染树中，但 `visibility: hidden` 的元素会）。

#### 12. 布局（Layout / Reflow）
*   也称为**重排**。浏览器计算渲染树中每个节点的**几何信息**（大小、位置、边距等）。
*   这是一个从根节点（`<html>`）开始，递归计算子节点位置的深度优先遍历过程。

#### 13. 绘制（Paint）
*   也称为**重绘**。将布局计算出的像素信息转换为屏幕上的实际像素。
*   包括文本、颜色、边框、阴影等。浏览器通常会将页面分成多个**图层（Layers）** 进行绘制。

#### 14. 合成（Composite）
*   浏览器的合成线程将绘制好的各个图层，按照正确的顺序（z-index 等）合并在一起，最终显示在用户的屏幕上。
*   *性能优化*：使用 `transform` 和 `opacity` 触发 GPU 加速，可以跳过布局和绘制阶段，直接在合成线程处理，大幅提升动画性能。

---

### 第五阶段：连接关闭

#### 15. TCP 四次挥手（断开连接）
如果 HTTP 响应头中没有 `Connection: keep-alive`，或者达到了 keep-alive 的超时时间，TCP 连接将关闭：
1.  **第一次挥手（FIN）**：客户端发送 FIN 报文，请求关闭连接。
2.  **第二次挥手（ACK）**：服务端收到 FIN，回复 ACK（此时客户端到服务端的连接关闭，但服务端可能还有数据没发完）。
3.  **第三次挥手（FIN）**：服务端数据发送完毕，发送 FIN 报文给客户端。
4.  **第四次挥手（ACK）**：客户端回复 ACK，进入 `TIME-WAIT` 状态，等待 2MSL（最长报文段寿命）后彻底关闭。服务端收到 ACK 后立即关闭。

### 💡 进阶补充（面试加分项）

1.  **HTTP/2 的多路复用**：如果是 HTTP/2，TCP 连接建立后，所有的 HTTP 请求和响应都在同一个 TCP 连接上**并发、交错**传输，解决了 HTTP/1.1 的队头阻塞问题，且支持头部压缩（HPACK）和服务器推送。
2.  **HTTP/3 与 QUIC**：HTTP/3 放弃了 TCP，改用基于 UDP 的 QUIC 协议。它将 TCP 的三次握手和 TLS 握手合并，实现了 **0-RTT 或 1-RTT** 建连，彻底解决了 TCP 层面的队头阻塞问题。
3.  **HSTS (HTTP Strict Transport Security)**：如果服务器配置了 HSTS，浏览器在第一次通过 HTTPS 访问后，会在本地记录。后续即使输入 HTTP，浏览器也会在**本地自动将其转换为 HTTPS**，省去了一次 301 重定向的网络开销。

### 总结全景图
**输入 URL** ➔ **查缓存** ➔ **DNS 解析** ➔ **TCP 三次握手** ➔ **(TLS 握手)** ➔ **发 HTTP 请求** ➔ **服务器处理并响应** ➔ **解析 HTML/CSS/JS** ➔ **构建 DOM/CSSOM** ➔ **生成渲染树** ➔ **布局 (Layout)** ➔ **绘制 (Paint)** ➔ **合成显示** ➔ **TCP 四次挥手**。
