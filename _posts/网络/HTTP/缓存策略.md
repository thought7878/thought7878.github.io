>前端开发中的缓存策略全景指南

`缓存`是前端性能优化的核心杠杆，也是工程化架构中最易引发线上事故的环节。现代前端缓存已从单一的“浏览器存文件”演进为**网络层、浏览器层、应用层、边缘层协同的多维体系**。掌握缓存策略，本质是掌握**性能、一致性、复杂度之间的权衡艺术**。

---

## 一、缓存分层架构全景

```
用户请求
  ├─ 1. 内存缓存（JS变量/Map/框架Store）        → 毫秒级，页面刷新丢失
  ├─ 2. Service Worker Cache API                → 拦截网络，离线可用，独立于HTTP缓存
  ├─ 3. 浏览器 HTTP 缓存（Memory/Disk Cache）    → 强缓存 + 协商缓存，自动管理
  ├─ 4. 浏览器客户端存储（LS/SS/IndexedDB）      → 手动读写，适合结构化/持久化数据
  └─ 5. CDN / 边缘节点缓存                      → 地理就近，分担源站，可主动刷新
```

> 💡 核心认知：**缓存不是单一机制，而是分层协作系统**。不同层解决不同问题，配置冲突会导致“缓存击穿”或“永久旧版”。

---

## 二、**HTTP 缓存**策略（网络层核心）

`HTTP 缓存`由浏览器*自动管理*，分为**强缓存**与**协商缓存**两阶段。

### 🔹 强缓存（直接返回，不请求服务器）
| 头部              | 作用           | 现代用法                                                                  |
| --------------- | ------------ | --------------------------------------------------------------------- |
| `Cache-Control` | 核心控制指令，优先级最高 | `max-age=31536000`、`no-cache`、`no-store`、`private/public`、`immutable` |
| `Expires`       | 绝对过期时间（GMT）  | ⚠️ 已废弃，仅兼容旧端。`Cache-Control` 存在时自动忽略                                  |

**关键指令语义**：
- `max-age=秒`：强缓存有效期
- `no-cache`：**不是不缓存**，而是“每次使用前必须向服务器验证”（走协商缓存）
- `no-store`：绝对不缓存，每次完整请求
- `private`：仅浏览器可缓存（默认）；`public`：CDN/代理也可缓存
- `immutable`：内容永不改变，跳过协商验证（配合哈希文件名使用）

### 🔹 协商缓存（强缓存失效后触发）
| 机制 | 请求头 | 响应头 | 特点 |
|------|--------|--------|------|
| ETag | `If-None-Match` | `ETag` | 内容级验证，精度高，现代标准 |
| Last-Modified | `If-Modified-Since` | `Last-Modified` | 时间级验证，秒级精度，仅兼容 |

**完整决策流**：
```
发起请求 → 检查强缓存(max-age/Expires)
  ├─ 命中 → 直接返回 (Size: memory cache / disk cache)
  └─ 失效 → 携带协商头(If-None-Match/If-Modified-Since) 请求服务器
      ├─ 未变化 → 304 Not Modified (浏览器读本地缓存)
      └─ 已变化 → 200 OK + 新内容 + 新缓存头
```

## 三、浏览器客户端存储缓存（手动控制层）

| 存储方式             | 容量     | 生命周期      | 同步/异步  | 适用场景                        | 致命限制                          |
| ---------------- | ------ | --------- | ------ | --------------------------- | ----------------------------- |
| **内存变量/Map**     | 受限于JS堆 | 页面刷新/关闭丢失 | 同步     | 请求去重、短期状态、高频计算缓存            | 无法持久，需手动管理淘汰策略                |
| `sessionStorage` | ~5MB   | 同源同标签页会话期 | 同步     | 多步表单、临时草稿、单页路由状态            | 跨标签页不共享，刷新保留但关闭丢失             |
| `localStorage`   | ~5MB   | 永久（需手动清除） | 同步     | 用户偏好、非敏感配置、UI主题             | **阻塞主线程**，配额超限抛异常，不防XSS       |
| `IndexedDB`      | GB级    | 永久        | 异步（事务） | 离线数据、大文件、复杂查询、PWA           | API 繁琐，需封装或库（idb/localForage） |
| `Cache Storage`  | 大容量    | 永久（SW控制）  | 异步     | 存储 Request/Response 对，PWA离线 | 仅 Service Worker 或安全上下文可用     |

> ⚠️ **工程红线**：
> 1. 绝不在 `localStorage` 存 Token/密码/PII（XSS 直接窃取）
> 2. 避免在主线程频繁读写 LS（同步阻塞渲染）
> 3. 存储前必做 `try/catch` 防 `QuotaExceededError`

## 四、应用层与**数据请求缓存**（现代前端核心）

前后端分离后，`API 数据缓存`成为体验优化主战场。传统“手动存 localStorage”已被声明式数据获取库取代。

### 🔹 现代数据缓存模式（SWR / React Query / Apollo）
参考：[[为什么需要 TanStack Query？]]

| 概念                         | 说明                             | 价值         |
| -------------------------- | ------------------------------ | ---------- |
| **Stale-While-Revalidate** | 先返回旧数据（Stale），后台静默请求新数据，更新后重渲染 | 零加载等待，体验流畅 |
| `staleTime`                | 数据“新鲜”有效期，期内不请求                | 减少冗余请求     |
| `cacheTime` / `gcTime`     | 数据在内存中保留时间，超时垃圾回收              | 控制内存占用     |
| **请求去重**                   | 同一 Key 并发请求只发一次                | 防重复提交/瀑布请求 |
| **乐观更新**                   | 先改 UI，请求失败回滚                   | 交互即时响应     |
| **后台重试/聚焦刷新**              | 网络恢复/窗口聚焦时自动同步                 | 移动端弱网友好    |

```js
// React Query 示例
const { data } = useQuery({
  queryKey: ['user', id],
  queryFn: () => fetchUser(id),
  staleTime: 1000 * 60 * 5,   // 5分钟内视为新鲜
  gcTime: 1000 * 60 * 30,     // 30分钟未使用则清理
  refetchOnWindowFocus: true  // 切回标签页自动同步
});
```

> 📌 注意：**应用层缓存 ≠ HTTP 缓存**。前者在 JS 内存，后者在浏览器网络层。两者可协同（如 HTTP 短 TTL + SWR 后台更新）。

## 五、Service Worker 与 PWA 缓存策略

SW 拦截网络请求，提供**独立于 HTTP 缓存**的编程式缓存能力。

### 🔹 Workbox 五大经典策略
| 策略                     | 逻辑                    | 适用资源                  |
| ---------------------- | --------------------- | --------------------- |
| `CacheFirst`           | 先读缓存，命中返回；未命中走网络并写入缓存 | 静态资源（JS/CSS/Img/Font） |
| `NetworkFirst`         | 先走网络，成功返回并更新缓存；失败读缓存  | API 接口、动态内容           |
| `StaleWhileRevalidate` | 立即返回缓存，后台请求更新缓存，下次生效  | 平衡实时性与性能的内容           |
| `NetworkOnly`          | 仅网络，不缓存               | 支付、敏感操作、实时性要求极高接口     |
| `CacheOnly`            | 仅缓存，不请求网络             | 离线包、预缓存核心资源           |

### 🔹 预缓存 vs 运行时缓存
- **Precaching**：构建时生成清单，SW 安装阶段批量下载。适合核心壳资源。
- **Runtime Caching**：按路由/正则匹配，首次访问后缓存。适合长尾资源。

> ⚠️ SW 缓存陷阱：未处理 `activate` 阶段旧缓存清理 → 用户永久停留在旧版本。必须配合版本号或哈希清单自动淘汰。

## 六、CDN 与边缘缓存策略

`CDN 缓存`位于用户与源站之间，遵循源站 `Cache-Control`，但可覆盖。

| 配置项                | 说明                               | 最佳实践                                          |
| ------------------ | -------------------------------- | --------------------------------------------- |
| **缓存键（Cache Key）** | 决定哪些请求共享缓存                       | 默认 URL+Query；忽略无关参数（如 `?utm_source`）；按设备/语言分片 |
| **TTL 覆盖**         | CDN 可忽略源站 max-age                | 静态资源设长 TTL；HTML 设 0 或短 TTL                    |
| **缓存刷新（Purge）**    | 主动失效 CDN 节点缓存                    | 发版后调用 Purge API；优先用哈希文件名替代刷新                  |
| **边缘计算**           | Cloudflare Workers / Vercel Edge | 动态 SSR 缓存、A/B 测试路由、个性化边缘缓存                    |

## 七、缓存更新与失效策略（核心难点）

> “计算机科学只有两件难事：缓存失效和命名。” —— Phil Karlton

| 策略               | 原理                       | 适用场景             | 优缺点                                 |
| ---------------- | ------------------------ | ---------------- | ----------------------------------- |
| **文件名内容哈希**      | `app.a1b2c3.js`，内容变则文件名变 | 构建产物（JS/CSS/Img） | ✅ 彻底免失效，配合 `immutable`<br>❌ 需构建工具支持 |
| **查询参数版本**       | `app.js?v=1.2.3`         | 老旧系统/无构建流程       | ⚠️ 部分 CDN 默认不缓存带参 URL；需配置忽略参数或按参缓存  |
| **主动 Purge**     | 调用 CDN/源站 API 清除缓存       | 紧急修复、无版本化资源      | ❌ 延迟高、成本高、易遗漏节点                     |
| **短 TTL + 后台更新** | `max-age=60` + SWR/轮询    | API 数据、配置接口      | ✅ 平衡实时与性能<br>❌ 仍有短暂不一致窗口            |
| **事件驱动失效**       | WebSocket/SSE 推送更新指令     | 协同编辑、实时行情、聊天     | ✅ 强一致性<br>❌ 架构复杂，需长连接               |

### 🚨 生死线：HTML 入口文件缓存策略
```http
# index.html 必须配置
Cache-Control: no-cache, max-age=0
# 或
Cache-Control: public, max-age=0, must-revalidate
```
- **原因**：HTML 是资源加载的“指挥棒”。若被强缓存，用户将永远加载旧版 JS/CSS，导致白屏/功能缺失。
- **正确做法**：HTML 走协商缓存（ETag）或极短 TTL；子资源用哈希文件名 + 长缓存。

## 八、工程化决策矩阵（直接可用）

| 资源类型                | HTTP 缓存策略                       | 客户端存储            | CDN 策略             | 更新机制          |
| ------------------- | ------------------------------- | ---------------- | ------------------ | ------------- |
| `index.html`        | `no-cache` / `max-age=0` + ETag | 不存               | 不缓存或 TTL=0         | 协商验证          |
| `app.[hash].js/css` | `max-age=31536000, immutable`   | 不存               | 长 TTL，按文件名缓存       | 文件名哈希变更       |
| 图片/字体/媒体            | `max-age=1y, immutable`         | 不存               | 长 TTL，WebP/AVIF 转换 | 文件名哈希或 CDN 刷新 |
| 公开 API 数据           | `no-cache` 或短 `max-age`         | SWR 内存缓存         | 边缘缓存（按用户/地区分片）     | 后台静默更新 / 事件推送 |
| 用户私有数据              | `private, no-store` 或短 TTL      | 内存/IndexedDB（加密） | 不缓存                | 登录后拉取，登出清理    |
| 第三方 SDK             | 视提供商策略                          | 不主动存             | 透传或独立 CDN          | 依赖供应商版本更新     |

## 九、常见陷阱与排查指南

### 🚫 高频线上事故
| 现象                | 根因                         | 修复                                   |
| ----------------- | -------------------------- | ------------------------------------ |
| 发版后用户白屏/功能缺失      | HTML 被强缓存，加载旧版 JS          | HTML 设 `no-cache`；子资源哈希化             |
| 接口数据不更新           | `max-age` 过长 + 无后台刷新       | 改用 `no-cache` 或 SWR 模式               |
| CDN 缓存命中率低        | 缓存键包含动态参数（如 `?t=169...`）   | CDN 配置忽略无关参数；改用路径版本化                 |
| LocalStorage 报错崩溃 | 未 try/catch，配额超限或无痕模式      | 封装安全读写；降级到内存或 IndexedDB              |
| SW 更新不生效          | 未处理 `skipWaiting` / 旧缓存未清理 | 监听 `controllerchange`；Workbox 自动清理策略 |

### 🛠 调试 checklist
1. **DevTools Network**：勾选 `Disable cache` 对比；查看 `Size` 列（`memory cache`/`disk cache`/`304`）
2. **Application 面板**：检查 Cache Storage、IndexedDB、LS 实际内容
3. **命令行验证**：`curl -I https://example.com/app.js` 查看响应头
4. **CDN 日志**：查看 `HIT/MISS/EXPIRED` 状态码与缓存键
5. **SW 调试**：`chrome://serviceworker-internals/` 强制更新/注销

## 十、总结与演进趋势

### 🔑 核心原则
1. **HTML 永不强缓存**，子资源哈希化 + `immutable`
2. **明确区分“静态资源”与“动态数据”**，采用不同缓存模型
3. **默认短缓存 + 后台更新（SWR）**，优于长缓存 + 手动刷新
4. **缓存失效靠版本，不靠 Purge**
5. **所有客户端存储必须防 XSS、防超限、防阻塞**

### 🌐 演进方向
- **边缘缓存智能化**：CDN 支持动态 SSR 缓存、个性化分片、AI 预测预加载
- **框架内置缓存**：Next.js App Router / Remix / Nuxt 3 将数据缓存与渲染生命周期深度绑定
- **浏览器默认收紧**：第三方存储分区化、LS 同步操作警告、SW 更新策略标准化
- **缓存可观测性**：Lighthouse CI 缓存评分、RUM 监控缓存命中率、自动化缓存头校验门禁

> 💡 工程心法：缓存不是“配完即忘”的静态配置，而是**与发版流程、数据流架构、用户体验指标绑定的动态策略**。用哈希解决静态资源，用 SWR 解决动态数据，用协商缓存兜底入口文件，用可观测性保障一致性。

如需针对具体场景输出**可落地方案**（如：Vite/Webpack 哈希缓存配置模板、React Query + HTTP 缓存协同架构、Service Worker 灰度更新流程、CDN 缓存键优化与 Purge 自动化脚本、或缓存命中率监控埋点方案），可提供技术栈与业务形态，我将提供完整代码与部署清单。
