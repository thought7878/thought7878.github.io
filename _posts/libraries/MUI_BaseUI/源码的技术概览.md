>Base UI — 全面技术概览

## 1. 项目目标与核心功能

Base UI（`@base-ui/react`，v1.6.0）是一个**无头（无样式）React 组件库**，由 Radix、Material UI 和 Floating UI 背后的团队创建。其目标是提供一套完整、可访问、可组合、样式无关的 UI 基础组件——约 35 个组件系列——负责处理行为、可访问性和状态管理，同时将所有视觉样式留给使用者决定。

**关键特性：**
- **无头设计**：不捆绑任何 CSS；使用者可使用 Tailwind、CSS Modules、CSS-in-JS 或纯 CSS。
- **可访问性优先**：组件实现了 [WAI-ARIA 设计模式](https://www.w3.org/WAI/ARIA/apg/patterns/)，内置键盘导航、焦点管理和屏幕阅读器支持。
- **可 Tree-shaking**：每个组件都是独立的入口点（如 `@base-ui/react/popover`），因此只有实际使用的代码会被包含在打包产物中。
- **兼容 React 17+**：支持 React 17、18 和 19。

## 2. 架构设计与组件结构

### 2.1 Monorepo 布局

```
base-ui/
├── packages/
│   ├── react/          → @base-ui/react（组件库）
│   └── utils/          → @base-ui/utils（共享 hooks 和工具函数）
├── docs/               → Next.js 文档站点
├── examples/           → Vite + CSS 示例项目
├── playground/         → 交互式演练场
└── test/               → E2E、回归测试和性能测试
```

项目使用 **pnpm workspaces** + **Lerna** 进行 monorepo 管理，并使用 **Nx** 进行构建缓存。

### 2.2 复合组件模式

每个组件系列都遵循 **复合组件** 架构，由一个 `Root` 加若干命名 `Parts` 组成：

```
packages/react/src/popover/
├── index.ts            → 从 index.parts.ts 重新导出
├── index.parts.ts      → 导出 { Root, Trigger, Portal, Positioner, Popup, Arrow, Title, Description }
├── root/PopoverRoot.tsx
├── trigger/PopoverTrigger.tsx
├── portal/PopoverPortal.tsx
├── positioner/PopoverPositioner.tsx
├── popup/PopoverPopup.tsx
├── arrow/PopoverArrow.tsx
├── title/PopoverTitle.tsx
└── description/PopoverDescription.tsx
```

`Root` 组件创建共享状态（通过 Context 或 Store），并将其向下传递。每个 `Part` 只消费它所需要的部分。这使使用者可以完全控制组合方式——他们可以自由地添加、移除或重新排列各部分。

### 2.3 `render` Prop 系统

一个标志性的架构特性是 **`render` prop**，它允许使用者将任何部件的默认 DOM 元素替换为自定义组件：

```tsx
<Menu.Trigger render={<MyCustomButton size="md" />}>
  打开菜单
</Menu.Trigger>
```

在内部，这由两个 hook 驱动：
- [useRender](file:///Users/ll/Desktop/%E8%B5%84%E6%96%99/%E7%BC%96%E7%A8%8B/%E4%BB%93%E5%BA%93/react/base-ui/packages/react/src/use-render/useRender.ts) — 公开 API，委托给 `useRenderElement`
- [useRenderElement](file:///Users/ll/Desktop/%E8%B5%84%E6%96%99/%E7%BC%96%E7%A8%8B/%E4%BB%93%E5%BA%93/react/base-ui/packages/react/src/internals/useRenderElement.tsx) — 解析 `render` prop，合并内部 props（事件处理器、ARIA 属性、refs、data 属性、CSS 变量），并生成最终的 React 元素

`render` prop 接受 **React 元素**（`<MyButton />`）或 **渲染函数**（`(props, state) => <span {...props}>...</span>`），从而完全控制 prop 的展开和基于状态的渲染。

### 2.4 样式钩子架构

组件暴露三种样式接口：
1. **`className`** — 接受字符串或 `(state) => string` 函数
2. **`style`** — 接受 CSS 对象或 `(state) => CSSObject` 函数
3. **Data 属性** — `[data-checked]`、`[data-highlighted]`、`[data-popup-open]`、`[data-starting-style]`、`[data-ending-style]` 等
4. **CSS 变量** — `--available-height`、`--transform-origin`、`--popup-width` 等

这种多层接口意味着任何样式方案都可以干净地集成。

## 4. 数据流与交互模式

### 4.1 Store 系统

架构中最具特色的部分是**自定义 Store 模式**，实现在 `@base-ui/utils/store/` 中：

```
Store（基类）
  └── ReactStore（React 感知扩展）
        ├── MenuStore
        ├── ComboboxStore（通过 store.ts）
        ├── SelectStore（通过 store.ts）
        ├── ToastStore
        └── ...其他组件 Store
```

#### `Store<State>` — [Store.ts](file:///Users/ll/Desktop/%E8%B5%84%E6%96%99/%E7%BC%96%E7%A8%8B/%E4%BB%93%E5%BA%93/react/base-ui/packages/utils/src/store/Store.ts)

一个轻量级的观察者模式 Store：
- **`state`** — 当前状态对象（同步读取）
- **`setState(newState)`** — 替换状态，通知监听器（通过 `updateTick` 处理递归调用保护）
- **`update(changes)`** — 浅合并部分变更，仅在实际发生变化时通知（`Object.is` 检查）
- **`set(key, value)`** — 设置单个键，仅在值变化时通知
- **`subscribe(fn)`** — 注册监听器，返回取消订阅函数
- **`getSnapshot()`** — 返回当前状态（供 `useSyncExternalStore` 使用）
- **`use(selector, ...args)`** — React hook，用于订阅派生状态

#### `ReactStore<State, Context, Selectors>` — [ReactStore.ts](file:///Users/ll/Desktop/%E8%B5%84%E6%96%99/%E7%BC%96%E7%A8%8B/%E4%BB%93%E5%BA%93/react/base-ui/packages/utils/src/store/ReactStore.ts)

扩展 `Store`，增加 React 特有能力：
- **`context`** — 非响应式值（refs、回调）与响应式状态一起存储
- **`selectors`** — 命名选择器函数，用于类型化的状态派生
- **`useControlledProp(key, value)`** — 将受控 props 同步到 Store，开发模式下会在受控↔非受控切换时发出警告
- **`useSyncedValue(key, value)`** — 通过 `useIsoLayoutEffect` 将单个外部 prop 同步到 Store
- **`useSyncedValues(partial)`** — 一次同步多个 props
- **`select(key)`** — 同步读取派生状态（不触发重渲染）
- **`useState(key)`** — 订阅派生状态（变化时触发重渲染）
- **`useContextCallback(key, fn)`** — 使用 `useStableCallback` 包装回调以获得稳定引用
- **`useStateSetter(key)`** — 返回一个稳定的 setter 函数
- **`observe(selector, listener)`** — 在 React 之外订阅选择器变化

#### `useStore` hook — [useStore.ts](file:///Users/ll/Desktop/%E8%B5%84%E6%96%99/%E7%BC%96%E7%A8%8B/%E4%BB%93%E5%BA%93/react/base-ui/packages/utils/src/store/useStore.ts)

Store 与 React 之间的桥梁。它有两条代码路径：
- **React 19+**：使用自定义的 **fast-hooks** 系统（`fastHooks.ts`），将多个 `useStore` 调用批处理为单个 `useSyncExternalStore` 订阅，减少重渲染。
- **React 17/18**：回退到 shim 包中的 `useSyncExternalStoreWithSelector`。

#### `createSelector` — [createSelector.ts](file:///Users/ll/Desktop/%E8%B5%84%E6%96%99/%E7%BC%96%E7%A8%8B/%E4%BB%93%E5%BA%93/react/base-ui/packages/utils/src/store/createSelector.ts)

创建记忆化的选择器函数，最多可将 6 个输入选择器组合为一个派生计算。与 Redux 风格的 `createSelector` 不同，这个实现**默认不做记忆化**——它是一个纯组合工具。记忆化发生在 `useStore` 层，通过 `Object.is` 比较选择器结果来实现。

### 4.2 数据流示例（Combobox）

```
用户在 Input 中输入
  → Combobox.Trigger 处理 onChange
  → store.set('searchTerm', value)
  → Store 通知订阅者
  → useStore(selector) 仅重渲染所选状态发生变化的组件
  → Listbox 过滤可见项
  → 活动后代通过 store.set('activeIndex', ...) 更新
  → 键盘导航（ArrowDown/Up）更新 activeIndex
  → Enter 选择：store.set('value', item)
  → onValueChange 回调触发
```

### 4.3 Context 与 Store 的选择

并非所有组件都使用 Store。较简单的组件（Accordion、Tabs、Separator）直接使用 React Context。Store 模式保留给具有复杂、频繁变化状态的组件（Menu、Combobox、Select、Toast），其中细粒度订阅可以减少不必要的重渲染。

## 5. 值得关注的实现细节与算法

### 5.1 动画/过渡系统

动画系统是 CSS 驱动的，由三个 hook 协调：

- [useTransitionStatus](file:///Users/ll/Desktop/%E8%B5%84%E6%96%99/%E7%BC%96%E7%A8%8B/%E4%BB%93%E5%BA%93/react/base-ui/packages/react/src/internals/useTransitionStatus.ts) — 管理生命周期状态机：`idle → starting → idle → ending → idle`。设置 `data-starting-style` 和 `data-ending-style` 属性，CSS 可以针对这些属性进行样式化：
  ```css
  [data-starting-style], [data-ending-style] {
    opacity: 0;
    transform: scale(0.98);
  }
  ```

- [useAnimationsFinished](file:///Users/ll/Desktop/%E8%B5%84%E6%96%99/%E7%BC%96%E7%A8%8B/%E4%BB%93%E5%BA%93/react/base-ui/packages/react/src/internals/useAnimationsFinished.ts) — 监听元素上的 `transitionend` 和 `animationend` 事件，在所有动画完成时触发回调。处理中止的动画。

- [useOpenChangeComplete](file:///Users/ll/Desktop/%E8%B5%84%E6%96%99/%E7%BC%96%E7%A8%8B/%E4%BB%93%E5%BA%93/react/base-ui/packages/react/src/internals/useOpenChangeComplete.tsx) — 结合上述两者，在打开/关闭动画完成时调用 `onOpenChangeComplete`。

关键洞察：Base UI **不**使用 JavaScript 动画库。它设置 CSS 属性，让浏览器原生的 CSS 过渡处理动画，然后通过 DOM 事件检测完成。

### 5.2 Shadow DOM 安全性

[shadowDom.ts](file:///Users/ll/Desktop/%E8%B5%84%E6%96%99/%E7%BC%96%E7%A8%8B/%E4%BB%93%E5%BA%93/react/base-ui/packages/react/src/internals/shadowDom.ts) 提供三个工具函数：
- `getActiveElement()` — 遍历 shadow root 以找到真正的活动元素
- `contains()` — 跨 shadow 边界检查包含关系
- `getTarget()` — 从组合事件中提取正确的事件目标

这些在整个代码库中替代了原始的 `document.activeElement`、`node.contains()` 和 `event.target` 使用。

### 5.3 弹出层定位

[usePositioner](file:///Users/ll/Desktop/%E8%B5%84%E6%96%99/%E7%BC%96%E7%A8%8B/%E4%BB%93%E5%BA%93/react/base-ui/packages/react/src/utils/usePositioner.tsx) 封装了 Floating UI 的定位引擎。它暴露 CSS 变量（`--available-height`、`--available-width`、`--anchor-width`、`--anchor-height`、`--transform-origin`），组件使用这些变量进行响应式尺寸调整和基于 transform-origin 的动画。

### 5.4 滑动关闭

[useSwipeDismiss](file:///Users/ll/Desktop/%E8%B5%84%E6%96%99/%E7%BC%96%E7%A8%8B/%E4%BB%93%E5%BA%93/react/base-ui/packages/react/src/utils/useSwipeDismiss.ts) 实现了 Drawer 及类似组件的触摸手势处理。它跟踪指针移动，应用 CSS 变换提供视觉反馈，并在超过阈值时触发关闭。

### 5.5 受控/非受控模式

`useControlled` hook 和 `ReactStore.useControlledProp` 一致地实现了受控/非受控模式。如果提供了 `value` prop，组件为受控模式；如果提供了 `defaultValue`，则为非受控模式。在两种模式之间切换会触发开发模式警告。

### 5.6 Fast Hooks（自定义 Hook 编排）

`fastHooks.ts` 模块实现了一个自定义 hook 编排层，将多个 `useSyncExternalStore` 调用合并为一个。这是一个重要的性能优化：不是每个 `useStore` 调用都创建自己的订阅（和潜在的重渲染），而是将组件内的所有 Store 订阅合并为单个 `useSyncExternalStore` 调用，使用统一的 `getSnapshot` 来检查所有选择器。

### 5.7 错误码系统

错误使用了对压缩友好的模式。在生产环境中，错误被缩减为数字代码。`extract-error-codes` 脚本维护着 `docs/src/error-codes.json` 中的映射。所有错误遵循三部分格式：发生了什么、为什么这是个问题、以及如何修复。

**已知的架构限制：**
- `render` prop 要求自定义组件转发 `ref` 并展开所有接收到的 props——这已有文档说明，但可能成为陷阱。
- fast-hooks 优化仅限 React 19+；React 17/18 使用较慢的 `useSyncExternalStoreWithSelector` shim。
- 部分组件（Navigation Menu、Autocomplete）是相对较新的 additions，可能缺乏足够的社区实战检验。
- Store 系统是内部的——没有公开 API 用于在组件使用之外创建自定义 Store。

## 7. 潜在的未来增强与扩展方向

基于代码库结构和发展轨迹：

1. **更多组件系列**：`internals/temporal/` 目录（带有 `date-fns` 和 `luxon` 适配器作为 peer 依赖）强烈暗示**日期选择器/日历**组件正在开发中。

2. **`unstable-use-media-query`**：已作为不稳定入口点导出，暗示未来可能扩展**响应式工具**。

3. **React 19+ 优化**：fast-hooks 系统仅限 React 19。随着 React 18 使用率下降，旧代码路径可以被移除，简化 Store 订阅层。

4. **Web Components / Shadow DOM**：现有的 Shadow DOM 工具暗示未来可能将 **Web Component 包装器**或更好的 Shadow DOM 支持作为一等特性。

5. **框架扩展**：虽然目前仅限 React，但 Store 模式在核心层面是框架无关的。Vue 或 Svelte 移植在架构上是可行的，尽管尚未公布。

6. **预样式生态系统**：文档已经引用了 [shadcn/ui](https://ui.shadcn.com/create?base=base) 作为预样式层。未来可能出现更多社区驱动的设计系统集成。

7. **性能工具**：`test/bundle-size` 和 `test/performance` 目录表明对包体积和运行时性能基准测试的持续投入。

## 总结

Base UI 是一个精密的、架构良好的无头组件库，优先考虑可访问性、性能和开发者控制力。其带有细粒度选择器的自定义 Store 系统、CSS 驱动的动画管道、Shadow DOM 安全工具以及 `render` prop 组合模型，代表了对构建无头 UI 组件核心挑战的深思熟虑的解决方案。代码库成熟（v1.6.0），经过跨浏览器充分测试，并由来自 React UI 生态系统的经验丰富的贡献者维护。
