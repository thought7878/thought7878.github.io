## 文件结构
>`/base-ui/packages/react/src/navigation-menu/trigger/NavigationMenuTrigger.test.tsx` 详细解释这个测试文件

这个测试文件用于测试 `<NavigationMenu.Trigger />` 组件，主要验证导航菜单触发器的行为和交互。

### 1. 辅助工具

**`createSuspender()`** (10-37行)
创建一个会延迟渲染的组件，用于测试 React Suspense 边界的情况。首次渲染时会抛出 Promise，10ms 后 resolve。

**`rapidHoverAnimationStyles`** (39-47行)
定义 CSS 过渡动画样式，用于测试快速 hover 时的尺寸变化行为。

**`TestNavigationMenuRapidHoverSizing`** (49-89行)
测试专用组件，包含两个菜单项（Product 和 Solutions），Product 的内容可以动态展开（从 700x420 变为 760x460）。

**辅助函数** (91-99行)
- `getPopupWidthCalls`: 过滤出 popup 宽度相关的 CSS 变量设置调用
- `getPositionerWidthCalls`: 过滤出 positioner 宽度相关的调用

### 2. 一致性测试 (104-117行)

```typescript
describeConformance(<NavigationMenu.Trigger />, () => ({
  refInstanceof: window.HTMLButtonElement,
  testComponentPropWith: 'button',
  button: true,

```

验证组件符合 Base UI 的标准规范：
- ref 应该指向 HTMLButtonElement
- 支持 `button` 组件 prop
- 能在 NavigationMenu 结构中正确渲染

### 3. 测试用例详解

#### 测试 1: RTL 模式下的方向键 (119-150行)
**目的**: 验证在从右到左（RTL）语言环境下，方向键行为是否正确镜像。

**流程**:
1. 渲染 RTL 方向的垂直导航菜单
2. 聚焦到 "Overview" 按钮
3. 按左方向键（在 RTL 中应该相当于 LTR 的右键）
4. 验证菜单内容是否打开

#### 测试 2: 焦点和 positioner 高度处理 (152-251行)
**标记**: `it.skipIf(isJSDOM)` - 仅在浏览器环境运行（需要真实布局计算）

**目的**: 验证键盘导航时，positioner 的高度能正确适配内容。

**流程**:
1. 渲染包含两个菜单项的导航
2. 聚焦 "Overview"，按下方向键进入内容
3. 验证 `--positioner-height` CSS 变量约为 18px（单行高度）
4. Tab 回到触发器，右键移动到 "Handbook"
5. 下方向键进入内容，验证高度约为 36px（两行高度）
6. 重复验证左右方向键的导航行为

#### 测试 3: positioner 宽度处理 (253-301行)
**标记**: `it.skipIf(isJSDOM)`

**目的**: 验证 positioner 宽度能正确匹配内容宽度。

**流程**:
1. 渲染一个无内容的触发器和一个有内容的触发器
2. 快速 hover 切换：withContent → noContent → withContent
3. 验证 `--positioner-width` 约为 183px（内容实际宽度）

#### 测试 4: hover 切换时重新定位 (303-352行)
**标记**: `it.skipIf(isJSDOM)`

**目的**: 验证在不同触发器间 hover 切换时，positioner 能正确重新定位。

**流程**:
1. 渲染两个并排的菜单项
2. hover "Overview"，记录 positioner 的 left 位置
3. hover "Handbook"，验证 positioner 的 left 位置变化超过 20px

#### 测试 5: 快速 hover 不会错误应用尺寸 (354-398行)
**标记**: `it.skipIf(isJSDOM)`

**目的**: 验证快速切换 hover 目标时，之前触发器的尺寸计算不会污染后续状态。

**流程**:
1. hover "Product"，等待 popup 宽度设为 700px
2. 快速 hover "Solutions"，记录宽度变化
3. 验证出现 500px（Solutions 宽度）后，不会再出现 700px
4. 最终宽度应该是 'auto'

**关键点**: 防止异步尺寸计算在切换后仍然生效。

#### 测试 6: 中断的 mutation 不会影响后续切换 (400-477行)
**标记**: `it.skipIf(isJSDOM)`

**目的**: 验证当内容尺寸正在变化时（mutation），切换 hover 目标不会导致尺寸计算错误。

**流程**:
1. hover "Product"，等待初始尺寸 700x420
2. 模拟尺寸变化到 760x460（通过修改 offsetWidth/offsetHeight）
3. 点击 "Expand Product" 按钮触发展开
4. 验证 positioner 宽度更新为 760px
5. 快速切换到 "Solutions"（尺寸 500x320）
6. 验证出现 500px 后，不会再出现 760px

**关键点**: 确保中断的尺寸观察回调不会在切换后重新应用。

#### 测试 7: **Suspense 边界不会导致无限循环 (479-536行)**

**目的**: 验证当 Content 使用 Suspense 包裹时，hover 切换不会导致无限更新循环。

**流程**:
1. 渲染包含 Suspense 边界的菜单，第一个项使用 Suspender 组件
2. hover "Item 1"，应该显示 "Loading..." fallback
3. 等待 suspension resolve
4. hover "Item 2"，验证正常显示内容
5. 再次 hover "Item 1"，验证能正常工作

**关键点**: 确保 Suspense 的异步加载不会与导航菜单的状态管理产生冲突。

## 技术要点

1. **环境区分**: 大部分涉及布局计算的测试使用 `skipIf(isJSDOM)` 跳过 jsdom，因为 jsdom 不支持真实的布局计算。

2. **CSS 变量**: 通过监听 `style.setProperty` 的调用来验证尺寸变化，而不是直接检查 DOM 尺寸。

3. **异步测试**: 大量使用 `waitFor`、`flushMicrotasks` 和 `act` 来处理异步状态更新。

4. **用户交互模拟**: 使用 `userEvent` 模拟真实的用户操作（hover、click、keyboard、tab）。

5. **spies**: 使用 `vi.spyOn` 监听样式属性的设置，验证尺寸计算逻辑的正确性。
