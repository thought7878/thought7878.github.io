### 1.2 三层架构

Base UI 采用清晰的三层架构：

```
┌─────────────────────────────────────────────────────────┐
│  组件层（packages/react/src/<component>/）              │
│  Menu / Dialog / Popover / Tabs / Combobox / ...        │
├─────────────────────────────────────────────────────────┤
│  领域工具层（packages/react/src/utils/ + internals/）   │
│  usePositioner / popups/store / composite / merge-props │
│  useRenderElement / Store 集成 / shadowDom / events     │
├─────────────────────────────────────────────────────────┤
│  基础设施层（packages/utils/src/）                      │
│  Store / platform / Hooks / 通用工具 / fastHooks        │
└─────────────────────────────────────────────────────────┘
```

- **基础设施层**：与 React 无强耦合的状态管理、平台检测、通用 Hooks，可独立发布。
- **领域工具层**：组件库内部共享的原语，包括渲染引擎 `useRenderElement`、弹出层状态模型 `PopupStoreState`、复合导航 `composite`、props 合并引擎 `mergeProps` 等。
- **组件层**：40+ 个面向用户的 headless 组件，统一遵循 Compound Component 模式。

### 1.3 核心设计理念

1. **Headless（无样式）**：组件不内置 CSS，通过 `data-*` 属性、CSS 变量、render prop *让用户完全掌控样式*。
2. **渲染统一**：所有 DOM 产出汇聚到 `useRenderElement`，支持 `render` prop 替换默认元素、state 函数式 `className`/`style`。
3. **状态外部化**：复杂组件采用 `ReactStore` + Context 注入，避免 props drilling，支持 detached trigger 与 imperative API。
4. **事件可取消**：`BaseUIEvent` + `reason` 体系*让用户能精细控制组件行为，所有交互都有明确归因*。
5. **全场景兼容**：Shadow DOM 安全工具、RTL 方向、CSP nonce、无障碍（VoiceOver/WebKit 适配）全面支持。

## 3. packages/utils — 基础设施层

`@base-ui/utils` 是 Base UI 的基石，提供状态管理、平台抽象、React Hooks 工具和通用工具函数。

### 3.1 状态管理系统（store/）

采用观察者模式实现与 React 深度集成的外部状态管理，让状态可跨组件共享而无需 props 逐层传递。

#### `Store.ts` — 基础状态存储类

泛型类 `Store<State>`，与 React 无强耦合（除 `use` 方法）。

| API | 说明 |
|-----|------|
| `state: State` | 当前状态，调用 setter 后立即更新，可在 effect/事件中直接读取 |
| `subscribe(fn)` | 注册监听器，返回取消订阅函数（`Set<Listener>` 存储） |
| `getSnapshot()` | 返回当前 state，供 `useSyncExternalStore` 使用 |
| `setState(newState)` | 替换整个状态，用 `Object.is` 比较；通过 `updateTick` 防护递归调用 |
| `update(changes)` | 浅合并 `Partial<State>`，逐字段比较无变化则不触发 |
| `set(key, value)` | 更新单个字段，仅值变化时触发 |
| `notifyAll()` | 给 state 新引用并通知所有监听器（强制刷新） |
| `use(selector, ...args)` | 便捷方法，内部调用 `useStore` |

导出 `ReadonlyStore<State>` = `Pick<Store, 'getSnapshot' \| 'subscribe' \| 'state'>`。

#### `ReactStore.ts` — React 专用扩展

继承 `Store`，增加 `<State, Context, Selectors>` 三个泛型参数。

| API | 说明 |
|-----|------|
| `context: Context` | 存放非响应式值（ref、稳定回调），不触发重渲染 |
| `selectors` | 选择器集合，配合 `useState`/`select`/`observe` 使用 |
| `useSyncedValue(key, value)` | 将单个外部值同步到 store（`useIsoLayoutEffect`） |
| `useSyncedValueWithCleanup(key, value)` | 同上，卸载时将 key 设为 `undefined` |
| `useSyncedValues(statePart)` | 批量同步多个外部值，开发模式校验 keys 形状一致 |
| `useControlledProp(key, controlled)` | 注册受控 prop 对，检测受控/非受控切换 |
| `select(key, ...args)` | 非响应式用选择器派生值 |
| `useState(key, ...args)` | 响应式订阅选择器结果 |
| `useContextCallback(key, fn)` | 用 `useStableCallback` 包装函数赋到 `context[key]` |
| `useStateSetter(key)` | 返回稳定的 setter 函数（用 `useRef` 缓存） |
| `observe(selector, listener)` | 观察选择器派生值变化，首次立即触发一次 |

#### `useStore.ts` — React 订阅桥接

将 Store 接入 React 的核心 Hook，支持 0-3 个额外参数传给 selector。

**双实现策略**：
- React 19+：`useStoreR19`，用原生 `useSyncExternalStore`
- React 18 及以下：`useStoreLegacy`，用 `useSyncExternalStoreWithSelector`（shim）

**Fast Hooks 优化**（`useStoreFast`）：当处于 `fastComponent` 上下文时，将所有 `useStore` 调用收集到 `instance.syncHooks`，在 `after` 钩子中用**单个** `useSyncExternalStore` 订阅所有 store，大幅减少 Hook 数量。

#### `createSelector.ts` / `createSelectorMemoized.ts`

- `createSelector`：纯函数组合（无 memoization），支持最多 6 个选择器组合
- `createSelectorMemoized`：基于 `reselect` 的 `lruMemoize`（`maxSize: 1`），按 state 缓存（`WeakMap`），适用于计算成本高的选择器

#### `StoreInspector.tsx`

开发调试工具，浮动可拖拽窗口可视化 Store 的 state/context，支持 Portal 渲染、`useTimeout` 节流、JSON 序列化（处理循环引用和 DOM Element）。

### 3.2 平台抽象层（platform/）

提供 SSR 安全的静态平台检测标志，所有标志在模块加载时一次性求值，`navigator` 未定义时（SSR）为 `false`。

| 文件 | 检测内容 |
|------|----------|
| `shared.ts` | 数据读取层，`readRawData()` 读取 UA/platform/maxTouchPoints |
| `engine.ts` | `webkit`（CSS.supports 检测）、`gecko`、`blink`，互斥构造 |
| `os.ts` | `ios`（含 iPadOS 13+ 识别）、`android`、`mac`、`windows`、`linux`、`apple` |
| `screen-reader.ts` | `voiceOver`（等于 `apple`，仅标识可能运行平台） |
| `env.ts` | `jsdom`（测试环境判断） |
| `parts.ts` / `index.ts` | 聚合导出，`platform.os.mac`、`platform.engine.webkit` 等访问方式 |

**设计原则**：仅检测运行时不变的特征；tree-shaking 友好（命名空间聚合）。

### 3.5 Fast Hooks 框架（fastHooks.ts）

渲染期上下文传递与 Hook 批量化优化框架，是性能优化核心机制。

| API | 说明 |
|-----|------|
| `fastComponent(fn)` | 高阶函数包装组件，渲染时设置 `currentInstance`，调用 before/after 钩子 |
| `fastComponentRef(fn)` | 结合 `React.forwardRef` 的版本 |
| `register(hook)` | 注册 `{ before, after }` 钩子 |
| `getInstance()` | 获取当前渲染中的 instance（非 fastComponent 上下文返回 `undefined`） |

**工作原理**：`fastComponent` 包装的组件内，所有 `useStore` 调用走 `useStoreFast` 路径，将 N 次 `useSyncExternalStore` 合并为 1 次，显著减少 Hook 数量和订阅开销。对订阅大量 store 字段的复杂组件（Menu、Select、Combobox）性能提升明显。

### 4.2 核心内部机制（internals/）

`internals/` 是整个库的"引擎室"，存放所有组件共享的低层机制。

#### `useRenderElement` — 渲染引擎

文件：`internals/useRenderElement.tsx`

整个库最核心的 Hook，所有组件最终都通过它产出 DOM 元素。

**输入参数三件套**：
- `element`：默认 HTML 标签名（如 `'div'`、`'button'`），可被 `render` prop 覆盖
- `componentProps`：含 `className`、`style`、`render` 三个用户可定制属性
- `params`：内部状态与配置（`state`、`ref`、`props`、`stateAttributesMapping`、`enabled`）

**核心流程**：
1. **className/style 解析**：通过 `resolveClassName`/`resolveStyle` 支持"字符串"或"基于 state 的函数"两种形式
2. **状态属性映射**：`getStateAttributesProps(state, stateAttributesMapping)` 把 state 对象转为 `data-*` 属性
3. **props 合并**：`mergeObjects(stateProps, resolvedProps)` 合并状态属性与显式 props
4. **ref 合并**：`useMergedRefs`/`useMergedRefsN` 合并内部 ref、forwardedRef、render 元素的 ref（服务端跳过）
5. **render prop 求值**：
   - 函数：`render(props, state)` 返回元素
   - ReactElement：`React.cloneElement` 克隆并合并 props
   - 都没有：用 `renderTag` 创建默认标签（`button` 自动加 `type="button"`，`img` 自动加 `alt=""`）
6. **enabled 短路**：`enabled === false` 时返回 `null`

**特殊处理**：React Lazy 元素兼容（issue #32392）、开发环境 `warnIfRenderPropLooksLikeComponent` 警告。

#### `useRender` — 公开包装

文件：`use-render/useRender.ts`

`useRenderElement` 的精简公开版本，接收单一 `params` 对象（含 `defaultTagName`，默认 `'div'`），通过 `namespace useRender` 暴露类型，适合用户自定义组件时使用。

#### `RequestQueue` — 并发请求队列

通用并发控制类，用于异步数据加载（如 Combobox 远程搜索）。
- 维护 `pendingRequests` 与 `queuedRequests` 两个 Map
- `maxConcurrentRequests` 限制并发数（默认 Infinity）
- `pickEntries(count)` 默认 FIFO，子类可重写实现优先级
- `getRequestStatus(key)` 返回 `'queued' \| 'pending' \| 'unknown'`

#### `clamp` / `filter` / `shadowDom` — 工具函数

- **`clamp.ts`**：`Math.max(min, Math.min(val, max))`，默认边界为 `MIN/MAX_SAFE_INTEGER`
- **`filter.ts`**：基于 `Intl.Collator` 的本地化字符串过滤工厂 `getFilter(options)`，返回 `{ contains, startsWith, endsWith }`
- **`shadowDom.ts`**（**AGENTS.md 强制使用**）：
  - `activeElement(doc)`：递归穿透 shadow root 找真实 active 元素
  - `contains(parent, child)`：沿 `parentNode`/`host` 链穿越 shadow boundary
  - `getTarget(event)`：优先 `composedPath()[0]` 获取真实事件目标

#### `getStateAttributesProps` + `stateAttributesMapping`

- `getStateAttributesProps(state, customMapping)`：遍历 state，`true` → `data-key=""`，其他真值 → `data-key="value"`，假值跳过
- `transitionStatusMapping`：预定义映射，`'starting'`/`'ending'` → `data-starting-style`/`data-ending-style`

#### `createBaseUIEventDetails` — 事件系统

- `ReasonToEventMap`：reason 字符串映射到原生事件类型
- `BaseUIChangeEventDetails<Reason, CustomProperties>`：含 `reason`、`event`、`cancel()`、`allowPropagation()`、`trigger` 及自定义属性
- `createChangeEventDetails(reason, event, trigger, customProperties)`：工厂函数创建可取消事件详情
- `createGenericEventDetails`：不可取消版本
- `reasons.ts`：所有 reason 常量（`REASONS.triggerPress`、`REASONS.escapeKey` 等）

#### `useTransitionStatus` — 过渡状态机

管理组件进出场动画状态机：`undefined → 'starting' → 'idle' → 'ending' → undefined`
- `mounted` 跟踪是否在 DOM 中（open 关闭时延迟 unmount 播放退出动画）
- `enableIdleState`：在 starting 与 ending 之间插入 idle 态
- `deferEndingState`：用 `AnimationFrame` 延迟一帧进入 ending 态
- 所有状态切换通过 `useIsoLayoutEffect` + `AnimationFrame.request` 调度，避免 `flushSync`（Firefox 兼容）

#### `merge-props` — Props 合并引擎

文件：`merge-props/mergeProps.ts`

整个库 props 合并的核心，被 `useRenderElement` 内部使用。

**合并规则**：
- **普通属性**：右覆盖左（Object.assign 语义）
- **事件处理器**：右到左链式调用，最右的先执行；`event.preventBaseUIHandler()` 阻止左侧（内部）处理器
- **className**：右到左拼接（`their + ' ' + our`），外部类优先
- **style**：`mergeObjects` 深合并，外部覆盖内部
- **ref**：不合并（需用 `useMergedRefs` 单独处理）

**函数式 props**：支持 `(previousProps) => props` 形式。`mergeProps` 固定 5 参数重载避免数组开销；`mergePropsN` 处理任意数量。`isEventHandler` 通过字符码判断（`on` + 大写字母）。

### 4.4 组件通用设计模式

#### Compound Component 模式 + 命名空间导出

每个组件目录有两个入口文件：
- **`index.ts`**：`export * as ComponentName from './index.parts'`（命名空间）+ `export type *`（类型）
  - 用户用法：`import { Menu } from '@base-ui/react'; <Menu.Root>...`
- **`index.parts.ts`**：导出各 part 的具名别名（`Root`、`Trigger`、`Popup`、`Item` 等）
  - 内部用法：`import { Root, Trigger } from './index.parts'`

#### Context 共享 + Store 模式

**Root 不渲染 DOM，只提供 Context**：
- `MenuRoot`、`DialogRoot`、`ComboboxRoot`、`PopoverRoot` 等都是"逻辑组件"
- 它们创建 Store 实例并通过 `<XxxRootContext.Provider value={{ store, ... }}>` 注入

**Context 文件约定**：
- 导出 `XxxRootContext`（`React.createContext`）
- 导出 `useXxxRootContext(optional?)` 重载：`optional=true` 返回 `undefined`，否则缺失时抛错
- 支持"向上查找父级"（如 `useMenuRootContext(true)` 用于子菜单探测父菜单）

**Store 模式**（基于 `@base-ui/utils/store` 的 `ReactStore`）：
- 每个弹出层组件有独立 Store 类（`MenuStore`、`DialogStore`、`PopoverStore` 等）
- Store 持有 `state`（不可变快照）+ `context`（refs、回调等可变引用）+ `selectors`（记忆化选择器）
- 组件通过 `store.useState('fieldName')` 或 `store.useState(selector, ...args)` 订阅
- `store.useControlledProp('openProp', openProp)` 同步受控 prop
- `store.useSyncedValues({ ... })` 同步多个派生值
- `store.useContextCallback('onOpenChange', onOpenChange)` 把外部回调挂到 context 上
- `XxxStore.useStore(externalStore, initialState)` 支持 `handle` 注入外部 store（detached trigger）

**父子 Store 传播**：`MenuStore` 构造时通过 `this.observe('parent', ...)` 监听父级变化，自动订阅父 store 的 `rootId`、`floatingTreeRoot`、`keyboardEventRelay` 等字段，实现嵌套菜单状态穿透。

#### parts 文件的作用

- **逻辑分文件**：每个 part（`Root`/`Trigger`/`Popup`/`Positioner`/`Item` 等）独立成文件
- **`*.test.tsx`**：与源码同目录，文件名 `Xxx.test.tsx`
- **`*.spec.tsx`**：更高层集成测试
- **`*Context.ts`**：part 向子 part 暴露数据
- **`*DataAttributes.ts`**：声明该 part 暴露的 `data-*` 属性常量
- **`*CssVars.ts`**：声明 CSS 自定义属性
- **`stateAttributesMapping.ts`**：组件私有的 state→data-* 映射
- **`store/` 子目录**：复杂组件的 Store 与 Handle（imperative API）分开存放

#### render prop + state 函数式 className/style

所有组件统一支持：
- `render={<CustomElement />}` 或 `render={(props, state) => <CustomElement {...props} />}` 替换默认元素
- `className` 既可是字符串，也可是 `(state) => string`
- `style` 同理

#### BaseUIEvent 可取消事件

所有事件回调接收 `eventDetails`，包含 `reason`、`event`、`cancel()`、`trigger`，让用户能精细控制组件行为（如阻止 ESC 关闭）。

#### floating-ui 集成

`packages/react/src/floating-ui-react/` 是 floating-ui 的内嵌定制版：
- `hooks/`：`useClick`、`useDismiss`、`useFloating`、`useFocus`、`useHover`、`useListNavigation`、`useTypeahead`
- `components/`：`FloatingRootStore`、`FloatingTreeStore`、`FloatingFocusManager`、`FloatingNode`、`FloatingTree`
- `utils/`：`tabbable`、`enqueueFocus`、`markOthers`、`nodes` 等

弹出层组件通过 `useSyncedFloatingRootContext` 把 popup store 与 floating-ui 连通，再用 `useDismiss`/`useListNavigation`/`useTypeahead` 装配交互。

### 4.5 代表性组件内部结构剖析

#### Menu（最复杂的弹出层组件）

**parts 列表**：Root、SubmenuRoot、Trigger、SubmenuTrigger、Positioner、Popup、Portal、Item、LinkItem、CheckboxItem、CheckboxItemIndicator、RadioGroup、RadioItem、RadioItemIndicator、Group、GroupLabel、Arrow、Backdrop、Viewport、Separator、Handle/createHandle

**MenuRoot 关键设计**：
- 用 `fastComponent`（而非 `React.forwardRef`）包装，因为 Root 不渲染 DOM
- **多父级类型判定**：通过 `useMenuRootContext(true)` / `useMenubarContext(true)` / `useContextMenuRootContext(true)` / `useMenuSubmenuRootContext()` 探测父级，构造 `MenuParent` 联合类型
- **Store 初始化**：`MenuStore.useStore(handle?.store, { open: defaultOpen, openProp, activeTriggerId, triggerIdProp, parent })`
- **floating-ui 装配**：`useSyncedFloatingRootContext` 桥接、`useDismiss`（context-menu 有 500ms grace period）、`useListNavigation`（支持 loop、orientation、RTL、嵌套）、`useTypeahead`
- **props 预合并**：在 Root 中就把 `activeTriggerProps`/`inactiveTriggerProps`/`popupProps`/`itemProps` 算好存入 store
- **setOpen 状态机**：处理 touch 延迟点击、键盘点击、dismiss 关闭，设置 `instantType`（`'click'|'dismiss'|'group'|'trigger-change'`）控制 CSS 过渡
- **FloatingTree 包装**：顶层 Root 用 `<FloatingTree externalTree={floatingTreeRoot}>` 包裹

**MenuStore 关键设计**：
- `State<Payload>` 扩展 `PopupStoreState<Payload>`，增加 `disabled`、`modal`、`openMethod`、`activeIndex`、`hoverEnabled`、`instantType`、`openChangeReason`、`floatingTreeRoot`、`itemProps`、`closeDelay`、`keyboardEventRelay`、`hasViewport` 等
- `Context` 持有大量 refs：`positionerRef`、`popupRef`、`typingRef`、`itemDomElements`、`itemLabels`、`allowMouseUpTriggerRef`、`triggerFocusTargetRef`、`beforeContentFocusGuardRef`
- **选择器继承与扩展**：`selectors = { ...popupStoreSelectors, ... }`，新增 `disabled`、`modal`、`rootId`、`floatingTreeRoot`、`keyboardEventRelay`、`isActive(itemIndex)` 等
- **`setOpen` 方法**：通过 `floatingRootContext.context.events.emit('setOpen', ...)` 派发，让 floating-ui 事件链先处理

**MenuPositioner**：调用 `useAnchorPositioning` 计算定位，根据 `parent.type` 调整默认 side/align/collisionAvoidance，通过 `floatingTreeRoot.events` 监听 `menuopenchange`/`itemhover` 实现"打开同级 submenu 时关闭兄弟 submenu"。

**MenuPopup**：用 `useRenderElement('div', ...)` 渲染，包裹 `<FloatingFocusManager>` 处理焦点陷阱，`useHoverFloatingInteraction` 处理 hover 开关延迟。

**MenuItem**：典型 leaf part，`useCompositeListItem({ label })` 注册到列表、`useBaseUiId` 生成 id、`useMenuItem` 获取交互 props，`store.useState('isActive', listItem.index)` 订阅"自己是否高亮"。

#### Dialog（模态对话框）

**parts**：Root、Trigger、Portal、Popup、Backdrop、Title、Description、Close、Viewport

**关键设计**：
- `DialogRoot` 极简，仅判断 `IsDrawerContext` 决定 mode（`'dialog' | 'drawer'`），委托给 `useRenderDialogRoot`
- `useRenderDialogRoot`：组装 Store、处理 `defaultOpen` 首帧同步、条件渲染 `<DialogInteractions>`、用 `<IsDrawerContext.Provider value={false}>` 阻止子级误判
- `DialogPopup`：state 含 `nestedDialogOpen`（嵌套对话框计数），映射到 `data-nested-dialog-open`，通过 `DialogPopupCssVars.nestedDialogs` 暴露为 CSS 变量
- **Drawer 复用 Dialog**：通过 `IsDrawerContext` 切换 mode，共享 Store 与大部分逻辑

#### Tabs（非弹出层 + 复合导航）

**parts**：Root、List、Tab、Panel、Indicator

**关键设计**：
- **Root 渲染 DOM**（与 Menu/Dialog 不同），用 `useRenderElement('div', ...)` 输出容器
- **受控/非受控值**：`useControlled({ controlled: valueProp, default: defaultValueProp, name: 'Tabs', state: 'value' })`
- **tabMap 注册体系**：用 `Map<Node, CompositeMetadata<TabsTab.Metadata> | null>` 跟踪所有 Tab 元素
- **激活方向计算**（`computeActivationDirection`）：值变化时通过 `getBoundingClientRect` 比较新旧 Tab 位置，推断 `'left'|'right'|'up'|'down'|'none'`，供 Indicator 动画使用
- **自动回退逻辑**：非受控模式下，当选中 Tab 被禁用/移除时自动切换到首个启用 Tab

#### Combobox（输入 + 弹出层 + 多选）

**parts**（最多）：Root、Label、Trigger、Input、InputGroup、Popup、Positioner、List、Item、ItemIndicator、Value、Icon、Arrow、Backdrop、Portal、Empty、Group、GroupLabel、Row、Chips、Chip、ChipRemove、Clear、Status、Collection

**关键设计**：
- **泛型组件**：`ComboboxRoot<Value, Multiple extends boolean | undefined = false>`，通过 `Multiple` 类型参数区分单选/多选
- **委托给 `AriaCombobox`**：Root 仅做 prop 适配，真正逻辑在 `AriaCombobox`（基于 aria 规范的实现）
- **`root/utils/`**：`useFilter`（封装 `internals/filter.ts`）、`useFilteredItems`（过滤 + 排序 + 高亮）
- **复杂 part**：`ComboboxChip`/`ComboboxChips`/`ComboboxChipRemove` 处理多选标签；`ComboboxStatus` 显示过滤结果数；`ComboboxCollection` 支持虚拟化数据源

#### NavigationMenu（导航菜单）

**parts**：Root、List、Item、Content、Trigger、Portal、Positioner、Viewport、Backdrop、Popup、Arrow、Link、Icon

**关键设计**：与 Menu 类似的弹出层结构，但专为顶部导航栏场景优化。`Viewport` part 是其特色：所有子菜单内容共享一个 viewport 区域，实现切换动画。

|------|------|

```bash

# 多浏览器

# E2E
pnpm test:e2e
# 视觉回归
pnpm test:regressions
```

**测试约定**（AGENTS.md）：
- 文件命名：`name.test.tsx`（与源文件同目录）
- 仅用 Vitest API：`expect()`、`vi.fn()`、`@testing-library/jest-dom`
- 浏览器专属测试：`it.skipIf(isJSDOM)` 或 `describe.skipIf(isJSDOM)`
- 不在 `await render(...)` 后直接 `await flushMicrotasks()`
- 每个 `expect()` 独立 `waitFor()`

## 8. 关键设计模式与约定

### 8.1 AGENTS.md 强制规范

#### 必须使用的工具函数

| 场景 | 必须使用 | 替代 |
|------|----------|------|
| 定时器 | `useTimeout`（`@base-ui/utils/useTimeout`） | `window.setTimeout` |
| 动画帧 | `useAnimationFrame`（`@base-ui/utils/useAnimationFrame`） | `requestAnimationFrame` |
| effect/事件中的回调 | `useStableCallback`（`@base-ui/utils/useStableCallback`） | `React.useCallback` |
| 布局 effect | `useIsoLayoutEffect`（`@base-ui/utils/useIsoLayoutEffect`） | `React.useLayoutEffect` |
| Shadow DOM 遍历 | `contains`、`getTarget`、`activeElement`（`internals/shadowDom`） | 原生 DOM API |
| owner 查找 | `ownerDocument`、`ownerWindow`（`@base-ui/utils/owner`） | 全局 `document`/`window` |

> **注意**：`useStableCallback` 不能用于渲染期间调用的函数（会抛错），渲染期间仍用 `React.useCallback`。

#### 样式规范

- CSS Modules 中使用 Tailwind `@theme` 块的原始颜色值（如 `oklch(14.5% 0 0deg)` 而非 `var(--color-neutral-950)`）
- `user-select: none;` 需同时添加 `-webkit-user-select: none;`（Safari 支持）

#### 错误消息规范

每个错误消息必须：
1. **说发生了什么** — 清晰描述问题
2. **说为什么是问题** — 解释后果
3. **指向解决方法** — 给出可操作指导

格式：
- 前缀 `Base UI:`
- 字符串拼接
- 包含文档链接（`https://base-ui.com/...`）

修改错误消息后**必须**运行 `pnpm extract-error-codes` 更新 `docs/src/error-codes.json`。

#### 提交规范

格式：`[scope] Imperative summary`（如 `[popover] Fix focus trap`）
- 广泛影响使用 `[all components]` scope
- scope 镜像包或组件名

### 8.2 文档贡献流程

#### 添加新组件文档页面

1. 创建 `docs/src/app/(docs)/react/components/{component}/page.mdx`
2. 创建 `types.ts`：
   - 单部件组件：`createTypes(import.meta.url, Component)`
   - 多部件组件：`createMultipleTypes(import.meta.url, Component)`
3. 在 MDX 中使用：`import { TypesToggle } from './types'; <TypesToggle />`
4. 运行 `pnpm docs:dev` 访问页面自动生成 `types.md`

#### 添加演示（Demo）

1. 在 `demos/` 创建组件：`demos/basic/ButtonBasic.tsx`
2. 创建 `index.ts`：`createDemo(import.meta.url, ButtonBasic)`
3. 多变体演示用 `createDemoWithVariants`

#### 页面结构约定

- **标题**：来自 `page.mdx` 顶部的 `# h1`
- **描述**：来自 h1 后的 `<Subtitle>` 组件
- **metadata**：放置在 `page.mdx` **末尾**的 `export const metadata = {}`
- 隐藏页面：`robots: { index: false }`

#### 自动生成文件（不可手动编辑）

- `types.md` 文件
- 部分 `page.mdx` 页面索引（含自动生成注释）

### 8.3 命名空间类型规范

对于命名空间暴露的类型，必须同时作为全局可访问类型暴露，否则 `createTypes` 无法工作：

```ts
export type AlertDialogPopupProps = { /* ... */ };
export declare namespace AlertDialogPopup {
  type Props = AlertDialogPopupProps;
}
```

症状：`types.md` 显示 `AlertDialogPopupProps` 而非 `AlertDialog.Popup.Props`。
