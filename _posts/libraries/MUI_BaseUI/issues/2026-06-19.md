基于对 https://github.com/mui/base-ui/issues 的系统性分析，以下是符合高质量标准的未解决 bug 总结。筛选标准包括：清晰的复现步骤、完整的环境信息、错误截图/录屏、已确认非用户配置问题、具有普遍性或严重影响用户体验。

## 一、严重级别：高（Critical）

### 1. [select] React StrictMode 下鼠标点击无法选中选项
- **链接**: https://github.com/mui/base-ui/issues/4698
- **问题描述**: 在 React StrictMode 开发环境（如 Next.js dev server）中，使用鼠标点击 `Select.Item` 完全无效，不会触发 `onValueChange`，但键盘导航正常。生产环境正常。
- **复现步骤**:
  1. 在 Next.js dev 环境（默认启用 StrictMode）中打开 Select 组件
  2. 点击触发器打开下拉菜单
  3. 点击任意选项，触发器保持显示占位符
- **预期行为**: 鼠标点击选项应正常提交值
- **实际行为**: 点击无效，下拉菜单可能关闭但不提交值
- **环境信息**: Base UI v1.4.1, Chrome 130, macOS, Next.js 14 dev
- **当前状态**: 已确认 bug，有 PR 在修复中
- **严重程度**: 高 - 影响所有使用 Next.js 开发环境的用户，鼠标交互完全失效

### 2. [dialog] 3+ 个兄弟对话框同时打开时 Escape 键关闭多个对话框
- **链接**: https://github.com/mui/base-ui/issues/4583
- **问题描述**: 当 3 个或更多非嵌套的兄弟模态对话框同时打开时，按 Escape 键会关闭多个对话框而不是仅关闭最顶层的一个。2 个对话框时正常。
- **复现步骤**:
  1. 同时打开 3 个或更多兄弟对话框
  2. 按 Escape 键
  3. 观察到多个对话框同时关闭
- **预期行为**: 每次按 Escape 只关闭最顶层的对话框
- **实际行为**: 3+ 个对话框时，第一次 Escape 关闭最顶层，剩余的一起关闭
- **环境信息**: Base UI v1.3.0, Chrome/Safari/Firefox, macOS
- **当前状态**: 已确认 bug，有相关 PR
- **严重程度**: 高 - 破坏模态对话框的基本交互逻辑，影响多对话框场景

### 3. [dialog] 滚动锁定机制与非 Base UI 模态覆盖层交互问题
- **链接**: https://github.com/mui/base-ui/issues/4640
- **问题描述**: 当从非 Base UI 的模态覆盖层（如 Ariakit Menu）打开 Base UI Dialog 时，页面滚动可能进入无法恢复的锁定状态。原因是不同库的滚动锁定清理时序不协调。
- **复现步骤**:
  1. 打开 Ariakit Menu（设置 `overflow: hidden`）
  2. 点击菜单项，Menu 开始关闭但有动画延迟
  3. 在 Menu 的滚动锁定清理前打开 Base UI Dialog
  4. Dialog 错误地恢复了页面滚动
  5. 关闭 Dialog 后，页面保持滚动锁定状态
- **预期行为**: 跨库的滚动锁定应正确协调和交接
- **实际行为**: 页面滚动被永久锁定
- **环境信息**: Base UI v1.4, Chrome, macOS
- **当前状态**: 已确认 bug，有 PR 在修复中
- **严重程度**: 高 - 导致页面完全无法滚动，严重影响用户体验

## 二、严重级别：中（High）

### 4. [tabs] Tabs.Indicator 在 3D 旋转父容器中位置错乱
- **链接**: https://github.com/mui/base-ui/issues/4837
- **问题描述**: 当 `Tabs.Indicator` 嵌套在 3D 变换过渡的容器中（如卡片翻转动画的 `rotateY`），指示器的视觉位置会失去同步并卡在错误位置。
- **复现步骤**:
  1. 将 Tabs 组件放在带有 `transform: rotateY(...)` 过渡的容器中
  2. 触发卡片翻转动画
  3. 观察指示器位置错乱
- **预期行为**: 过渡完成后指示器应与活动标签保持对齐
- **实际行为**: 指示器位置错乱并卡在错误位置
- **环境信息**: Base UI v1.4.1, Firefox Developer Edition, macOS
- **当前状态**: 已确认 bug，有 PR 在修复中
- **严重程度**: 中 - 影响使用 3D 动画的场景

### 5. [menu] **父菜单动画期间子菜单位置跳动**
- **链接**: https://github.com/mui/base-ui/issues/4224
- **问题描述**: 当 `SubmenuRoot` 设置 `defaultOpen={true}` 时，子菜单在父菜单进行过渡动画期间就确定位置，导致子菜单先出现在错误位置，动画结束后"瞬移"到正确位置。
- **复现步骤**:
  1. 设置子菜单 `defaultOpen={true}`
  2. 打开父菜单
  3. 观察子菜单位置跳动
- **预期行为**: 子菜单应在父菜单动画完成后打开
- **实际行为**: 子菜单位置跳动，视觉体验差
- **环境信息**: Base UI v1.2.0, Chrome, macOS
- **当前状态**: 已确认 bug，有 PR 在修复中
- **严重程度**: 中 - 影响视觉体验和动画流畅性

### 6. [avatar] SSR 水合时缓存图片会闪烁 Fallback
- **链接**: https://github.com/mui/base-ui/issues/4468
- **问题描述**: 在 SSR 水合期间，即使图片已在浏览器缓存中，`Avatar.Fallback` 仍会产生可见闪烁。`useImageLoadingStatus` 没有同步的缓存图片快速路径，总是异步初始化。
- **复现步骤**:
  1. 使用 SSR 框架（Next.js/TanStack Start）
  2. 渲染带缓存图片的 Avatar 组件
  3. 观察到红色 fallback 闪烁
- **预期行为**: 缓存图片应立即渲染，不显示 Fallback
- **实际行为**: Fallback 闪烁，SSR 场景下持续约 304ms（TanStack Start）或 52ms（Next.js）
- **环境信息**: Base UI v1.3.0, Chrome/Firefox, Windows/macOS/Linux
- **当前状态**: 已确认 bug，有 PR 在修复中
- **严重程度**: 中 - 影响 SSR 应用的视觉体验

### 7. [all components] 包结构破坏 Sandpack 和 CDN 解析
- **链接**: https://github.com/mui/base-ui/issues/4335
- **问题描述**: v1.2.0 和 v1.3.0 的构建输出和导出配置完全破坏了 Sandpack 等浏览器环境。v1.3.0 在 CDN 上无法加载，v1.2.0 因内部 `.js` 导入导致模块解析错误。
- **复现步骤**:
  1. 在 Sandpack 的 react-ts 模板中使用 Base UI v1.2.0 或 v1.3.0
  2. 导入任意组件
  3. 观察到模块解析错误
- **预期行为**: Base UI 应可在 CodeSandbox/Sandpack 中直接使用
- **实际行为**: 需要手动注入虚拟文件或自定义 esbuild 步骤
- **环境信息**: Base UI v1.2.0/v1.3.0, Sandpack
- **当前状态**: 已确认问题，等待包输出结构重构完成
- **严重程度**: 中 - 影响在线演示和代码沙箱的使用

## 三、严重级别：中低（Medium-Low）

### 8. [RSC] **styled-components 在 RSC 中传递 render prop 验证失败**
- **链接**: https://github.com/mui/base-ui/issues/4155
- **问题描述**: 在 React Server Components 中，当将带有样式的 styled-component 传递给 `render` prop 时，Base UI 抛出无效 React element 错误。无样式或全客户端组件时正常。
- **复现步骤**:
  1. 在 RSC 中使用 styled-components 创建组件
  2. 将 styled-component 传递给 Base UI 的 `render` prop
  3. 观察到验证错误
- **预期行为**: styled-component 应可正常传递给 render prop
- **实际行为**: 抛出 "invalid React element" 错误
- **环境信息**: Base UI v1.0.2, Firefox, Windows, Next.js RSC
- **当前状态**: 已确认 bug，暂无 PR
- **严重程度**: 中低 - 影响 RSC + styled-components 用户

### 9. [field] **data-focused 在 data-disabled 时持续存在**
- **链接**: https://github.com/mui/base-ui/issues/3987
- **问题描述**: 原生 HTML 元素在 disabled 时不可聚焦，但 Base UI 表单组件（Field、Input、Select 等）在 `data-disabled` 存在时 `data-focused` 仍持续存在。
- **复现步骤**:
  1. 使用 Base UI 表单组件
  2. 聚焦组件后禁用
  3. 观察到 `data-focused` 和 `data-disabled` 同时存在
- **预期行为**: `data-disabled` 时应移除 `data-focused`
- **实际行为**: 两个属性同时存在，需要用 `not-data-disabled` 修饰符 workaround
- **环境信息**: 所有环境
- **当前状态**: 已确认 bug，有 PR 在修复中
- **严重程度**: 中低 - 影响样式逻辑，有 workaround

### 10. [navigation menu] **水合前发生的 hover 事件不会被 React 重放**
- **链接**: https://github.com/mui/base-ui/issues/3570
- **问题描述**: SSR 渲染 NavigationMenu 时，水合完成前的鼠标悬停事件会丢失，用户需要再次悬停才能打开菜单。
- **复现步骤**:
  1. SSR 渲染 NavigationMenu
  2. 页面加载后立即将鼠标移到触发器上（水合完成前）
  3. 观察到菜单不打开
  4. 移开鼠标再次悬停，菜单正常打开
- **预期行为**: 水合完成时如果指针已在触发器上，菜单应立即打开
- **实际行为**: 首次悬停事件丢失，需要二次悬停
- **环境信息**: Base UI v1.0.0, Chrome, macOS
- **当前状态**: 已确认 bug，有 workaround，有 PR 但存在问题
- **严重程度**: 中低 - 影响首次交互体验，有 workaround

### 11. [combobox] iOS 打开键盘时 --available-height 意外变化
- **链接**: https://github.com/mui/base-ui/issues/3227
- **问题描述**: 在 iOS 上，当 combobox 的 filter input 聚焦并打开键盘时，菜单大小会意外变化，可能是 `--available-height` 计算问题。
- **复现步骤**:
  1. 在 iOS 设备上访问 combobox 示例
  2. 聚焦 filter input 打开键盘
  3. 观察菜单大小异常变化
- **预期行为**: 键盘打开时菜单应调整大小
- **实际行为**: 菜单大小意外变化
- **环境信息**: Base UI v1.0.0-beta.4, Safari iOS, iOS 26/26.1
- **当前状态**: 已确认 bug，有相关 PR
- **严重程度**: 中低 - 仅影响 iOS 移动端

### 12. [menu] 使用绝对定位元素扩展触发器点击区域时菜单立即关闭
- **链接**: https://github.com/mui/base-ui/issues/3184
- **问题描述**: 当在 `Menu.Trigger` 内使用绝对定位元素扩展点击区域时，点击扩展部分会导致菜单打开后立即关闭。
- **复现步骤**:

  2. 点击扩展的点击区域
  3. 菜单打开后立即关闭
- **预期行为**: 点击触发器的任何部分都应正常打开菜单
- **实际行为**: 菜单立即关闭
- **环境信息**: Base UI v1.0.0-beta.4, Chrome/Safari/Firefox, macOS
- **当前状态**: 已确认 bug，有 workaround（使用伪元素）
- **严重程度**: 中低 - 影响无障碍设计，有 workaround

## 五、关键发现

1. **React StrictMode 兼容性问题**: Select 组件在 StrictMode 下的严重 bug 影响所有 Next.js 开发环境用户
2. **多模态对话框协调**: Dialog 组件在多实例场景下的 Escape 键和滚动锁定处理存在缺陷
3. **SSR/水合问题**: Avatar 和 Navigation Menu 在 SSR 场景下存在交互问题
4. **动画/过渡问题**: Tabs 和 Menu 在动画期间的定位和渲染存在问题
5. **移动端兼容性**: iOS 键盘打开时的视口高度计算问题
6. **跨库互操作**: 与其他 UI 库（如 Ariakit）的滚动锁定协调问题

所有列出的 bug 均已由维护者确认，大部分有相关 PR 在修复中。建议优先关注严重级别为"高"的 bug，特别是影响核心交互功能的 Select、Dialog 问题。
