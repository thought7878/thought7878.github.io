>在radix中 ，是否有shadcn 10979这个类型的issue？

是的，radix-ui/primitives 中存在这个类型的 issue。我找到了两个高度相关的：

参考：[[10979]]

# 

## 1. Issue \#3461（最匹配）
[Select inside Dialog or Popover closes both components unexpectedly on mobile](https://github.com/radix-ui/primitives/issues/3461)（仍为 Open 状态）

**症状完全一致**：在 Dialog/Popover 内打开 Select，然后点击 Dialog 内、Select 外的区域时，Select 和 Dialog **同时关闭**。该 issue 归因于 `@radix-ui/react-dismissable-layer`，提到与 portal 挂载/卸载和 `pointerdown`/`mousedown` 事件冒泡有关。注：报告者强调此问题在移动端/触控设备上更易复现。

## 2. Issue \#3142（根本原因完全一致）
[[Bug] Dialog and Popover conflict #3142](https://github.com/radix-ui/primitives/issues/3142)

这个 issue 的建议解决方案与 shadcn #10979 中 TylerGennaro 的修复方案**几乎一字不差**：

> "Setting `pointer-events: none` on the body is an excellent solution for preventing body scrolling while the Dialog is open, but it causes other modals to malfunction... `pointer-events: auto` should be applied to Popovers or third-party modals for proper functionality."

区别在于 #3142 是关于 Dialog + Popover，而 shadcn #10979 是关于 Dialog + Select，但**根因相同**：Dialog 打开子组件时给 body/DialogContent 设置了 `pointer-events: none`，导致点击穿透到 DialogOverlay，被识别为"外部点击"从而关闭 Dialog。修复方案也一致：在子组件的 Content 上强制设置 `pointer-events-auto`。

## 结论
该类型的 issue 在 radix-ui/primitives 中确实存在（#3461 和 \#3142），根因（`pointer-events: none` 穿透导致误判外部点击）和解决方案（`pointer-events: auto` 覆盖）都已记录在案。shadcn #10979 本质上是同一底层问题在 shadcn 封装层的表现。

# 为什么 Select 打开时会给 DialogContent 加 `pointer-events: none`

这不是 Select 主动对 DialogContent 做的操作，而是两者共用的 `DismissableLayer` 机制的副作用。下面是基于源码的解析。

## 一、谁在设置 `pointer-events: none`

是 [dismissable-layer.tsx](file:///workspace/packages/react/dismissable-layer/src/dismissable-layer.tsx) 在 `disableOutsidePointerEvents` 为 `true` 时做的。看 [dismissable-layer.tsx:145-170](file:///workspace/packages/react/dismissable-layer/src/dismissable-layer.tsx#L145-L170)：

```js
React.useEffect(() => {
  if (!node) return;
  if (disableOutsidePointerEvents) {
    if (context.layersWithOutsidePointerEventsDisabled.size === 0) {
      originalBodyPointerEvents = ownerDocument.body.style.pointerEvents;
      ownerDocument.body.style.pointerEvents = 'none';   // ① 把 body 置为 none
    }
    context.layersWithOutsidePointerEventsDisabled.add(node);
  }
  ...
});
```

而 **Dialog 和 Select 都开启了 `disableOutsidePointerEvents`**：
- Dialog: [dialog.tsx:287](file:///workspace/packages/react/dialog/src/dialog.tsx#L287) `disableOutsidePointerEvents={context.open}`
- Select: [select.tsx:856](file:///workspace/packages/react/select/src/select.tsx#L856) `disableOutsidePointerEvents`

## 二、为什么只有最顶层能 `auto`，下层变 `none`

关键在 [dismissable-layer.tsx:91-96](file:///workspace/packages/react/dismissable-layer/src/dismissable-layer.tsx#L91-L96)：

```js
const [highestLayerWithOutsidePointerEventsDisabled] =
  [...context.layersWithOutsidePointerEventsDisabled].slice(-1);   // 取最后一个（最新挂载的）
const highestLayerWithOutsidePointerEventsDisabledIndex =
  layers.indexOf(highestLayerWithOutsidePointerEventsDisabled!);
const index = node ? layers.indexOf(node) : -1;
const isPointerEventsEnabled =
  index >= highestLayerWithOutsidePointerEventsDisabledIndex;       // 只有最顶层为 true
```

再在 [dismissable-layer.tsx:197-204](file:///workspace/packages/react/dismissable-layer/src/dismissable-layer.tsx#L197-L204) 应用到自身 style：

```js
style={{
  pointerEvents: isBodyPointerEventsDisabled
    ? isPointerEventsEnabled
      ? 'auto'     // 顶层：恢复
      : 'none'     // 下层：禁用 ← Dialog 在 Select 打开后落到这里
    : undefined,
  ...props.style,
}}
```

所以当 Select 在 Dialog 中打开时：
1. `body { pointer-events: none }`（本来 Dialog 就已经设置了，Select 不会重复设置）
2. Select 是后挂载的，成为 `highestLayerWithOutsidePointerEventsDisabled`，自身 content 拿到 `auto`
3. **Dialog 的 DismissableLayer 不再是顶层，`isPointerEventsEnabled` 变 `false` → 它的 content 被设为 `pointer-events: none`**

## 三、这套机制的目的（为什么要禁用 body 的指针事件）

源码注释见 [dismissable-layer.tsx:35-40](file:///workspace/packages/react/dismissable-layer/src/dismissable-layer.tsx#L35-L40)：

> When `true`, hover/focus/click interactions will be disabled on elements outside the `DismissableLayer`. Users will need to click twice on outside elements to interact with them: once to close the `DismissableLayer`, and again to trigger the element.

设计意图有三点：

1. **强制模态行为**：模态层打开时，背景的所有元素（包括其它下层模态）都不应被点到，避免误操作。
2. **保证"点外部即关闭"的可靠性**：因为除顶层外，`body` 及所有元素都 `pointer-events: none`，点击会穿透到 `document`，被 DismissableLayer 的 `pointerdown` 监听器捕获 → 判定为"外部点击" → 触发 `onPointerDownOutside` → 关闭。
3. **避免下层元素意外触发**：比如点 Select 外部时，不想顺手把 Dialog 里某个按钮也激活。

## 四、为什么这正好就是 shadcn #10979 的根因

由于 Dialog 的 content 被设成了 `pointer-events: none`，鼠标点 DialogContent 时事件**穿透到下层的 DialogOverlay**。而 DialogOverlay 通过 [dialog.tsx:207](file:///workspace/packages/react/dialog/src/dialog.tsx#L207) 的 `useDismissableLayerSurface()` 注册成了 dismissable surface，于是这个穿透下来的点击同时满足两个条件：
- 对 Select 而言：是一次"外部点击" → 关 Select
- 对 Dialog 而言：是点在了自己的 overlay（dismiss surface）上 → 关 Dialog

所以两者一起关闭。shadcn 那个 issue 里 TylerGennaro 给的修复 `pointer-events-auto!` 之所以有效，就是强制让 DialogContent 重新接收事件，阻断上面那条"穿透到 overlay"的路径。

参考已有 issue [radix-ui/primitives#3142](https://github.com/radix-ui/primitives/issues/3142) 和 [#3461](https://github.com/radix-ui/primitives/issues/3461) 也都提到了同一根因与同样的解法思路。
