## 问题背景

### 技术背景

`useEffect` 是 React 中用于处理副作用的 Hook，其基本模式为：

```javascript
useEffect(() => {
  // 副作用 setup 代码
  return () => {
    // cleanup 清理代码
  };
}, []);
```

根据 React 文档，**cleanup 函数应该在组件卸载（unmount）时被调用**，以清理副作用，如：
- 取消定时器 (`setTimeout`, `setInterval`)
- 移除事件监听器 (`addEventListener`)
- 取消网络请求
- 清理 DOM 操作

### 问题触发场景

该问题发生在以下条件同时满足时：
1. **启用了 `React.StrictMode`**
2. **使用了并发特性**（如 `startTransition`）
3. **组件在过渡期间卸载**

## 复现步骤

### 1. 环境准备

```javascript
import { StrictMode } from 'react';
import { createRoot } from 'react-dom/client';

createRoot(document.getElementById('root')).render(
  <StrictMode>
    <App />
  </StrictMode>
);
```

### 2. 创建带 useEffect 的组件

```javascript
import { useEffect, startTransition, useState } from 'react';

function MyComponent() {
  useEffect(() => {
    const handler = () => console.log('event fired');
    window.addEventListener('resize', handler);
    return () => {
      console.log('cleanup called'); // ⚠️ This does not log on unmount
      window.removeEventListener('resize', handler);
    };
  }, []);
  
  return <div>Hello</div>;
}
```

### 3. 通过 startTransition 触发卸载

```javascript
function App() {
  const [show, setShow] = useState(true);
  
  const handleUnmount = () => {
    startTransition(() => {
      setShow(false); // 在过渡期间卸载 MyComponent
    });
  };
  
  return (
    <div>
      {show && <MyComponent />}
      <button onClick={handleUnmount}>Unmount</button>
    </div>
  );
}
```

### 4. 观察结果

点击按钮后，`MyComponent` 会被卸载，但控制台**不会输出 "cleanup called"**。

---

## 当前行为 vs 预期行为

| 行为类型 | 描述 |
|---------|------|
| **当前行为** | 在 StrictMode + 并发特性下，组件卸载时 useEffect 的 cleanup 函数**不会**被调用 |
| **预期行为** | cleanup 函数**应该始终**被调用，无论 StrictMode 或并发特性是否启用 |
| **实际影响** | 内存泄漏、重复的事件监听器、悬停的定时器等 |

---

## 可能的原因分析

### 原因 1：StrictMode 的 Double-Invocation 机制

React 18 引入的 StrictMode 会在**开发环境**下**故意重复执行**某些操作*以帮助发现副作用问题*。但问题可能出在：

```
正常卸载流程：
setup → (组件保持) → unmount → cleanup ✓ 正确调用

Concurrent + StrictMode 可能的异常流程：
setup → (StrictMode 标记为"待卸载") → (过渡被中断?) → cleanup ✗ 未调用
```

### 原因 2：Concurrent Mode 的调度机制

`startTransition` 允许 React 在过渡期间**中断和恢复**工作。当一个过渡被中断时：

1. React 可能标记组件为"可恢复"状态而非立即卸载
2. 卸载调度可能被推迟
3. cleanup 的调用时机被错过

### 原因 3：Fiber 架构的边界情况

React Fiber 将组件工作拆分成多个"工作单元"（fiber nodes）。在并发模式下：

```
current tree → work in progress → commit phase
                    ↑
              可能在这里被中断
```

当组件的 fiber 在**提交阶段之前**被中断时，其 effect cleanup 可能被跳过。

### 原因 4：useEffect 调度的优先级

React 19 改进了 `useEffect` 的调度机制。在并发过渡期间：

```
User clicks → Transition starts → Component marked as "hidden"
                                        ↓
                            Fiber cleanup scheduled (low priority)
                                        ↓
                            Transition interrupted/cancelled
                                        ↓
                            Cleanup never executed ⚠️
```

## 技术细节分析

### 涉及的 React 内部机制

1. **`enableStrictMode` (Fiber 内部标志)**
   - 控制 StrictMode 的行为
   - 在开发环境下会重复执行 setup/cleanup

2. **`startTransition` API**
   - 标记状态更新为非紧急的过渡
   - 允许 React 中断和恢复工作

3. **Fiber Effects 链表**
   - React 使用链表管理 effect
   - `useEffect` 被组织在 `updateQueue` 中
   - 卸载时遍历此链表调用 cleanup

4. **Scheduler (调度器)**
   - React 19 使用新的调度机制
   - cleanup 任务可能被调度但未执行

## 问题影响

### 短期影响

| 影响类型 | 描述 |
|---------|------|
| **内存泄漏** | 事件监听器未移除，对象引用保留 |
| **重复监听器** | 组件重新挂载时创建新的监听器而不清理旧的 |
| **状态不一致** | setInterval 等持续运行，不受控制 |
| **性能下降** | 累积的监听器和定时器拖累应用性能 |

- 长时间运行的 SPA 会出现内存持续增长
- 用户设备资源被耗尽（电池寿命缩短、浏览器变慢）

## 临时解决方案

### 方案 1：避免在过渡中直接卸载

```javascript
// 使用 flushSync 确保同步卸载
import { flushSync } from 'react-dom';

const handleUnmount = () => {
  flushSync(() => {
    setShow(false);
  });
};
```

### 方案 2：使用 ref 跟踪清理状态

```javascript
function MyComponent() {
  const cleanupExecuted = useRef(false);
  
  useEffect(() => {
    if (cleanupExecuted.current) return;
    
    const handler = () => console.log('event fired');
    window.addEventListener('resize', handler);
    
    return () => {
      if (cleanupExecuted.current) return;
      cleanupExecuted.current = true;
      console.log('cleanup called');
      window.removeEventListener('resize', handler);
    };
  }, []);
  
  return <div>Hello</div>;
}
```

### 方案 3：手动管理事件监听器

```javascript
function MyComponent() {
  useEffect(() => {
    let mounted = true;
    const handler = () => console.log('event fired');
    
    window.addEventListener('resize', handler);
    
    return () => {
      mounted = false;
      window.removeEventListener('resize', handler);
    };
  }, []);
  
  return <div>Hello</div>;
}
```

## 总结

**Issue #36284** 揭示了 React 在 **StrictMode + Concurrent Mode** 组合使用时的边界情况 bug。问题的本质是：

> **在并发过渡期间卸载组件时，React 可能会跳过 useEffect cleanup 的执行**

这违反了 React 的核心契约——**副作用清理应该可靠执行**。

该问题值得深入研究，因为：
1. 它触及 React 并发架构的核心机制
2. 影响所有使用 React 18+ 新特性的应用
3. 修复需要仔细平衡性能和正确性

如果你想深入研究这个问题，建议从 React Fiber 的 `completeWork` 和 `commitWork` 阶段开始，重点关注 `useEffect` cleanup 的调度逻辑。
