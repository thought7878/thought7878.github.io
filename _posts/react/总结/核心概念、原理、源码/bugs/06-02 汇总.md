### 2. 性能优化类

#### #**36176** - [Fiber] Apps that have data-heavy props or state use hundreds of MBs of extra memory
- **创建时间**: 2026-04-01
- **状态**: Open
- **🔥 高关注度**
- **核心问题**: Fiber架构下,当新数据替换旧数据时,旧数据仍保留在内存中,导致双倍内存占用

- **讨论要点**: 这是Fiber设计的已知tradeoff,社区期待核心团队给出明确声明
- **变通方案**: 使用useRef或手动清理前一个数据

## 四、React当前面临的主要挑战

### 1. **React Compiler成熟度问题**
- AST序列化合同违反(#36327)暴露Compiler内部实现泄漏
- 条件渲染处理不完善(#36245)
- 解构props优化不足(#36281)影响性能优化效果
- **发展方向**: 需要加强Compiler的稳定性和优化覆盖率

### 2. **性能回归与内存管理**
- React 19的更新性能回归(#36229)显示版本迭代中的质量保障挑战
- Fiber架构的内存占用问题(#36176)长期存在,数据密集型应用受影响严重
- 开发模式下TypedArray处理性能问题(#36200)
- **发展方向**: 需要平衡新特性与性能稳定性,优化Fiber内存管理

### 3. **DevTools稳定性**
- 多个"Cannot remove/set node"相关Bug(#36315, #36375, #36577)表明DevTools与Suspense/Activity的集成存在系统性问题
- **发展方向**: 需要重构DevTools的节点同步机制

### 4. **SSR/RSC生态完善**
- Fizz的preloadModule崩溃(#36440)和referrerPolicy丢失(#36563)反映SSR功能仍在打磨
- React.lazy的300ms延迟(#36569)影响用户体验
- **发展方向**: 持续完善SSR管线,优化加载策略

### 5. **项目治理挑战**
- AI生成的低质量PR涌入(#36461)增加维护负担
- **发展方向**: 需要建立更完善的贡献质量把控机制

## 三、高关注度关键议题详解

### 🔴 极高关注度（100+ 👍）

#### [#35758](https://github.com/facebook/react/issues/35758) — eslint-plugin-react-hooks 不支持 ESLint 10
- **创建时间：** 2026-02-10 | **状态：** Open (Unconfirmed) | **👍 124 | 💬 21**
- **核心问题：** `eslint-plugin-react-hooks@7.0.1` 的 peerDependencies 仅声明支持 ESLint 3-9，ESLint 10 发布后无法安装（`ERESOLVE` 错误）。
- **主要讨论观点：**
  - 大量用户反馈 ESLint 10 已是稳定版本，React 作为社区核心工具应及时跟进
  - 社区成员提出可用 `overrides`/`--force` 绕过，但这不应该是长期方案
  - 此 issue 被 3 个其他 issue 标记为 blocking
- **潜在方案：** 将 peerDependencies 范围扩展为 `"^3.0.0 || ... || ^9.0.0 || ^10.0.0"`，并确认 ESLint 10 无 breaking changes 影响插件 API。

### 🟠 高关注度（30+ 👍 或 15+ 💬）

#### [#35395](https://github.com/facebook/react/issues/35395) — ESLint 插件规则极慢（👍 34 | 💬 19）
- **创建时间：** 2025-12-19 | **状态：** Open | **标签：** Feature Request, Compiler, ESLint Rules
- **核心问题：** 在 ~15k 文件代码库中，`react-hooks/static-components` 单条规则耗时 **393 秒（6.5 分钟）**，占 lint 总时间的 **56%**，使总体 lint 时间翻倍。
- **主要讨论：** 该问题在 CI 管道和本地开发中显著影响效率；即使并发 4 进程也无法改善。部分团队考虑禁用该规则或单独并行运行。
- **潜在方案：** 优化 `static-components` 规则的 AST 遍历算法；考虑增量分析或缓存策略。

### 🟡 中等关注度（10-30 👍）

#### [#36143](https://github.com/facebook/react/issues/36143) — RSC 渲染管道性能（👍 10 | 💬 4）
- **创建时间：** 2026-03-25 | **状态：** Open
- **核心问题：** RSC 完整管道（Flight 序列化 → SSR → 压缩）在单线程环境下比纯 `renderToString` **慢近 10 倍**。Node Streams 比 Web Streams 快 35%，但 CPU 单线程瓶颈是核心矛盾。
- **关键发现：**
  - Flight 序列化 + Fizz SSR 的三阶段处理在单线程上相互背压
  - 作者提出"融合管道"（Fused Pipeline）POC——让 Fizz 直接识别 client component 边界，跳过 Flight 序列化/反序列化中间步骤，实现 **1.8 倍吞吐提升**，峰值内存从 297MB 降至 60MB
- **讨论观点：** 该问题长期未被发现因为 serverless 部署中每个请求有独立线程；传统 Node 服务器才会暴露 CPU 争用问题。
- **潜在方案：** 实现 Flight + Fizz 融合渲染；减少 Flight payload 中纯服务端树的序列化开销；支持 Node Streams 作为一等公民。

#### [#35402](https://github.com/facebook/react/issues/35402) — DevTools Commit tree fiber 重复（👍 17 | 💬 4）
- **创建时间：** 2025-12-23 | **已分配：** @hoxyq | **标签：** Bug, DevTools
- **核心问题：** Profiler 翻页时持续抛出 `Commit tree already contains fiber "XXX". This is a bug in React DevTools`。
- **影响：** 多个相似 issue（#36048, #36006, #35640, #36124, #35818, #35421）表明这是 DevTools v7.0.1 的系统性 Store 状态管理缺陷。

#### [#35713](https://github.com/facebook/react/issues/35713) — DevTools Store 节点匹配失败（👍 12 | 💬 10）
- **创建时间：** 2026-02-06 | **标签：** Bug, DevTools
- **核心问题：** Profiler 运行时持续抛出 `Cannot remove/add/reorder node "XXX" because no matching node was found in the Store`。
- **潜在方案：** 审查 DevTools Store 的节点注册/注销生命周期逻辑，修复 fiber 与 Store 节点的同步不一致。

### 🟢 值得关注的技术深度 Issues

#### [#36176](https://github.com/facebook/react/issues/36176) — Fiber 内存泄漏（👍 4 | 💬 2）
- **创建时间：** 2026-04-01 | **报告者：** Grafana 工程师 @leeoniya
- **核心问题：** 自 Fiber 架构以来（React 16+），数据密集型 props/state 在旧数据替换时不会释放，导致 **2 倍内存占用**。此问题自 2018 年 issue #14380 就存在但被 stale-close。
- **现实影响：** Grafana 等数据可视化平台通过 props 传递大量查询结果，无法改变此架构合约。
- **潜在方案：** 渲染完成后释放旧 fiber 状态；或提供显式 API（如 `useWeakRef`）让开发者主动管理。

#### [#35460](https://github.com/facebook/react/issues/35460) — Suspense 对大型子元素渲染 fallback（👍 4 | 💬 6）
- **创建时间：** 2026-01-07 | React 19.2 行为变更
- **核心问题：** React 19.2 中，`Suspense` 在服务端渲染时，当子元素"太大"时不再渲染 children 而渲染 fallback。破坏了 React 19.1 中非 JS 用户能获取页面内容的行为。
- **潜在方案：** 确认此行为变更是有意还是回退；如果是有意的，提供替代方案确保无 JS 用户的 SSR 内容可访问。

#### [#36200](https://github.com/facebook/react/issues/36200) — 大型 TypedArray 冻结浏览器（👍 1 | 💬 6）
- **创建时间：** 2026-04-02 | React >= 19.2
- **核心问题：** 将 2500 万元素的 TypedArray 作为 prop 传递时，dev 模式渲染挂起 >10 秒。与 19.2 新增的 `logComponentRender`（组件性能 track）相关。
- **潜在方案：** 对大型数组/TypedArray 采用采样日志而非全量深比较。

## 五、React 当前面临的主要挑战与发展方向

### 🎯 挑战一：React Compiler 的成熟度鸿沟

React Compiler（v1.0.0）虽已正式发布，但近半年涌现出大量严重缺陷：

- **闭包完整性破坏**（#35386, #36128, #35762）：Compiler 的代码提升优化在工厂函数、`.map()` 回调、`useEffect` 等场景中破坏 JavaScript 闭包语义，导致运行时 `ReferenceError` 或 `TypeError`。
- **AST 契约违反**（#36327）：将 `Symbol()` 写入 Babel AST 的 `Node.loc` 字段，导致下游工具（jest-worker、v8.serialize）序列化失败。
- **JSX pragma 冲突**（#36460）：Compiler 在文件顶部插入 import 语句，覆盖 `@jsxImportSource` 等 pragma 注释，使 Oxc/SWC/TypeScript 的原生 JSX 转换失效。
- **ESLint 规则与 Compiler 理念冲突**（#35499, #35326, #35576）：`exhaustive-deps` 要求手动 `useCallback`，但 Compiler 承诺自动处理——两者矛盾导致困扰。

**方向判断：** React Compiler 需要至少 6-12 个月的"稳定期"，重点解决与现有生态（Babel 替代品、工厂模式、ESLint 规则集）的兼容性问题。

### 🎯 挑战二：React Server Components 的性能天花板

RSC 架构带来了巨大的开发者体验提升，但服务端渲染吞吐量问题（#36143, 关联 #34727）暴露了架构设计中的效率瓶颈：

- **10 倍吞吐衰减**（vs 纯 SSR）在单线程 Node 环境下不可接受
- **Flight 序列化 → Fizz 渲染**的两阶段设计导致数据"膨胀"和 CPU 争用
- **Web Streams vs Node Streams** 的 35% 性能差距影响部署选择
- **Fiber 内存占用**（#36176）自 2018 年悬而未决，在数据密集型应用中尤为突出

**方向判断：** 需要从架构层面重新审视 RSC 渲染管道。社区的"融合管道"提案表明可以大幅减少中间序列化开销，这可能是 React 20 的重要优化方向。

### 🎯 挑战三：开发者工具（DevTools）的稳定性危机

DevTools v7.0.1 的 Profiler 组件存在系统性缺陷，至少 8 个独立 issue 报告了同类 Store 状态管理错误：

- "Commit tree already contains fiber" 系列：**#35402, #35640, #36006, #36124**
- "Cannot add/remove/reorder node" 系列：**#36048, #35713, #36315, #35421, #35818**
- "Cannot set rects for suspense node"：**#36577**

这些错误表明 DevTools 的核心 Store 在 fiber 节点同步方面存在竞态条件或状态一致性问题，在生产级应用中几乎必然触发。

### 🎯 挑战四：React 19.2 的性能诊断功能的副作用

React 19.2 引入的 `logComponentRender`（组件渲染性能跟踪）是开发体验的重要补充，但带来了以下意外副作用：

- **MobX 警告**（#35475）：深比较触发了 observable 在 reaction 外读取
- **跨域 Window 崩溃**（#36430）：遍历 props 时对 cross-origin `Window` 对象读取 `$$typeof` 抛出 `SecurityError`，导致整个 fiber 树损坏
- **大型 TypedArray 冻结**（#36200）：全量遍历数千万元素的 TypedArray 导致渲染挂起

### 🎯 挑战五：工程项目治理

- **垃圾 issue 泛滥**（#36400）：大量空白 "Bug:" issue 被提交，需要自动化 bot 过滤
- **AI 垃圾 PR**（#36461）：社区呼吁安装 PR 质量门禁
- **版本号管理不一致**（#35722）：插件版本号硬编码而非从 `package.json` 动态读取
- **文档过时**（#36549）：多个 README 与实际 API 签名不一致

## 六、总结

| 维度 | 核心发现 |
|------|---------|
| **最紧迫问题** | ESLint 10 兼容性（124👍）——影响面最广，修复成本最低 |
| **最严重性能问题** | ESLint 插件性能（6.5 分钟单规则）和 RSC 管道吞吐（10x 衰减） |
| **最系统性风险** | DevTools Profiler Store 状态管理缺陷（8+ 独立 issue） |
| **最大技术债务** | Fiber 内存占用（2018 年至今未解决）和 Compiler 闭包语义破坏 |
| **最受期待方向** | RSC 融合渲染管道、React Compiler 稳定性提升 |
| **质量信号** | 136 个开放 issue 中约 15% 为空白或低质量，反映 issue 治理需加强 |

React 19.x 系列在引入 Server Components、React Compiler 等重大创新的同时，也积累了相当的技术债务。社区反馈表明，当前最需要的是**稳定性加固**而非更多新功能——特别是在工具链兼容性（ESLint 10、Oxc/SWC）、Compiler 成熟度、DevTools 可靠性以及 RSC 性能方面的系统化改进。
