# Bug: `<ViewTransition>` name-prop change does not trigger view transition without child DOM mutation  #35983
Bug: `<ViewTransition>` name-prop 变化不触发视图转换而无子 DOM 变更 

A `<ViewTransition>` whose `name` prop changes between renders does **not** trigger the view transition update pipeline unless a child DOM mutation also occurs. Hero morphs via name-prop toggling on mounted components silently produce no animation.  
一个 `<ViewTransition>` 的 `name` 属性在渲染之间发生变化时，不会触发视图转换更新管道，除非同时发生子 DOM 变化。在挂载组件上通过名称属性切换实现的英雄形态转换会无声地产生无动画效果。

## React version  React 版本

- `react`: `19.3.0-canary-46103596-20260305`
- `react-dom`: `19.3.0-canary-46103596-20260305`

## Steps to reproduce  复现步骤

1. Mount two `<ViewTransition>` boundaries wrapping a button and a dialog  
    挂载两个 `<ViewTransition>` 边界，其中包含一个按钮和一个对话框
2. Toggle the `name` prop between them (one gets `"camera-hero"`, the other `undefined`) via `startTransition` + `addTransitionType`  
    通过 `startTransition` + `addTransitionType` 在它们之间切换 `name` prop（一个得到 `"camera-hero"` ，另一个得到 `undefined` ）
3. No child DOM mutations occur during the transition  
    过渡期间没有发生子 DOM 变化

## Expected behavior  预期行为

Opening the modal morphs smoothly from the trigger button to the dialog (shared `camera-hero` element with CSS view-transition animations).  
点击触发按钮后，模态框平滑地从按钮过渡到对话框（共享 `camera-hero` 元素，使用 CSS view-transition 动画）。

## Actual behavior  实际行为

Modal appears instantly -- no hero morph animation. See [root cause §1](https://github.com/facebook/react/issues/35983#1-name-only-changes-do-not-produce-update) for the code path that leads to a zero-opacity cancellation.  
模态框立即出现——没有英雄形态动画。查看根因 §1 了解导致透明度取消的代码路径。

### Workaround

Two changes required simultaneously (visible in workaround mode in the reproduction):  
同时需要两个更改（在复现的 workaround 模式下可见）：

1. **`update` prop** -- unblocks VT processing (otherwise `default='none'` gates it out):  
    `update` prop -- 解锁 VT 处理（否则 `default='none'` 会阻止它）：
    
    ```tsx
    update={{ default: 'none', 'camera-modal': 'camera-hero-morph' }}
    ```
    
2. **Forced child DOM mutation** -- sets the internal `Update` flag on the VT fiber:  
    强制子 DOM 变更 -- 在 VT fiber 上设置内部 `Update` 标志：
    
    ```tsx
    <button data-hero-owner={heroOwner}>Use live camera</button>
    ```
    

The `data-hero-owner` attribute serves no purpose other than forcing a child DOM mutation that sets a flag which logically should depend on the name change itself.  
`data-hero-owner` 属性除了强制子 DOM 变更是没有其他作用的，这个变更会设置一个标志，而这个标志在逻辑上应该依赖于名称变更本身。

## Root cause (source references)  
根本原因（来源参考）

All references below are GitHub permalinks to React commit [`4610359651fa10247159e2050f8ec222cb7faa91`](https://github.com/facebook/react/commit/4610359651fa10247159e2050f8ec222cb7faa91), matching `react-dom@19.3.0-canary-46103596-20260305`.  
以下所有参考都是指向 React 提交 `4610359651fa10247159e2050f8ec222cb7faa91` 的 GitHub 永久链接，与 `react-dom@19.3.0-canary-46103596-20260305` 匹配。

### 1. Name-only changes do not produce `Update`  
1. 仅名称变更不会触发 `Update`

In [`measureViewTransitionHostInstancesRecursive`](https://github.com/facebook/react/blob/4610359651fa10247159e2050f8ec222cb7faa91/packages/react-reconciler/src/ReactFiberCommitViewTransitions.js#L642-L705), the `Update` flag gates whether `applyViewTransitionName` fires for the new snapshot:  
在 `measureViewTransitionHostInstancesRecursive` 中， `Update` 标志控制 `applyViewTransitionName` 是否在新快照中触发：

```js
(parentViewTransition.flags & Update) !== NoFlags
  && applyViewTransitionName(instance, newName, className);
```

In this code path, the `Update` flag is set in the mutation phase ([`ReactFiberCommitWork.js#L2634-L2645`](https://github.com/facebook/react/blob/4610359651fa10247159e2050f8ec222cb7faa91/packages/react-reconciler/src/ReactFiberCommitWork.js#L2634-L2645)) when `viewTransitionMutationContext` is `true`:  
在这个代码路径中，当 `viewTransitionMutationContext` 为 `true` 时， `Update` 标志在变异阶段（ `ReactFiberCommitWork.js#L2634-L2645` ）被设置：

```js
viewTransitionMutationContext && (finishedWork.flags |= Update);
```

`viewTransitionMutationContext` becomes `true` when a DOM mutation (attribute change, text change, element insertion/removal) occurs within the VT's child subtree during the mutation commit. The `Update` flag can also be set during measurement if `hasInstanceChanged()` detects a bounding-rect change ([L676-681](https://github.com/facebook/react/blob/4610359651fa10247159e2050f8ec222cb7faa91/packages/react-reconciler/src/ReactFiberCommitViewTransitions.js#L676-L681)), but in this scenario (name-only prop change, no layout shift), measurements are identical and this path returns false.  
`viewTransitionMutationContext` 在视图过渡（VT）的子树中发生 DOM 变更（属性变更、文本变更、元素插入/移除）时，在变更提交过程中会变成 `true` 。如果 `hasInstanceChanged()` 检测到边界矩形变更（L676-681）， `Update` 标志也可以在测量时被设置，但在这种情况下（仅属性变更，无布局偏移），测量结果相同，此路径返回 false。

When only the VT's `name` prop changes, with no child mutations and no layout change, the flag stays `NoFlags`. Without `Update`, the instance is not passed to `applyViewTransitionName` but is instead pushed to `viewTransitionCancelableChildren`, which leads to a zero-opacity cancellation path in [`ReactFiberConfigDOM.js#L1616-L1640`](https://github.com/facebook/react/blob/4610359651fa10247159e2050f8ec222cb7faa91/packages/react-dom-bindings/src/client/ReactFiberConfigDOM.js#L1616-L1640):  
当仅是视图过渡的 `name` 属性变更，且无子元素变更和布局变更时，标志保持 `NoFlags` 。如果没有 `Update` ，该实例不会被传递给 `applyViewTransitionName` ，而是被推送到 `viewTransitionCancelableChildren` ，这会导致在 `ReactFiberConfigDOM.js#L1616-L1640` 中触发零不透明度取消路径：

```js
current.animate(
  { opacity: [0, 0], pointerEvents: ['none', 'none'] },
  { duration: 0, fill: 'forwards', pseudoElement: '::view-transition-group(' + oldName + ')' },
);
```

> **Note:** this cancellation path is only reachable when the `update` prop unblocks before-mutation snapshot capture and after-mutation measurement. With `default='none'` and no `update` prop, host instances never receive VT names in the first place, so they never reach `viewTransitionCancelableChildren`. The zero-opacity cancellation manifests specifically in the partial-workaround state (setting `update` without also forcing a DOM mutation).  
> 注意：这条取消路径只有在 `update` prop 在 mutation 前快照捕获和 mutation 后测量之前解除阻塞时才可达。对于 `default='none'` 且没有 `update` prop 的情况，宿主实例根本不会接收到 VT 名称，因此它们永远不会达到 `viewTransitionCancelableChildren` 。零不透明度的取消现象特别体现在部分解决方案状态中（设置 `update` 而不强制 DOM mutation）。

React already detects the name change in `beginWork` at [`ReactFiberBeginWork.js#L3617-L3622`](https://github.com/facebook/react/blob/4610359651fa10247159e2050f8ec222cb7faa91/packages/react-reconciler/src/ReactFiberBeginWork.js#L3617-L3622):  
React 已经在 `ReactFiberBeginWork.js#L3617-L3622` 检测到 `beginWork` 的名称变化：

```js
if (current !== null && current.memoizedProps.name !== pendingProps.name) {
  workInProgress.flags |= Ref | RefStatic;
}
```

This sets `Ref | RefStatic` for ref lifecycle, but it does not propagate into the commit phase as a VT update signal.  
这为 ref 生命周期设置了 `Ref | RefStatic` ，但它不会作为 VT 更新信号传播到提交阶段。

### 2. `default='none'` gates out the update path (compounding)

The `share` prop name suggests it controls shared-element/hero morphs, but in the source it is consulted in the **enter/exit pair** path ([`commitEnterViewTransitions`](https://github.com/facebook/react/blob/4610359651fa10247159e2050f8ec222cb7faa91/packages/react-reconciler/src/ReactFiberCommitViewTransitions.js#L287-L326) and [`commitExitViewTransitions`](https://github.com/facebook/react/blob/4610359651fa10247159e2050f8ec222cb7faa91/packages/react-reconciler/src/ReactFiberCommitViewTransitions.js#L409-L451)) and not in the update path. When both VTs stay mounted and only change their `name` prop, React routes through the **update** path, which calls `getViewTransitionClassName(props.default, props.update)` in two phases:  
`share` prop 名称表明它控制共享元素/英雄变形，但在源码中，它在 enter/exit 对路径（ `commitEnterViewTransitions` 和 `commitExitViewTransitions` ）中被查询，而不是在更新路径中。当两个 VT 都保持挂载且仅更改它们的 `name` prop 时，React 会通过更新路径，该路径在两个阶段调用 `getViewTransitionClassName(props.default, props.update)` ：

|Phase  阶段|Source permalink  源链接|Gate  门控|
|---|---|---|
|Before-mutation  突变前|[`ReactFiberCommitViewTransitions.js#L489-L513`](https://github.com/facebook/react/blob/4610359651fa10247159e2050f8ec222cb7faa91/packages/react-reconciler/src/ReactFiberCommitViewTransitions.js#L489-L513)|Class name `=== 'none'` → skip old-snapshot VT name application  <br>类名 `=== 'none'` → 跳过旧快照 VT 名称应用|
|After-mutation  变异后|[`ReactFiberCommitViewTransitions.js#L768-L803`](https://github.com/facebook/react/blob/4610359651fa10247159e2050f8ec222cb7faa91/packages/react-reconciler/src/ReactFiberCommitViewTransitions.js#L768-L803)|Class name `=== 'none'` → skip measurement and new-snapshot VT name  <br>类名 `=== 'none'` → 跳过测量和新快照 VT 名称|

The mutation phase ([`ReactFiberCommitWork.js#L2623-L2633`](https://github.com/facebook/react/blob/4610359651fa10247159e2050f8ec222cb7faa91/packages/react-reconciler/src/ReactFiberCommitWork.js#L2623-L2633)) also checks the class name to set `inUpdateViewTransition`, but this variable controls Portal context propagation in the traced code path -- the `Update` flag assignment at [L2634-L2645](https://github.com/facebook/react/blob/4610359651fa10247159e2050f8ec222cb7faa91/packages/react-reconciler/src/ReactFiberCommitWork.js#L2634-L2645) is gated by `viewTransitionMutationContext`, not by the class name.  
变异阶段（ `ReactFiberCommitWork.js#L2623-L2633` ）还会检查类名来设置 `inUpdateViewTransition` ，但这个变量控制着在跟踪的代码路径中 Portal 上下文的传播——L2634-L2645 处的 `Update` 标志赋值是由 `viewTransitionMutationContext` 控制的，而不是由类名控制的。  
So `default='none'` blocks the before/after-mutation phases, while the missing DOM mutation independently blocks the `Update` flag.  
所以 `default='none'` 会阻塞 before/after-mutation 阶段，而缺少的 DOM 变化会独立阻塞 `Update` 标志。

### 3. Separate observation: `share` not consulted in update path

The `share` prop is only consulted in the enter/exit pair path and not in the update path. This may be by-design (the update path handles DOM mutations, not shared-element pairing), but the documentation is silent on the expected behavior when only `name` changes on a mounted `<ViewTransition>`.  
`share` 属性仅在 enter/exit 对应路径中被查询，而不在更新路径中。这可能是有意为之（更新路径处理 DOM 变化，而不是共享元素配对），但文档对仅当 `name` 在已挂载的 `<ViewTransition>` 上变化时的预期行为保持沉默。

## Suggested fix  建议的修复方案

The name-change detection in [`beginWork`](https://github.com/facebook/react/blob/4610359651fa10247159e2050f8ec222cb7faa91/packages/react-reconciler/src/ReactFiberBeginWork.js#L3617-L3622) (`current.memoizedProps.name !== pendingProps.name`) could additionally set the `Update` flag or an equivalent signal, so the commit phase treats the VT as updated even without child DOM mutations.  
`beginWork` （ `current.memoizedProps.name !== pendingProps.name` ）中的名称变化检测可以额外设置 `Update` 标志或等效信号，以便提交阶段将 VT 视为已更新，即使没有子 DOM 变化。

A cleaner alternative is to keep the fix local to the mutation-phase commit handler for `ViewTransitionComponent`, adding a name-change check alongside `viewTransitionMutationContext`:  
一个更简洁的替代方案是将修复保持在 `ViewTransitionComponent` 的突变阶段提交处理程序中，同时添加一个名称变更检查，如 `viewTransitionMutationContext` :

```js
} else if (
  viewTransitionMutationContext
  || current.memoizedProps.name !== finishedWork.memoizedProps.name
) {
  finishedWork.flags |= Update;
}
```

This would keep the `Update` flag closer to its apparent current usage (set during mutation phase, consumed in measurement) without changing `beginWork` flag behavior.  
这将使 `Update` 标志更接近其当前明显用法（在变异阶段设置，在测量时消耗），而不会改变 `beginWork` 标志的行为。

Additionally, the `share` className may need to be consulted in the update path when a name change is detected, not only in the enter/exit pair path.  
此外，当检测到名称变更时，在更新路径中可能还需要参考 `share` className，而不仅限于进入/退出路径对。

## Docs note  文档说明

The `<ViewTransition>` documentation distinguishes `share` (enter/exit pairing) from `update` (DOM mutations / layout effects), but does not clarify what should happen when only the `name` prop changes across already-mounted `<ViewTransition>` boundaries. It's unclear whether mounted name toggling is expected to produce a transition or is intentionally unsupported.  
`<ViewTransition>` 文档区分了 `share` （进入/退出配对）和 `update` （DOM 变更/布局效果），但并未明确说明当仅跨已挂载的 `<ViewTransition>` 边界更改 `name` 属性时应如何处理。不清楚是否期望挂载名称切换能触发过渡，或是有意不支持。
