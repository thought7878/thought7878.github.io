你好！很高兴能为你深度拆解 `L1: 基础设施层 (Infrastructure Layer)`。

在前端架构中，L1 层是最容易被初级开发者忽视，却又最能让高级架构师“救命”的一层。*很多项目到了后期，换个网络库要改几百个文件，或者在 Node.js (SSR) 环境下跑不起来报错 `window is not defined`，都是因为 L1 层没有设计好*。

作为架构师，我们要把 L1 层定义为：**系统的“物理接触面”与“变化隔离带”**。它直接与浏览器、操作系统、网络和第三方服务打交道，负责处理所有“脏活、累活、底层活”。

下面，我将从**核心模块、设计模式、代码落地、以及架构避坑**四个维度，为你全景式剖析 L1 层。

---

## 一、 L1 层的五大核心模块

一个健壮的 L1 层，通常包含以下五个子模块：

### 1. 网络通信引擎 (Network Engine)
参考：[[网络通信引擎（基于Axios）]]

*   **职责**：封装底层的 HTTP/HTTPS、WebSocket、SSE (Server-Sent Events) 等通信协议。
*   **架构要求**：上层（L2/L3）绝对不允许直接 `import axios from 'axios'`。L1 必须提供一个统一的 `HttpClient` 接口。
*   **关键能力**：
    *   **统一拦截**：*请求前统一注入 Token、TraceID（链路追踪）；响应后统一处理 HTTP 状态码（如 401 跳登录，500 弹全局错误）*。
    *   **请求重试与取消**：内置弱网环境下的自动重试机制，以及基于 `AbortController` 的请求取消能力（防止组件卸载后更新 State 报错）。
    *   **并发控制**：限制同时发出的请求数量，防止打爆后端或耗尽浏览器连接池。

### 2. 持久化存储引擎 (Storage Engine)
*   **职责**：统一管理客户端的本地数据存储。
*   **架构要求**：抹平 `Cookie`、`LocalStorage`、`SessionStorage`、`IndexedDB` 和 `Cache API` 的差异，对外提供统一的 KV (键值对) 或文档型 API。
*   **关键能力**：
    *   **容量与降级策略**：当 LocalStorage 满了（通常 5MB），自动降级到 IndexedDB，或者清理过期数据。
    *   **序列化/反序列化**：自动处理 JSON 转换、Date 对象还原、BigInt 处理等。
    *   **过期时间控制**：原生 Storage 不支持过期时间，L1 层需要自己实现 TTL (Time To Live) 机制。

### 3. 设备与浏览器能力抽象 (Device & Browser APIs)
*   **职责**：封装浏览器原生 API（如 Geolocation、Clipboard、Camera、Notifications、Web Workers）。
*   **架构要求**：处理浏览器兼容性问题，处理用户授权（Permissions）状态，将异步的回调 API 封装为 Promise/Async-Await。
*   **关键能力**：
    *   **环境抹平**：判断当前是 Web、PWA、小程序还是 SSR 环境，提供安全的 API 调用（防止在 SSR 时调用 `document` 报错）。
    *   **权限管理**：在调用敏感 API（如摄像头）前，统一检查并引导用户授权。

### 4. 第三方 SDK 集成 (Third-party Integrations)
*   **职责**：接入外部服务，如监控 (Sentry)、埋点 (神策/GrowingIO)、地图 (高德/Google)、图表 (ECharts)、支付等。
*   **架构要求**：**按需加载 (Lazy Load)** 与 **防污染**。第三方 SDK 通常体积巨大且可能污染全局变量。
*   **关键能力**：
    *   **动态导入**：使用 `import()` 在需要时才加载 SDK，不阻塞首屏。
    *   **统一事件总线**：将不同 SDK 的回调统一转化为前端的事件或状态，隔离第三方 SDK 升级带来的 API 变动。

### 5. 环境与基础抹平 (Environment & Polyfills)
*   **职责**：确保代码在任何目标环境中都能以相同的方式运行。
*   **关键能力**：
    *   **核心 Polyfill**：按需引入 `core-js` 或 `core-js-pure`，补齐老旧浏览器的 ES6+ 语法。
    *   **全局环境注入**：在应用启动的最早期，注入全局的环境变量（如 `__DEV__`, `__VERSION__`）。

## 二、 架构设计模式与代码落地 (Show me the code)

L1 层的设计核心是 **“面向接口编程”** 和 **“依赖倒置”**。我们来看两个具体的代码示例。

### 示例 1：网络通信引擎的抽象 (依赖倒置)

**错误做法（强耦合）**：
```typescript
// 业务层直接依赖 axios，如果以后换 fetch，业务层全得改
import axios from 'axios'; 
export const getUser = () => axios.get('/api/user');
```

**架构师做法（L1 层封装接口与实现）**：
```typescript
// 1. 定义抽象接口 (L1 层暴露给 L2/L3 的契约)
export interface IHttpClient {
  get<T>(url: string, config?: RequestConfig): Promise<T>;
  post<T>(url: string, data?: any, config?: RequestConfig): Promise<T>;
}

// 2. 具体实现 (L1 层内部实现，对外隐藏)
class AxiosHttpClient implements IHttpClient {
  private instance: AxiosInstance;

  constructor() {
    this.instance = axios.create({ timeout: 10000 });
    // 统一拦截器：注入 Token、处理全局错误等
    this.instance.interceptors.request.use(this.addAuthHeader);
    this.instance.interceptors.response.use(this.handleResponse, this.handleError);
  }

  async get<T>(url: string, config?: RequestConfig): Promise<T> {
    const res = await this.instance.get(url, config);
    return res.data; // 直接返回业务数据，剥离 axios 的 response 外壳
  }
  
  // ... post 方法类似
}

// 3. 依赖注入/单例导出 (L5 应用层或 L1 层提供)
export const httpClient: IHttpClient = new AxiosHttpClient();
```
*收益：L2/L3 层只依赖 `IHttpClient` 接口。如果明天老板要求换成 `ky` 或原生 `fetch`，你只需要在 L1 层写一个 `FetchHttpClient` 替换掉 `AxiosHttpClient`，上层业务代码**一行都不用改**。*

### 示例 2：持久化存储引擎 (策略模式与降级)

```typescript
// L1 层：统一存储引擎
export class StorageEngine {
  // 根据数据类型和大小，自动选择底层存储介质
  async set(key: string, value: any, options?: { ttl?: number }) {
    const data = {
      value,
      expireAt: options?.ttl ? Date.now() + options.ttl : null,
    };
    const serialized = JSON.stringify(data);

    try {
      // 优先尝试 LocalStorage (同步，快)
      if (serialized.length < 1024 * 1024) { // 小于 1MB
        localStorage.setItem(key, serialized);
      } else {
        // 大数据降级到 IndexedDB (异步，容量大)
        await this.saveToIndexedDB(key, data);
      }
    } catch (e) {
      if (e.name === 'QuotaExceededError') {
        // 容量超限，清理过期数据后重试，或降级到 IndexedDB
        this.clearExpiredItems();
        await this.saveToIndexedDB(key, data);
      } else {
        throw e;
      }
    }
  }

  async get<T>(key: string): Promise<T | null> {
    // 1. 先查 LocalStorage
    // 2. 查不到或过期，查 IndexedDB
    // 3. 校验 TTL，过期则删除并返回 null
    // ... 具体实现略
  }
}
```
*收益：业务层只需要调用 `storage.set('cart', cartData)`，完全不需要关心数据存在哪里、会不会爆满、有没有过期。*

## 三、 架构师的“避坑指南” (Pro Tips)

在设计 L1 层时，我见过无数团队踩坑，请务必牢记以下四条铁律：

### 1. 严禁在 L1 层写业务逻辑！
*   **坑**：在 Axios 的响应拦截器里写 `if (url === '/api/login') { ... }`。
*   **正解**：L1 层是**通用基础设施**，它不应该知道“登录”、“购物车”是什么。拦截器只处理通用逻辑（如 401 统一跳登录页，500 统一弹 Toast）。特定的业务错误处理，应该抛给 L2 (防腐层) 或 L3 (业务层) 去处理。

### 2. 警惕“内存泄漏”与“幽灵事件”
*   **坑**：封装 WebSocket 或 `window.addEventListener` 时，没有在组件卸载时清理。
*   **正解**：L1 层封装的长连接或全局监听，必须提供明确的 `destroy()` 或 `unsubscribe()` 方法，并在架构规范中强制要求上层在 `useEffect` 的 cleanup 中调用。

### 3. 第三方 SDK 必须“懒加载”与“隔离”
*   **坑**：在 `main.ts` 里直接 `import * as echarts from 'echarts'`，导致首屏 JS 体积暴增 1MB。
*   **正解**：L1 层必须提供动态加载能力。
    ```typescript
    // L1 层封装图表 SDK
    let echartsInstance: typeof import('echarts') | null = null;
    export const getEcharts = async () => {
      if (!echartsInstance) {
        echartsInstance = await import('echarts'); // 按需加载
      }
      return echartsInstance;
    };
    ```

### 4. 关注 Tree-Shaking 与包体积
*   **坑**：L1 层的工具库（如 `lodash`, `moment`）全量引入。
*   **正解**：L1 层的代码会被所有业务引用，它的体积会被放大无数倍。必须使用 `lodash-es`、`dayjs`，或者在构建工具中配置好按需引入。L1 层的每一 KB，都关乎首屏性能。

## 四、 总结：L1 层的终极价值

如果把前端应用比作一家餐厅：
*   **L4 (视图层)** 是前厅，负责装修和端盘子（UI 展示）。
*   **L3 (业务层)** 是厨师，负责炒菜（业务逻辑）。
*   **L2 (防腐层)** 是采购员，负责把买来的带泥的萝卜洗干净、切好（数据清洗）。
*   **L1 (基础设施层)** 则是**水电煤、燃气管道和物流车队**。

顾客（用户）和厨师（业务逻辑）都不需要知道水管是怎么接的、燃气是什么牌子的。但如果水管爆了（网络断开）、燃气断了（存储满了），整个餐厅就会瘫痪。

**L1 层的终极价值在于：通过高度的封装和抽象，把底层环境的“不确定性”和“复杂性”死死锁在这一层，为上层提供一个极其干净、稳定、可靠的“确定性”接口。** 这就是基础设施层存在的意义。
