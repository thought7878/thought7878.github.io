
# 原理

>你是顶级的前端架构师，详细介绍前端架构（Frontend Architecture）的分层设计图


你好！作为前端架构师，我非常乐意为你*深度拆解`前端架构的分层设计（Layered Architecture）`*。

**在初级开发者眼中，前端代码往往是 *“一锅炖”*** ：组件里直接写 API 请求，请求完直接塞进组件的 `state` 里，样式和逻辑混杂。这种 ***“面条代码”*** 在项目初期跑得很快，*但随着业务膨胀，必然走向 **“改一个 Bug 引发三个新 Bug”** 的熵增深渊*。

**分层设计的核心哲学只有八个字：*控制复杂度，隔离变化***。

现代前端架构早已超越了早期的 MVC/MVVM，演进为融合了 `整洁架构（Clean Architecture）` 和 `领域驱动设计（DDD）` 思想的 `“现代前端五层架构模型”`。

下面，我将通过**架构全景图**和**逐层深度剖析**，为你展现这套工业级的分层设计。

## 一、 五层架构全景图

请注意这张图的**依赖方向**：**上层依赖下层，下层绝不能反向依赖上层。** *核心业务逻辑（中间层）应当尽可能独立于具体的 UI 框架（React/Vue）和底层网络请求库*。

```text
                  [ 用户交互 / 浏览器环境 / 路由跳转 ]
                                 ⬇
======================================================================
  L5: 应用编排层 (Application & Orchestration Layer)
  [职责] 路由控制、全局布局、权限拦截、生命周期管理、全局异常捕获
======================================================================
                 ⬆ (全局事件)               ⬇ (挂载页面) 
======================================================================
  L4: 视图组件层 (View & UI Components Layer)
  [职责] 纯 UI 渲染、交互反馈、设计系统 (Design System)、Headless UI
  [原则] "哑巴组件" (Dumb Components)，只接收 Props，通过 Events 向上通信
======================================================================
                 ⬆ (传递数据/Props)         ⬇ (触发 Hooks/Actions)
======================================================================
  L3: 状态与业务逻辑层 (State & Domain Logic Layer)  <-- 核心大脑
  [职责] 状态管理、核心业务规则、数据聚合、跨组件通信
  [细分] Server State (服务端状态) vs Client State (客户端状态)
======================================================================
                 ⬆ (返回干净的业务模型)      ⬇ (调用具体服务)
======================================================================
  L2: 服务与防腐层 (Service & Anti-Corruption Layer)
  [职责] API 接口封装、DTO (Data Transfer Object) 到 Domain Model 的转换、缓存策略、Mock 拦截
  [原则] 隔离后端糟糕的数据结构，防止其污染前端核心逻辑
======================================================================
                 ⬆ (Raw JSON / 二进制流)   ⬇ (发起底层请求)
======================================================================
  L1: 基础设施层 (Infrastructure Layer)
  [职责] 网络底层 (Axios/Fetch)、本地存储 (IndexedDB/LocalStorage)、
         第三方 SDK (埋点/监控/地图)、底层工具库 (Crypto/Date)
======================================================================
                 ⬆ (Raw JSON / 二进制流)   ⬇ (发起底层请求)
======================================================================
                  
                  [ 后端 BFF / 微服务 / 本地硬盘 / 外部第三方 API ]
```

## 二、 **逐层深度剖析**与落地指南

### L1: 基础设施层 (Infrastructure Layer)
参考：[[L1. 基础设施层 (Infrastructure Layer)]]

这是系统的“地基”，负责*与外部世界进行最底层的 I/O 交互*。
*   **架构师视角**：这一层的代码**极少变动**。*如果哪天公司决定从 Axios 换成原生的 Fetch，或者从 LocalStorage 换成 IndexedDB，只需要修改这一层，上面的所有代码**一行都不用改***。
*   **技术落地**：
    *   **网络引擎**：封装统一的 `HttpClient`，内置拦截器（统一处理 Token 刷新、全局 Loading、统一错误码弹窗）。
    *   **存储引擎**：封装 `StorageManager`，处理数据序列化/反序列化、过期时间控制、容量超限降级策略。
    *   **环境感知**：判断当前是 Web、Node (SSR) 还是小程序环境，提供抹平差异的底层 API。

### L2: 服务与防腐层 (Service & ACL)
**防腐层（Anti-Corruption Layer, ACL）** 是高级架构的标志性设计。后端 API 返回的数据结构（DTO (Data Transfer Object)）往往是为了数据库查询方便而设计的，并不适合前端 UI 渲染。
*   **架构师视角**：永远不要直接把后端返回的 JSON 扔给 UI 组件！这会导致 UI 组件里充斥着 `data?.user?.info?.name` 这种丑陋的容错代码和格式转换逻辑。
*   **技术落地**：
    *   **API 契约化**：使用 OpenAPI/Swagger 自动生成 TS 接口类型和请求函数。
    *   **数据适配器 (Adapter)**：在 Service 层将后端的 `snake_case`（如 `created_at`）转换为前端的 `camelCase`（如 `createdAt`），将扁平数据组装成树状结构，将枚举值（`status: 1`）映射为前端语义（`status: 'ACTIVE'`）。
    *   **Mock 拦截**：在开发环境，这一层可以无缝切换为读取本地 Mock 数据，实现前后端完全并行开发。

### L3: 状态与业务逻辑层 (State & Domain Logic Layer)
这是前端架构的**分水岭**。现代前端架构*必须在这里做严格的“状态分流”*。
*   **架构师视角**：放弃“一个巨大的 Redux Store 装下所有东西”的旧思维。*状态必须分类治理*。
*   **技术落地 (双轨制状态管理)**：
    1.  **Server State (服务端状态)**：即*后端数据库里的数据*（如商品列表、用户信息）。
        *   **工具**：`React Query`, `SWR`, `Apollo Client`。
        *   **职责**：处理缓存、去重、后台静默刷新、乐观更新（Optimistic UI）、分页/无限滚动。***不要把服务端数据放进 Redux/Zustand！***
    2.  **Client State (客户端状态)**：*纯粹的前端 UI 状态*（如弹窗开关、侧边栏折叠、表单草稿）。
        *   **工具**：`Zustand`, `Pinia`, `Jotai`, `Recoil`。
        *   **职责**：极简、同步、跨组件共享。
    3.  **Domain Logic (领域逻辑)**：*抽离出与 UI 无关的纯业务函数*（如：计算购物车总价、满减规则校验、复杂表单联动逻辑）。*这些函数应该是纯函数 (Pure Functions)*，极其容易编写单元测试。

### L4: 视图组件层 (View & UI Components Layer)
用户直接看到和触摸的地方。架构师在这里的核心目标是**极致的复用性**和**设计一致性**。
*   **架构师视角**：*组件必须分为`“基础组件”`和`“业务组件”`，并推崇 `Headless UI（无头组件）` 理念*。
*   **技术落地 (三级组件体系)**：
    1.  **L4.1 基础设计系统 (Design System)**：
        *   如 Button, Input, Modal。
        *   **架构要求**：*必须是“无头”的*（如 Radix UI, Headless UI），即只封装无障碍访问 (a11y) 和键盘交互逻辑，*不写死任何 CSS 样式*，样式由上层的 Tailwind/CSS-in-JS 注入。
    2.  **L4.2 业务区块组件 (Business Blocks)**：
        *   如 `UserCard`, `ProductSkuSelector`。
        *   **架构要求**：***只依赖 L3 层的 Hooks 获取数据，自身不发起网络请求***。
    3.  **L4.3 页面容器 (Pages/Views)**：
        *   ***将多个区块组件拼装成一个完整的页面，负责处理页面的 Loading 骨架屏和 Error 降级 UI***。

### L5: 应用编排层 (Application & Orchestration Layer)
***顶层指挥官，负责把上述所有层串联起来，形成一个可运行的 App***。
*   **架构师视角**：这里不写具体的业务逻辑，只做“胶水”和“路由调度”的工作。
*   **技术落地**：
    *   **路由编排**：配置 React Router / Vue Router，定义路由表。
    *   **权限网关 (Auth Guards)**：在路由切换前，拦截并校验 Token 和角色权限，决定是放行、重定向到登录页，还是显示 403。
    *   **全局布局 (Layouts)**：管理顶栏、侧边栏、底部版权等全局不变的 UI 框架。
    *   **全局错误边界 (Error Boundaries)**：捕获底层抛出的未处理异常，展示友好的“系统崩溃”降级页面，并自动上报 Sentry。

## 三、 跨层通信的“架构铁律”

分层设计最怕的是“层与层之间乱连线”，导致架构退化为“大泥球”。作为架构师，必须制定严格的跨层通信规则：

### 1. 依赖倒置原则 (DIP - Dependency Inversion Principle)
*   **痛点**：如果 L3 (业务逻辑) 直接 `import axios from 'axios'`，那么*业务逻辑就和网络库死死绑定了*。
*   **解法**：L3 层定义一个**接口 (Interface)**（例如 `IUserRepository`），L1/L2 层去**实现**这个接口。在应用启动时（L5层），通过**依赖注入 (Dependency Injection)** 或 **Context/Provide** 将具体实现传给 L3。
*   **收益**：写单元测试时，你可以轻松注入一个 Mock 的 Repository，而不需要发起真实的网络请求。

### 2. 命令与查询分离 (CQRS 在前端的变体)
*   **Query (查)**：L4 视图层通过 L3 提供的 Hooks（如 `useUserList`）读取数据。数据流是**响应式**的（Reactive），数据变了，UI 自动更新。
*   **Command (改)**：L4 视图层触发用户操作（如点击“删除”），调用 L3 提供的 Action（如 `deleteUser(id)`）。Action 执行完毕后，去更新 Server State 的缓存，从而**自动触发** Query 的响应式更新，UI 随之刷新。**严禁在 Action 里直接操作 DOM 或强制刷新组件！**

### 3. 避免“越级调用”
*   L4 (视图) **绝对不允许**直接调用 L1 (Axios) 发起请求。
*   L4 (视图) **绝对不允许**直接读写 LocalStorage。
*   所有的外部交互，必须经过 L2/L3 层的统一接口。

## 四、 架构师的终极思考：物理目录映射

再好的逻辑分层，如果没有合理的**物理目录结构（Folder Structure）** 支撑，最终也会崩溃。

传统的按文件类型分类（`/components`, `/services`, `/utils`）在大型项目中是反人类的。现代前端架构师通常推崇 **Feature-Sliced Design (FSD - 特性切片设计)** 或 **基于领域 (Domain-based)** 的目录结构：

```text
src/
 ├── app/             # L5: 应用编排层 (路由、全局Provider、Layout)
 ├── pages/           # L4/L5: 页面级容器
 ├── widgets/         # L4: 复杂的业务区块组合 (如 侧边栏、头部导航)
 ├── features/        # L3/L4: 核心业务特性 (按业务模块划分，如 Cart, Auth)
 │    ├── cart/
 │    │   ├── ui/          # 购物车相关的 UI 组件
 │    │   ├── model/       # 购物车的状态、Action、领域逻辑 (Store/Hooks)
 │    │   └── api/         # 购物车的 API 请求 (Service)
 ├── entities/        # L3: 核心业务实体 (如 User, Product 的模型和基础UI)
 ├── shared/          # L1/L2: 基础设施、全局UI库、工具函数 (跨业务复用)
```
*这种目录结构保证了：**当你需要修改“购物车”功能时，你只需要在 `features/cart` 目录下工作，而不需要在十几个不同的文件夹之间反复横跳。***

## 总结

前端分层架构不是为了炫技，而是为了**在团队规模扩大、业务逻辑指数级增长时，系统依然能保持优雅和可控**。

*   **L1/L2** 帮你挡住后端的朝令夕改和底层 API 的变动；
*   **L3** 帮你沉淀核心业务资产，让逻辑可测试、可复用；
*   **L4** 帮你统一设计语言，提升 UI 研发效能；
*   **L5** 帮你掌控全局，保障系统的安全与稳定。

掌握了这套分层设计图，你就不再是一个“写页面的切图仔”，而是一个真正能够驾驭大型复杂系统的**前端架构师**。

## L1: 基础设施层 (Infrastructure)
***目标*：封装底层 I/O，上层业务绝不直接感知 Axios 或 Fetch。**

```typescript
// src/shared/api/httpClient.ts
import axios, { AxiosInstance, AxiosRequestConfig } from 'axios';

// 架构师批注：这里只负责最底层的网络通信、Token 注入、全局拦截。
// 不包含任何业务逻辑（如“登录过期跳转”应放在 L5 拦截器或 L2 处理）。
class HttpClient {
  private instance: AxiosInstance;

  constructor() {
    this.instance = axios.create({ baseURL: '/api/v1', timeout: 10000 });
    
    // 请求拦截：注入 Token
    this.instance.interceptors.request.use((config) => {
      const token = localStorage.getItem('token');
      if (token) config.headers.Authorization = `Bearer ${token}`;
      return config;
    });
  }

  async get<T>(url: string, config?: AxiosRequestConfig): Promise<T> {
    const response = await this.instance.get<T>(url, config);
    return response.data;
  }

  async put<T>(url: string, data: any, config?: AxiosRequestConfig): Promise<T> {
    const response = await this.instance.put<T>(url, data, config);
    return response.data;
  }
}

export const httpClient = new HttpClient();
```

## L2: 服务与防腐层 (Service & ACL)
***目标*：定义 API 契约，隔离后端糟糕的数据结构，转换为前端友好的领域模型。**

```typescript
// src/features/user-profile/api/types.ts
// 架构师批注：严格区分后端 DTO 和前端 Domain Model

// 1. 后端返回的 DTO (Data Transfer Object) - 可能是 snake_case，结构扁平
export interface UserDTO {
  user_id: number;
  user_name: string;
  email_addr: string;
  created_at: string; // ISO 字符串
  role_type: number;  // 1: Admin, 2: User
}

// 2. 前端领域模型 (Domain Model) - 语义化，类型安全，符合前端习惯
export interface User {
  id: number;
  name: string;
  email: string;
  createdAt: Date;    // 转换为 Date 对象
  role: 'ADMIN' | 'USER'; // 转换为枚举字符串
}

// 3. 适配器 (Adapter) - 防腐层的核心
export const mapUserDTOToDomain = (dto: UserDTO): User => ({
  id: dto.user_id,
  name: dto.user_name,
  email: dto.email_addr,
  createdAt: new Date(dto.created_at),
  role: dto.role_type === 1 ? 'ADMIN' : 'USER',
});
```

```typescript
// src/features/user-profile/api/userApi.ts
import { httpClient } from '@/shared/api/httpClient';
import { UserDTO, User, mapUserDTOToDomain } from './types';

export const userApi = {
  // 获取用户信息
  async fetchUser(): Promise<User> {
    // 1. 调用底层 httpClient 获取原始 DTO
    const dto = await httpClient.get<UserDTO>('/user/me');
    // 2. 通过防腐层转换，向上层返回干净的 Domain Model
    return mapUserDTOToDomain(dto); 
  },

  // 更新用户信息 (注意：入参和出参都必须是前端 Domain Model)
  async updateUser(updates: Partial<Pick<User, 'name' | 'email'>>): Promise<User> {
    // 如果需要，这里也可以写一个 mapDomainToDTO 转换回去
    const payload = {
      user_name: updates.name,
      email_addr: updates.email,
    };
    const dto = await httpClient.put<UserDTO>('/user/me', payload);
    return mapUserDTOToDomain(dto);
  }
};
```

## L3: 状态与业务逻辑层 (State & Domain Logic)
***目标*：状态分流（Server vs Client），沉淀*纯业务逻辑*。**

### Server State
```typescript
// src/features/user-profile/model/useUser.ts (Server State)
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query';
import { userApi } from '../api/userApi';
import { User } from '../api/types';

// 架构师批注：使用 React Query 管理服务端状态。
// 自动处理缓存、去重、后台刷新，彻底告别手动写 isLoading 和 useEffect 获取数据。
export const useUser = () => {
  const queryClient = useQueryClient();

  // 查询 (Query)
  const { data: user, isLoading, error } = useQuery<User>({
    queryKey: ['user', 'me'],
    queryFn: userApi.fetchUser,
  });

  // 变更 (Mutation)
  const updateMutation = useMutation({
    mutationFn: userApi.updateUser,
    // 乐观更新或成功后刷新缓存
    onSuccess: (updatedUser) => {
      queryClient.setQueryData(['user', 'me'], updatedUser);
    },
  });

  return {
    user,
    isLoading,
    error,
    updateUser: updateMutation.mutate,
    isUpdating: updateMutation.isPending,
  };
};
```

### Client State
```typescript
// src/features/user-profile/model/userStore.ts (Client State)
import { create } from 'zustand';

// 架构师批注：Zustand 只管理纯粹的 UI 状态。
// 不要把后端返回的 User 数据放进这里！！！
interface UserProfileUIState {
  isEditing: boolean;
  setEditing: (isEditing: boolean) => void;
}

export const useUserProfileUIStore = create<UserProfileUIState>((set) => ({
  isEditing: false,
  setEditing: (isEditing) => set({ isEditing }),
}));
```

### Domain Logic
```typescript
// src/features/user-profile/model/userDomain.ts (Domain Logic)
import { User } from '../api/types';

// 架构师批注：纯函数！不依赖任何 React Hooks，极易编写单元测试。
export const getUserDisplayName = (user: User): string => {
  if (user.role === 'ADMIN') return `[管理员] ${user.name}`;
  return user.name;
};

export const validateEmail = (email: string): boolean => {
  return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);
};
```

// src/features/user-profile/ui/EditProfileForm.tsx (业务区块组件)
import { useState } from 'react';
import { User } from '../api/types';
import { validateEmail } from '../model/userDomain';
import { Button } from '@/shared/ui/Button';
import { Input } from '@/shared/ui/Input'; // 假设存在

interface Props {
  initialUser: User;
  isSubmitting: boolean;
  onSubmit: (data: { name: string; email: string }) => void;
  onCancel: () => void;
}

export const EditProfileForm = ({ initialUser, isSubmitting, onSubmit, onCancel }: Props) => {
  const [name, setName] = useState(initialUser.name);
  const [email, setEmail] = useState(initialUser.email);
  const [error, setError] = useState('');

  const handleSubmit = () => {
    if (!validateEmail(email)) {
      setError('邮箱格式不正确');
      return;
    }
    onSubmit({ name, email });
  };

  return (
    <div className="form-card">
      <Input value={name} onChange={setName} placeholder="姓名" />
      <Input value={email} onChange={setEmail} placeholder="邮箱" />
      {error && <span className="error">{error}</span>}
      
      <div className="actions">
        <Button onClick={onCancel} variant="secondary">取消</Button>
        <Button onClick={handleSubmit} disabled={isSubmitting}>
          {isSubmitting ? '保存中...' : '保存'}
        </Button>
      </div>
    </div>
  );
};

## L5: 应用编排层 (Application & Orchestration)

### 页面容器
```tsx
// src/pages/ProfilePage.tsx (页面容器)
import { useUser } from '@/features/user-profile/model/useUser';
import { useUserProfileUIStore } from '@/features/user-profile/model/userStore';
import { UserProfileCard } from '@/features/user-profile/ui/UserProfileCard';
import { EditProfileForm } from '@/features/user-profile/ui/EditProfileForm';

export const ProfilePage = () => {
  // 1. 获取 Server State
  const { user, isLoading, error, updateUser, isUpdating } = useUser();
  // 2. 获取 Client State
  const { isEditing, setEditing } = useUserProfileUIStore();

  // 3. 处理全局降级 (Error Boundary 的简化版)
  if (isLoading) return <div className="skeleton">加载用户信息中...</div>;
  if (error || !user) return <div className="error-page">系统异常，请稍后再试</div>;

  // 4. 编排业务动作
  const handleUpdate = (data: { name: string; email: string }) => {
    updateUser(data, {
      onSuccess: () => {
        setEditing(false); // 成功后关闭编辑模式
      }
    });
  };

  // 5. 渲染视图
  return (
    <div className="page-container">
      <h1>我的账户</h1>
      
      {isEditing ? (
        <EditProfileForm 
          initialUser={user} 
          isSubmitting={isUpdating}
          onSubmit={handleUpdate} 
          onCancel={() => setEditing(false)} 
        />
      ) : (
        <UserProfileCard 
          user={user} 
          onEditClick={() => setEditing(true)} 
        />
      )}
    </div>
  );
};
```

## 架构师复盘：这套代码为什么“高级”？

如果你**把上面这段代码交给一个初级开发，他可能会问：“*为什么要搞这么复杂？我直接在 `ProfilePage.tsx` 里写个 `useEffect` 调 `axios`，然后 `setState` 渲染不就行了？*”**

*作为架构师，我会这样回答他，这套分层设计带来的**三大核心收益**：*

### 1. 极致的可测试性 (Testability)
*   **初级写法**：组件里混杂了网络请求和 UI，写单元测试必须 Mock 整个 DOM 和 HTTP 库，极其痛苦。
*   **架构写法**：
    *   `userDomain.ts` 是纯函数，直接 `expect(getUserDisplayName(mockUser)).toBe(...)`，毫秒级测试。
    *   `UserProfileCard` 是哑巴组件，测试时只需传入 Mock Props，断言渲染出的 HTML 即可。
    *   `userApi.ts` 可以脱离 React 环境，直接测试 DTO 到 Domain 的转换逻辑。

### 2. 完美的关注点分离 (Separation of Concerns)
*   如果后端把 `user_name` 改成了 `nick_name`，**只需要修改 L2 层的 `mapUserDTOToDomain`**。L3 的状态、L4 的组件、L5 的页面**一行代码都不用改**！这就是防腐层的威力。
*   如果 UI 设计师要求把“编辑按钮”从卡片里移到顶部导航栏，**只需要在 L5 层调整组件编排**，L4 的 `UserProfileCard` 依然保持纯粹。

### 3. 状态管理的“降维打击”
*   通过引入 React Query，我们彻底消灭了前端最恶心的问题之一：**“服务端数据与 UI 状态不同步”**。
*   在另一个页面（比如“订单页”）如果也展示了用户名，当用户在“资料页”修改了名字，React Query 的缓存机制会自动让“订单页”的名字也无缝更新，无需手动跨组件通信。

## 总结

*这套代码展示的不是“如何写出能跑的代码”，而是 **“如何写出能在 10 人团队、50 万行代码规模下，依然让新人快速上手、让老手不骂娘的工程化代码”***。

***架构的本质**，就是**用前期的设计成本，去换取后期的维护自由***。
