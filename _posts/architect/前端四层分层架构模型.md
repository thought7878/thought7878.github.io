你好！我是前端架构师。在前端工程化日益复杂的今天，前端早已不再是“画页面”和“调接口”的简单堆砌。一个优秀的前端架构，其核心目的在于**控制复杂度、隔离变化、提升可维护性与可扩展性**。

为了实现代码职责的彻底分离，我通常会采用基于 **“整洁架构（Clean Architecture）”** 和 **“六边形架构（Hexagonal Architecture）”** 思想演进而来的`前端四层分层架构模型`。

下面，我将为你详细拆解这套分层设计。

## 一、 前端分层架构全景设计图

我们将前端应用从外到内（或从底到顶）划分为四个核心层级，外加一个横切关注点层。核心原则是：**依赖只能由外向内，内层绝不依赖外层（依赖倒置）**。

```text
========================================================================================
                            【 横切关注点层 (Cross-Cutting Concerns) 】
      [ 路由 Router ]  [ 鉴权 Auth ]  [ 国际化 i18n ]  [ 监控 Monitor ]  [ 主题 Theme ]
========================================================================================
                                      ▲ (被各层按需调用)
                                      │
┌────────────────────────────────────────────────────────────────────────────────────────┐
│  Layer 4: 视图/展示层 (Presentation Layer)                                               │
│  [ 页面组件 Pages ]  [ 业务组件 Business Components ]  [ 基础UI组件 UI Components ]        │
│  职责：UI渲染、用户交互、动画、样式、局部UI状态管理。                                          │
└──────────────────────────────────────┬─────────────────────────────────────────────────┘
                                       │ 调用 (依赖)
                                       ▼
┌────────────────────────────────────────────────────────────────────────────────────────┐
│  Layer 3: 领域/状态层 (Domain / State Layer)                                             │
│  [ 业务逻辑 Business Logic ]  [ 状态管理 State Store ]  [ 领域模型 Domain Models ]         │
│  职责：核心业务规则、全局/模块状态管理、数据流转控制。                                          │
└──────────────────────────────────────┬─────────────────────────────────────────────────┘
                                       │ 调用 (依赖)
                                       ▼
┌────────────────────────────────────────────────────────────────────────────────────────┐
│  Layer 2: 服务/数据层 (Service / Data Layer)                                             │
│  [ API 接口封装 ]  [ 数据转换器 (DTO <-> Entity) ]  [ 缓存策略 Cache ]                      │
│  职责：与外部系统通信、数据清洗与格式化、屏蔽底层网络细节。                                      │
└──────────────────────────────────────┬─────────────────────────────────────────────────┘
                                       │ 调用 (依赖)
                                       ▼
┌────────────────────────────────────────────────────────────────────────────────────────┐
│  Layer 1: 基础设施层 (Infrastructure / Core Layer)                                       │
│  [ HTTP 客户端 Axios/Fetch ]  [ 本地存储 LocalStorage/IndexedDB ]  [ 纯工具函数 Utils ]    │
│  职责：提供最底层的原子能力，与具体业务完全解耦。                                              │
└────────────────────────────────────────────────────────────────────────────────────────┘
```

## 二、 各层级详细职责与设计规范

### 1. 基础设施层 (Infrastructure / Core Layer)
这是整个应用的地基，**最稳定、最底层、与业务完全无关**。
*   **核心职责**：提供网络请求、本地存储、日期处理、正则校验等原子能力。
*   **设计原则**：
    *   **零业务逻辑**：绝对不能出现类似 `getUserInfo` 这种业务词汇，只能有 `http.get('/user')`。
    *   **高复用性**：这一层的代码可以直接复制到公司的另一个项目中而无需修改。
*   **代码示例**：
    ```typescript
    // core/http.ts
    import axios from 'axios';
    export const http = axios.create({ baseURL: '/api', timeout: 10000 });

    // core/storage.ts
    export const storage = {
      get: (key: string) => JSON.parse(localStorage.getItem(key) || 'null'),
      set: (key: string, value: any) => localStorage.setItem(key, JSON.stringify(value)),
    };
    ```

### 2. 服务/数据层 (Service / Data Layer)
这一层是**外部世界与内部业务的边界（防腐层）**。
*   **核心职责**：
    *   封装具体的 API 请求。
    *   **数据转换（核心）**：将后端返回的 DTO（数据传输对象）转换为前端领域模型（Entity/VO）。后端数据结构变了，只改这一层，上层无感知。
    *   处理请求/响应拦截、Token 注入、统一错误处理。
*   **设计原则**：
    *   **按业务模块划分**：如 `userService`, `orderService`。
    *   **屏蔽网络细节**：上层调用 `userService.getUser()`，不需要关心底层是用 Fetch 还是 Axios，也不关心具体的 URL。
*   **代码示例**：
    ```typescript
    // services/userService.ts
    import { http } from '@/core/http';
    import { UserDTO, UserEntity } from '@/types';

    // 数据转换函数
    const mapDtoToEntity = (dto: UserDTO): UserEntity => ({
      id: dto.user_id, // 将后端的 user_id 映射为前端的 id
      fullName: `${dto.first_name} ${dto.last_name}`,
    });

    export const userService = {
      async getUser(id: string): Promise<UserEntity> {
        const response = await http.get<UserDTO>(`/users/${id}`);
        return mapDtoToEntity(response.data); // 返回纯净的领域模型
      }
    };
    ```

### 3. 领域/状态层 (Domain / State Layer)
这是前端的**大脑**，承载核心业务逻辑和状态流转。
*   **核心职责**：
    *   管理应用的全局状态或模块状态（使用 Redux, Zustand, Pinia 等）。
    *   编排业务流程（如：先校验权限 -> 再获取数据 -> 最后更新状态）。
    *   定义领域模型和枚举。
*   **设计原则**：
    *   **UI 框架无关**：理想情况下，这一层的代码不应该包含 React/Vue 的特定语法（如 JSX 或 Template），它应该是纯 JS/TS 逻辑。
    *   **单向数据流**：状态更新必须通过明确的 Action/Mutation/Reducer，禁止在组件中直接修改状态。
*   **代码示例 (以 Zustand 为例)**：
    ```typescript
    // store/userStore.ts
    import { create } from 'zustand';
    import { userService } from '@/services/userService';
    import { UserEntity } from '@/types';

    interface UserState {
      currentUser: UserEntity | null;
      isLoading: boolean;
      fetchUser: (id: string) => Promise<void>;
    }

    export const useUserStore = create<UserState>((set) => ({
      currentUser: null,
      isLoading: false,
      fetchUser: async (id) => {
        set({ isLoading: true });
        try {
          const user = await userService.getUser(id); // 调用服务层
          set({ currentUser: user, isLoading: false });
        } catch (error) {
          set({ isLoading: false });
          throw error;
        }
      }
    }));
    ```

### 4. 视图/展示层 (Presentation Layer)
这是**最外层、变化最频繁**的一层，直接面向用户。
*   **核心职责**：
    *   组件拆分与组合（Pages, Features, UI Components）。
    *   处理用户交互事件（点击、输入、拖拽）。
    *   管理**局部 UI 状态**（如：弹窗的显隐、下拉菜单的展开、表单的临时输入值）。
*   **设计原则**：
    *   **尽量无状态（Dumb Components）**：基础 UI 组件只接收 Props 和触发 Events，不连接全局 Store。
    *   **关注点分离**：将复杂的视图逻辑抽离为自定义 Hooks（React）或 Composables（Vue），保持组件模板的简洁。
*   **代码示例**：
    ```tsx
    // presentation/pages/UserProfile.tsx
    import { useEffect } from 'react';
    import { useUserStore } from '@/store/userStore';
    import { UserProfileUI } from '@/components/UserProfileUI'; // 纯展示组件

    export const UserProfilePage = ({ userId }: { userId: string }) => {
      const { currentUser, isLoading, fetchUser } = useUserStore();

      useEffect(() => {
        fetchUser(userId);
      }, [userId, fetchUser]);

      // 视图层只负责：1. 触发状态层 2. 消费状态层数据 3. 渲染UI
      return <UserProfileUI user={currentUser} loading={isLoading} />;
    };
    ```

## 三、 横切关注点 (Cross-Cutting Concerns)

有些功能不属于特定的业务层级，而是贯穿整个应用的，我们需要将其独立管理：
1.  **路由 (Router)**：管理页面跳转、路由守卫、懒加载。
2.  **鉴权 (Auth)**：Token 管理、权限指令/高阶组件、路由拦截。
3.  **监控与埋点 (Monitor)**：错误捕获（ErrorBoundary）、性能监控、业务埋点。
4.  **国际化 (i18n) & 主题 (Theme)**：多语言配置、CSS 变量/Design Token 管理。

## 四、 推荐的工程化目录结构

为了配合上述分层架构，推荐采用**基于特性（Feature-based）结合分层**的目录结构。这比传统的按文件类型（components/, views/, store/）划分更具可维护性。

```text
src/
├── core/                   # 【Layer 1】基础设施层
│   ├── http/               # axios 封装、拦截器
│   ├── storage/            # 本地存储封装
│   ├── utils/              # 纯工具函数 (date, string, math)
│   └── constants/          # 全局常量、枚举
│
├── shared/                 # 【跨层共享】跨模块复用的资产
│   ├── ui/                 # 基础 UI 组件 (Button, Input, Modal)
│   ├── hooks/              # 通用自定义 Hooks
│   └── types/              # 全局公共 TypeScript 类型
│
├── features/               # 【Layer 2, 3, 4 的聚合】按业务特性划分
│   ├── user-management/    # 用户管理模块
│   │   ├── api/            # (Layer 2) 该模块专属的 API 和数据转换
│   │   ├── store/          # (Layer 3) 该模块专属的状态管理
│   │   ├── components/     # (Layer 4) 该模块专属的业务组件
│   │   └── index.ts        # 模块统一对外暴露的 API (门面模式)
│   │
│   └── order-processing/   # 订单处理模块 (结构同上)
│
├── app/                    # 【应用壳】应用入口与全局配置
│   ├── router/             # 路由配置
│   ├── providers/          # 全局 Context/Provider (Theme, i18n, Auth)
│   ├── App.tsx             # 根组件
│   └── main.tsx            # 入口文件
│
└── assets/                 # 静态资源 (图片, 字体, 全局样式)
```

## 五、 架构师视角的落地建议

1.  **严格遵守依赖方向**：
    *   **绝对禁止**：`core` 层 import `features` 层的代码。
    *   **尽量避免**：`features` 之间直接互相 import。如果模块 A 需要模块 B 的数据，应该通过顶层路由状态传递，或者将公共逻辑下沉到 `shared` 层。
2.  **拥抱 TypeScript**：
    *   分层架构的精髓在于**契约（Contract）**。利用 TS 的 Interface 和 Type，在层与层之间定义严格的数据契约，这是保证架构不腐化的最强武器。
3.  **不要过度设计**：
    *   如果项目只是一个简单的后台管理系统（CRUD），不需要强行引入复杂的领域模型（Layer 3 可以简化）。
    *   **架构是演进而来的**，初期可以先做好 `core`（基础设施）和 `shared`（公共组件）的分离，随着业务复杂度增加，再逐步完善 `features` 内部的分层。
4.  **数据流向的单向性**：
    *   用户操作 (UI) -> 触发 Action (State) -> 调用 Service (Data) -> 更新 State -> 驱动 UI 重新渲染。保持这个单向数据流，能让 Bug 的追踪变得极其简单。

通过这套分层设计，你的前端代码将实现 **“高内聚、低耦合”**。当后端接口变更时，你只需修改数据层；当 UI 重构时，领域层和基础设施层毫发无损；当需要替换状态管理库时，视图层也只需做最小幅度的适配。这就是优秀架构带来的长期红利。
