## 本地手动发包配置流程 
[00:33]
    
- **前端项目内置registry支持**
	*前端项目默认内置registry，仅需配置即可推送至GitLab*。
- **scope定义与官方说明**
	`package.json`中名称需含scope（如`@my-org/pkg-demo`）；scope可为用户名或自建group name；链接跳转至NPM官网scope说明页。
- **.npmrc文件核心配置**
	配置项作用：指定包下载源地址与推送目标地址；含`//<domain>/api/v4/groups/<group-id>/-/npm/:_authToken=`字段；`_authToken`值为GitLab生成的NPM token，用于身份鉴权；下载时凭此token访问scope对应仓库。
- 变量参考说明
	变量含义参考文档末尾三句说明。

## 本地发包实操演示 
[02:03](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=1cd97e72b13147e4aca8217600c5deb1#?seek_t=123)
    
- 新建GitLab项目
	创建空白项目，项目名即为scope（如`root`）；包名示例为`pkg-demo`；可选私有/公开；私有需token，公开无需；组织内包通常设为私有。
- 初始化NPM包
	终端执行`git clone` → `cd pkg-demo` → `npm init -y`完成初始化。
- 编辑package.json
	修改`name`为`@root/pkg-demo`；填写`version`、`description`（"this is demo of his npm registry"）、`author`（用户名+邮箱）、`license`（MIT）。
- 添加入口文件与首次提交
	创建`index.js`，内容为`console.log("hello world")`；执行`git add . && git commit -m "first commit"`。
- 创建并配置.npmrc文件
	新建`.npmrc`文件；粘贴GitLab提供的模板代码。
- 替换.npmrc中的动态参数
	替换`https://`为实际GitLab协议（如`http://`）；替换域名（如`gitlab.example.com`）；替换`<your-domain>`为同域名；替换`<your-project-id>`为项目ID（示例ID为`6`）。
- 生成并配置NPM token
	进入GitLab“Settings → Deploy Tokens”；创建令牌，命名如`project-pkg-publish`；勾选`read_package_registry`（只读）与`write_package_registry`（读写）；说明：读写权限存在安全风险，后续推荐仅读；复制生成的token值填入`.npmrc`对应位置。
- 执行发布与验证
	`npm publish`命令推送；输出显示`Publishing to <GitLab-URL>`；成功后提示`+ @root/pkg-demo@1.0.0`；GitLab界面“Packages & Registries”可见已发布包。
- 本地安装测试
	新建空目录 → `npm init -y` → 执行`npm install @root/pkg-demo@1.0.0` → 验证下载成功且为刚上传包。

## CI/CD自动化发布方案 
[08:24]
    
- CI/CD配置优势与必要性 
	手动发布不安全且繁琐；CI/CD可实现自动化、标准化发布。
- CI专用.npmrc模板
	使用GitLab文档提供的CI模板；替换scope为`@root`；保留`CI_PROJECT_ID`占位符（由CI环境注入）；删除无关行；`PROJECT_ID`仍需手动改为`6`。
- CI/CD变量来源说明
	`CI_PROJECT_ID`与`CI_JOB_TOKEN`由GitLab CI运行时自动注入，无需手动配置。
- 流水线YAML配置
	在“CI/CD → Editor”中粘贴YAML；将`image: node`改为`node:18-alpine`；`deploy`阶段执行`npm publish`；移除冗余的CI server host填充逻辑（因已配置.npmrc）。
- 提交流水线配置
	先提交`.npmrc`文件 → 再提交流水线YAML文件 → 点击“Commit changes”。
- 流水线执行与版本迭代
	流水线首次失败因重复发布`1.0.0`；本地升级`package.json`中`version`为`1.0.1`；执行`git pull origin main`合并线上配置；`git add && git commit && git push`触发新流水线；成功后软件包库显示`1.0.1`版本；流水线日志确认`npm publish`执行成功。

## 安全权限最佳实践 
[11:49]
    
- 只读令牌策略
	下载端使用只读令牌（`read_package_registry`），写权限交由CI/CD统一管控；只读令牌可安全存于本地。
- 令牌时效与范围控制
	可设置令牌过期时间；可限定特定用户组；可限制私有仓库下载有效期（如仅限指定日期前下载旧版，新版禁止下载）。
