### **双分支简化模型与核心操作命令 [03:03](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=183)**

- **简化原则：两条分支即够用** [03:03](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=183)  
    主张将所有分支概念统一简化为**主分支（main）** 和**开发分支（dev）**，掌握此二者即可举一反三。
- **核心Git命令集** [03:25](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=205)  
    分支管理仅需以下命令：
    - 初始化：`git clone`, `git init`
    - 提交：`git add`, `git commit`
    - 创建分支：`git branch <name>` 或 `git checkout -b <name>`（从当前节点创建dev分支）
    - 切换与合并：`git checkout main` → `git merge dev`（将dev代码合并至main）

### **典型开发流程：远程dev为主干场景 [04:41](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=281)**

- **场景设定** [04:41](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=281)  
    公司远程仓库默认分支为`dev`（非main），`main`仅限有权限者操作。
- **标准开发流程** [05:18](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=318)
    - 克隆远程`dev`到本地；
    - 直接在本地`dev`分支上开发、`add`、`commit`；
    - 推送至**远端个人fork的dev分支**（非公司仓库）；
    - 创建PR（Pull Request）请求合并至公司`dev`分支。
- **反对过度分支化** [05:41](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=341)  
    警告勿在本地额外创建feature分支：因最终仍需合并回`dev`，徒增本地管理负担，属“自我挖坑”。
- **流程优势** [06:39](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=399)  
    避免污染公司仓库，不产生大量冗余feature分支，保持远程仓库整洁。

### **小团队轻量流程：直推公司feature分支 [07:12](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=432)**

- **适配二人团队** [07:12](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=432)  
    若团队极小且保留PR机制，可改为：
    - 将本地开发推送至**公司远程仓库的临时feature分支**；
    - 创建PR合并至`dev`；
    - 合并后**立即删除该feature分支**（`git branch -d <branch>`）。

    此方式虽简化流程，但需人工清理远端feature分支，否则造成仓库污染。

### **紧急Hotfix流程与同步机制 [09:43](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=583)**

- **触发条件** [09:43](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=583)  
    线上`main`分支出现紧急bug需修复。
- **四步修复流程** [09:58](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=598)
    1. 提交本地未完成开发（避免丢失）；
    2. 切换至远程`main`分支：`git checkout main` → `git pull`；
    3. 基于`main`创建本地hotfix分支：`git checkout -b hotfix/urgent-fix`；
    4. 修改、`add`、`commit`、测试后，创建PR**合并至`main`**。
- **dev分支同步机制** [12:02](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=722)
    - 修复合并至`main`后，需将`main`变更同步至`dev`：
        - 修复者：直接`git merge main`至本地`dev`；
        - 其他开发者：`git fetch origin main` → `git merge origin/main`。
- **自动化提示机制** [13:23](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=803)  
    Git分布式特性可智能检测冲突：若管理员已将`main`变更合并至远端`dev`，后续PR提交时系统将强制提示“需先拉取远端dev最新代码”。

### **分支模型最佳实践：两类高效范式 [16:16](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=976)**

- **CI/CD驱动模型（左侧）** [16:18](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=978)
    - 所有开发提交至`master`；
    - 稳定节点由权限者合并至`preproduction`分支 → CI工具自动部署测试环境；
    - 测试通过后，再合并至`production`分支 → CI工具自动发布生产版本；
    - `master`、`preproduction`、`production`均为线上分支。
- **开源主流模型（右侧）** [17:12](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=1032)
    - 开发集中于单一主干分支（如`main`）；
    - 管理员审核合并；
    - 版本发布通过**打Tag**（如`v2.1.0`）实现，支持任意Tag间切换；
    - 必要时可从Tag切出长期维护分支（如`maintenance/v1.x`），用于旧版bug修复。
- **核心原则重申** [18:09](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=1089)  
    两种模型均为实践经验总结，团队应依自身规模、流程成熟度、CI/CD能力灵活选择，目标始终是**高效交付**。
