## 第二种场景：存在冲突的协同流程 
[02:46](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=166)
    
- **冲突产生前提 [02:46](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=166)**  
	A与B同时修改同一文件（如`A.vue`）：A添加`H1 from a user`，B添加`HE from b user`；此情形常见于共用工具类等场景。
- **首次推送无冲突 [03:35](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=215)**  
	先推送者（如A）因远端尚无B代码，可直推成功；若`main`为保护分支，则B后续PR将被拒绝并提示`conflict`。
- **B同学PR被驳回后的本地冲突解决 [05:20](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=320)**  
	B收到驳回后执行`git pull`失败，提示“无法在偏离分支上快进”；需显式执行`git merge --no-ff`触发冲突检测。
- **Git pull命令本质解析 [06:02](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=362)**  
	`git pull` = `git fetch` + `git merge`；`git merge`含两种策略：`fast-forward`（无新提交，仅移动指针）与`--no-ff`（强制创建新merge提交）。
- **冲突文件手动解决 [07:02](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=422)**  
	冲突标记显示双方修改（A的`H1 from a user`与B的`HE from b user`）；选择“保留双方更改”，编辑后执行`git add`与`git commit -m "fix conflict"`。
- **提交用户信息修正 [04:26](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=266)**  
	若提交用户配置错误（如`git config user.name`未设），需用`git commit --amend --author="to m <email>"`重写提交作者信息。
