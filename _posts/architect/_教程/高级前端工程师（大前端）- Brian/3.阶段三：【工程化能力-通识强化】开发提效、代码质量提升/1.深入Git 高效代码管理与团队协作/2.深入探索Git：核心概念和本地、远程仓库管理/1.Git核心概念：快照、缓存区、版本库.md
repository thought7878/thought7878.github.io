### 快照生成流程与缓存区作用 
[00:45](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=45)

- **快照非一次性写入** [00:50](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=50)  
    快照并非直接将工作区内容写入版本库，而是经由`缓存区（Stage / Index）`中转，*提供二次确认机会*。
- **文件进入缓存区的操作** [01:03](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=63)  
    使用 `git add <file>` 将工作区文件添加至缓存区，触发版本库准备对该文件生成快照。
- **快照的本质与哈希标识** [01:23](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=83)  
    快照即对文件内容的完整拷贝，并为其计算唯一哈希值（如 SHA-1 字符串），该哈希成为后续定位该版本的索引。
- **修改后的新快照生成** [01:40](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=100)  
    工作区文件被修改后，Git自动检测变更；再次执行 `git add` 时，版本库为新内容生成全新快照，覆盖旧快照引用。
- **提交完成快照固化** [02:02](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=122)  
    执行 `git commit` 后，暂存区中所有文件的当前快照被永久保存至版本库，形成可回退的历史节点。

### 核心命令与三区域状态定义 
[02:08](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=128)

- **正向操作流程** [02:08](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=128)  
    `git add`：将工作区文件加入暂存区；  
    `git commit`：将暂存区快照固化为版本库中的一个提交（commit）。
- **三区域状态分类** [03:16](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=196)
    - **工作区（等待监视）**：未执行 `git add` 的文件，Git 不追踪其变更；
    - **暂存区（半监控）**：已 `git add` 但未 `git commit` 的文件，可随时用 `git reset` 撤回至工作区，或用 `git checkout --` 丢弃修改恢复原始版本；
    - **版本库（全监控）**：已 `git commit` 的快照集合，支持任意历史版本间切换（如 `git checkout <commit-hash>` 或 `git reset --hard <commit-hash>`）。
- **监视生效前提** [02:44](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=164)  
    文件仅在首次成功 `git commit` 后才被 Git 版本库正式监视；此前所有修改均不进入版本历史。

### Git操作命令全景与协作场景映射 
[03:47](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=227)

- **三区域对应的核心命令集** [03:47](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=227)
    - 工作区 → 暂存区：`git add`, `git rm --cached`, `git restore --staged`；
    - 暂存区 → 版本库：`git commit`, `git commit --amend`；
    - 版本库 ↔ 工作区：`git checkout`, `git restore`, `git reset`；
    - 版本库 ↔ 远程仓库：`git clone`, `git push`, `git pull`, `git fetch`, `git merge`, `git rebase`。
- **实际协作能力检验问题** [04:33](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=379ebc04bb6e45bbbc27b6cc8d806f80#?seek_t=273)
    - 下载团队代码（`git clone` / `git pull`）；
    - 修改后提交（`git add` + `git commit` + `git push`）；
    - 获取同事最新完成代码（`git pull` 或 `git fetch` + `git merge`）；
    - 回退至指定历史版本（`git checkout <commit>`或 `git reset --hard <commit>`）；
    - 合并已测试代码与本地开发中代码（`git merge` / `git rebase`）；
    - 线上故障时的标准化修复工作流（如基于 `git revert` 或 hotfix 分支）。

    - **快照机制**：每次 `git commit` 生成不可变快照，以哈希值唯一标识文件版本；
    - **三区域模型**：工作区（未追踪）、暂存区（待确认）、版本库（已提交/全控制）；
    - **监视逻辑**：仅已提交文件受 Git 全生命周期管理，切换版本即应用对应快照；
    - **缓存区价值**：作为提交前的检查点，支持选择性提交与暂存内容修正。
