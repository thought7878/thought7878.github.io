# 分层架构

00:00
通过将应用程序分为`表现层`、`业务逻辑层`和`数据访问层`，可以实现关注点分离，降低认知负荷，简化测试过程，使业务逻辑与UI框架解耦，从而提高代码的可维护性和可扩展性。各层逻辑独立，便于单独测试和理解，同时支持灵活的物理部署策略，适应不同应用场景需求。

---

![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/1029d32c6125881f8ec572c72aac525e_MD5.png]]
- `呈现层`负责**展示信息和捕捉用户交互**，*不包含业务逻辑，确保界面的独立性和测试性*。
- `业务逻辑层`**处理核心业务规则和操作**，如*计算总额和验证规则，使业务处理与界面分离*。
- `数据层`**管理外部数据源的API集成和状态管理**，*专注于数据的获取、转换和缓存，保持数据处理的独立性*。

- *每层专注于单一职责*，简化测试流程，允许独立测试，**增强代码的可理解性和可维护性**。
- *类比建筑楼层的特定功能，强调各层间的清晰界限，确保系统结构的清晰和功能的专一性*。


![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/8556691c9f6fba09680b82ef547adfc0_MD5.png]]
- **应用程序结构分为三层**：`表现层`（如尽可能无状态的React组件）、`业务逻辑层`（含模型、业务规则、计算）和`数据访问层`（API客户端、数据格式转换与映射），以*保持代码组织性和可维护性*。
- 对于**简单的静态网站或应用**，可能*无需业务逻辑和数据访问层，所有逻辑可放在后端处理*。
- 随着应用复杂度增加，**分层架构有助于保持代码清晰**，但*应根据实际问题而非预设模式添加层*。
- `表现层`应*保持纯净，避免包含状态和业务逻辑，专注于UI展示*。
- `数据访问层`*负责与后端通信；转换数据格式，确保业务逻辑层能理解接收的数据*。


## 代码的运行位置

![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/8fc17c2742e6811d54cd35f7cf9fa973_MD5.png]]
- 层与物理设备之间存在逻辑分离，*层可以分布在不同的设备上，这取决于部署策略*。
- 前端应用的`展示层`**通常在浏览器中运行**，而`业务逻辑层`和`数据访问层`**可以留在浏览器端以JavaScript形式运行，或放置在后端**。
- 架构设计不强制特定的部署策略，可以*根据应用类型选择将所有层部署在同一台机器或VPC上，或分散到多台设备*。
- 选择*将业务逻辑放在前端（丰富客户端）或后端（瘦客户端），取决于应用设计需求和性能考量*。
- *这种层的分离，提供了灵活性，允许根据实际需求调整应用的部署和运行方式*。

## 分层架构的优势

![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/fcf698114e754bec89188069e0b47a4e_MD5.png]]
使用分层架构的优势 (Benefits of using separated layers)

**1. 降低认知负荷 (Reduced cognitive load)**
*   **原文：** Focus on one concern at a time without mental context switching
*   **翻译：** 一次只专注于一个关注点，无需进行思维上下文的切换。
**2. 测试业务逻辑 (Test business logic)**
*   **原文：** Test business logic without UI complexity & Easy mocking - Mock dependencies at layer boundaries
*   **翻译：** 在没有 UI 复杂度的情况下测试业务逻辑 & 易于模拟（Mock）——在层级边界处模拟依赖项。
**3. 框架无关性 (Framework Independence)**
*   **原文：** Business logic doesn't depend on the UI framework. We replace React with Vue without touching the domain code
*   **翻译：** 业务逻辑不依赖于 UI 框架。我们可以用 Vue 替换 React，而无需触碰领域代码（核心业务代码）。
**4. 独立扩展与代码复用 (Independent scaling & Code reusability)**
*   **原文：** Different layers can scale independently & Code reusability - Domain logic shared across web, mobile, AP
*   **翻译：** 不同层级可以独立扩展 & 代码复用性 —— 领域逻辑可以在 Web 端、移动端、API（原文为AP，推测指API或应用层）之间共享。

---

- 专注于单一关注点，避免认知负荷增加，如*在处理组件时只考虑渲染逻辑，处理业务逻辑时只考虑规则*，实现无干扰的开发环境。
- *通过简化测试方法，使得业务逻辑的测试变得直接且高效*，无需依赖复杂的测试库，只需调用函数并验证结果，提升测试效率。
- *业务逻辑独立于UI框架，确保代码的可移植性*，即使更换框架如从React到Vue，也不影响核心业务逻辑的稳定性。
- *不同模块的独立性允许并行开发和扩展，提高团队协作效率*，确保代码的可维护性和可扩展性，支持跨平台共享和复用。
- *通过清晰的分层架构*，实现代码的全局可见性和可重用性，*确保业务逻辑、UI、数据处理等各层之间良好的解耦合，提升整体系统的灵活性和响应速度*。

# **架构分层详解**（从上到下）
04:24
对话讨论了*Web应用中分层架构的构建，包括表现层、业务逻辑层和数据访问层*，强调了*各层的职责分离*及对测试的积极影响。**通过将业务逻辑和数据处理从UI组件中分离，简化了组件测试，提高了代码的可维护性和测试效率**。分层架构的实施应基于解决实际问题，而非盲目追求复杂性。

![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/d650f0787c44867e05d8fa13a5ee4863_MD5.png]]

该架构主要分为三个核心层级，以及一个底层的共享层：
- `展示层 (Presentation) `- 绿色部分
    - 这是用户直接交互的界面层，**也是React/Vue特定代码的边界（下面那些层不涉及UI库的代码/接口）**。
    - **UI Components (UI 组件):** 纯展示组件。它们*只负责渲染页面和处理用户的点击等交互，不包含复杂的业务逻辑*。
    - **View Logic (视图逻辑):** 这里通常指*自定义 Hooks*（在 React 等框架中）。它们*负责管理组件的状态（State）*，并*充当“适配器”的角色，将底层的领域对象（Domain Objects）转换为 UI 组件能理解的数据格式*。
- `业务逻辑层 (Business Logic) `- 蓝色部分 
    - 这是应用程序的核心“大脑”，包含具体的业务规则，**只包含纯JS/TS的代码，没有UI库的代码**。
    - **Entities (实体):** 核心业务对象。它们代表了领域中的主要概念（例如：用户、订单、产品），通常*包含数据及其相关的业务规则*。
    - **Services (服务):** 领域服务。它们*负责编排复杂的业务操作，协调多个实体来完成特定的任务*。
- `数据访问层 (Data Access) `- 橙色部分
    - 负责*与外部数据源（如后端 API、本地存储）进行交互*。
    - **Repositories (仓储):** *抽象接口*。它们*隐藏了数据存储的具体细节*（比如数据是来自 API 还是 LocalStorage），*对上层提供统一的数据存取方法*。
    - **Data Mappers (数据映射器):** *负责数据转换*。它们*在 API 返回的原始数据格式和内部的领域对象（Domain Objects）之间进行转换*。
    - **API:** *底层 HTTP 客户端。负责实际的网络通信、认证（Authentication）、重试机制（Retries）和错误处理*。
- `共享层 (Shared)` - 灰色部分
    - 位于最底部，*通常包含跨层级使用的通用工具函数、常量或基础类型定义*。

图片底部有一段重要的提示：
***含义：** 层级的数量和职责是灵活的。但一旦你决定了采用这种架构，就必须保持一致性。例如，如果你决定在 Data Mappers 中验证所有数据，就不要在 Repositories 中对部分数据进行验证，反之亦然。*

---

- `分层架构`定义了清晰的责任边界，包括`展示层`处理用户交互与渲染，`逻辑层`负责业务逻辑和数据操作，以及底层的`工具层`提供通用功能。
- `展示层`专注于UI组件和状态管理，`逻辑层`包含实体和服务，`数据层`处理数据转换和存储，确保各层职责分明，提升代码可维护性。
- 保持架构一致性，如选择数据验证方式时，应统一应用，避免混用，确保代码库的可预测性和易于理解。
- *分层设计有利于单元测试，每层独立的特性使得测试更加聚焦，提高测试效率和代码质量*。
- `工具层`虽提供通用功能，但需避免成为杂乱无章的代码仓库，保持其组织性和目的性，确保架构的清晰与高效。

---

- `展示层（Presentation）`
	- `UI Component`只处理渲染、绑定交互的纯UI组件
	- `渲染逻辑`，管理组件状态，并充当领域对象与 UI 组件之间适配器的自定义 Hook。自定义hook，管理组件状态，在领域对象与组件之间的适配器
	- **架构含义**：前端 UI 组件只负责展示和交互，不直接处理复杂的业务逻辑（领域对象）；而“自定义 Hook”在中间起到了桥梁（适配器）的作用，把底层的业务数据转换成 UI 组件能够理解和使用的“组件状态”。
- 

专业术语解析：
- **custom hooks**：自定义 Hook（React 等前端框架中的核心概念，通常不翻译为“钩子”，直接保留“Hook”更符合开发者习惯）。
- **component state**：组件状态（前端界面中用于驱动视图更新的数据）。
- **adapters**：适配器（设计模式之一，用于连接两个接口不兼容的系统）。
- **domain objects**：领域对象（后端或业务逻辑层的核心数据模型，包含业务规则）。
- **UI components**：UI 组件 / 界面组件（前端用于渲染页面的视觉元素）。

# 分层架构在测试中的优势
![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/bc0df823645b022e607ee8460ef4cfd0_MD5.png]]
- *传统组件测试复杂度高，需模拟用户交互、网络请求，影响测试效率*。
- 组件内*业务逻辑、API调用与渲染耦合，导致测试准备繁琐且耗时*。
- 传统架构下，*测试执行时间长，主要源于组件挂载和网络调用等待*。
- Lair架构*通过解耦组件逻辑与渲染*，简化测试流程，提升效率。
- 采用Lair架构*可显著减少测试前的环境搭建与等待时间*，优化整体测试体验。

![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/9a2077918848fad3bd6446bd712a76c5_MD5.png]]
- 组件简化设计，*通过接收购物车对象并在函数中添加物品，仅负责数据渲染，分离了业务逻辑与展示逻辑，提高了测试的效率与准确性*。
- *测试策略聚焦于组件的展示逻辑，确保数据被正确呈现，而复杂的业务逻辑则在独立的单元测试中验证*，无需React框架介入，提升了测试速度。
- 通过隔离测试，验证组件在不同数据输入下的表现，确保其展示逻辑的正确性，同时避免了因业务逻辑错误导致的测试失败，增强了测试的稳定性。
- 采用快速运行的脚本测试，专注于验证业务逻辑的正确性，无需构建完整的React应用环境，显著减少了测试执行时间，提高了开发效率。
- 设计上强调*组件与业务逻辑的解耦，使得组件测试更加简单直接*，同时*也便于后续的维护与功能扩展，确保了项目的长期可持续发展*。
