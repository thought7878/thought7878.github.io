# 高内聚和低耦合
: 模型中结构化特征的关键  

High Cohesion & Low Coupling: Key to Structuring Features in Models
01:48 

高内聚和低耦合是在模型中组织特征的基本概念，确保相关元素被分组在一起并且独立的特征彼此不依赖，从而促进团队之间更容易的维护、开发和协作。  

![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/77dc94f69efa2a9042f8277fd35762b5_MD5.png]]

**架构规则 (Rule of Architecture)：高内聚与低耦合 (High cohesion and Low coupling)**

`高内聚`意味着*一起变化的事物应该在一起*；
`低耦合`意味着*独立变化的事物不应该相互依赖*。
**功能内部高内聚，功能之间低耦合。** 

**具体规则：**
- **功能内部的高关联（High coupling within features），所有相关内容放在一起** —— 更容易理解、修改和删除完整的功能，而无需在多个文件夹中翻找。 _(注：这里原文用词比较特殊，通常指功能模块内部紧密相关，即“高内聚”的概念)_
- **功能之间的低耦合（Low coupling between features）** —— 修复“结账”功能不会意外破坏“搜索”功能，团队*可以同时处理不同的功能而不会产生冲突*。
**优势 (Benefits)：**
- **支持并行开发** —— 多个开发者/团队可以在不同的功能上工作，而不会互相阻塞，也不需要频繁协调共享代码的更改。
- **使变更范围可预测** —— 当修改一个功能时，你确切知道会影响什么，因为所有内容都集中在一个地方，而不是分散在不同的架构层中。

右侧图示说明：
- **左图 (Coupling - 耦合)**：
    - 图示展示了三个“Module 1”（模块1）互相连接。
    - **解释**：耦合指的是模块之间相互依赖的程度。
- **右图 (Cohesion - 内聚)**：
    - 图示展示了 Element 1, 2, 3（元素1, 2, 3）互相连接。
    - **解释**：内聚指的是模块内的元素为了完成单一、明确的目标而协同工作的程度。


![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/2d9aacea1ba39c62fa69a00f6f4b4b05_MD5.png]]  
**高内聚与水平扩展**

- 每个垂直切片（Vertical slice）都包含 UI、实体（Entity）、服务（Service）、仓库（Repository）。
- 它们共享底层的 HTTP 客户端实现。
- 这在保持功能独立性的同时，避免了基础设施代码的重复。

**优势 (Benefits)**
- **功能内部高内聚 (High cohesion within features)** - 所有与“结账”相关的代码（从 UI 层到 Repository 层）都集中在一起。
- **功能之间低耦合 (Low coupling between features)** - “认证”模块不依赖于“产品”服务。
- **共享的技术关注点 (Shared technical concerns)** - HTTP 请求处理、错误日志记录、缓存机制等可以集中管理。

# 垂直切片
: 增强团队可扩展性和代码组织  
Vertical Slices: Enhancing Team Scalability and Code Organization
04:27 
`软件架构中的垂直切片`***按业务功能组织代码**，增强团队可扩展性，并行开发*和代码删除。 虽然它可能导致代码重复和一致性挑战，但仔细平衡可以缓解这些问题，使其**成为复杂项目的强大组织模式**。  

## 垂直切片的定义

![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/e787dc76bc4e4ac86767de2eaca4cb6a_MD5.png]]
**垂直切片 (Vertical Slices)**

`垂直切片`是*一种架构模式，它按**业务功能**（垂直切片）而不是**技术层**（水平分层）来**组织代码***。*每个切片包含一个功能所需的所有代码——从 UI 界面到业务逻辑，再到数据访问*。

```
src/
├── domain/                     # 共享 - 领域实体
│   ── entities/
│       ├── User.ts             (用户)
│       ├── Product.ts          (产品)
│       ├── Cart.ts             (购物车)
│       └── Order.ts            (订单)
│
├── infrastructure/             # 共享 - 基础设施
│   ├── http/
│   │   └── httpClient.ts       (HTTP 客户端)
│   └── repositories/
│       ├── UserRepository.ts   (用户仓库)
│       ├── ProductRepository.ts(产品仓库)
│       └── CartRepository.ts   (购物车仓库)
│
└── features/                   # 垂直切片
    └── user-authentication/    (用户认证)
        ├── login/              (登录)
        │   ├── LoginForm.tsx   (登录表单组件)
        │   ├── useLogin.ts     (登录 Hook)
        │   ├── loginService.ts (登录服务)
        │   └── loginValidation.ts (登录验证)
        └── register/           (注册)
            ├── RegisterForm.tsx(注册表单组件)
            ├── useRegister.ts  (注册 Hook)
            └── registerService.ts (注册服务)
```

## 架构中的垂直切片
![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/4b7fc4d7b7ea7a943fc4706a176404df_MD5.png]]
**垂直切片 (Vertical Slices)**

分层架构：
- **Presentation**：展示层
- **Business Logic**：业务逻辑层
- **Data Access**：数据访问层
- **Vertical Slice**：垂直切片（贯穿各层的竖条）
整洁架构：
- **Entities**：实体（最核心层）
- **Use Cases**：用例
- **Controllers**：控制器
- **Presenters**：展示器
- **UI**：用户界面
- **Gateways**：网关
- **Database**：数据库
- **Vertical Slice**：垂直切片（从外到内贯穿各层的三角形）

`垂直切片`可以应用于**任何**架构——无论是分层架构、整洁架构、六边形架构还是洋葱架构。
它是一种**组织模式**，而不是用来替代架构层的。

## 团队合作中的垂直切片
![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/c48348a4e10439a129492b212bf241ab_MD5.png]]
**垂直切片 (Vertical Slices)**

左侧图示文字翻译：
- Vertical Slice 1 / 2 / 3：垂直切片 1 / 2 / 3
- Presentation：表示层（最上层绿色块）
- Business Logic：业务逻辑层（中间蓝色块）
- Data Access：数据访问层（最下层橙色块）
- **图示展示了两个开发者通过 Git（GitHub 图标）分别开发不同的垂直切片，最后合并代码**。

**右侧文本内容：使用版本控制进行并行开发 (Parallel development with version control)**
- **开发者 1** 负责 **垂直切片 1**（例如：登录功能）。
- **开发者 2** 负责 **垂直切片 3**（例如：结账功能）。
- 他们分别提交到不同的**功能分支** (feature branches)。
- **合并冲突极少**，因为他们在不同的切片（代码区域）中工作。
- 每个切片都可以**独立部署**。

## 优点、缺点
![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/f6a2adcd5dfc89f8dd7bfad040a207fa_MD5.png]]  

**优点 (Pros)：**
- **易于导航 (Easy navigation)**：所有相关代码都在同一个文件夹中。
- **团队可扩展性 (Team scalability)**：不同的团队负责不同的功能，互不冲突。
- **易于删除 (Easy deletion)**：只需删除一个文件夹即可移除整个功能。
- **并行开发 (Parallel development)**：团队独立工作，互不阻塞。

**缺点 (Cons)：**
- **代码重复 (Code duplication)**：验证、格式化等逻辑可能会在不同功能中重复出现。
- **一致性挑战 (Consistency challenges)**：每个功能可能会以不同的方式实现相似的功能。
- **需要自律 (Requires discipline)**：团队必须能够识别何时提取共享代码（以避免过度重复）。


# 尖叫架构
: 将代码结构与业务领域对齐，以增强理解和协作  

Screaming Architecture: Aligning Code Structure with Business Domain for Enhanced Understanding and Collaboration
08:36 

尖叫的架构主张使用商业领域语言来命名功能，以立即揭示应用程序的目的，减少认知的负载，改进入职，并促进开发人员、产品经理和设计师之间更好的沟通。 这种方法在文件夹名称中优先考虑业务术语而不是技术术语，确保整个代码库中的一致性和清晰度，从而使系统的功能立即为所有利益相关者所理解，无论他们是否熟悉该项目。  

![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/d6a21f07dd91d5a5450716cfb23b52bf_MD5.png]]  
**尖叫架构 (Screaming Architecture)**

**定义：** “`尖叫架构`”意味着你的文件夹结构应该能让人一眼看出这个应用是**做什么的（业务目的）**，而不是它**使用了什么技术**。

**优势 (Benefits)：**
- **更快的入职上手 (Faster Onboarding)**。新开发者只需浏览一下文件夹名称，几分钟内就能理解系统的用途。
- **降低认知负荷 (Reduced Cognitive Load)**。开发者不需要耗费脑力将技术术语映射到业务概念上。
- **更好的沟通 (Better Communication)**。产品经理、设计师和开发者使用的是同一种语言（业务语言）。

**最佳实践 (Best Practice)：**
- **使用业务领域语言**。例如，使用 `cancel-booking`（取消预订）而不是 `deleteBooking`（删除预订）。
- **在顶层文件夹中避免使用技术术语**。*仅在功能文件夹内部按需使用这些技术术语*。
- **保持一致性**。如果你使用了 `booking-` 前缀，请在所有与预订相关的地方都使用它。
- **记录通用语言 (Ubiquitous Language)**。确保代码中的命名与业务讨论中的术语相匹配。

```
// 错误示范 (BAD)
src/
├── controllers/    (控制器层)
├── services/       (服务层)
├── models/         (模型层)
└── views/          (视图层)

// 正确示范 (GOOD)
src/
├── book-room/          (预订房间)
├── cancel-reservation/ (取消预订)
├── process-payment/    (处理支付)
└── manage-reviews/     (管理评论)
```


# 分层架构

: 平衡成本和收益以提高可维护性和可扩展性  
Layered Architecture: Balancing Costs and Benefits for Enhanced Maintainability and Scalability
11:09 

分层架构将表示与业务逻辑分离，增强了可测性和可维护性。 它提倡为特定目的使用正确的工具，从简单开始，根据需要增加复杂性，确保功能和独立开发的高度凝聚力。 虽然它需要初步的学习和设置，但它被证明对长期稳定性和团队可扩展性有益，在优先考虑可维护性和可扩展性而非初始开发速度的情况下，它的成本超过了成本。

![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/e4ceb03bb5f92ae553e4551716134e0b_MD5.png]]
**权衡（分层架构）**

**成本 (Costs)：**
- **需要浏览更多的文件** (More files to navigate)
- **初始学习曲线与搭建时间** (Initial learning curve & Setup time)
- **过度工程化的风险** (Over-engineering risk) —— *很容易添加不必要的抽象层*。

**何时应避免使用 (When to avoid)：**
- **MVP**（最小可行性产品）
- **简单的 CRUD 应用**（增删改查应用）
- **工期非常紧张的项目**（少于 2 周）


![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/a1a4b49589cc57d33b5bffa923a35ae7_MD5.png]]

**关键要点 (Key Takeaways)**

1. **将展示层与业务逻辑分离。** 业务逻辑应存在于与框架无关（framework-agnostic）的服务和实体中。
2. **在不挂载组件的情况下测试业务逻辑。** 组件测试仅用于验证渲染效果。
3. **在合适的场景使用合适的工具。** 使用 React Query 进行`服务器数据`缓存。使用 Redux 管理`全局 UI 状态`。使用`局部状态（Local state）`*管理仅组件内部使用的数据*。
4. **从简单开始，按需增加复杂度。** 如果是简单的 CRUD（增删改查）应用，可以跳过某些架构层。*当需要协调（orchestration）逻辑时再添加服务层*。
5. **功能内部高内聚，功能之间低耦合。** *一起变化的代码应该放在一起。功能之间不应相互依赖内部实现*。
