**分层架构与状态管理器 (Layers Architecture and State Manager)**

**定义：** **状态管理器 (State Manager)** —— 一种管理应用状态（数据）的工具，用于在多个组件之间共享数据，或在用户交互过程中保持数据持久化。

**它解决的问题 (The Problem it Solves)**
- 跨多个组件层级传递数据（属性钻取 / Prop Drilling）
- 在多个位置重复维护同一状态
- 手动在组件之间同步状态

**状态管理器负责处理的内容 (What State Managers Handle)**
- 多个组件所需的全局数据
- 认证状态（如登录/登出状态）
- UI 状态（弹窗开关、主题切换、导航状态等）
- 跨页面持久化的表单数据


![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/f7e3e22c35338d385ee27f0a5093a3ac_MD5.png]]

Redux（以及其他状态管理器）通常管理**与 UI 相关的状态**：
- 主题设置
- 弹窗（模态框）的显示/隐藏状态
- 认证状态（如登录状态）
- 用户偏好设置
- 导航状态
- 表单数据
**这些都是属于表示层（展示层）关注的问题，而不是业务逻辑。**

**状态存储管理器 vs 查询管理器 (Store manager vs Query managers)**

**为什么 API 缓存（服务器响应）不应该存储在 Redux 中：**
- **生命周期不同 (Different lifecycles)**。缓存需要自动过期、后台刷新和垃圾回收机制——而 Redux 的状态是手动管理且持久化的。
- **工具用错了 (Wrong tool for the job)**。Redux 缺乏缓存库所提供的内置缓存失效（cache invalidation）、过期时重新验证（stale-while-revalidate）以及请求去重（request deduplication）功能。
- **复杂性开销 (Complexity overhead)**。在 Redux 中手动管理缓存键（cache keys）、过期时间和后台更新，会产生大量的样板代码（boilerplate）。
