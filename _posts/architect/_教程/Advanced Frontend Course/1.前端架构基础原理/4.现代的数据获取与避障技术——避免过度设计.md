# 数据访问层与离线应用设计  
00:00 
讨论了数据访问层如何抽象数据源，使业务逻辑层无需关心数据实际来源。例如，当用户在线时从API获取数据并存储到IndexedDB，离线时则从IndexedDB读取数据，业务逻辑保持不变，仅由数据访问层决定数据来源，实现无缝切换。


![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/a36d5b646677b43efc2ca3522e76ae55_MD5.png]]

- Universal IT's data access layer is versatile, working with any data source, including APIs, IndexedDB, local storage, session storage, and file systems, abstracting the specifics from the business logic.
- Repositories in the data access layer handle data retrieval and storage, deciding between online API calls and offline IndexedDB usage based on the user's connection status, ensuring consistent business logic.
- The system allows applications to function seamlessly offline by caching data in IndexedDB when online and reading from it when offline, maintaining the same interface while changing the underlying data source.
- By using the same interface for different data sources, the system simplifies development, as business rules and service logic remain unchanged, with the repository determining the appropriate data access method.
- This approach to data access ensures efficient, scalable, and adaptable applications, capable of handling various data sources and user scenarios without altering core business functionalities.


# 数据获取策略与技术选型探讨  
01:17 
讨论了在不同规模和需求的应用中选择合适的数据获取方法。对于小型应用或轻量级数据处理，推荐使用React工具和API如fetch；中大型应用需重数据流管理时，建议采用Redux等状态管理器；针对中大型应用的复杂数据需求，推崇使用如React Query等现代数据获取库，因其提供乐观缓存、数据验证等功能，更专业且高效。

![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/014eb2e6a9cf034e55564f0fc6d30e54_MD5.png]]

- 对于小型应用或不需要复杂数据处理的场景，推荐使用React工具、Hooks和fetch API进行数据获取，尽管这种方式代码量较大且维护性较低。
- 对于已经采用Redux或类似状态管理库的中大型应用，推荐继续使用这些库，如Redux Toolkit，以保持数据流的统一控制和转换。
- 针对数据获取需求较高的中大型应用，推荐使用专门的数据获取库如React Query，因其具备乐观缓存、数据验证及自动重试等特性，能有效提升数据处理效率。
- 强调根据项目实际需求选择合适的数据获取方案，避免过度设计或功能冗余，特别是对于不涉及全局状态管理的应用，无需引入复杂状态管理库。
- 建议在现有项目中考虑引入如React Query这样的数据获取库，即使项目已使用Redux等状态管理库，因为这些库在数据获取方面更为专业和高效。


# 数据获取与缓存管理的最佳实践  
02:47 
讨论了在数据获取和缓存管理方面，推荐使用特定库如React Query以简化代码、提升性能和增强应用的健壮性。强调了这些库在减少样板代码、自动处理加载和错误状态、智能缓存更新以及提高UI响应速度方面的作用。此外，提到了这些工具的架构优势，即易于替换和框架无关性，使开发者能够灵活选择最适合当前项目需求的解决方案。

![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/b9196e9f371f8a03df0defe7c094ef29_MD5.png]]
- 通过缓存技术，系统能够立即返回之前请求的数据，无需重新加载，显著提升响应速度和用户体验。
- 自动处理重试逻辑，当数据发生变化时，系统能够智能判断并更新或移除缓存数据，保持数据的实时性和准确性。
- 简化UI更新流程，一旦数据更新，UI能够立即反映变化，同时在服务器调用失败时提供回滚机制，确保应用的稳定运行。
- 消除冗余代码，开发者无需手动管理加载状态、缓存逻辑或重试机制，大大减少了代码量和维护成本。
- 提供开箱即用的解决方案，无需额外配置，即可享受高效的数据管理和查询优化，显著提升开发效率和应用性能。

![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/cf62d390c710efdad6c81c966cd5d265_MD5.png]]
- 通过使用car服务调用和处理数据，而非在内部进行，实现了组件间的解耦，使得系统更易于维护和扩展。
- 采用React Query作为数据获取工具，但其仅为前端React组件的交互层，服务端逻辑保持不变，便于未来技术栈的灵活调整。
- 当需要更换数据获取库时，只需修改React Query相关的hook，而服务端代码和组件逻辑无需改动，体现了良好的分离设计。
- 这种分离架构不仅增强了系统的可替换性和可维护性，还简化了不同技术栈之间的迁移过程，提升了开发效率。
- 强调了在架构设计中分离关注点的重要性，确保了系统各部分的独立性和可替换性，从而增强了整体的健壮性和灵活性。


# 软件架构设计与层分离的重要性  
05:27 
对话强调了在软件设计中分离不同层的重要性，特别是避免在业务逻辑和数据访问层使用React等呈现库，以保持代码的可移植性和可测试性。同时，讨论了在不同场景下可以省略某些层，如服务层、仓库层或实体层，以简化架构，避免过度设计。

![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/059c38113ea6eddd1e0859ae4ae1400f_MD5.png]]
- 为了保持业务逻辑和数据访问层的纯净与可移植性，应避免在这些层中使用React或任何带有React前缀的库，确保它们仅限于展示层使用。
- 导入React相关的Hooks到业务逻辑中被视为警告信号，表明可能违反了分层原则，应谨慎处理以维护架构的清晰性。
- 推荐使用如Ladies、Data Files和Exercise这样的工具库，而非展示库，因为它们不依赖于React，更适合业务逻辑和数据处理。
- 展示层可以自由使用React及其相关库，但业务逻辑和数据访问层应保持为纯TypeScript或JavaScript，以增强可测试性和独立性。
- 虽然并非所有项目都需要完全的分层架构，但遵循这些指导原则有助于创建更清晰、更易于维护的代码结构。

![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/00c1ccce622826879966e30255576d3b_MD5.png]]
- 在操作简单且无需协调或业务工作流时，可直接从实体到仓库进行操作，无需服务层，以简化架构。
- 对于无需数据转换或简单应用中的直接查询，若仅需获取数据，使用简单的数据展示，可省略重复的业务层，提升效率。
- 当应用程序无需业务逻辑或规则，仅需显示服务器发送的简单数据时，实体层可能并非必要，减少复杂度。
- 在简单粗暴的应用场景下，如仅需直接调用API进行查询，可直接跳过服务层和仓库层，简化架构至最基础状态。
- 不应盲目添加层次结构，仅在解决实际问题时引入服务、仓库或实体层，避免过度设计，保持系统简洁高效。


# 如何理解架构层数：何时需要增加或减少  
07:59 
讨论了识别何时需要增加或减少架构层数的方法，指出当遇到复杂业务规则分散、难以测试和复用逻辑等问题时，可能需要调整层数。强调最佳架构应简化代码工作流程，而非增加复杂性，提醒开发者应根据实际问题确定架构，确保其服务于开发人员，使开发过程更加便捷。

![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/3ac47ce3b7b48159e92c406fbedd811a_MD5.png]]
- 了解何时需要增加或减少软件架构中的层次，特别是在复制验证逻辑、散乱复杂的业务规则、难以测试业务逻辑或在不同上下文中复用逻辑时，架构应简化而非复杂化代码。
- 架构设计应以解决实际问题为出发点，避免不必要的层次，如直接调用仓库而非通过无附加值的服务层，确保架构的最小结构满足需求。
- 良好的架构虽初期构建和学习成本高，但长期应提升开发效率，使代码易于理解和维护，最终为开发者带来便利。
- 架构的核心价值在于服务于开发者，简化工作流程，而非增加复杂度，应时刻铭记这一原则指导架构决策。
- 过度复杂的架构，如难以定位逻辑归属，可能表明架构设计存在问题，需反思是否真正服务于开发需求。

![[_posts/architect/_教程/Advanced Frontend Course/1.前端架构基础原理/media/af99f809c16c720b6926f030478ede4d_MD5.png]]
- 识别代码放置位置的困难源于架构层次过多且职责不明确，需要通过责任划分来确定代码归属，如业务逻辑归实体或服务，数据转换归复制器，展示相关归组件或钩子。
- 当架构无法满足应用需求时，可能是结构过于复杂或过于简单，需审视并调整架构，可能需要增加或减少层次。
- 在决定代码归属时，应考虑其功能属性，如是否涉及业务逻辑、数据处理或界面展示，以此判断其在架构中的合适位置。
- 若现有架构层次无法合理容纳新代码，可能需要重新设计架构，增加新的层次或调整现有层次的职责范围。
- 架构调整应以清晰的责任分配和层次边界为原则，确保每一部分都有明确的功能定位，避免职责重叠或缺失。
