实际应用程序中存在哪些不同的数据访问用例？  
答：实际应用中的不同数据访问用例包括理解应用所需的数据、如何使用数据以及数据检索方法的效率。  
  
为什么可能REST并不总是适合数据访问中的每种情况？  
答：REST可能并不总是合适的，因为它可能会导致诸如长而复杂的查询字符串、调试和实现自定义解析器困难以及随着过滤器数量的增加而缺乏缓存效率和性能等问题。  
  
web应用程序中与查询字符串复杂性相关的潜在问题和解决方案是什么？  
答：查询字符串的问题包括可读性、缓存效率、后端验证复杂性和URL长度限制。 解决方案包括使用旧数据样式过滤器来实现更具表现力的查询，或者使用带有JSON正文的发布请求来实现复杂查询和嵌套条件。  
  
在优点和缺点方面，使用JSON正文的post请求与其他方法相比如何？  
答：带有JSON正文的Post请求允许无限的查询复杂性和长度，为复杂的嵌套过滤器维护干净的结构，并增强安全性。 然而，它打破了其他约定，在CDN和代理级别缺乏HTTP缓存，并且需要针对API构建客户端的条件逻辑。  
  
在REST API中获取不足的挑战是什么？它如何影响用户体验？  
答：当并非所有数据都在单个服务器请求中获得时，会发生提取不足，需要多个请求并增加延迟。 这导致 “瀑布效应”，用户必须等待所有数据加载，由于错误处理和数据集成的等待时间和复杂性增加，导致用户体验不佳。  
  
在web应用程序的上下文中，什么是过度获取？  
答：过度提取是指单个API端点返回比特定请求所需更多数据的情况，导致带宽浪费和响应时间增加。  
  
过度获取的后果是什么？  
答：过度获取的后果包括浪费带宽、减慢响应时间、增加服务器负载以及更高的CPU和内存使用率，导致系统容量减少和可能更多的数据消耗，尤其是在移动网络上。  
  
“稀疏字段集” 解决方案地址如何过度获取？  
答：“稀疏字段集” 解决方案通过允许客户端仅请求所需的特定字段来解决过度获取问题，从而减少负载大小并消除对多个专用端点的需求。  
  
稀疏字段集的限制是什么？  
答：稀疏字段集的限制包括缓存复杂性增加，缓存命中率降低，处理选定字段内的复杂过滤的表现力有限，以及在需要来自多个相关资源的数据时无法解决未获取问题。  
  
对于过度获取问题的第二种解决方案是什么？  
答：过度获取问题的第二个解决方案是创建多个专门的端点来满足不同的需求，例如搜索结果、详细信息页面和比较，而不是一个通用端点。  
  
创建多个专门端点的利弊是什么？  
答：创建多个专门的端点的优点包括解决过度和欠获取问题，具有简单的客户端代码，并且不需要API编排。 然而，这种方法需要维护许多相似但略有不同的端点，增加了操作的复杂性，并且可能需要更改数据模型，这对于实施来说可能是一个挑战。  
  
什么是 “前端的后端” (BFF) 模式？  
答：“前端的后端” (BFF) 模式是一种服务器架构，其中位于客户端和后端服务之间的专用服务器聚合、转换和优化特定客户端类型的数据，从而提高性能并减少过度和不足的获取。  
  
实施BFF服务器的好处是什么？  
答：BFF服务器的好处包括减少过度获取，过滤对客户端需求的后端响应，减少不足获取，以及通过将多个后端服务调用聚合为单个客户端请求来提高性能。 此外，它允许每个客户端独立发展，并从智能缓存策略中受益。  
  
使用BFF服务器有哪些权衡？  
答：使用BFF服务器的权衡包括由于需要部署和维护单独的服务器而产生的基础设施开销，需要多个端点来实现不同功能的端点问题的持续存在，增加的操作复杂性以及后端更改可能仍需要更新多个BFF的协调问题。

# API的过滤策略
: 查询参数和JSON正文的获取与发布  
02:18 Exploring API Filtering Strategies: GET vs. POST for Query Parameters and JSON Body
02:18
讨论了**三种用于API过滤的方法:** 使用查询字符串的简单GET，使用类似SQL的查询的高级GET，以及使用JSON body的POST。 重点介绍了优点和缺点，包括*可缓存性、复杂性、安全性、遵守REST原则*，最后采用混合方法结合GET for简单性和POST for复杂查询。  

## 1. 简单过滤 (Simple Filtering)

![[_posts/architect/_教程/Advanced Frontend Course/4.数据访问层/media/e6ca64e91c633a3e0104f51adcfcb4df_MD5.png]]

选项 #1. 简单过滤 (Simple Filtering)

最简单的选项是在查询（query）中发送过滤参数。
`GET /products?type=car`

**为什么这种方式行之有效 (Why This Works Well)**
*   **每个查询参数都是一个过滤器** - 每个 query parameter 都充当过滤条件。
*   **URL 是人类可读且可缓存的** - 链接清晰易懂，且支持浏览器缓存。
*   **易于实现和调试** - 开发和维护成本低。

右侧图表部分翻译：
*   API Design: Filters
    *   API 设计：过滤器
*   步骤 1：
    *   客户端发送请求 `GET /products?type=car`
*   步骤 2：
    *   服务器向数据库发送请求：`SELECT * FROM products WHERE type = 'car'`


![[_posts/architect/_教程/Advanced Frontend Course/4.数据访问层/media/3caec75ce1c580173ee201b7c4177b64_MD5.png]]

选项 #1. 简单过滤 (Simple Filtering)

**如果我们有很多过滤参数怎么办？**
`GET /cars?make=BMW&price_max=30000&year_min=2018&transmission=automatic`
*   **URL 变得难以阅读且容易出错**
*   **缓存效率不高**（因为会产生太多唯一的 URL）
*   **后端验证变得复杂**
*   **浏览器/服务器限制了 URL 的长度**（通常为 2048-8192 个字符）

## 2. OData 风格过滤器 (OData-style filter)

![[_posts/architect/_教程/Advanced Frontend Course/4.数据访问层/media/65f42dd92c76d951f0b241a5b33826d4_MD5.png]]

选项 #2. OData 风格过滤器 (OData-style filter)

**代码示例：**
`GET /cars?filter=(make eq 'BMW' or make eq 'Audi') and price le 30000`
*(注：这是一个 GET 请求，过滤条件为：品牌是 BMW 或 Audi，且价格小于等于 30000)*

**优点 (Pros)**
*   **比简单的键值对更具表现力** - 可以表达更复杂的逻辑（如 or, and）。
*   **URL（后端 API 地址）保持可分享性** - 仍然可以通过链接分享查询结果。

**缺点 (Cons)**
*   **需要实现自定义解析器** - 后端需要专门编写代码来解析这种复杂的过滤语法。
*   **调试 URL 中的语法错误很痛苦** - 在长 URL 中找语法错误非常困难。
*   **复杂查询仍然会触及 URL 长度限制** - 如果过滤条件太多，URL 还是会变得太长。


## 3. POST 请求与 JSON 请求体
![[_posts/architect/_教程/Advanced Frontend Course/4.数据访问层/media/3247621b9ba9cd4229d67e38715714d9_MD5.png]]

选项 #3. POST 请求与 JSON 请求体 (POST & JSON Body)

代码示例：
```json
POST /cars/search
Content-Type: application/json

{
  "filters": {
    "make": "BMW",
    "priceMax": 30000
  }
}
```

**优点 (Pros)**
*   **查询复杂度和长度不受限制** - 可以处理非常长和复杂的查询。
*   **清晰的 JSON 结构** - 非常适合处理复杂的嵌套过滤条件。
*   **更高的安全性** - 请求参数对代理服务器不可见，且通常不会被保存在日志文件中。

**缺点 (Cons)**
*   **违背 REST 规范** - 使用 POST 方法来进行读取（查询）操作，不符合 RESTful 设计原则（通常 GET 用于读取）。
*   **无法利用 HTTP 缓存** - 在 CDN 或代理层无法进行 HTTP 缓存（因为 POST 请求通常不被缓存）。


## 4. 混合方法
![[_posts/architect/_教程/Advanced Frontend Course/4.数据访问层/media/c8da194a11281bbef64bb7e7130217fa_MD5.png]]  

选项 #4. 混合方法 (Hybrid approach)

左侧图示说明：
*   **Simple filters via GET** (通过 GET 请求进行简单过滤)
    *   代码示例：`GET /cars?make=BMW&priceMax=30000&location=10001`
*   **Complex filters via POST** (通过 POST 请求进行复杂过滤)
    *   代码示例：`POST /cars/search`，并在 JSON 请求体中包含复杂的嵌套过滤条件（如品牌列表、价格范围、地理位置半径、特定功能需求等）。

**优点 (Pros)**
*   **两全其美 (Best of both worlds)** - GET 请求可以在 CDN/代理层被缓存，适用于常见搜索；而 POST 请求则负责处理高级查询的复杂性。
*   **性能优化 (Performance optimization)** - 常用的简单过滤器可以利用 HTTP 缓存基础设施的优势。

**缺点 (Cons)**
*   **API 不一致性 (API inconsistency)** - 同一个概念操作（搜索/过滤）使用了两种不同的模式。
*   **文档负担 (Documentation overhead)** - 需要向使用者解释何时该使用哪种方法。
*   **客户端复杂性 (Client complexity)** - 开发者必须实现条件逻辑，根据情况决定是使用 GET 还是 POST。


# 获取不足与过度获取（REST API）
07:03 Addressing Under Fetching in REST APIs: The Challenge of Incomplete Data Retrieval
当一个请求无法检索所有必要的数据，需要多个请求时，会发生在REST APIs中获取不足的情况。 例如，为汽车构建全面的产品页面需要单独的详细信息、照片和历史记录，这说明了这个问题带来的低效率和复杂性。  

![[_posts/architect/_教程/Advanced Frontend Course/4.数据访问层/media/a28fdfa56bf02119f66c20c18549fa1b_MD5.png]]

**获取不足与过度获取问题**
*(注：Under-fetching 指获取的数据不足，通常需要多次请求；Over-fetching 指获取了超出需求的数据，造成浪费。)*

## 获取不足 (Under-fetching)

![[_posts/architect/_教程/Advanced Frontend Course/4.数据访问层/media/abefde1831a128438e1b56a1db7ebfef_MD5.png]]  

标题：获取不足 (Under-fetching)

为了构建一个完整的产品页面，你可能需要：
代码块内容（左侧为请求，右侧为注释）：
*   `GET /cars/123` —— # 汽车详情
*   `GET /cars/123/photos` —— # 图片画廊
*   `GET /cars/123/history` —— # 车辆历史报告
*   `GET /dealers/456` —— # 经销商信息
*   `GET /financing/quotes` —— # 贷款选项
*   `GET /insurance/quotes` —— # 保险估价
*   `GET /cars/similar/123` —— # 相似车辆

底部总结：
这个问题被称为 **获取不足 (under-fetching)** —— 我们*无法通过一次向服务器的请求接收所有数据，而必须进行多次请求*。


![[_posts/architect/_教程/Advanced Frontend Course/4.数据访问层/media/98ddae7a1d662a93f3b3e51d743e4c75_MD5.png]]

获取不足 (Under-fetching)

为什么这是一个问题？
*   **网络延迟 (Network Latency)** - 每个额外的 HTTP 请求都会增加 50-200 毫秒的往返时间，而顺序依赖意味着用户必须等待“瀑布流”效应（即数据像瀑布一样逐个加载，必须等前一个完成才能开始下一个）。
*   **错误处理的复杂性 (Error Handling Complexity)** - 当五个请求中的一个失败时，你必须决定是显示部分数据、重试所有请求，还是进行优雅降级（graceful degradation，即部分功能不可用时保持系统基本可用）。
*   **移动数据成本 (Mobile Data Costs)** - 多个 HTTP 请求会增加请求头（header）的开销，对于获取相同的信息会消耗更多的移动数据流量。
*   **开发开销 (Development Overhead)** - 客户端应用程序变成了编排层，需要管理复杂的加载状态和竞态条件（race conditions）。

右侧图示：
展示了客户端（笔记本电脑）向服务器依次发送多个 GET 请求（如获取汽车详情、照片、历史记录等），形象地说明了多次请求的过程。


## 过度获取 (Over-fetching)
![[_posts/architect/_教程/Advanced Frontend Course/4.数据访问层/media/5031a0d68947cc100afe27a4e7e639c9_MD5.png]]

过度获取 (Over-fetching)

过度获取的例子
*   **搜索结果页需要：** 品牌、型号、价格、缩略图 (2KB)
*   **详情页需要：** 完整规格、历史记录、经销商信息、照片 (50KB)
*   **REST 解决方案：** 让 `/cars/123` 返回所有数据 (50KB)，以便两个页面都能使用它。
*   **结果：** 搜索结果页每辆车多获取了 48KB × 20 辆车 = **浪费了 960KB**。

代码块内容翻译：
```text
# 原始问题：需要多次调用
GET /cars/123          # 仅返回汽车数据
GET /dealers/456       # 仅返回经销商数据

# “解决方案”：让端点返回更多数据
GET /cars/123          # 现在返回 汽车 + 经销商 + 融资 + 历史
                       # 但搜索结果页只需要汽车基本信息
```


![[_posts/architect/_教程/Advanced Frontend Course/4.数据访问层/media/c2bdb50323a00c312284c472f78ab697_MD5.png]]

过度获取 (Over-fetching)

为什么这是一个问题？
*   **浪费带宽 (Wasted Bandwidth)** - `/cars/123` 端点返回了所有汽车字段，而搜索结果页实际上只需要品牌、型号、年份、价格和缩略图。
*   **响应时间变慢 (Slower Response Times)** - 当你只需要 2KB 的摘要信息时，却发送了 50KB 的完整汽车数据，这意味着用户等待时间更长，服务器的工作负担也更重。
*   **移动端性能受损 (Mobile Performance Hit)** - 在移动网络上，过度获取尤其令人痛苦，因为流量费用昂贵且带宽有限，这会直接影响用户体验。
*   **服务器负载增加 (Increased Server Load)** - 服务器浪费了 CPU 和内存来序列化（serialize）那些客户端拿到后立刻就会丢弃的数据，从而降低了整体系统的处理能力。

## 解决方案 1：稀疏字段集 (Sparse Field-sets)
![[_posts/architect/_教程/Advanced Frontend Course/4.数据访问层/media/04630adebcb8517be500d1b94be5fbbf_MD5.png]]  

解决方案 #1. 稀疏字段集 (Sparse Field-sets)

代码示例：
```text
# 移动端：列表视图所需的最小数据
GET /cars?fields=id,make,model,price,thumbnail

# 桌面端：相同列表的更丰富数据
GET /cars?fields=id,make,model,year,price,mileage,thumbnail,dealer.name
```

**优点 (Pros)**
*   **解决过度获取** - 客户端仅请求所需字段，减少负载大小。
*   **单一端点** - 无需为不同用例创建多个专用端点。

**缺点 (Cons)**
*   **未解决获取不足** - 仍需多次请求相关资源。
*   **缓存复杂性** - 每种字段组合创建唯一的缓存键，降低缓存命中率。
*   **表达能力有限** - 难以轻松处理嵌套关系或所选字段内的复杂过滤。

> **💡 提示框内容：**
> *稀疏字段集在单个资源内存在**过度获取**时效果很好，但当你需要从多个相关资源获取数据时，它们就无能为力了。你仍需多次 API 调用，这又带来了**获取不足**的问题*。


  
## 解决方案 2：多个端点 (Multiple Endpoints)  

![[_posts/architect/_教程/Advanced Frontend Course/4.数据访问层/media/22297fd189b86408785c4828d8aee97b_MD5.png]]  

解决方案 #2. 多个端点 (Multiple Endpoints)

与其使用一个试图满足所有用例的通用 `/cars/123` 端点，不如创建多个专用端点。

代码块内容翻译：
```text
// 摘要端点 - 针对搜索结果网格优化
GET /cars/123/summary

// 完整端点 - 针对详情页优化
GET /cars/123/full

// 对比端点 - 针对并排对比优化
GET /cars/123/comparison
```

**优点 (Pros)**
*   **解决两个问题** - 既没有过度获取（over-fetching），也没有获取不足（under-fetching）。
*   **客户端代码简单** - 前端只需调用合适的端点，无需进行复杂的编排。

**缺点 (Cons)**
*   **端点众多** - 每个新屏幕/功能都需要一个新的端点（如 summary, full, comparison, mobile-summary 等）。
*   **后端代码重复** - 相似的逻辑在多个端点间重复，仅有细微差别。
*   **维护成本高** - 汽车数据模型的变更需要更新多个端点。
*   **文档负担** - 必须记录和维护许多相似但略有不同的端点。

> **💡 提示框内容：**
> 这种方法适用于**高流量、稳定的应用**，且客户端集合固定（如移动应用、网页、合作伙伴集成），此时性能优化足以证明维护多个相似端点的合理性。
> 它**不太适用于快速迭代的产品**或需要频繁更改 UI 的场景，因为每个新视图都需要一个新端点。


  
## 解决方案 3：面向前端的后端 (Backend For Frontend)
前端后端 (BFF) 设计模式，用于优化特定于客户端的数据传输  
14:41 Backend for Frontend (BFF) Design Pattern for Optimized Client-Specific Data Delivery
BFF充当客户端和真正的后端服务之间的专用服务器，聚合、转换和优化特定客户端类型的数据，减少过度获取和不足获取，实现独立的客户端演进，并实施智能缓存，尽管它引入了基础设施开销和操作复杂性。

解决方案 #3. 面向前端的后端 (Backend For Frontend)

解决方案 #3. 面向前端的后端 (Backend For Frontend)

**优点 (Pros)**
*   **减少过度获取 (Reduces over-fetching)** - BFF 可以根据客户端的需求过滤后端的响应数据。
*   **减少获取不足 (Reduces under-fetching)** - BFF 将多个后端服务的调用聚合为单个客户端请求。
*   **独立演进 (Independent evolution)** - 移动团队可以更改其 BFF 的实现（例如添加新数据源、修改响应格式、优化带宽），而无需与 Web 团队协调，也无需担心破坏 Web 客户端。
*   **智能缓存 (Smart caching)** - BFF 可以缓存频繁访问的后端数据，并应用针对特定客户端的缓存策略。

**缺点 (Cons)**
*   **基础设施开销 (Infrastructure overhead)** - 需要部署和维护独立的 BFF 服务器。
*   **未完全解决端点问题 (Doesn't fully solve the endpoint problem)** - 每个 BFF 仍然需要为不同的屏幕或功能提供多个端点。
*   **运维复杂性 (Operational complexity)** - 需要监控、扩展和调试更多的服务。
*   **团队协调 (Team coordination)** - 后端的变更可能仍然需要更新多个 BFF。

> **💡 提示框内容：**
> 最适合那些不同客户端有不同限制和需求的组织——例如需要 10KB 响应的移动应用，与可以处理 100KB 丰富元数据的管理后台。
> 当你拥有截然不同的客户端应用时非常有效：公共 Web 端 vs 内部工具，面向客户 vs 合作伙伴 API。
