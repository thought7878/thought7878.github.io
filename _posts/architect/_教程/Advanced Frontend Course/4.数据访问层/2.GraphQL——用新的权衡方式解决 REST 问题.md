使用GraphQL多个REST调用的主要优点是什么？  
答：使用GraphQL而不是多个REST调用的主要优点是它允许编写一个简单的查询，该查询可以在单个请求中完全请求所需的数据，从而避免了多个API调用的需要。  
  
GraphQL在客户端请求和服务器响应方面与传统的REST有何不同？  
答：使用GraphQL，客户端控制要获取的确切数据，并使用HTTP上的查询语言在一个请求中获取它。 这与REST形成了对比，REST由服务器提供一组固定的端点，客户端必须管理如何从中获取所需的数据。  
  
GraphQL比REST有什么优点？  
答：GraphQL相对于REST的优势包括性能优势，例如仅获取必要的数据，在不破坏现有客户端的情况下进行API演进，为强大的工具提供强大的类型系统支持，客户端灵活性以及解决过时API文档问题的自文档化模式。  
  
使用GraphQL有哪些挑战和缺点？  
答：使用GraphQL的挑战和缺点包括实现的复杂性，由于单点查询请求而导致的缓存的潜在问题，由于其灵活的性质而引起的安全问题，以及对速率限制和复杂性分析的需求。 此外，以GraphQL格式上传文件不是内置功能，并且由于其强类型性质，不是发送二进制数据的理想用例。  
  
在什么情况下应该考虑使用GraphQL而不是传统的REST？  
答：如果您的客户需要显著不同的数据集，特别是在您的移动应用程序和具有不同数据要求的web应用程序的情况下，您应该考虑使用GraphQL而不是传统的REST。 这可以显着减少服务器上的负载，并允许每个客户端仅获取所需的数据，从而使开发人员的工作更轻松。  
  
在移动应用程序中使用GraphQL比REST有什么好处？  
答：使用GraphQL，移动客户端可以请求更少的字段，web请求可以有更多的字段，并且不需要重复的端点，因为新字段可以添加到架构中一次并立即使用，而不需要最新进展到移动应用程序。  
  
你什么时候考虑使用GraphQL作为后端服务？  
答：对于单个源连接或简单连接，REST是可以的。 但是，如果需要从多个服务中提取，您可以考虑使用GraphQL或BFF。  
  
为什么休息通常是外部或第三方开发人员的更好选择？  
答：Rest通常是外部或第三方开发人员的更好选择，因为几乎每个开发人员都熟悉使用REST API，将GraphQL作为公共端点的附加解决方案可能会引入复杂性，尤其是在安全性方面。  
  
数据关系对选择GraphQL和REST有什么影响？  
答：如果数据关系很简单，比如产品和品类，其余部分可以很好地处理这个问题。 但是对于深度嵌套的数据关系，例如每个作者的其他帖子都有评论的博客帖子，复杂度会随着REST的增加而增加，而GraphQL可以通过一个请求来处理这个问题。  
  
在什么情况下，对于有类似需求的客户，您可能不需要GraphQL？  
答：如果移动和桌面客户端可以使用相同的端点而不会出现获取或获取不足的问题，并且它们的要求没有明显差异，那么即使有多个具有类似需求的客户端，其他客户端也可能仍然没问题。  
  
在GraphQL和REST之间进行选择时，应该考虑哪些关于团队经验和项目时间表的因素？  
答：要考虑的因素包括需要快速发货、团队对GraphQL的熟悉程度、客户UI需求的稳定性和可预测性，以及是否需要GraphQL提供的灵活性。  
  
GraphQL的灵活性如何带来安全风险？  
答：GraphQL的灵活性带来了安全风险，因为客户端可以控制查询结构和深度，这可能导致资源耗尽的请求，如无限循环或获取大量数据。 解决方案包括限制查询深度、实施查询复杂性分析以及使用基于成本的查询限制。  
  
为什么我们不能对GraphQL使用简单的浏览器缓存？有什么工具可以优化缓存？  
答：简单的浏览器缓存对于GraphQL是不可能的，因为有一个单一的GraphQL端点没有基于URL的缓存机制。 需要像阿波罗这样的特殊工具来进行服务器端和客户端缓存。  
  
什么是阿波罗？它如何帮助GraphQL？  
答：Apollo是一组工具，可帮助构建GraphQL API服务器，包括查询解析和执行、与缓存系统集成以及支持持久性查询等功能，以减少网络使用并提高性能。  
  
主题持久性查询 (APQP) 如何提高GraphQL的性能？  
答：主题持久性查询 (APQP) 通过允许客户端发送查询标识符而不是完整的查询字符串来减少网络使用并提高性能，从而在不影响响应大小的情况下显著减少请求大小。  
  
在阿波罗中使用短哈希查询作为get请求的目的是什么？  
答：在阿波罗中使用短哈希查询作为get请求的目的是利用CDN缓存来处理这些get请求，减少通过网络发送的数据量，并使服务器能够使用预先存储的映射来识别和执行持久性查询字符串。  
  
阿波罗服务器如何利用缓存来提高查询响应时间？  
答：阿波罗服务器通过在执行时将查询响应存储在内存缓存中来利用缓存。 随后，对于相同数据的请求可以从此缓存中提供，而不是再次执行查询，从而减少了服务器上的工作负载，并缩短了所有客户端的响应时间。  
  
客户端缓存在阿波罗中起什么作用？  
答：客户端缓存在阿波罗中起着减少不必要的网络请求的作用，通过将GraphQL查询的结果存储在一个特定于一个客户端的规范化内存缓存中。 这允许客户端从缓存中返回数据以进行重复查询，使应用程序对用户来说感觉非常即时。  
  
客户端和服务器端缓存如何在GraphQL设置中协同工作？  
答：客户端和服务器端缓存在GraphQL设置中协同工作，通过减少跨客户端重复查询的服务器工作负载并利用CDN缓存进行get请求。 阿波罗服务器的后端缓存可以在所有客户端之间共享，并可以使用哈希的持久性查询来进一步优化网络数据传输。 这两种缓存的组合可以更快地检索数据并降低系统的复杂性。  
  
与GraphQL相比，使用REST有哪些挑战？  
答：与使用GraphQL相比，使用REST的挑战包括可能过度获取或无法获取数据，这可能分别是浪费或缓慢的。 此外，REST需要管理多个类似的端点或复杂的URL参数，并且缺乏必须手动构建的自动缓存和安全保护。  
  
与休息相比，GraphQL存在哪些权衡？  
答：GraphQL提出了一些权衡，例如更陡峭的学习曲线以及需要为查询哈希、客户端和服务器缓存进行深思熟虑的架构设计，这增加了复杂性，但也提供了更大的客户端灵活性和在单个请求中获取精确数据的控制。  
  
在决定使用REST和GraphQL进行项目时，应该考虑什么？  
答：在项目中决定使用REST还是GraphQL时，要考虑客户的多样性、数据需求的复杂性以及是否每个人都需要相同的数据。 如果项目需要在最少的GraphQL知识的情况下快速发货，从REST开始可能是优选的，并且有可能在复杂性证明其合理性时转换到GraphQL。

GraphQL允许客户在一个请求中指定确切的数据需求，将多个REST调用组合到一个查询中，以提高效率并降低复杂性。

代码块（保持原样，因为这是代码语法）：
```graphql
query CarDetails($id: ID!) {
  car(id: $id) {

  }
```

`GraphQL` 是一种*查询语言和运行时*，它**允许客户端通过编写声明式查询，在单个请求中精确获取所需的数据**。
*与返回固定数据结构的多个 REST 端点不同，GraphQL 仅提供一个端点*。在该端点中，*客户端可以指定他们需要的精确字段和关系，而服务器则会根据需要从多个数据源获取数据来解析该查询*。


![[_posts/architect/_教程/Advanced Frontend Course/4.数据访问层/media/25698f2380663c53c900ba34d0c92a3c_MD5.png]]

**GraphQL**
*   **不是新的网络协议。** GraphQL *使用标准的 HTTP **POST 请求**发送到单一的 `/graphql` 端点，查询和响应均使用 **JSON 格式**的数据负载（payloads）*。
*   **对多客户端场景特别有价值。** 不同的客户端可以*从同一个端点请求不同的字段*，而*不需要为每种客户端类型维护单独的 API 或 BFF（Backend for Frontend，服务于前端的后端）服务器*。
*   **不是替代品。** *GraphQL、REST 和 BFF 可以根据具体需求在同一个系统中共存协作*。

# GraphQL 对比 REST
: 一个灵活的客户端驱动的数据获取解决方案  
01:03
GraphQL通过标准的HTTP POST请求进行操作，允许多个客户端从单个端点请求不同的数据字段。 与REST提供固定端点和结构化数据不同，GraphQL使客户端能够控制在一个请求中获取的确切数据。 这种灵活性使其非常适合需要动态数据需求的场景，尽管由于使用POST请求，它缺乏自动浏览器缓存。 GraphQL是补充而不是替代REST，以满足系统中的特定需求。

左侧文本（REST）：
**REST**
服务器提供*固定的端点*。*客户端需要自行决定如何从这些端点获取所需的数据*。
*   **面向资源的端点** (Resource-oriented endpoints)
*   **服务器决定包含哪些数据** (Server decides data inclusion)
*   **获取相关数据需要多次请求** (Multiple requests for related data)
*   **基于 HTTP 动词的语义** (HTTP semantics with verbs，指 GET, POST 等)
*   **URL 可缓存** (Cacheable URLs)

右侧文本（GraphQL）：
**GraphQL**
*客户端控制获取什么数据，并且通过一次请求即可获取*。
*   **面向查询的数据获取** (Query-oriented data fetching)
*   **客户端决定确切的字段** (Client decides exact fields)
*   **一次请求获取所有数据** (Single request for all data)
*   **基于 HTTP 的查询语言** (Query language over HTTP)
*   **灵活的单一端点** (Flexible single endpoint)
