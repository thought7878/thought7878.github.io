持续集成和部署 (CIC) 循环工具的目的是什么？  
答：持续集成和部署 (CIC) 循环工具的目的是确定代码是否准备好合并。  
  
Jenkins如何促进持续集成？它的限制是什么？  
答：Jenkins通过从GitHub运行webhooks触发的构建、测试和部署来促进持续集成。 它的限制包括新项目的复杂设置，需要维护Jenkins服务器和插件，潜在的插件冲突以及维护负担。  
  
为什么新项目不太可能选择Jenkins，Jenkins和GitHub Actions和GitLab CI等替代品目前的市场份额是多少？ 
答：新项目不太可能选择Jenkins，因为它的复杂性和服务器和插件的维护负担。 尽管Jenkins因传统安装而拥有显著的市场份额，但它并不是新项目的首选。 许多团队更喜欢GitHub Actions和GitLab CI。  
  
GitHub行动与GitLab CI的比较优势是什么？  
答：对于GitLab来说，主要优点是它提供了一个功能全面的工具，包括CI/CD、问题跟踪、安全扫描和项目管理。 对于GitHub来说，其优势包括与更大的第三方工具和服务生态系统集成，并且GitHub操作已经改进，以更简单的用户界面提供类似的功能。  
  
团队什么时候可以选择GitHub或GitLab来满足他们的CI/CD需求，应该考虑什么？  
答：选择GitHub可能是因为其全面的集成生态系统和易用性，特别是对于复杂的企业工作流。 然而，团队在做出选择之前应该考虑他们的特定需求和潜在问题，例如学习曲线和全面的解决方案的必要性。

# Jenkins

![[_posts/architect/_教程/Advanced Frontend Course/8.面向前端开发者的 CICD 基础知识/media/bbaf5d0aab04f77938e215f045a1d9d9_MD5.png]]

Jenkins 是一个独立于 GitHub 的工具。  
Jenkins 通过使其对常规开发团队（而不仅限于配备专职构建工程师的大型企业）可用，*推动了 CI/CD 的普及化*。  
- 你需要配置 GitHub 向 Jenkins 服务器发送 webhooks（网络钩子）  
- Jenkins 会在触发时从 GitHub 拉取代码  
- Jenkins 在其独立的基础设施上运行构建（而非 GitHub 的基础设施）  

流程图：CI/CD 工作流  
1. **推送代码到仓库会触发 Jenkins**  
   （`Pushing code to the repository triggers Jenkins`）  
   *→ 箭头指向：App Code → Build*  
2. **Jenkins 负责：**  
   - 构建（Build）  
   - 测试（Testing）  
   - 创建制品（Create an artifact）  
   - 部署（Deploy）  
   *→ 箭头指向：Build → Test → Artifact Repo*  

图示关键元素：  
- **CODE**（编码）→ **BUILD**（构建）→ **TEST**（测试）→ **RELEASE**（发布）  
- **ARTIFACT REPO**（制品库）：存储测试通过并可部署的构建产物  
- **DEPLOY**（部署）→ **OPERATE**（运维）→ **MONITOR**（监控）→ **PLAN**（规划）→ **CODE**（闭环）

## 问题

![[_posts/architect/_教程/Advanced Frontend Course/8.面向前端开发者的 CICD 基础知识/media/688d382f072267309f01c8ec86702561_MD5.png]]

Jenkins 的现状 (Jenkins Today)

**问题 (Problems)**
*   新项目的**设置复杂**。此外，你需要维护 Jenkins 服务器。*它不像 GitHub Actions 或 GitLab CI 那样是内置的解决方案*。
*   你**需要维护插件**。Jenkins 是一个插件系统，每个插件都必须被检查、安装和更新。插件之间可能会相互冲突。

**仍在使用 (Still use)**
*   已经在使用 Jenkins 的大型企业。
*   需要本地部署（on-premise）CI/CD 的强监管组织。（GitHub Actions 支持自托管）
*   拥有复杂、自定义构建需求的团队。插件系统允许创建非常灵活的 CI/CD 解决方案。

底部蓝色框结论：
**结论：** *新项目很少选择 Jenkins。许多团队正在积极地从 Jenkins 迁移出去*。由于遗留系统的安装，Jenkins 仍然占有显著的市场份额。


# GitHub 与 GitLab

![[_posts/architect/_教程/Advanced Frontend Course/8.面向前端开发者的 CICD 基础知识/media/e39e498902c70eac0fd86eea71654d0b_MD5.png]]  

GitHub 与 GitLab 的选择 (GitHub vs GitLab Choice)

**GitLab**
*   **多合一 (All-in-one)。** 如果你希望在一个工具中包含所有功能（CI/CD、问题跟踪、安全扫描、项目管理），GitLab 仍然提供更全面的解决方案。
*   **更强大的 CI/CD。** 在某些特定场景下，GitLab 的 CI/CD 仍然更强大，特别是对于复杂的企业工作流。

**GitHub**
*   **卓越的集成 (Superior integrations)：** 第三方工具、包和服务通常优先与 GitHub 集成。其生态系统更大、更成熟。
*   **GitHub Actions 的成熟度：** 虽然 GitLab CI 最初更胜一筹，但 GitHub Actions 已经迎头赶上，现在提供相当的功能，且更易于使用。

绿色提示框：
✅ **现实检验 (Reality check)：** 在 2025 年，选择任何一个平台都不会犯大错。

底部建议：
***建议 (Recommendation)：*** 对于大多数团队来说，GitHub 是更安全的选择；对于复杂的企业需求，选择 GitLab（只有当你清楚为什么需要它时才选它）。
