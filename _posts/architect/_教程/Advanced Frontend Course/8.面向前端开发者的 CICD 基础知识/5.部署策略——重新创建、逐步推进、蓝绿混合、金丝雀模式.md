问：NJS应用程序或web应用程序在服务器端呈现或创建静态水疗中心的部署策略是什么？  
答：NJS应用程序或web应用程序的部署策略，具有服务器端渲染或创建静态水疗中心的部署策略，包括通过云解决方案进行简单部署，如没有复杂部署策略的S3存储桶、VPS (虚拟专用服务器) 、VPS上的下午2、VPS上的Docker和VPS上的Kubernetes。  
  
问：什么是VPS，它如何促进NoJS应用程序的部署？  
答：VPS是服务器上的私人空间，与他人共享，但拥有自己的资源，如CPU、内存和存储。 它通过SSH连接提供安全的环境，方便部署NoJS应用程序，应用程序可以直接部署并运行在监听本地主机端口3000的VPS上。  
  
问：在部署NoJS应用程序时，引擎扮演什么角色？  
答：引擎的作用是处理诸如HTTPS的SSL证书之类的事情，有效地提供静态文件，并在用户访问应用程序时充当反向代理。 它将请求从前端转发到在VPS上运行的NoJS应用程序。  
  
问：使用VPS进行部署的优点和缺点是什么？  
答：使用VPS的优点包括最小的设置复杂性，直接访问服务器进行调试，并且没有额外的阻塞层。 缺点包括进程管理问题，其中应用程序崩溃导致停机和需要手动重新启动，以及开发和生产之间的数学环境不一致。  
  
问：什么是下午2，它如何改进基本的VPS部署？  
答：下午2是一个内置负载均衡器的NoJS应用程序生产流程管理器，允许应用程序保持活动状态并在崩溃时自动重启，而不会停机。 它通过提供其他功能，如监控CPU和内存使用情况、零停机时间部署和简化管理，改进了基本的VPS部署。  
  
问：在VPS上使用Docker有哪些权衡和好处？  
答：在VPS上使用Docker的好处包括所有阶段的环境一致性，通过版本控制的Docker镜像轻松回滚到以前的版本，以及简化的部署过程。 权衡是学习Docker的额外复杂性，以及它仍然依赖于具有有限扩展选项的单个服务器的事实。  
  
问：Kubernetes如何促进应用程序的部署和管理？  
答：Kubernetes通过自动处理扩展、故障转移和部署来简化应用程序的部署和管理。 它可以添加更多的容器来处理增加的流量，用新的容器替换旧的容器，监视容器的健康状况，并在它们之间分配流量。  
  
问：使用Kubernetes的优点和缺点是什么？  
答：使用Kubernetes的优点包括自动扩展和负载平衡、自我修复、零停机时间的滚动最新进展和多服务器高可用性。 缺点是复杂的设置和维护要求、运营开销以及管理复杂的分布式系统。  
  
问：什么是AWS弹性计算服务（ECS），它如何简化部署过程？  
答：AWS弹性计算服务（ECS） (亚马逊弹性容器服务) 是一种托管的Docker容器服务，通过处理复杂的设置和维护来简化部署过程。 它允许您专注于更新和部署应用程序，而无需担心底层基础设施。  
  
问：与管理自托管Kubernetes集群相比，在PPS上使用Kubernetes有哪些权衡？  
答：在PPS上使用Kubernetes可以消除管理底层基础设施和编排系统的需要，提供扩展、负载均衡和健康检查等好处，而无需管理开销。 然而，这种便利是有代价的，由于支付AWS服务费用、有限的定制选项以及供应商锁定在AWS基础设施中，可能会产生更高的费用。  
  
问：无服务器计算是如何工作的，它的优点和缺点是什么？  
答：无服务器计算允许运行代码，而无需通过编写部署到AWS等云提供商的小函数来管理服务器。 供应商处理其他一切，包括服务器、扩展和监控。 其优势包括零服务器管理、基于使用情况的自动扩展和按需付费定价。 然而，由于函数执行的时间限制，它也存在诸如供应商锁定和长时间运行的流程限制等缺点。  
  
问：在成本、控制和复杂性方面，平台即服务 (PaaS) (PaaS) 与其他部署选项相比如何？  
答：像Heroku或铁路这样的平台即服务 (PaaS) 提供了一种方便且经济的方式来开发和部署应用程序，内置监控和扩展功能，只需要最少的基础设施知识。 然而，它是最昂贵的选项，并提供对基础设施的最少控制。 权衡表明，虽然它易于使用且非常适合快速原型设计，但它比自我管理选项的成本更高。  
  
问：无服务器计算有哪些用例受益？  
答：无服务器计算使涉及短暂、不可预测流程的使用案例受益，例如文件处理、用户上传、Webhook、计划任务和事件驱动的应用程序。 这些使用案例需要可扩展的解决方案，而不需要长期的服务器承诺。  
  
问：使用平台即服务 (PaaS) (PaaS) 的权衡是什么？  
答：使用PaaS的权衡包括: 与管理VPS相比成本最高，由于在平台的限制内工作而导致的有限定制，以及无法微调基础架构。 它最适合快速原型制作和没有运营专业知识的团队。  
  
问：无服务器计算的权衡是什么？  
答：无服务器计算的权衡包括与其他选项相比最昂贵的定价模式，有限的定制，因为它在平台的限制内运行，以及以应用为中心的开发方法，与平台的方法紧密集成。  
  
问：什么时候使用VPS比Kubernetes等更复杂的解决方案更可取？  
答：对于需要在可扩展性、复杂性和成本之间取得平衡的团队，VPS更受欢迎，以更低的成本提供最多的控制。 它适用于希望专注于开发而没有基础设施问题的小团队或单独开发人员，以及需要以最少的基础设施快速上市的应用程序。  
  
问：无服务器计算什么时候是合适的选择？  
答：无服务器计算适用于需要类似Kubernetes的好处而不需要操作复杂性、持续编排的初创公司和团队，以及需要快速原型设计和快速上市时间且基础设施问题最小的应用程序。

部署与运维。在 VPS 上使用 PM2

部署与运维。在 VPS 上使用 Docker

**优点 (Pros)**
*   **所有阶段的环境一致性**：确保开发、测试和生产环境完全一致。
*   **轻松回滚到之前的版本**：如果新版本有问题，可以快速切换回旧版本的镜像。

部署与运维。在 VPS 上使用 Kubernetes

**优点 (Pros)**
*   **自动伸缩和负载均衡**：根据流量自动增加或减少容器数量，并分配流量。
*   **自愈能力（自动重启失败的容器）**：如果容器崩溃，Kubernetes 会自动重新启动它。
*   **滚动更新，零停机时间**：在更新应用时，逐步替换旧版本，确保服务不中断。
*   **多服务器高可用性**：可以在多台服务器上运行，一台坏了其他的还能继续工作。
*   **高级部署策略**：支持蓝绿部署、金丝雀发布等复杂的发布方式。

**缺点 (Cons)**
*   **设置和维护复杂**：搭建 Kubernetes 集群非常麻烦，后续维护也不简单。
*   **需要专门的 DevOps 专业知识**：团队里必须有懂 Kubernetes 的专家。
*   **显著的运维开销**：管理集群本身就需要消耗大量的资源和精力。

**使用场景 (Use cases)**
*   **需要跨多台服务器实现高可用性的大型应用程序**：适合不能宕机的大项目。
*   **需要复杂自动伸缩和负载均衡的应用程序**：适合流量波动大、架构复杂的应用。

### 优点、缺点、使用场景
![[_posts/architect/_教程/Advanced Frontend Course/8.面向前端开发者的 CICD 基础知识/media/ebc92fafdc11968499664c77726f105c_MD5.png]]

部署与运维。托管容器服务

**优点 (Pros)**
*   **拥有类似 Kubernetes 的所有优势，且无需承担管理开销**：*享受容器编排的好处，但不用自己操心集群维护*。
*   **无需维护基础设施**：云服务商（如 AWS）会处理底层的服务器和系统更新。

**缺点 (Cons)**
*   **供应商锁定 (Vendor lock-in)**：一旦使用了特定云服务商的服务，很难迁移到其他平台。
*   **对基础设施的控制力较弱**：你无法像管理自己的服务器那样深入配置底层系统。
*   **在大规模使用时可能很昂贵**：随着应用规模扩大，托管服务的费用可能会显著增加。
*   **自定义选项有限**：只能使用服务商提供的配置选项，灵活性不如自建集群。

**使用场景 (Use cases)**
*   **需要 Kubernetes 优势但不想面对运维复杂性的成长型初创公司**：想要快速扩展，但人手不足。
*   **想要容器编排但缺乏 DevOps 专业知识的团队**：团队主要关注代码开发，不想花大量时间学习复杂的集群管理。

`无服务器 (Serverless)`**意味着你运行代码而无需管理服务器**。

部署与运维。无服务器 (Serverless)

**优点 (Pros)**
*   **零服务器管理，无需担心基础设施**：完全不需要操心服务器的配置和维护。
*   **自动伸缩**：系统会根据流量自动调整资源。
*   **按实际使用量付费**：用多少付多少，没有请求时不产生费用。
*   **内置高可用性**：云服务商通常会自动处理容错和备份。

**缺点 (Cons)**
*   **冷启动延迟 (Cold start latency)**：如果函数很久没被调用，第一次调用时会有延迟（因为需要重新初始化环境）。
*   **供应商锁定 (Vendor lock-in)**：代码和架构通常深度绑定特定的云服务商（如 AWS Lambda），很难迁移。
*   **不适合长时间运行的进程**：Serverless 函数通常有执行时间限制（例如几分钟），不适合跑很久的任务。

**使用场景 (Use cases)**
*   **流量不可预测的 API 端点**：有时候流量很大，有时候没人用，Serverless 能很好地应对这种波动。
*   **事件驱动的应用程序**：例如文件处理（上传文件后触发处理）、Webhook 接收、定时任务等。
