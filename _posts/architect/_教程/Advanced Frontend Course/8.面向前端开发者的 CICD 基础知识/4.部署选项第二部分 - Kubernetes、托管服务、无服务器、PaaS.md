部署静态网站需要哪些主要组件？  
答：部署静态网站所需的三个主要组件是文件、存储 (如AWS S3) 和用于快速交付的引擎或CDN。  
  
有哪些CI/CD管道的优化技术？  
答：CI/CD管道的优化技术包括使用日志文件哈希的依赖项缓存，在多个运行器之间拆分作业以并行运行任务，在关键故障时尽早退出管道，以及通过删除旧工件和临时资源来清理自动化。  
  
在软件发布流程中，手动部署和自动部署有什么区别？  
答：手动部署涉及人类做出触发部署的决策，例如合并拉取请求或运行命令，而自动部署则在系统根据特定情况进行部署时发生，例如时间表或被合并的分支，无需人为干预。  
  
有哪些手动和自动发布策略的例子？  
答：手动发布策略的示例包括将拉取请求与系统触发的后续自动部署合并到主分支，或者创建触发主题发布的新标签。 自动发布策略包括创建可能触发自动部署的发布分支，计划在固定日期发布而不考虑代码更改，以及从暂存到生产的环境升级，人类可以决定生产部署新功能的准备情况。  
  
哪些特征定义了选择自动与手动部署策略的团队？  
答：选择自动部署的团队通常具有强大的测试管道，部署小的频繁更改，使用功能标志实现受控的功能可见性，具有出色的监控和快速回滚功能。 相比之下，选择手动部署的团队通常需要在面向客户的更改之前进行利益相关者审查，在需要审计跟踪的受监管行业工作，或者对自动化测试的覆盖范围缺乏信心。

# 优化技术
![[_posts/architect/_教程/Advanced Frontend Course/8.面向前端开发者的 CICD 基础知识/media/5df3eb87c932cf51c2dbadc64ae2ccae_MD5.png]]

**优化技术 (Optimisation Techniques)**
*   **依赖缓存**：在多次运行之间存储 `node_modules` 依赖，使用 lockfile（锁文件）的哈希值作为缓存键。*这可以将构建时间减少 30-50%*。
*   **在多个 Runner 上拆分任务**：*同时运行代码检查 (linting)、测试和构建，而不是按顺序依次执行*。
*   **提前退出**：当遇到编译错误或安全漏洞等关键失败时，立即停止流水线 (pipeline)。
*   **清理自动化**：移除旧的构建产物 (artifacts)、Docker 镜像和临时资源，*以防止存储空间膨胀*。

# 何时、如何 发布新版本？
![[_posts/architect/_教程/Advanced Frontend Course/8.面向前端开发者的 CICD 基础知识/media/9f52869d8cfcb41ad82f86cff7803db1_MD5.png]]  

**我们何时（以及如何）发布新版本**

我们有多种发布方式，包括`自动发布`和`手动发布`。
*   **(手动决策或自动)** `将 PR 合并到主分支`（master, prod, production —— 名称不限）。*由人工决定何时合并 PR，这可能会触发自动部署。我们将使用该分支的应用代码进行构建并部署*。
*   **(手动决策)** `创建新标签` (Tag)。*由人工决定何时创建一个标签（例如 v2.0）*，该标签指向特定的提交 (commit) 或分支。*创建标签会触发自动发布。我们将使用标签所指向的提交或分支中的应用代码*。
*   **(手动决策)** `发布分支策略`。*由人工决定何时创建专用的发布分支（例如 release/v2.1）。创建或更新该分支可能会触发自动部署*。
*   **(手动决策)** `API/Webhook 触发`。人工通过外部 API 调用、项目管理工具或监控系统发出的 Webhook，或者通过部署仪表盘上的手动触发器来启动部署。
*   **(自动决策)** `定时发布`。*按照固定的时间表进行部署*（例如，每周二下午 2 点），无论代码是否有变更或是否有人工干预。
*   **(手动决策)** `环境晋级` (Environment promotion)。由人工决定何时手动*将代码从预发布环境 (staging) 晋级到生产环境*。

**核心问题是：当新功能准备就绪时，我们是希望自动将新版本部署到生产环境，还是希望手动启动部署流程？**


# 自动部署还是手动部署？
04:28
自动部署适合具有强大测试、频繁的小更改、功能标志和快速回滚的团队，而需要利益相关者审查、在受监管部门工作或缺乏自动测试可信度的人更喜欢手动部署。


![[_posts/architect/_教程/Advanced Frontend Course/8.面向前端开发者的 CICD 基础知识/media/55bdf6aec7e191b9e1f6a9dbcfd0a151_MD5.png]]

自动部署还是手动部署

**自动部署 (Automatic)**
**选择此方式的团队：**
*   拥有他们完全信任的成熟测试流水线。
*   部署小而频繁的变更。
*   使用功能开关（Feature Flags）来控制功能的可见性。
*   拥有出色的监控和快速回滚能力。

**手动部署 (Manual)**
**选择此方式的团队：**
*   在面向客户的变更发布前，需要利益相关者（Stakeholder）进行审查。
*   在受监管的行业工作，需要审计追踪（Audit Trails）。
*   拥有复杂的集成，需要人工验证。
*   对自动化测试的覆盖率缺乏信心。
