应用程序部署的重新创建策略是什么？  
答：重新创建策略，也称为 “大爆炸”，是部署新应用版本的最简单方法。 它涉及停止旧应用程序版本，部署新应用程序，然后启动它。 在过渡期间，应用程序不可用，此策略需要停机。  
  
重新创作策略的优点和缺点是什么？  
答：重新创建策略的优点包括实施简单，过程直观，不需要复杂的编排，以及资源效率高，因为它不需要同时运行两个应用程序版本。 缺点包括需要停机时间和部署期间服务不可用的高风险，特别是在部署失败并需要手动回滚的情况下。 此外，这种策略不允许逐步测试，因为所有流量都会立即命中新版本。  
  
重新创建策略通常在哪些场景中使用？  
答：重新创建策略通常用于可接受停机时间的方案，例如在开发或暂存环境中，用户期望偶尔会停机，具有定期维护的内部工具，以及用户群有限和灵活可用性要求的小型应用程序。  
  
滚动更新策略是什么？  
答：滚动更新策略是为运行多个实例的生产应用程序设计的，以实现可靠性和负载平衡。 它逐渐逐一或批量替换所有实例，确保至少有一些实例始终运行旧版本，以维护应用程序可用性。  
  
蓝绿部署如何工作？  
答：蓝绿部署涉及两个环境-蓝色代表当前实时应用程序，绿色代表新应用程序版本。 最初，流量继续变成蓝色，而绿色随着新版本的推出而变暖。 一旦绿色被验证为健康状态，流量就会立即切换到它，允许立即转换到新版本，并在需要时提供快速回滚到蓝色的选项。  
  
部署金丝雀的目的是什么？  
答：金丝雀部署用于逐步向一定比例的用户 (如煤矿中的金丝雀) 介绍新的应用版本，管理费率、响应时间和其他指标，并在完全部署到每个人之前检测任何问题。 如果新版本表现良好，流量会逐渐增加; 如果有问题，可以启动快速回滚到旧稳定版本的操作。  
  
SPA (单页应用程序) 如何部署？  
答：通过使用静态文件 (如JavaScript和CSS) 构建应用程序的不可变版本，然后从静态文件存储桶中提供这些文件来部署水疗中心。 此过程使用GitHub操作等工具将应用程序构建上传到Amazon S3，其中文件以唯一的哈希名称存储，以供浏览器缓存。 用户的请求通过AWS CloudRoute定向到Amazon CloudFront，如果可用，它会返回缓存文件，或者在需要时获取新文件，从而提供全球分发和快速加载时间。

部署策略 (Deployment Strategies)

**重建（“大爆炸”式） (Recreate ("Big Bang"))**
停止旧版本 -> 部署新版本 -> 启动它。

流程图解：
1.  **停止旧应用 (Stop the old app)**：
    *   首先停止正在运行的 Node.js 应用（图标变灰）。*此时 Nginx 无法将请求转发给应用，服务中断*。
2.  **运行新应用 (Run the new app)**：
    *   部署并启动新的应用代码（图标显示加载/旋转状态）。
3.  **新版本 (New Version)**：
    *   *新应用启动成功（图标变绿并打勾），Nginx 开始将流量转发给新版本*。

*(注：这种策略在部署过程中会有明显的停机时间，因为旧版本停止后，新版本还没完全启动。)*

## 优点、缺点、使用场景
![[_posts/architect/_教程/Advanced Frontend Course/8.面向前端开发者的 CICD 基础知识/media/692fa5641846a98d557dedcba40c228b_MD5.png]]  

部署策略。重建式 (Recreate)

**工作原理：** *停止旧版本的所有实例，然后启动新版本的所有实例*。

**优点 (Pros)**
*   **实现最简单** -  shutdown（关闭）和 startup（启动）的过程非常直接。
*   **资源高效** - 不需要重复的基础设施（因为旧版本停了之后才启新版本，不需要同时运行两套）。

**缺点 (Cons)**
*   **需要停机** - *部署期间服务不可用*。
*   **高风险** - 如果部署失败，整个服务都会宕机。
*   **没有渐进式测试** - 所有流量会立即打到新版本上（没有缓冲期）。

**使用场景 (Use Cases)**
*   **开发/预发布 (staging) 环境**，其中停机是可以接受的。
*   **内部工具**，且有预定的维护时间窗口。
*   **小型应用程序**，用户群有限且对可用性（在线率）的要求比较灵活。

# SPA 部署策略
![[_posts/architect/_教程/Advanced Frontend Course/8.面向前端开发者的 CICD 基础知识/media/3540daee1c9a7dd42481e436f5c185d5_MD5.png]]  


`SPA 部署 (SPA Deployment)`，部署 SPA（单页应用）意味着构建**不可变的、带版本号的静态文件**（HTML, JS, CSS），**并通过文件存储桶（File Bucket）+ CDN 来提供服务**。

**1. 部署 (Deployment)**
*   **流程**：*GitHub Actions -> 上传应用构建产物 (Upload application build) -> Amazon S3*。
*   **Amazon S3 内容**：
    *   `app` 文件夹
    *   `index.[hash].html`
    *   `index.[hash].js`
    *   `index.[hash].css`
    *   *(注：文件名中包含哈希值，用于版本控制)*
*   **注释**：如果文件名发生变化，浏览器/CDN 总是会获取新文件 (Browsers/CDNs always fetch new files if the name changes)。

**2. 访问 (Access)**
*   **流程**：用户访问 `my-app.com` -> 经过 `AWS Route53` (DNS 解析) -> 到达 `AWS CloudFront` (***CDN***)。
*   **判断逻辑**：文件在缓存中吗？ (Files in cache?)
    *   **Yes (是)** -> 返回文件 (Return files)。
    *   **No (否)** -> 请求 `Amazon S3` (源站获取文件)。


# 使用AWS和GitHub操作优化Web应用程序交付  
06:24
GitHub Actions将使用缓存破坏文件名的应用程序构建上传到亚马逊S3，通过AWS CloudFront CDN增强全球分发和快速加载时间。 当用户访问站点时，CloudFront会从最近的边缘站点提供缓存文件，确保在全球范围内高效地交付内容。 每次更新都会自动更改文件名，提示浏览器提取新版本，为用户维护最新内容。
