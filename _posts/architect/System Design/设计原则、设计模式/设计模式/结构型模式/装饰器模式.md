`装饰器模式（Decorator Pattern） `是软件工程中**结构型设计模式（Structural Pattern）** 中最灵活、最常用的一种。

它的`核心思想`是：***扩展一个对象的功能*，动态地给一个对象添加一些额外的功能/职责，而不改变其原有结构。** *就功能而言，装饰器模式比生成子类更为灵活*。
*通俗地说，就是 **“包装”***。就像你穿衣服：先穿内衣，再穿衬衫，再穿外套。*每加一件衣服（装饰器），你就多了一层功能*（保暖/美观），但你还是你（核心对象），而且你可以随时脱掉外套（移除装饰）。

装饰器模式是*实现 **开闭原则（OCP）** 和 **组合优于继承** 的典范*。

---

### 1. 为什么要用装饰器模式？（解决的问题）

*在没有装饰器模式时，如果要**扩展**一个对象的功能，我们通常使用**继承***。

#### ❌ 违反装饰器模式的写法（继承爆炸）
假设我们有一个咖啡系统，有各种配料（牛奶、糖、奶油）。

```typescript
// 基础咖啡
class Coffee {
  cost(): number { return 5; }
}

// 加牛奶的咖啡
class MilkCoffee extends Coffee {
  cost(): number { return super.cost() + 2; }
}

// 加糖的咖啡
class SugarCoffee extends Coffee {
  cost(): number { return super.cost() + 1; }
}

// 加牛奶又加糖的咖啡
class MilkSugarCoffee extends Coffee { // 或者 extends MilkCoffee?
  cost(): number { return 5 + 2 + 1; }
}

// 问题：如果有 10 种配料，组合起来会有 2^10 个子类！类爆炸！
```
**问题：**
1.  **类爆炸（Class Explosion）**：每增加一种组合，就要新增一个类。
2.  **违反开闭原则（OCP）**：修改功能需要修改类层次结构。
3.  **静态耦合**：继承是静态的，编译时就确定了，运行时无法动态移除“牛奶”。

#### ✅ 使用装饰器模式的写法（动态组合）
```typescript
// 基础咖啡
let coffee = new SimpleCoffee();
// 动态加牛奶
coffee = new MilkDecorator(coffee);
// 动态加糖
coffee = new SugarDecorator(coffee);

console.log(coffee.cost()); // 5 + 2 + 1 = 8
```
***好处：** 不需要为每种组合创建子类，运行时自由组装。*

---

### 2. 核心角色

1.  **Component（抽象组件）**：定义一个接口，可以给这些对象动态地添加职责。
2.  **ConcreteComponent（具体组件）**：定义一个对象，可以给这个对象添加一些职责。
3.  **Decorator（抽象装饰器）**：维持一个指向 Component 对象的引用，并定义一个与 Component 接口一致的接口。
4.  **ConcreteDecorator（具体装饰器）**：向组件添加具体的职责。

---

### 3. TypeScript 实现

#### (1) 经典实现（对象装饰）

```typescript
// 1. 抽象组件接口
interface Coffee {
  cost(): number;
  getDescription(): string;
}

// 2. 具体组件：基础咖啡
class SimpleCoffee implements Coffee {
  cost(): number {
    return 5;
  }
  getDescription(): string {
    return "Simple Coffee";
  }
}

// 3. 抽象装饰器：持有组件引用
abstract class CoffeeDecorator implements Coffee {
  protected coffee: Coffee;

  constructor(coffee: Coffee) {
    this.coffee = coffee;
  }

  cost(): number {
    return this.coffee.cost();
  }

  getDescription(): string {
    return this.coffee.getDescription();
  }
}

// 4. 具体装饰器：牛奶
class MilkDecorator extends CoffeeDecorator {
  cost(): number {
    return super.cost() + 2;
  }
  getDescription(): string {
    return super.getDescription() + ", Milk";
  }
}

// 5. 具体装饰器：糖
class SugarDecorator extends CoffeeDecorator {
  cost(): number {
    return super.cost() + 1;
  }
  getDescription(): string {
    return super.getDescription() + ", Sugar";
  }
}

// 6. 客户端使用
const myCoffee: Coffee = new SimpleCoffee();
console.log(`${myCoffee.getDescription()}: $${myCoffee.cost()}`);

const milkCoffee: Coffee = new MilkDecorator(myCoffee);
console.log(`${milkCoffee.getDescription()}: $${milkCoffee.cost()}`);

const sweetMilkCoffee: Coffee = new SugarDecorator(milkCoffee);
console.log(`${sweetMilkCoffee.getDescription()}: $${sweetMilkCoffee.cost()}`);

/* 输出:
Simple Coffee: $5
Simple Coffee, Milk: $7
Simple Coffee, Milk, Sugar: $8
*/
```

#### (2) **函数装饰器**（TS 语言特性 vs 设计模式）

**注意区分**：TypeScript 语言*本身支持 `@Decorator` 语法*（用于类、方法、属性），*这与`装饰器设计模式`是两个概念，但思想相通*。

*   **设计模式**：运行时对象包装（Composition）。
*   **TS 语法**：编译时元数据修改（Metadata/Reflection）。

但*在函数式编程中，我们可以用高阶函数实现装饰器模式：*

```typescript
// 定义函数类型
type Handler = (input: string) => string;

// 装饰器函数：添加日志
function withLogging(handler: Handler): Handler {
  return (input: string) => {
    console.log(`[LOG] Processing: ${input}`);
    return handler(input);
  };
}

// 装饰器函数：添加加密
function withEncryption(handler: Handler): Handler {
  return (input: string) => {
    const result = handler(input);
    return `[ENCRYPTED] ${result}`;
  };
}

// 核心业务逻辑
const coreHandler: Handler = (input) => `Processed ${input}`;

// 动态组合装饰器
const decoratedHandler = withEncryption(withLogging(coreHandler));

console.log(decoratedHandler("Data"));
/* 输出:
[LOG] Processing: Data
[ENCRYPTED] Processed Data
*/
```

***这种写法在前端非常常见**（如 Redux 中间件、Express 中间件、React HOC）*。

### 4. 装饰器模式与 SOLID 原则

装饰器模式是 SOLID 原则的**教科书级实现**。

| 原则           | 装饰器模式如何体现                                       |
| :----------- | :---------------------------------------------- |
| **O (开闭原则)** | **核心受益者**。新增功能（如加奶油）只需新增一个装饰器类，无需修改原有咖啡类或客户端代码。 |
| **S (单一职责)** | 每个装饰器类只负责一种职责（如只负责加牛奶），核心类只负责核心功能。              |
| **D (依赖倒置)** | 装饰器和具体组件都依赖抽象接口 `Coffee`，而不是具体实现。               |
| **组合优于继承**   | 通过组合（持有引用）来扩展功能，避免了继承的僵化。                       |

---

### 5. 优缺点分析

#### ✅ 优点
1.  **灵活性高**：比继承更灵活，可以动态添加/移除职责。
2.  **避免类爆炸**：不需要为每种功能组合创建子类。
3.  **符合开闭原则**：新增功能无需修改现有代码。
4.  **透明性**：客户端面对装饰后的对象和原对象，接口是一致的。

#### ❌ 缺点
1.  **多层嵌套复杂**：如果装饰器太多，调试和阅读代码会比较困难（“洋葱皮”结构）。
2.  **对象标识问题**：装饰后的对象虽然接口一致，但 `instanceof` 检查可能会失败（取决于实现）。
3.  **初始化繁琐**：需要层层 `new` 来包裹对象，代码量可能较多（可以通过工厂模式简化）。
4.  **移除困难**：虽然可以动态添加，但要移除中间的某个装饰器比较麻烦（通常设计为不可移除，或需要维护链表）。

### 6. 现代前端开发中的**应用**

装饰器模式在前端生态中无处不在，只是形态各异。

#### (1) **React 高阶组件（HOC）**
HOC 本质就是装饰器模式。
```typescript
// 一个日志装饰器
function withLogging(WrappedComponent) {
  return class extends React.Component {
    componentDidMount() {
      console.log('Component mounted');
    }
    render() {
      return <WrappedComponent {...this.props} />;
    }
  };
}

// 使用
const LoggedUser = withLogging(UserProfile);
```

#### (2) Express / Koa 中间件
中间件链就是典型的装饰器链。
```javascript
// app.use 本质是将函数包装起来
app.use(loggingMiddleware); // 装饰一层
app.use(authMiddleware);    // 再装饰一层
app.use(handler);           // 核心逻辑
```

#### (3) TypeScript 编译器装饰器
TS 的 `@Component`, `@Injectable` 等语法糖，用于在类定义时修改类的行为或元数据（如 Angular 框架）。
```typescript
@Component({
  selector: 'app-root',
  template: '<h1>Hello</h1>'
})
class AppComponent {} // 被装饰器修改了元数据
```

#### (4) IO 流（Java/Node.js）
Node.js 的 `Stream` 管道（`.pipe()`）也是装饰器思想。
```javascript
// 文件流 -> 压缩流 -> 加密流 -> 输出流
readStream.pipe(zlib.createGzip()).pipe(cryptoStream).pipe(writeStream);
```

### 7. 装饰器模式 vs 其他模式

| 模式 | 区别 |
| :--- | :--- |
| **继承** | 继承是静态的，装饰器是动态的。继承改变整个类，装饰器只改变单个对象。 |
| **代理模式 (Proxy)** | **结构相似**（都持有引用）。**目的不同**：代理控制访问（如权限、延迟加载），装饰器增强功能（如加日志、加功能）。 |
| **策略模式 (Strategy)** | 策略是**替换**算法（换一个行为），装饰器是**增强**算法（在原行为上加东西）。 |
| **组合模式 (Composite)** | 组合模式构建树形结构，装饰器模式构成链式结构。 |

---

### 8. 最佳实践与注意事项

1.  **接口一致性**：装饰器必须实现与被装饰对象相同的接口，保证透明性。
2.  **抽象装饰器**：建议提供一个抽象装饰器基类，减少重复代码（如上面的 `CoffeeDecorator`）。
3.  **避免过度嵌套**：如果装饰器超过 3-4 层，考虑重构或使用配置对象。
4.  **配合工厂模式**：使用工厂来创建复杂的装饰链，简化客户端调用。
    ```typescript
    class CoffeeFactory {
      static createSpecialCoffee(): Coffee {
        let c = new SimpleCoffee();
        c = new MilkDecorator(c);
        c = new SugarDecorator(c);
        c = new CreamDecorator(c);
        return c;
      }
    }
    ```
5.  **TS 语法 vs 模式**：在团队沟通中，明确是指“设计模式”还是"TS 的 `@` 语法”，避免混淆。

---

### 9. 总结

**装饰器模式的核心价值在于“动态增强”与“组合复用”。**

*   **何时使用**：
    *   需要在不影响其他对象的情况下，给单个对象添加职责。
    *   功能组合太多，继承会导致类爆炸。
    *   功能可以动态添加或撤销。
*   **核心思想**：**Composition over Inheritance（组合优于继承）**。
*   **前端地位**：它是 React HOC、Vue 混入（Mixins）、Express 中间件、TS 装饰器语法的理论基础。

结合你之前的学习路径：
*   用 **单例模式** 管理全局装饰器配置。
*   用 **工厂模式** 简化装饰链的创建。
*   用 **SOLID 原则** 确保装饰器的接口规范。
*   用 **TypeScript** 的类型系统保证装饰链的类型安全。

掌握装饰器模式，意味着你掌握了**“在不修改源代码的前提下扩展功能”**的终极武器。
