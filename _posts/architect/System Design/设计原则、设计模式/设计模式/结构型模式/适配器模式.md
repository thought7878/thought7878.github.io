`适配器模式（Adapter Pattern） `是软件工程中**结构型设计模式（Structural Pattern）** 的一种。

它的`核心思想`是：**将一个类的接口转换成客户希望的另外一个接口。适配器模式使得原本由于接口不兼容而不能一起工作的那些类可以一起工作。**
通俗地说，就是 **“转换器”** 或 **“翻译官”**。就像你出国旅行，需要*把国内的插头（适配者）通过一个转换头（适配器）插到国外的插座（目标接口）上*。你*不需要修改插头，也不需要修改插座，只需要一个中间件*。

`适配器模式`是*解决**遗留代码集成**、**第三方库对接** 和 **接口标准化** 的关键工具*。

---

### 1. 为什么要用适配器模式？（解决的问题）

*在没有适配器模式时，如果两个系统的接口不匹配，我们通常面临两难：*
1.  修改原有代码以适配新接口（可能无法修改，如第三方库）。
2.  在新代码中写大量转换逻辑（导致耦合度高，违反 OCP）。

#### ❌ 违反适配器模式的写法（硬编码转换）
假设我们有一个旧的支付系统（Legacy Payment），现在要接入新的订单系统（New Order System）。

```typescript
// 旧系统接口（无法修改）
class LegacyPayment {
  oldPay(amount: number, currency: string) {
    console.log(`Paying ${amount} ${currency} via Legacy`);
  }
}

// 新系统期望的接口
interface NewPayment {
  process(amount: number): void;
}

// 新订单系统
class OrderSystem {
  constructor(private payment: NewPayment) {}
  
  checkout(amount: number) {
    this.payment.process(amount); // 期望调用 process
  }
}

// ❌ 错误用法：接口不匹配
// const legacy = new LegacyPayment();
// const order = new OrderSystem(legacy); // 类型错误！
```
**问题：** *`LegacyPayment` 没有 `process` 方法，且参数也不对。如果直接修改 `LegacyPayment`，可能破坏原有功能或根本不允许（第三方库）。*

#### ✅ 使用适配器模式的写法
```typescript
// 创建适配器
class PaymentAdapter implements NewPayment {
  private legacyPayment: LegacyPayment;

  constructor(legacyPayment: LegacyPayment) {
    this.legacyPayment = legacyPayment;
  }

  // 实现新接口，内部调用旧方法
  process(amount: number): void {
    // 转换逻辑：默认货币为 USD
    this.legacyPayment.oldPay(amount, 'USD');
  }
}

// 使用
const legacy = new LegacyPayment();
const adapter = new PaymentAdapter(legacy);
const order = new OrderSystem(adapter); // 成功！
order.checkout(100);
```
**好处：** *新旧系统解耦，无需修改原有代码*，符合 **开闭原则（OCP）**。

### 2. 核心角色

1.  **Target（目标接口）**：客户所期待的接口（如 `NewPayment`）。
2.  **Adaptee（适配者）**：需要被适配的现有接口（如 `LegacyPayment`）。
3.  **Adapter（适配器）**：实现目标接口，并持有适配者的引用，负责转换调用。

---

### 3. TypeScript 实现

适配器模式主要有两种实现方式：**类适配器** 和 **对象适配器**。在 TypeScript/JavaScript 中，**对象适配器** 是首选。

#### (1) 对象适配器（Object Adapter）—— 推荐
通过**组合（Composition）** 实现。适配器持有适配者的实例。

```typescript
// 1. 目标接口
interface Target {
  request(): string;
}

// 2. 适配者（现有类）
class Adaptee {
  specificRequest(): string {
    return 'Adaptee method';
  }
}

// 3. 适配器
class Adapter implements Target {
  private adaptee: Adaptee;

  constructor(adaptee: Adaptee) {
    this.adaptee = adaptee;
  }

  request(): string {
    // 转换逻辑
    return `Adapter: ${this.adaptee.specificRequest()}`;
  }
}

// 4. 客户端
const client = (target: Target) => {
  console.log(target.request());
};

client(new Adapter(new Adaptee()));
```
**优点**：灵活，可以同时适配多个适配者（多继承效果），符合 **组合优于继承** 原则。

#### (2) 类适配器（Class Adapter）
通过**继承（Inheritance）** 实现。适配器继承适配者，并实现目标接口。

```typescript
// 注意：TS 不支持多继承，如果 Target 也是类，这招就行不通了
class Adapter extends Adaptee implements Target {
  request(): string {
    return `Adapter: ${this.specificRequest()}`;
  }
}
```
**缺点**：TS/JS 单继承限制，耦合度高（适配器与适配者强绑定）。**不推荐在 TS 中使用。**

#### (3) 函数式适配器（Functional Adapter）
在 TS 中，很多时候不需要类，一个函数就是适配器。

```typescript
// 旧 API 返回
type OldResponse = { user_name: string; user_age: number };

// 新组件期望
type NewUser = { name: string; age: number };

// 适配器函数
const userAdapter = (old: OldResponse): NewUser => ({
  name: old.user_name,
  age: old.user_age
});

// 使用
const oldData = { user_name: 'Alice', user_age: 25 };
const newData = userAdapter(oldData);
```
**场景**：数据格式转换、API 响应标准化。

### 4. 适配器模式与 SOLID 原则

| 原则 | 适配器模式如何体现 |
| :--- | :--- |
| **O (开闭原则)** | **核心受益者**。无需修改现有类（Adaptee），通过新增适配器类来兼容新接口。 |
| **S (单一职责)** | 适配器只负责接口转换，不包含业务逻辑。 |
| **D (依赖倒置)** | 客户端依赖抽象 `Target` 接口，不依赖具体 `Adaptee`。 |
| **L (里氏替换)** | 适配器实现了 Target 接口，可以替换任何 Target 实现。 |

---

### 5. 适配器模式 vs 其他模式

这是最容易混淆的部分，特别是与 **装饰器** 和 **代理**。

| 模式 | 核心意图 | 接口变化 | 关系 |
| :--- | :--- | :--- | :--- |
| **适配器 (Adapter)** | **兼容**。让不兼容的接口能一起工作。 | **改变接口**（A -> B）。 | **翻译官**。 |
| **装饰器 (Decorator)** | **增强**。给对象添加新功能。 | **保持接口**（A -> A+）。 | **化妆师**。 |
| **代理 (Proxy)** | **控制**。控制对对象的访问。 | **保持接口**（A -> A）。 | **守门员**。 |
| **外观 (Facade)** | **简化**。为复杂系统提供统一入口。 | **简化接口**（Complex -> Simple）。 | **前台接待**。 |

**关键区别口诀：**
*   **适配器**：接口不一样，我要**改**接口。
*   **装饰器**：接口一样，我要**加**功能。
*   **代理**：接口一样，我要**控**访问。
*   **外观**：接口太乱，我要**简**化它。

### 6. 前端开发中的实际应用

适配器模式在前端开发中**无处不在**，尤其是数据层和库集成。

#### (1) API 响应数据标准化（最常见）
后端返回的数据格式（蛇形命名）与前端组件期望（驼峰命名）不一致。

```typescript
// 后端返回
interface ApiResponse {
  user_id: number;
  user_name: string;
  is_vip: boolean;
}

// 前端组件期望
interface UserViewModel {
  id: number;
  name: string;
  isVip: boolean;
}

// 适配器
function adaptUser(api: ApiResponse): UserViewModel {
  return {
    id: api.user_id,
    name: api.user_name,
    isVip: api.is_vip
  };
}

// 使用
const userData = adaptUser(apiResponse);
```
**进阶**：可以使用库如 `axios` 的拦截器作为全局适配器。

#### (2) 第三方库集成
将 jQuery 插件适配为 React/Vue 组件。

```typescript
// 旧 jQuery 插件
declare const $: any;
function initDatePicker(element: any) {
  $(element).datepicker();
}

// React 适配器组件
class DatePickerAdapter extends React.Component {
  componentDidMount() {
    // 适配 jQuery 逻辑到 React 生命周期
    initDatePicker(this.ref);
  }
  render() {
    return <div ref={el => this.ref = el} />;
  }
}
```

#### (3) 遗留代码迁移
逐步重构老项目时，新模块调用老模块，通过适配器屏蔽老代码的怪异接口。

#### (4) 地图服务切换
如果系统需要从 高德地图 切换到 百度地图，它们的 API 不同。
*   定义统一 `MapProvider` 接口。
*   编写 `AmapAdapter` 和 `BmapAdapter`。
*   业务代码只依赖 `MapProvider`，切换地图只需换适配器。

#### (5) Vue/React Props 适配
子组件期望的 props 与父组件提供的数据名称不一致。
```tsx
// 子组件期望
function Child({ userName }: { userName: string }) {}

// 父组件适配
function Parent() {
  const data = { name: 'Alice' };
  return <Child userName={data.name} />; // 适配器逻辑
}
```

### 7. 优缺点分析

#### ✅ 优点
1.  **复用旧代码**：无需修改原有类，即可利用其功能。
2.  **解耦**：客户端与适配者解耦，通过适配器交互。
3.  **灵活透明**：客户端面对统一的目标接口，不知道背后是适配器。
4.  **符合 OCP**：新增适配只需新增适配器类。

#### ❌ 缺点
1.  **增加复杂度**：引入了额外的类或函数，系统结构变复杂。
2.  **过度使用风险**：如果系统设计初期就规划好接口，后期就不需要大量适配器。适配器多是“补救”措施。
3.  **性能开销**：多了一层调用（通常可忽略）。

---

### 8. 最佳实践与注意事项

1.  **优先使用对象适配器**：在 TS/JS 中，利用组合（持有实例）比继承更灵活。
2.  **函数即适配器**：对于简单的数据转换，不需要写类，一个纯函数就是很好的适配器。
3.  **不要滥用**：如果两个接口差异巨大，强行适配可能导致逻辑扭曲。此时应考虑重构原有接口。
4.  **双向适配**：有时需要双向通信（如两个系统同步数据），可能需要两个适配器或一个双向适配器。
5.  **类型安全**：在 TS 中，确保适配器的输入输出类型定义清晰，利用泛型提高复用性。
    ```typescript
    function createAdapter<T, U>(fn: (input: T) => U) {
      return fn;
    }
    ```
6.  **与外观模式区分**：如果目的是简化复杂接口，用外观模式；如果目的是兼容不同接口，用适配器模式。

---

### 9. 总结

**`适配器模式`的*核心价值*在于“兼容”与“复用”。**

*   **何时使用**：
    *   *需要使用现有的类，但其接口不符合要求*。
    *   想创建一个可以复用的类，处理一些不相关的类或接口。
    *   *集成第三方库或遗留系统*。
*   **核心思想**：***接口转换***。
*   **前端地位**：它是 **API 数据层**、**库集成**、**系统迁移** 的标准解决方案。
*   **与 SOLID 的关系**：是 **OCP（开闭原则）** 的典型应用（不修改旧代码）。
*   **与装饰器/代理的区别**：
    *   **适配器** = ***改接口***（为了兼容）。
    *   **装饰器** = ***加功能***（为了增强）。
    *   **代理** = ***控访问***（为了安全/延迟）。
*   **TS 最佳实践**：
    *   简单数据转换用 **函数**。
    *   复杂对象交互用 **对象适配器（类）**。
    *   避免使用 **类适配器（继承）**。

结合你之前的学习路径：
*   用 **工厂模式** 创建适配器。
*   用 **策略模式** 选择不同的适配器（如不同地图服务商）。
*   用 **装饰器模式** 增强适配器功能（如加日志）。
*   用 **AOP** 监控适配器调用。
*   用 **TypeScript 类型系统** 保证适配过程的类型安全。

掌握适配器模式，意味着你具备了**“整合异构系统”**的能力，能够在不破坏现有架构的前提下，灵活地引入新技术或兼容旧代码。它是架构演进中的“润滑剂”。
