`代理模式（Proxy Pattern）` 是软件工程中**结构型设计模式（Structural Pattern）** 的一种。

它的`核心思想`是：**为另一个对象提供一个替身或占位符，以控制对这个对象的访问。**

*通俗地说，就是 **“中间人”***。就像你找明星代言，你不能直接联系明星，*必须通过经纪人（代理）*。经纪人可以`控制访问`（筛选请求）、`增加功能`（谈价格、签合同），而明星（真实对象）*只需要专注表演*。

*代理模式与装饰器模式结构非常相似*，但**意图不同**。
*`代理模式`是实现 **AOP** 和 **控制访问** 的核心手段之一*。

---

### 1. 为什么要用代理模式？（解决的问题）

在没有代理模式时，客户端直接访问真实对象。但在以下场景中，直接访问不合适：

1.  **权限控制**：某些用户不能访问特定对象。
2.  **延迟加载（Lazy Loading）**：对象创建成本很高（如大图、大文件），只有在真正需要时才创建。
3.  **远程调用**：对象在远程服务器上，本地需要代理来模拟网络通信。
4.  **额外逻辑**：需要在调用前后添加日志、缓存、计数等逻辑，但不想修改原对象。
5.  **保护对象**：防止对象被意外修改或销毁。

**核心意图**：**控制访问**（Control Access）。

---

### 2. 核心角色

1.  **Subject（抽象主题）**：定义真实对象和代理对象的*共同接口*。
2.  **RealSubject（真实主题）**：真正的业务逻辑对象。
3.  **Proxy（代理主题）**：*持有真实对象的引用，控制对真实对象的访问，并可能添加额外操作*。

---

### 3. 代理模式的常见类型

| 类型 | 英文 | 场景 | 例子 |
| :--- | :--- | :--- | :--- |
| **虚拟代理** | **Virtual Proxy** | 延迟加载，节省资源。 | 图片懒加载（先显示占位图，滚动到视野再加载原图）。 |
| **保护代理** | **Protection Proxy** | 控制权限。 | 只有管理员才能调用删除方法。 |
| **远程代理** | **Remote Proxy** | 代表远程对象。 | RPC 调用、HTTP 客户端代理。 |
| **智能引用** | **Smart Reference** | 添加额外操作。 | 记录调用次数、自动垃圾回收、缓存。 |

### 4. TypeScript 实现

在 TypeScript/JavaScript 中，代理模式可以*通过**类**实现*，也可以直接*使用语言特性的 **`Proxy` 对象** 实现*。

#### (1) 经典实现（基于类）

```typescript
// 1. 抽象主题接口
interface Image {
  display(): void;
}

// 2. 真实主题：大图加载
class RealImage implements Image {
  private filename: string;

  constructor(filename: string) {
    this.filename = filename;
    this.loadFromDisk(); // 耗时操作
  }

  private loadFromDisk() {
    console.log(`Loading ${this.filename}...`);
    // 模拟耗时
    for (let i = 0; i < 100000000; i++) {} 
    console.log(`Loaded ${this.filename}`);
  }

  display() {
    console.log(`Displaying ${this.filename}`);
  }
}

// 3. 代理主题：虚拟代理（延迟加载）
class ProxyImage implements Image {
  private realImage: RealImage | null = null;
  private filename: string;

  constructor(filename: string) {
    this.filename = filename;
    // 注意：这里没有调用 loadFromDisk
  }

  display() {
    // 只有真正需要显示时，才创建真实对象
    if (this.realImage === null) {
      this.realImage = new RealImage(this.filename);
    }
    this.realImage.display();
  }
}

// 4. 客户端使用
const image: Image = new ProxyImage('huge_photo.jpg');
console.log('Proxy created (no load yet)');
// 输出：Proxy created (no load yet)

image.display(); 
// 输出：
// Loading huge_photo.jpg...
// Loaded huge_photo.jpg
// Displaying huge_photo.jpg
```

#### (2) 现代实现（基于 ES6 `Proxy`）

ES6 的 `Proxy` 是语言层面的代理机制，非常适合实现**智能引用代理**（如日志、权限）。

```typescript
// 真实对象
class UserService {
  getUser(id: string) {
    console.log(`Fetching user ${id}`);
    return { id, name: 'Alice' };
  }
}

// 代理处理器
const handler = {
  // 拦截方法调用
  get(target: any, prop: string, receiver: any) {
    const originalMethod = target[prop];

    if (typeof originalMethod === 'function') {
      return function (...args: any[]) {
        console.log(`[Proxy] Before calling ${prop}`);
        
        // 权限检查示例
        if (prop === 'deleteUser' && !this.isAdmin) {
          throw new Error('Permission denied');
        }

        const result = originalMethod.apply(target, args);
        
        console.log(`[Proxy] After calling ${prop}`);
        return result;
      };
    }

    return Reflect.get(target, prop, receiver);
  }
};

// 创建代理对象
const userService = new UserService();
const proxyService = new Proxy(userService, handler);

// 使用
proxyService.getUser('123');
// 输出:
// [Proxy] Before calling getUser
// Fetching user 123
// [Proxy] After calling getUser
```

### 5. 代理模式 vs 装饰器模式

这是*最容易混淆*的两个模式，因为它们的**结构几乎一样**（都持有引用，都实现相同接口）。

| 特性 | 代理模式 (Proxy) | 装饰器模式 (Decorator) |
| :--- | :--- | :--- |
| **核心意图** | **控制访问**（Control Access）。 | **增强功能**（Add Responsibility）。 |
| **创建时机** | 通常在对象创建时就确定（如延迟加载）。 | 通常在运行时动态组合（如加牛奶、加糖）。 |
| **依赖关系** | 代理通常完全控制真实对象的生命周期。 | 装饰器通常不控制对象生命周期，只是包装。 |
| **关注点** | 关注“能不能调用”、“何时调用”。 | 关注“调用后做什么额外事”。 |
| **例子** | 图片懒加载、权限网关。 | 咖啡加配料、React HOC。 |
| **关系** | **代理模式是“守门员”**。 | **装饰器模式是“化妆师”**。 |

**简单判断法**：
*   如果是为了**保护、延迟、远程**，选 **代理**。
*   如果是为了**添加新功能、组合行为**，选 **装饰器**。

---

### 6. 代理模式与 SOLID 原则

| 原则 | 代理模式如何体现 |
| :--- | :--- |
| **S (单一职责)** | 将“访问控制”逻辑从业务逻辑中分离出来。代理管访问，真实对象管业务。 |
| **O (开闭原则)** | 新增访问控制逻辑（如新增缓存代理），无需修改真实对象代码。 |
| **D (依赖倒置)** | 客户端依赖抽象接口 `Subject`，不知道背后是真实对象还是代理。 |
| **L (里氏替换)** | 代理对象可以无缝替换真实对象，客户端无感知。 |

### 7. 代理模式与 AOP 的关系

**代理模式是实现 AOP 的核心技术手段之一。**

*   **AOP** 是一种编程思想（面向切面）。
*   **代理模式** 是一种设计模式（结构实现）。
*   **关系**：AOP 的“织入（Weaving）”过程，通常就是通过**动态代理**（Dynamic Proxy）来实现的。
    *   在 Java Spring 中，AOP 默认使用 JDK 动态代理或 CGLIB 代理。
    *   在 TypeScript 中，AOP 可以通过 **ES6 Proxy** 或 **装饰器** 实现。

你之前学习的 **TypeScript 装饰器** 和 **AOP**，底层往往都依赖了代理模式的思想。

### 8. 前端开发中的**实际应用**

#### (1) Vue 3 的响应式系统
Vue 3 使用 ES6 `Proxy` 来实现数据劫持，这是代理模式的典型应用。
```javascript
// 简化版 Vue 响应式原理
function reactive(target) {
  return new Proxy(target, {
    get(target, key, receiver) {
      track(target, key); // 依赖收集
      return Reflect.get(target, key, receiver);
    },
    set(target, key, value, receiver) {
      const result = Reflect.set(target, key, value, receiver);
      trigger(target, key); // 触发更新
      return result;
    }
  });
}
```

#### (2) 图片懒加载（Virtual Proxy）
网页中大量图片，先加载缩略图或占位符，当图片进入视口（Intersection Observer）再加载原图。
```typescript
class ImageLoader {
  load(url: string) { /* 加载原图 */ }
}

class LazyImageLoader {
  private loader: ImageLoader | null = null;
  private url: string;

  constructor(url: string) {
    this.url = url;
    this.showPlaceholder(); // 先显示占位图
  }

  load() {
    if (!this.loader) {
      this.loader = new ImageLoader();
    }
    this.loader.load(this.url);
  }
}
```

#### (3) API 请求缓存（Smart Reference）
```typescript
function createCachedProxy(apiService: any) {
  const cache = new Map();
  return new Proxy(apiService, {
    get(target, prop) {
      const originalMethod = target[prop];
      if (typeof originalMethod === 'function') {
        return function (...args: any[]) {
          const key = `${prop}-${JSON.stringify(args)}`;
          if (cache.has(key)) {
            console.log('Cache hit');
            return cache.get(key);
          }
          const result = originalMethod.apply(target, args);
          cache.set(key, result);
          return result;
        };
      }
      return originalMethod;
    }
  });
}
```

#### (4) 跨域代理（Remote Proxy）
在开发环境中，前端请求本地服务器，本地服务器代理转发到远程 API，解决跨域问题。这属于远程代理的网络层面应用。

### 9. 优缺点分析

#### ✅ 优点
1.  **控制灵活**：可以在不修改目标对象的前提下，控制访问、添加逻辑。
2.  **性能优化**：通过延迟加载、缓存等机制提高性能。
3.  **安全性**：可以隐藏真实对象的细节，提供保护层。
4.  **符合开闭原则**：新增代理逻辑无需修改核心业务代码。

#### ❌ 缺点
1.  **增加复杂度**：引入了额外的类或代理层，系统结构变复杂。
2.  **性能开销**：每次访问都要经过代理层，会有轻微的性能损耗（尤其是动态代理）。
3.  **调试困难**：调用栈中多了代理层，可能增加调试难度。
4.  **过度设计**：对于简单对象，引入代理可能没必要。

---

### 10. 总结

**代理模式的核心价值在于“控制”与“隔离”。**

*   **何时使用**：
    *   需要延迟加载重型对象（虚拟代理）。
    *   需要权限控制或安全保护（保护代理）。
    *   需要记录日志、缓存、计数（智能引用）。
    *   对象在远程地址（远程代理）。
*   **与装饰器的区别**：
    *   **代理** = **守门员**（控制访问，能不能做）。
    *   **装饰器** = **化妆师**（增强功能，做得更好）。
*   **与 AOP 的关系**：
    *   代理模式是 AOP 的**实现基石**。
*   **TS 最佳实践**：
    *   对于简单的访问控制，优先使用 **ES6 `Proxy`**。
    *   对于复杂的业务代理，可以使用 **类继承/接口实现**。
    *   在框架（如 Vue、NestJS）中，理解其背后的代理机制有助于深入掌握框架原理。

结合你之前的学习路径：
*   用 **单例模式** 管理代理对象。
*   用 **工厂模式** 创建代理。
*   用 **装饰器模式** 增强代理功能。
*   用 **AOP 思想** 指导代理逻辑的设计。

代理模式是连接**业务逻辑**与**基础设施**（安全、性能、日志）的桥梁，是构建健壮系统的关键模式之一。
