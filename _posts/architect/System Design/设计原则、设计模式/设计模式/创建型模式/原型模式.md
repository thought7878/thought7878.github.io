`原型模式（Prototype Pattern）` 是软件工程中**创建型设计模式（Creational Pattern）** 的一种。

它的`核心思想`是：**通过复制（克隆）一个现有的实例来创建新对象，而不是通过 `new` 关键字实例化类。**

`原型模式`*提供了一种更灵活、有时更高效的对象创建方式，特别是在对象初始化成本很高或具体类在运行时才能确定的场景下*。

---

### 1. 为什么要用原型模式？（解决的问题）

在常规开发中，我们通常使用 `new ClassName()` 创建对象。但在以下场景中，这种方式可能不够好：

1.  **初始化成本过高**：对象创建需要耗时操作（如读取数据库、加载大文件、复杂计算）。如果每次 `new` 都执行一遍，性能太差。
2.  **构造函数复杂**：对象有很多配置项，构造函数参数过多，难以记忆和使用。
3.  **运行时动态确定类型**：客户端代码在编译时不知道具体需要创建哪个类的实例，只知道需要一个“类似”的对象。
4.  **保护封装性**：不希望客户端直接依赖具体的类构造函数，而是依赖一个原型接口。

**核心意图**：用**复制**代替**新建**，以空间换时间，或解耦创建逻辑。

---

### 2. 核心角色

1.  **Prototype（抽象原型）**：声明一个 `clone()` 接口，用于克隆自身。
2.  **ConcretePrototype（具体原型）**：实现 `clone()` 接口，执行具体的复制逻辑。
3.  **Client（客户端）**：调用原型的 `clone()` 方法创建新对象。
4.  **Prototype Registry（原型注册表，可选）**：管理原型的集合，客户端通过 Key 获取原型并克隆（类似工厂）。

---

### 3. TypeScript 实现

在 TypeScript/JavaScript 中，实现原型模式需要注意**浅拷贝（Shallow Copy）**与**深拷贝（Deep Copy）**的区别。

#### (1) 基础实现（接口定义）

```typescript
// 1. 抽象原型接口
interface Prototype<T> {
  clone(): T;
}

// 2. 具体原型：游戏角色
class GameCharacter implements Prototype<GameCharacter> {
  public name: string;
  public level: number;
  public inventory: string[]; // 引用类型
  public equipment: { weapon: string; armor: string }; // 嵌套对象

  constructor(name: string, level: number, inventory: string[], equipment: any) {
    this.name = name;
    this.level = level;
    this.inventory = inventory;
    this.equipment = equipment;
  }

  // 3. 实现克隆方法
  clone(): GameCharacter {
    // 注意：这里需要决定是浅拷贝还是深拷贝
    // 浅拷贝示例 (Object.assign 或 展开运算符)
    // const copy = Object.assign({}, this); 
    
    // 深拷贝示例 (现代 TS/JS 推荐 structuredClone)
    const copy = structuredClone(this);
    
    return copy;
  }
}

// 4. 客户端使用
const original = new GameCharacter('Hero', 1, ['Potion'], { weapon: 'Sword', armor: 'Leather' });

// 创建新角色，基于原角色克隆
const newCharacter = original.clone();
newCharacter.name = 'Hero Copy';
newCharacter.level = 2;

console.log(original.name); // 'Hero'
console.log(newCharacter.name); // 'Hero Copy'

// 验证深拷贝：修改新角色的装备，不应影响原角色
newCharacter.equipment.weapon = 'Axe';
console.log(original.equipment.weapon); // 'Sword' (如果是深拷贝)
```

#### (2) 原型注册表（Prototype Manager）

通常我们会配合一个注册表来管理原型，这样客户端连具体类名都不知道，只通过 ID 获取。

```typescript
class PrototypeRegistry {
  private prototypes: Map<string, Prototype<any>> = new Map();

  // 注册原型
  set(key: string, prototype: Prototype<any>): void {
    this.prototypes.set(key, prototype);
  }

  // 获取并克隆原型
  clone(key: string): any {
    const prototype = this.prototypes.get(key);
    if (!prototype) {
      throw new Error(`Prototype with key "${key}" not found`);
    }
    return prototype.clone();
  }
}

// 使用注册表
const registry = new PrototypeRegistry();

// 预定义一些“模板”对象
const enemyTemplate = new GameCharacter('Enemy', 1, [], { weapon: 'Club', armor: 'None' });
registry.set('basic_enemy', enemyTemplate);

// 客户端只需要知道 key，不需要知道 GameCharacter 类
const enemy1 = registry.clone('basic_enemy');
enemy1.name = 'Goblin';

const enemy2 = registry.clone('basic_enemy');
enemy2.name = 'Orc';
```

### 4. 浅拷贝 vs 深拷贝（关键陷阱）

在 JS/TS 中实现原型模式，**最大的坑在于引用类型的复制**。

| 特性 | 浅拷贝 (Shallow Copy) | 深拷贝 (Deep Copy) |
| :--- | :--- | :--- |
| **定义** | 只复制对象的第一层属性。如果属性是对象，只复制引用。 | 递归复制对象的所有层级，完全独立。 |
| **实现** | `Object.assign()`, `{...obj}`, `Array.slice()` | `structuredClone()`, `JSON.parse(JSON.stringify())`, 递归函数 |
| **风险** | 修改新对象的嵌套属性，会影响原对象。 | 性能开销大，无法复制函数、DOM 节点等。 |
| **适用** | 对象结构简单，无嵌套引用。 | 对象结构复杂，需要完全隔离状态。 |

**代码对比：**

```typescript
// ❌ 浅拷贝风险
const original = { info: { level: 1 } };
const shallow = { ...original };
shallow.info.level = 10;
console.log(original.info.level); // 10 (被污染了！)

// ✅ 深拷贝安全
const deep = structuredClone(original);
deep.info.level = 10;
console.log(original.info.level); // 1 (安全)
```

### 5. 原型模式与 SOLID 原则

| 原则 | 原型模式如何体现 |
| :--- | :--- |
| **O (开闭原则)** | **核心受益者**。新增一种对象类型，只需新增一个具体原型类并注册，无需修改客户端代码或工厂逻辑。 |
| **D (依赖倒置)** | 客户端依赖的是 `Prototype` 接口，而不是具体的 `GameCharacter` 类。 |
| **S (单一职责)** | 对象的创建逻辑（克隆）被封装在对象内部或注册表中，与业务逻辑分离。 |
| **L (里氏替换)** | 任何实现了 `Prototype` 接口的类都可以被注册和克隆。 |

---

### 6. 原型模式 vs 工厂模式

你之前学过工厂模式，这两个模式经常一起使用，但侧重点不同：

| 特性 | 工厂模式 (Factory) | 原型模式 (Prototype) |
| :--- | :--- | :--- |
| **创建方式** | 通过 `new` 实例化类。 | 通过 `clone()` 复制实例。 |
| **适用场景** | 对象创建逻辑复杂，但初始化成本低。 | 对象初始化成本高，或需要保留特定状态。 |
| **性能** | 每次创建都走构造函数。 | 内存复制，通常比 `new` 快（取决于对象大小）。 |
| **依赖** | 需要知道具体类（或配置）。 | 只需要知道原型 Key。 |
| **结合** | **工厂方法可以返回克隆的原型**。 | **原型注册表本质上是一个工厂**。 |

---

### 7. JavaScript/TypeScript 中的特殊性

在 Java 或 C++ 中，原型模式是一个显式的设计模式（需要实现 `Cloneable` 接口）。

**但在 JavaScript 中，原型模式是“语言基因”：**
1.  **基于原型的语言**：JS 对象本身就是通过原型链继承的。
2.  **`Object.create()`**：这是 JS 原生支持的原型模式变体，它创建一个新对象，使用现有的对象来提供新创建的对象的 `__proto__`。
3.  **类语法糖**：ES6 的 `class` 只是语法糖，底层依然是原型。

**现代 TS 开发中的“隐形”原型模式：**
你其实每天都在用，只是没意识到：
*   **React/Vue 组件**：组件实例化某种程度上是原型的复用。
*   **对象字面量**：`const b = { ...a }` 就是最简单的原型模式。
*   **Redux/Immer**：状态更新时的不可变数据复制，本质上是原型克隆思想的体现。

### 8. 优缺点分析

#### ✅ 优点
1.  **性能提升**：避免重复的初始化操作（如数据库连接、复杂计算），直接内存复制通常更快。
2.  **简化创建**：对于构造参数极多的对象，克隆一个配置好的模板比传几十个参数简单。
3.  **动态运行**：可以在运行时动态增加或删除原型，灵活性高。
4.  **解耦**：客户端不需要知道具体类的构造函数。

#### ❌ 缺点
1.  **深拷贝复杂**：实现完美的深拷贝很难（循环引用、特殊对象如 RegExp/Date/Function）。
2.  **内存占用**：需要存储原型对象，如果对象很大，会占用内存。
3.  **隐藏初始化逻辑**：克隆对象可能隐藏了某些初始化副作用，导致调试困难。
4.  **API 侵入性**：每个类都需要实现 `clone()` 方法（除非使用通用工具函数）。

---

### 9. 实际应用场景

1.  **游戏开发**：
    *   生成大量相似的敌人、子弹、粒子效果。
    *   每个敌人有相同的属性模板，只需克隆后微调位置/ID。
2.  **复杂配置对象**：
    *   创建一个默认的“配置模板”，用户修改部分配置后，克隆一份作为最终配置。
3.  **撤销/重做（Undo/Redo）**：
    *   保存历史状态时，克隆当前对象快照，而不是引用。
4.  **数据库连接池**：
    *   初始化一个连接对象，后续需要时克隆（如果驱动支持）或复用，避免频繁握手。
5.  **前端状态管理**：
    *   不可变数据更新（Immutable.js, Immer），每次状态变更都是生成一个新的状态对象（类似克隆修改）。

---

### 10. 总结

**原型模式的核心价值在于“复制”与“效率”。**

*   **何时使用**：
    *   对象创建成本 > 对象复制成本。
    *   需要保留对象的特定状态作为模板。
    *   希望避免复杂的构造函数参数。
*   **TS 最佳实践**：
    *   优先使用 `structuredClone()` 进行深拷贝（现代环境）。
    *   对于简单对象，使用展开运算符 `{...obj}`。
    *   配合 **原型注册表** 使用，实现类似工厂的解耦效果。
    *   注意 **循环引用** 和 **特殊类型**（函数、DOM）无法被 `structuredClone` 处理。

结合你之前的学习路径：
*   用 **单例模式** 管理原型注册表。
*   用 **工厂模式** 封装克隆逻辑。
*   用 **SOLID 原则** 确保原型接口的规范性。

原型模式是创建型模式中**最轻量级**的一种，在 JavaScript/TypeScript 生态中，它往往隐藏在语言特性背后，理解它有助于你更好地掌控对象的生命周期和性能。
