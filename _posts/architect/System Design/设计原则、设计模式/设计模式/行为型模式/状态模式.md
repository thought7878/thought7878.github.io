`状态模式（State Pattern） `是软件工程中**行为型设计模式（Behavioral Pattern）** 的一种。

它的`核心思想`是：**允许一个对象在其内部状态改变时改变它的行为。对象看起来似乎修改了它的类。**
通俗地说，就是 **“境随心转”**。*同一个对象，在不同的状态下，面对同样的请求，会做出不同的反应*。就像一个人，在“童年”状态下只能玩耍，在“成年”状态下可以工作，在“老年”状态下可以退休。*状态变了，行为自然就变了*。

`状态模式`是*消除 **状态机复杂逻辑（if-else 地狱）**、实现 **开闭原则（OCP）** 的重要模式*。*它与`策略模式`结构非常相似*，但**意图完全不同**。

---

### 1. 为什么要用状态模式？（解决的问题）

在没有状态模式时，处理对象的不同状态通常使用大量的条件判断语句。

#### ❌ 违反状态模式的写法（状态机地狱）
假设一个订单系统，订单有 `待支付 `、` 已支付 `、` 已发货` 等状态。

```typescript
enum OrderStatus {
  PENDING = 'PENDING',
  PAID = 'PAID',
  SHIPPED = 'SHIPPED'
}

class Order {
  private status: OrderStatus;

  constructor() {
    this.status = OrderStatus.PENDING;
  }

  // 问题：违反开闭原则，新增状态要改这里
  // 问题：违反单一职责，Order 类承担了所有状态逻辑
  handleRequest(action: string) {
    if (this.status === OrderStatus.PENDING) {
      if (action === 'pay') {
        this.status = OrderStatus.PAID;
        console.log('Order Paid');
      } else {
        console.log('Invalid action for PENDING');
      }
    } else if (this.status === OrderStatus.PAID) {
      if (action === 'ship') {
        this.status = OrderStatus.SHIPPED;
        console.log('Order Shipped');
      } else {
        console.log('Invalid action for PAID');
      }
    } else if (this.status === OrderStatus.SHIPPED) {
      console.log('Order completed, no actions allowed');
    }
  }
}
```
**问题：**
1.  **违反开闭原则（OCP）**：新增一个状态（如 `REFUNDED`），需要修改 `handleRequest` 的所有分支。
2.  **违反单一职责（SRP）**：`Order` 类包含了所有状态的逻辑，代码臃肿。
3.  **难以维护**：状态越多，`if-else` 嵌套越深，容易出错。
4.  **状态转换逻辑分散**：状态流转规则硬编码在方法里。

#### ✅ 使用状态模式的写法（状态对象化）
```typescript
// 客户端只负责触发事件，不关心当前状态
order.handleEvent('pay'); 
// 内部自动根据当前状态决定行为
```
**好处：** 新增状态只需新增一个状态类，无需修改订单上下文。

---

### 2. 核心角色

1.  **Context（上下文）**：持有当前状态对象的引用，定义客户端感兴趣的接口。
2.  **State（抽象状态）**：定义一个接口，封装与 Context 特定状态相关的行为。
3.  **ConcreteState（具体状态）**：实现具体状态的行为，并可能负责状态转换（切换 Context 的当前状态）。

---

### 3. TypeScript 实现

在 TypeScript 中，状态模式有两种实现方式：**经典类实现** 和 **现代函数式/类型实现**（更适合前端）。

#### (1) 经典实现（基于类）
适用于状态逻辑复杂、需要维护内部状态的场景。

```typescript
// 1. 抽象状态接口
interface OrderState {
  handlePay(context: OrderContext): void;
  handleShip(context: OrderContext): void;
}

// 2. 上下文
class OrderContext {
  private state: OrderState;

  constructor() {
    // 初始状态
    this.state = new PendingState();
  }

  setState(state: OrderState) {
    this.state = state;
  }

  pay() {
    this.state.handlePay(this);
  }

  ship() {
    this.state.handleShip(this);
  }
}

// 3. 具体状态：待支付
class PendingState implements OrderState {
  handlePay(context: OrderContext) {
    console.log('Payment successful');
    context.setState(new PaidState()); // 状态流转
  }
  handleShip(context: OrderContext) {
    console.log('Cannot ship unpaid order');
  }
}

// 4. 具体状态：已支付
class PaidState implements OrderState {
  handlePay(context: OrderContext) {
    console.log('Already paid');
  }
  handleShip(context: OrderContext) {
    console.log('Order shipped');
    context.setState(new ShippedState());
  }
}

// 5. 具体状态：已发货
class ShippedState implements OrderState {
  handlePay(context: OrderContext) {
    console.log('Cannot pay shipped order');
  }
  handleShip(context: OrderContext) {
    console.log('Already shipped');
  }
}

// 6. 客户端
const order = new OrderContext();
order.pay();   // 输出：Payment successful
order.ship();  // 输出：Order shipped
order.ship();  // 输出：Already shipped
```

#### (2) 现代实现（基于 TS 类型 +  reducer 函数）
在前端（React/Redux/XState）中，更常用函数式状态机。利用 TS 的 **判别联合类型（Discriminated Unions）** 保证类型安全。

```typescript
// 1. 定义状态类型
type OrderStatus = 'PENDING' | 'PAID' | 'SHIPPED';

interface OrderState {
  status: OrderStatus;
  data: any;
}

// 2. 定义事件类型
type OrderEvent = 
  | { type: 'PAY' } 
  | { type: 'SHIP' };

// 3. 状态转换函数 (Reducer)
function orderReducer(state: OrderState, event: OrderEvent): OrderState {
  switch (state.status) {
    case 'PENDING':
      if (event.type === 'PAY') return { status: 'PAID', data: state.data };
      break;
    case 'PAID':
      if (event.type === 'SHIP') return { status: 'SHIPPED', data: state.data };
      break;
  }
  // 无效操作返回原状态
  console.log(`Invalid event ${event.type} for status ${state.status}`);
  return state;
}

// 4. 使用
let currentState: OrderState = { status: 'PENDING', data: {} };
currentState = orderReducer(currentState, { type: 'PAY' });
currentState = orderReducer(currentState, { type: 'SHIP' });
```
**优势**：不可变数据，类型安全，易于测试，符合 React 理念。

### 4. 状态模式 vs 策略模式

这是**最容易混淆**的两个模式，因为它们的**类图结构几乎完全一样**。

| 特性 | 状态模式 (State) | 策略模式 (Strategy) |
| :--- | :--- | :--- |
| **核心意图** | **状态驱动行为**。对象内部状态变化导致行为变化。 | **算法替换**。客户端主动选择使用哪个算法。 |
| **依赖关系** | 状态之间可能**相互依赖**（知道下一个状态是什么）。 | 策略之间通常**独立**，互不知道对方。 |
| **转换控制** | 状态转换通常由**Context 或 State 内部**控制。 | 策略切换通常由**客户端外部**控制。 |
| **生命周期** | 状态通常代表对象的**生命周期阶段**。 | 策略代表解决问题的**不同方法**。 |
| **例子** | 订单流转（待支付->已支付）。 | 支付方式（选支付宝 **或** 微信）。 |
| **口诀** | **状态是“现在是啥”（内部驱动）**。 | **策略是“选哪个”（外部选择）**。 |

**简单判断法**：
*   如果行为变化是因为**对象自身发生了变化**（如订单付钱了），用 **状态模式**。
*   如果行为变化是因为**用户想要不同的实现**（如用户选了 VIP 折扣），用 **策略模式**。

---

### 5. 状态模式与 SOLID 原则

| 原则 | 状态模式如何体现 |
| :--- | :--- |
| **O (开闭原则)** | **核心受益者**。新增状态只需新增状态类，无需修改 Context 或其他状态类。 |
| **S (单一职责)** | 每个状态类只负责该状态下的行为逻辑，职责单一。 |
| **D (依赖倒置)** | Context 依赖抽象 `State` 接口，不依赖具体状态实现。 |
| **L (里氏替换)** | 任何实现了 State 接口的类都可以替换当前状态。 |

### 6. 前端开发中的实际应用

状态模式在前端开发中**极其重要**，因为前端本质上是**UI 状态的管理**。

#### (1) 组件生命周期状态
React/Vue 组件的不同渲染状态（Loading, Success, Error, Empty）。
```typescript
type ComponentState = 'loading' | 'success' | 'error';

function render(state: ComponentState, data: any) {
  switch (state) {
    case 'loading': return <Spinner />;
    case 'success': return <List data={data} />;
    case 'error': return <ErrorMessage />;
  }
}
// 优化：使用状态模式将每种渲染逻辑封装成组件
```

#### (2) 表单步骤向导（Wizard）
多步骤表单（步骤 1 -> 步骤 2 -> 步骤 3），每一步的验证逻辑和提交逻辑不同。
*   每个步骤是一个状态对象。
*   “下一步”按钮触发状态流转。

#### (3) 用户认证状态
未登录 -> 登录中 -> 已登录 -> 过期。
*   不同状态下，API 请求拦截器的行为不同（未登录跳转登录页，已登录加 Token）。

#### (4) 游戏开发（最经典场景）
游戏角色状态： idle（待机）-> run（跑）-> jump（跳）-> attack（攻击）。
*   不同状态下，按键响应不同（待机时按空格是跳，攻击时按空格可能是终结技）。
*   状态机管理角色行为。

#### (5) 状态管理库（Redux / XState）
*   **Redux**：本质上是一个巨大的状态模式实现（Reducer 根据 action 切换 state）。
*   **XState**：专门用于前端的状态机库，完美实现了状态模式，支持可视化状态图。
    ```javascript
    // XState 示例
    const machine = createMachine({
      initial: 'idle',
      states: {
        idle: { on: { FETCH: 'loading' } },
        loading: { on: { RESOLVE: 'success', REJECT: 'failure' } },
        success: {},
        failure: {}
      }
    });
    ```

#### (6) 播放器控制
播放器的状态：Stopped -> Playing -> Paused。
*   在 `Playing` 状态下，点击按钮是 `Pause`。
*   在 `Paused` 状态下，点击按钮是 `Resume`。
*   按钮的文本和行为随状态改变。

### 7. 优缺点分析

#### ✅ 优点
1.  **消除条件语句**：完美替代庞大的 `switch-case` 状态判断。
2.  **符合开闭原则**：新增状态无需修改现有代码。
3.  **状态逻辑内聚**：特定状态的行为集中在一个类中，易于维护。
4.  **明确状态流转**：状态转换逻辑清晰可见。

#### ❌ 缺点
1.  **类/文件数量增加**：每个状态一个类，如果状态很多，会导致类爆炸。
2.  **状态转换复杂**：如果状态流转逻辑非常复杂（多对多），维护状态之间的转换关系可能变得困难。
3.  **性能开销**：频繁创建状态对象可能有开销（可用单例模式优化状态对象）。
4.  **调试困难**：状态隐藏在不同对象中，追踪当前状态可能需要查看 Context。

---

### 8. 最佳实践与注意事项

1.  **状态对象单例化**：如果状态对象无内部状态（无成员变量），可以使用 **单例模式** 复用状态对象，减少内存开销。
    ```typescript
    class PendingState implements OrderState {
      static instance = new PendingState();
      // 返回单例
    }
    ```
2.  **使用状态机库**：对于复杂的前端状态逻辑，不要手动实现状态模式，推荐使用 **XState** 或 **Redux Toolkit**。它们已经封装好了状态模式的最佳实践。
3.  **类型安全**：在 TS 中，利用 **判别联合类型** 确保状态和数据的匹配，避免非法状态组合。
4.  **明确状态转换**：最好在文档或代码注释中明确状态流转图（State Transition Diagram），防止出现死锁状态。
5.  **区分状态与属性**：不要把所有变量都当成状态。只有当变量变化会导致**行为本质变化**时，才使用状态模式。
    *   *是状态*：订单是否支付（行为完全不同）。
    *   *不是状态*：用户名字颜色（只是属性变化，行为不变）。
6.  **结合观察者模式**：状态变化时，通常需要通知 UI 更新。状态模式常与观察者模式结合使用（状态变了 -> 发布事件 -> UI 更新）。

---

### 9. 总结

**状态模式的核心价值在于“封装变化”与“状态流转”。**

*   **何时使用**：
    *   对象的行为依赖于它的状态，并且必须在运行时根据状态改变行为。
    *   代码中包含大量与对象状态相关的条件语句（`if-else` / `switch`）。
    *   状态流转逻辑复杂，需要明确管理。
*   **核心思想**：**状态对象化**，将状态逻辑从上下文剥离。
*   **前端地位**：它是 **UI 渲染逻辑**、**表单流程**、**游戏行为**、**状态管理库（Redux/XState）** 的理论基础。
*   **与策略模式的区别**：
    *   **状态** = **内部驱动**（生命周期变化，如订单流转）。
    *   **策略** = **外部选择**（算法替换，如支付方式）。
*   **与 SOLID 的关系**：是 **OCP（开闭原则）** 和 **SRP（单一职责）** 的典型应用。
*   **TS 最佳实践**：
    *   简单逻辑用 **Reducer 函数 + 联合类型**。
    *   复杂逻辑用 **类 + 状态机库（XState）**。
    *   无状态对象用 **单例模式** 优化。

结合你之前的学习路径：
*   用 **单例模式** 管理状态对象实例。
*   用 **观察者模式** 通知状态变化。
*   用 **策略模式** 处理状态内的具体算法。
*   用 **TypeScript 类型系统** 保证状态流转的合法性。
*   用 **AOP** 记录状态转换日志。

掌握状态模式，意味着你掌握了**管理复杂业务流转**的核心能力，能够写出逻辑清晰、易于扩展的状态驱动型代码。它是前端架构师处理复杂交互逻辑的必备工具。
