`职责链模式（Chain of Responsibility Pattern） `是软件工程中**行为型设计模式（Behavioral Pattern）** 的一种。

它的`核心思想`是：**使多个对象都有机会处理请求，从而避免请求的发送者和接收者之间的耦合关系。将这些对象连成一条链，并沿着这条链传递该请求，直到有一个对象处理它为止。**
通俗地说，就是 **“击鼓传花”** 或 **“层层审批”**。你提交一个申请，组长看能不能批，不能批就给经理，经理不能批就给总监。你不需要知道具体谁会批，只需要把申请交上去。

`职责链模式`是*实现 **业务逻辑解耦** 和 **流程编排** 的重要工具，也是 **中间件（Middleware）** 机制的理论基础*。

---

### 1. 为什么要用职责链模式？（解决的问题）

在没有职责链模式时，处理多种条件的请求通常使用大量的 `if-else` 或 `switch-case`。

#### ❌ 违反职责链模式的写法（高耦合）
假设一个电商订单系统，需要根据订单类型进行不同的验证。

```typescript
class OrderService {
  processOrder(order: Order) {
    if (order.type === 'NORMAL') {
      // 验证普通订单
      this.validateNormal(order);
    } else if (order.type === 'VIP') {
      // 验证 VIP 订单
      this.validateVIP(order);
    } else if (order.type === 'GROUP') {
      // 验证团购订单
      this.validateGroup(order);
    } else {
      throw new Error('Unknown type');
    }
    // 保存订单
    this.save(order);
  }
}
```
**问题：**
1.  **违反开闭原则（OCP）**：新增一种订单类型（如 `PRE_SALE`），必须修改 `processOrder` 方法。
2.  **违反单一职责（SRP）**：`OrderService` 既管流程，又管各种具体的验证逻辑。
3.  **代码臃肿**：随着类型增加，方法会变得极长且难以维护。

#### ✅ 使用职责链模式的写法（解耦）
```typescript
// 客户端只负责发起请求
chain.handle(order); 
// 具体哪个处理器处理，客户端不关心
```
**好处：** 新增验证逻辑只需新增一个处理器类，并加入链条，无需修改原有代码。

---

### 2. 核心角色

1.  **Handler（抽象处理者）**：*定义处理请求的接口*，通常*包含一个指向下一个处理者的引用*（`next`）。
2.  **ConcreteHandler（具体处理者）**：*实现处理逻辑*。如果能处理就处理，不能处理就转发给 `next`。
3.  **Client（客户端）**：组装链条，并向链头提交请求。

### 3. TypeScript 实现

职责链模式*在 TS 中有两种常见实现方式*：`经典链表式` 和 `现代中间件式（洋葱模型）`。

#### (1) 经典实现（链表式）
适用于“只有一个处理器能处理请求”的场景（如审批流）。

```typescript
// 1. 请求对象
interface Order {
  type: string;
  amount: number;
}

// 2. 抽象处理者
abstract class OrderHandler {
  protected nextHandler: OrderHandler | null = null;

  // 设置下一个处理者
  setNext(handler: OrderHandler): OrderHandler {
    this.nextHandler = handler;
    return handler; // 方便链式调用
  }

  // 处理请求
  handle(order: Order): void {
    if (this.nextHandler) {
      this.nextHandler.handle(order);
    } else {
      console.log('No handler found, request dropped.');
    }
  }
}

// 3. 具体处理者：普通订单
class NormalOrderHandler extends OrderHandler {
  handle(order: Order): void {
    if (order.type === 'NORMAL') {
      console.log('Handling Normal Order');
      // 处理完毕，不再传递
    } else {
      // 传递下去
      super.handle(order);
    }
  }
}

// 4. 具体处理者：VIP 订单
class VIPOrderHandler extends OrderHandler {
  handle(order: Order): void {
    if (order.type === 'VIP') {
      console.log('Handling VIP Order');
    } else {
      super.handle(order);
    }
  }
}

// 5. 客户端组装
const chain = new NormalOrderHandler();
chain.setNext(new VIPOrderHandler());
// chain.setNext(new GroupOrderHandler()); // 可随时扩展

// 使用
chain.handle({ type: 'VIP', amount: 100 }); 
// 输出：Handling VIP Order
```

#### (2) 现代实现（中间件/洋葱模型）
**适用于“所有处理器都要执行”或“可拦截”的场景**（如 `Express/Koa 中间件`、`Axios 拦截器`）。这是*前端更常见的形式*。

```typescript
// 中间件函数类型
type Middleware = (context: any, next: () => Promise<void>) => Promise<void>;

class MiddlewareChain {
  // 保存中间件
  private middlewares: Middleware[] = [];

  // 注册中间件
  use(middleware: Middleware): void {
    this.middlewares.push(middleware);
  }

  // 执行链条
  async execute(context: any): Promise<void> {
    let index = -1;

    const dispatch = async (i: number): Promise<void> => {
      if (i <= index) throw new Error('next() called multiple times');
      index = i;
      
      // 如果到了末尾，结束
      if (i === this.middlewares.length) return;

      const middleware = this.middlewares[i];
      // 传入 next 函数，指向下一个中间件
      await middleware(context, () => dispatch(i + 1));
    };

    await dispatch(0);
  }
}

// 使用示例
const chain = new MiddlewareChain();

// 中间件 1：日志
chain.use(async (ctx, next) => {
  console.log('1. Start Log');
  await next(); // 继续
  console.log('1. End Log');
});

// 中间件 2：权限验证（可拦截）
chain.use(async (ctx, next) => {
  console.log('2. Check Auth');
  if (!ctx.isAdmin) {
    console.log('2. Access Denied');
    return; // 不调用 next，中断链条
  }
  await next();
});

// 中间件 3：核心业务
chain.use(async (ctx, next) => {
  console.log('3. Business Logic');
});

// 执行
chain.execute({ isAdmin: true });
/* 输出 (洋葱模型):
1. Start Log
2. Check Auth
3. Business Logic
4. End Log
*/
```

### 4. 职责链模式与 SOLID 原则

| 原则 | 职责链模式如何体现 |
| :--- | :--- |
| **S (单一职责)** | **核心受益者**。每个处理器只负责一种特定的逻辑（如只验 VIP，只验团购）。 |
| **O (开闭原则)** | 新增处理逻辑只需新增一个 Handler 类并加入链条，无需修改客户端或原有 Handler。 |
| **D (依赖倒置)** | 客户端依赖抽象 `Handler`，不依赖具体实现。Handler 之间也依赖抽象。 |
| **L (里氏替换)** | 任何实现了 Handler 接口的类都可以加入链条。 |

---

### 5. 职责链模式 vs 其他模式

| 模式        | 区别                                                          |
| :-------- | :---------------------------------------------------------- |
| **观察者模式** | `观察者`是**一对多广播**（所有订阅者都收到通知）；职责链是**一对一传递**（通常只有一个处理，或按顺序传递）。 |
| **装饰器模式** | `装饰器`是**嵌套包裹**，增强功能；`职责链`是**线性传递**，处理请求。结构相似，意图不同。          |
| **命令模式**  | 命令模式封装请求为对象；职责链关注请求的**传递路径**。两者常结合使用。                       |
| **AOP**   | 职责链是 **同步/线性 的 AOP**。中间件模式本质上就是 AOP 的一种实现。                  |

### 6. 前端开发中的**实际应用**

职责链模式在前端生态中**极其普遍**，*甚至可以说是现代前端框架的基石之一*。

#### (1) DOM 事件流（Event Propagation）
DOM 的事件捕获（Capture）和冒泡（Bubble）机制就是典型的职责链。
*   **链条**：`window -> document -> html -> body -> div -> button`
*   **处理器**：每个节点上的事件监听器。
*   **控制**：`event.stopPropagation()` 相当于中断链条（不调用 `next`）。

#### (2) HTTP 库中间件（Axios Interceptors）
*Axios 的请求和响应拦截器就是一个职责链。*
```typescript
axios.interceptors.request.use(config => {
  // 添加 Token
  config.headers.Token = 'xyz';
  return config; // next(config)
});

axios.interceptors.response.use(response => {
  // 统一处理错误
  return response; 
});
```

#### (3) 前端框架中间件
*   **Koa / Express**：Node.js 后端框架的核心就是职责链（中间件）。
*   **NestJS**：Guards（守卫）、Interceptors（拦截器）、Pipes（管道）都是职责链的不同变体。
*   **React Router**：路由守卫（Before Enter）也是职责链。

#### (4) 表单验证
多个验证规则串联：非空验证 -> 格式验证 -> 长度验证 -> 唯一性验证。
```typescript
const validatorChain = new ValidatorChain();
validatorChain.use(required());
validatorChain.use(emailFormat());
validatorChain.use(minLength(6));

validatorChain.validate(input);
```

#### (5) 游戏开发
游戏对象更新循环：输入处理 -> 物理更新 -> 逻辑更新 -> 渲染。每个阶段是一个处理器。

### 7. 优缺点分析

#### ✅ 优点
1.  **降低耦合**：发送者不需要知道接收者的细节。
2.  **增强灵活性**：可以动态改变链条内的成员顺序或增减成员。
3.  **符合 SRP/OCP**：每个类职责单一，易于扩展。
4.  **简化对象**：对象不需要知道整个链条的结构，只需要知道下一个节点。

#### ❌ 缺点
1.  **性能问题**：链条过长时，请求传递耗时增加。
2.  **调试困难**：请求在链条中流转，断点调试可能比较繁琐，不易追踪最终是谁处理了请求。
3.  **未处理风险**：如果链条末端没有默认处理器，请求可能石沉大海（需在代码中处理 `null` 情况）。
4.  **循环依赖**：如果链条配置错误形成环，会导致死循环（栈溢出）。

---

### 8. 最佳实践与注意事项

1.  **默认处理器**：链条末尾最好有一个默认处理器（Default Handler），防止请求丢失。
2.  **避免过长**：链条节点不宜过多（建议 < 10 个），否则维护成本高。
3.  **明确中断机制**：在使用中间件模式时，明确 `next()` 的调用规则，避免忘记调用导致后续逻辑不执行，或重复调用导致逻辑执行多次。
4.  **类型安全**：在 TS 中，利用泛型确保 `context` 在链条传递过程中类型一致或按预期变化。
5.  **异步处理**：现代前端多为异步，确保链条支持 `Promise` 或 `async/await`（如上面的中间件示例）。

---

### 9. **总结**

**`职责链模式`的*核心价值*在于“解耦”与“编排”。**

*   **何时使用**：
    *   有多个对象可以处理同一请求，具体由谁处理运行时决定。
    *   想动态指定一组对象处理请求，并动态改变顺序。
    *   请求的处理逻辑需要分步骤、分层级（如审批、验证、中间件）。
*   **核心思想**：**请求发送者 vs 接收者解耦**。
*   **前端地位**：它是 **DOM 事件流**、**Axios 拦截器**、**Koa/NestJS 中间件** 的理论基础。
*   **与 AOP 的关系**：职责链是 AOP 的线性实现形式。
*   **与装饰器的关系**：结构相似，但`职责链`侧重*流程控制*，`装饰器`侧重*功能增强*。

结合你之前的学习路径：
*   用 **单例模式** 管理链条实例。
*   用 **工厂模式** 创建处理器节点。
*   用 **装饰器模式** 增强处理器功能。
*   用 **TypeScript 泛型** 保证链条数据流类型安全。
*   用 **AOP 思想** 设计中间件逻辑。

掌握职责链模式，意味着你*掌握了**流程控制**和**中间件架构**的核心逻辑，能够设计出高扩展性、低耦合的业务系统*。
