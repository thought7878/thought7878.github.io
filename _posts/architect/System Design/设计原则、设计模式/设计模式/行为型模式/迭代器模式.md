`迭代器模式（Iterator Pattern） `是软件工程中**行为型设计模式（Behavioral Pattern）** 的一种。

它的`核心思想`是：**提供一种方法顺序访问一个聚合对象（集合）中的各个元素，而又*不暴露该对象的内部*表示。**
通俗地说，就是 **“把遍历的逻辑从集合中分离出来”**。*你不需要知道集合底层是数组、链表还是树，你只需要通过迭代器接口，一个一个地取数据*。

*迭代器模式在现代前端开发中其实无处不在*（例如 `for...of`、`Array.map`、生成器函数）。

---

## 1. 为什么要用迭代器模式？（解决的问题）

*在没有迭代器模式时，遍历逻辑通常耦合在集合类内部，或者由客户端直接访问集合的内部结构*。

### ❌ 违反迭代器模式的写法（高耦合）
假设我们有一个书架（`BookShelf`），底层用数组存储书籍。

```typescript
class BookShelf {
  private books: string[] = [];

  addBook(book: string) {
    this.books.push(book);
  }

  // 问题 1: 客户端必须知道内部是数组
  // 问题 2: 如果想倒序遍历，得再写一个方法
  // 问题 3: 违反单一职责原则（SRP），书架既管存书，又管遍历
  getBooks(): string[] {
    return this.books; 
  }
}

// 客户端
const shelf = new BookShelf();
shelf.addBook('Book A');
shelf.addBook('Book B');

// 客户端直接依赖了内部实现（数组）
const books = shelf.getBooks(); 
for (let i = 0; i < books.length; i++) {
  console.log(books[i]);
}
```

**问题：**
1.  **暴露内部结构**：*如果以后把 `books` 改成链表，所有调用 `getBooks()` 的地方都要改*。
2.  **违反开闭原则（OCP）**：如果需要支持“倒序遍历”或“按价格遍历”，必须修改 `BookShelf` 类。
3.  **违反单一职责（SRP）**：`BookShelf` 承担了存储和遍历两种职责。

### ✅ 使用迭代器模式的写法（解耦）
引入一个迭代器对象，专门负责遍历。

```typescript
// 客户端只关心迭代器接口
const iterator = shelf.createIterator();
while (iterator.hasNext()) {
  console.log(iterator.next());
}
```
**好处：** 书架只管存书，迭代器只管走路。*底层结构变了，只要迭代器适配即可，客户端代码无需修改*。

## 2. 核心角色

1.  **Iterator（抽象迭代器）**：定义访问和遍历元素的接口（`next()`, `hasNext()`）。
2.  **ConcreteIterator（具体迭代器）**：实现抽象迭代器，负责记录当前遍历的位置。
3.  **Aggregate（抽象聚合/集合）**：定义创建迭代器的接口（`createIterator()`）。
4.  **ConcreteAggregate（具体聚合）**：实现创建迭代器的接口，返回具体的迭代器实例。

---

## 3. TypeScript 实现

为了让你理解原理，我们先手动实现一个经典版本，再展示 TypeScript 的原生支持。

### (1) 经典实现（手动定义接口）

```typescript
// 1. 抽象迭代器接口
interface Iterator<T> {
  next(): T | null;
  hasNext(): boolean;
  reset(): void;
}

// 2. 抽象集合接口
interface Aggregate<T> {
  createIterator(): Iterator<T>;
}

// 3. 具体集合：书架
class BookShelf implements Aggregate<string> {
  private books: string[] = [];

  addBook(book: string) {
    this.books.push(book);
  }

  // 工厂方法：创建迭代器
  createIterator(): Iterator<string> {
    return new BookShelfIterator(this);
  }

  // 供迭代器访问内部数据（通常设为包级私有或 friend）
  getBooks(): string[] {
    return this.books;
  }
}

// 4. 具体迭代器：书架迭代器
class BookShelfIterator implements Iterator<string> {
  private shelf: BookShelf;
  private index: number = 0;

  constructor(shelf: BookShelf) {
    this.shelf = shelf;
  }

  next(): string | null {
    if (this.hasNext()) {
      return this.shelf.getBooks()[this.index++];
    }
    return null;
  }

  hasNext(): boolean {
    return this.index < this.shelf.getBooks().length;
  }

  reset(): void {
    this.index = 0;
  }
}

// 5. 使用
const shelf = new BookShelf();
shelf.addBook('Design Patterns');
shelf.addBook('Clean Code');
shelf.addBook('Refactoring');

const iterator = shelf.createIterator();
while (iterator.hasNext()) {
  console.log(iterator.next());
}
```

### (2) 现代 TypeScript 实现（原生支持）

*在现代 TS/JS 中，迭代器模式已经内置在语言特性中（`Symbol.iterator`）。你只需要实现 `Iterable` 接口*。

```typescript
// 实现 Iterable 接口，让对象可以被 for...of 遍历
class BookShelf implements Iterable<string> {
  private books: string[] = [];

  addBook(book: string) {
    this.books.push(book);
  }

  // 核心：定义 [Symbol.iterator] 方法
  [Symbol.iterator](): Iterator<string> {
    let index = 0;
    const books = this.books;

    return {
      next(): IteratorResult<string> {
        if (index < books.length) {
          return { value: books[index++], done: false };
        } else {
          return { value: undefined as any, done: true };
        }
      }
    };
  }
}

// 使用：极其简洁
const shelf = new BookShelf();
shelf.addBook('Book A');
shelf.addBook('Book B');

// 1. for...of 循环
for (const book of shelf) {
  console.log(book);
}

// 2. 扩展运算符
console.log([...shelf]);

// 3. 解构
const [first, second] = shelf;
```

### (3) 生成器函数（Generator）简化版

使用 `function*` 可以进一步简化迭代器的编写，这是迭代器模式的语法糖。

```typescript
class BookShelf implements Iterable<string> {
  private books: string[] = [];

  addBook(book: string) {
    this.books.push(book);
  }

  // 使用生成器自动实现 Iterator 接口
  *[Symbol.iterator](): Iterator<string> {
    for (const book of this.books) {
      yield book; // 每次 yield 返回一个值，暂停函数
    }
  }
}
```

## 4. 迭代器模式与 SOLID 原则

| 原则 | 迭代器模式如何体现 |
| :--- | :--- |
| **S (单一职责)** | **核心受益者**。集合类只负责存储数据，迭代器类只负责遍历逻辑。职责分离。 |
| **O (开闭原则)** | 如果需要新的遍历方式（如倒序、过滤），只需新增一个迭代器类（`ReverseIterator`），无需修改集合类。 |
| **D (依赖倒置)** | 客户端依赖的是 `Iterator` 接口，而不是具体的 `BookShelf` 内部结构。 |
| **L (里氏替换)** | 任何实现了 `Iterator` 接口的类都可以替换使用。 |

## 5. 优缺点分析

### ✅ 优点
1.  **统一访问接口**：无论底层是数组、链表、树还是图，客户端都用同样的方式（`next()` 或 `for...of`）遍历。
2.  **支持多种遍历**：可以同时存在多个迭代器（如一个正向遍历，一个反向遍历），互不干扰。
3.  **封装内部结构**：客户端不需要知道集合内部是怎么存储的，方便后续修改底层实现。
4.  **符合 SRP**：遍历逻辑从集合中剥离。

### ❌ 缺点
1.  **类数量增加**：每个集合类可能需要对应一个迭代器类，增加了系统复杂度（但在现代语言中通常由语言内置解决）。
2.  **功能受限**：迭代器通常只支持单向遍历（虽然可以实现双向，但接口会变复杂）。
3.  **性能开销**：对于简单的数组遍历，使用迭代器可能比直接 `for` 循环稍微慢一点点（通常可忽略）。

## 6. 现代前端开发中的应用

*迭代器模式在前端开发中非常普遍，你可能每天都在用，但没意识到它是设计模式*。

1.  **数组方法**：
    *   `Array.prototype.forEach`, `map`, `filter`, `reduce` 本质上都是迭代器模式的变体。
    *   它们封装了遍历逻辑，你只需要提供回调函数。

2.  **DOM 遍历**：
    *   `document.querySelectorAll('div')` 返回的是一个 `NodeList`，它是可迭代的（Iterable）。
    *   你可以直接用 `for (const div of document.querySelectorAll('div'))`。

3.  **异步迭代器（Async Iterator）**：
    *   处理异步数据流（如读取文件流、WebSocket 消息）。
    *   使用 `for await...of` 语法。
    ```typescript
    async function* fetchDataStream() {
      yield await fetch('/api/page1');
      yield await fetch('/api/page2');
    }
    
    for await (const response of fetchDataStream()) {
      // 处理数据
    }
    ```

4.  **状态管理库**：
    *   Redux 的 `store.subscribe` 结合 selector，某种程度上也是迭代观察数据变化。
    *   RxJS 的 `Observable` 是迭代器模式的超级进化版（支持异步、多值、操作符链式调用）。

---

## 7. 迭代器模式 vs 其他模式

| 模式        | 区别                                          |
| :-------- | :------------------------------------------ |
| **观察者模式** | 观察者是被动的（等通知），迭代器是主动的（主动去 `next()` 拉取数据）。    |
| **工厂模式**  | 迭代器的创建通常使用工厂方法（`createIterator()`），两者常结合使用。 |
| **访问者模式** | 访问者模式用于对复杂对象结构执行操作，迭代器模式仅用于顺序访问。            |

---

## 8. 总结

**迭代器模式的核心价值在于“统一访问”和“解耦遍历”。**

*   **经典意义**：它教会我们将“遍历算法”从“数据结构”中分离出来（SRP）。
*   **现代意义**：在 TypeScript/JavaScript 中，它已经内化为语言特性（`Iterable`, `Iterator`, `Generator`）。
*   **最佳实践**：
    1.  如果你自定义了一个数据结构（如树、图、特殊链表），**务必实现 `[Symbol.iterator]`**，这样你的类就能无缝融入 JS 生态（支持 `for...of`, `...` 等）。
    2.  优先使用**生成器函数（`function*`）** 来实现迭代器，代码更简洁。
    3.  理解其背后的思想，即使语言内置了，也要知道为什么要这样设计（为了开闭原则和单一职责）。

掌握迭代器模式，意味着你不仅能写出能跑的代码，还能写出**符合语言习惯、易于集成**的代码。
