`策略模式（Strategy Pattern）` 是软件工程中**行为型设计模式（Behavioral Pattern）** 中*最常用、最实用的一种*。

它的`核心思想`是：**定义一系列算法，把它们一个个封装起来，并且使它们可相互替换。本模式使得算法可独立于使用它的客户而变化。**
通俗地说，就是 **“锦囊妙计”** 或 **“多方案选择”**。*比如*你要去北京，可以选择坐飞机、坐高铁或开车。这三种方式（策略）都能达到目的，但成本和时间不同。你可以*根据实际情况（预算、时间）动态选择一种，而不需要修改“去北京”这个核心目标*。

`策略模式`是*消除 **`if-else` / `switch-case`** 复杂度、实现 **开闭原则（OCP）** 的最佳实践之一*。

---

### 1. 为什么要用策略模式？（解决的问题）

*在没有策略模式时，处理多种业务规则或算法，通常使用大量的条件判断*。

#### ❌ 违反策略模式的写法（硬编码）
假设一个电商系统，根据用户等级计算折扣。

```typescript
class Order {
  calculatePrice(price: number, userLevel: string): number {
    // 问题：违反开闭原则，每次新增等级都要改这里
    // 问题：违反单一职责，订单类既管订单，又管折扣算法
    if (userLevel === 'NORMAL') {
      return price;
    } else if (userLevel === 'VIP') {
      return price * 0.9;
    } else if (userLevel === 'SVIP') {
      return price * 0.8;
    } else {
      throw new Error('Unknown level');
    }
  }
}
```
**问题：**
1.  **违反开闭原则（OCP）**：新增 `SUPER_VIP` 等级，必须修改 `calculatePrice` 方法。
2.  **违反单一职责（SRP）**：折扣算法逻辑耦合在订单类中。
3.  **难以测试**：测试订单逻辑时，必须同时测试所有折扣分支。
4.  **代码臃肿**：随着策略增多，方法变得极长。

#### ✅ 使用策略模式的写法（多态替换）
```typescript
// 客户端根据上下文选择策略
const context = new OrderContext(new VIPStrategy());
context.executePrice(100); 
```
**好处：** 新增折扣算法只需新增一个策略类，订单上下文无需修改。

---

### 2. 核心角色

1.  **Strategy（抽象策略）**：定义所有支持算法的公共接口。
2.  **ConcreteStrategy（具体策略）**：实现具体的算法逻辑。
3.  **Context（上下文/环境）**：持有一个 Strategy 的引用，提供给客户端调用。

---

### 3. TypeScript 实现

在 TypeScript 中，*策略模式有两种实现方式*：**经典类实现** 和 **函数式实现**（利用 TS 函数是一等公民的特性）。

#### (1) 经典实现（基于类）
适用于*策略逻辑复杂、需要维护状态的*场景。

```typescript
// 1. 抽象策略接口
interface DiscountStrategy {
  calculate(price: number): number;
}

// 2. 具体策略：无折扣
class NoDiscount implements DiscountStrategy {
  calculate(price: number): number {
    return price;
  }
}

// 3. 具体策略：VIP 9 折
class VIPDiscount implements DiscountStrategy {
  calculate(price: number): number {
    return price * 0.9;
  }
}

// 4. 具体策略：SVIP 8 折
class SVIPDiscount implements DiscountStrategy {
  calculate(price: number): number {
    return price * 0.8;
  }
}

// 5. 上下文：订单
class OrderContext {
  private strategy: DiscountStrategy;

  constructor(strategy: DiscountStrategy) {
    this.strategy = strategy;
  }

  // 动态切换策略
  setStrategy(strategy: DiscountStrategy) {
    this.strategy = strategy;
  }

  getPrice(price: number): number {
    return this.strategy.calculate(price);
  }
}

// 6. 客户端使用
const order = new OrderContext(new NoDiscount());
console.log(order.getPrice(100)); // 100

// 运行时动态切换策略
order.setStrategy(new VIPDiscount());
console.log(order.getPrice(100)); // 90
```

#### (2) 函数式实现（TS  idiomatic）
在 TS/JS 中，如果策略无状态，直接用**函数**代替类，更简洁。

```typescript
// 1. 策略类型定义
type DiscountStrategy = (price: number) => number;

// 2. 具体策略函数
const noDiscount: DiscountStrategy = (price) => price;
const vipDiscount: DiscountStrategy = (price) => price * 0.9;
const svipDiscount: DiscountStrategy = (price) => price * 0.8;

// 3. 上下文
class Order {
  constructor(private strategy: DiscountStrategy) {}

  setStrategy(strategy: DiscountStrategy) {
    this.strategy = strategy;
  }

  getPrice(price: number): number {
    return this.strategy(price);
  }
}

// 4. 使用
const order = new Order(noDiscount);
order.setStrategy(vipDiscount); // 直接传函数
```

#### (3) 结合工厂模式（最佳实践）
客户端通常不应该直接 `new` 具体策略，而是通过工厂获取。

```typescript
class StrategyFactory {
  static getStrategy(level: string): DiscountStrategy {
    switch (level) {
      case 'VIP': return new VIPDiscount();
      case 'SVIP': return new SVIPDiscount();
      default: return new NoDiscount();
    }
  }
}

// 客户端
const strategy = StrategyFactory.getStrategy('VIP');
const order = new OrderContext(strategy);
```
**优势**：客户端完全解耦具体策略类，符合 **依赖倒置原则（DIP）**。

### 4. 策略模式与 SOLID 原则

策略模式是 SOLID 原则的**典范应用**。

| 原则 | 策略模式如何体现 |
| :--- | :--- |
| **O (开闭原则)** | **核心受益者**。新增算法只需新增策略类，无需修改 Context 或客户端。 |
| **S (单一职责)** | 每个策略类只负责一种算法逻辑，Context 只负责调用。 |
| **D (依赖倒置)** | Context 依赖抽象 `Strategy` 接口，不依赖具体实现。 |
| **L (里氏替换)** | 任何实现了 Strategy 接口的类都可以无缝替换。 |

### 5. 策略模式 vs 其他模式

这是面试和实际开发中最容易混淆的部分。

#### (1) 策略模式 vs 状态模式（State Pattern）
**结构几乎一样**（都有 Context 和 状态/策略类），但**意图不同**。

| 特性       | 策略模式 (Strategy)          | 状态模式 (State)                    |
| :------- | :----------------------- | :------------------------------ |
| **意图**   | **选择算法**。*客户端主动选择用哪个策略*。 | **改变行为**。对象内部状态变化导致行为变化。        |
| **依赖关系** | 策略之间通常**独立**，互不知道对方。     | 状态之间可能**相互依赖**（如从 A 状态转到 B 状态）。 |
| **变化原因** | *为了复用算法，消除 `if-else`*。   | 为了管理复杂的状态流转逻辑。                  |
| **例子**   | 支付方式（选支付宝还是微信）。          | 订单状态（待支付 -> 已支付 -> 已发货）。        |
| **口诀**   | **策略是“选哪个”，状态是“现在是啥”**。  |                                 |

#### (2) 策略模式 vs 工厂模式
*   **工厂模式**：关注**对象的创建**（怎么 `new` 出来）。
*   **策略模式**：关注**对象的使用**（怎么执行算法）。
*   **结合**：通常用**工厂模式创建策略对象**，然后交给**策略模式使用**。

#### (3) 策略模式 vs 装饰器模式
*   **装饰器**：动态**叠加**功能（如：日志 + 权限 + 业务）。
*   **策略**：动态**替换**功能（如：算法 A **或** 算法 B）。

#### (4) 策略模式 vs 职责链模式
*   **职责链**：*请求沿着链传递，直到有人处理*（可能多个处理，可能一个）。
*   **策略**：*明确选定一个策略执行*（通常只有一个）。

### 6. 前端开发中的实际应用

策略模式在前端非常普遍，尤其是表单验证和动画。

#### (1) 表单验证（最经典场景）
不同字段有不同的验证规则（必填、邮箱、手机号、长度）。

```typescript
// 策略接口
type Validator = (value: any) => string | null; // 返回错误信息或 null

// 具体策略
const required: Validator = (v) => (!v ? '不能为空' : null);
const isEmail: Validator = (v) => (/^\w+@\w+\.\w+$/.test(v) ? null : '邮箱格式错误');
const minLength = (len: number): Validator => (v) => (v.length < len ? `至少${len}字` : null);

// 上下文：表单字段
class FormField {
  constructor(private value: string, private validators: Validator[]) {}

  validate(): string[] {
    const errors: string[] = [];
    for (const validator of this.validators) {
      const error = validator(this.value);
      if (error) errors.push(error);
    }
    return errors;
  }
}

// 使用
const username = new FormField('abc', [required, minLength(5)]);
console.log(username.validate()); // ['至少 5 字']
```
**优势**：新增验证规则（如“必须是数字”），只需新增一个函数，无需修改 `FormField` 类。

#### (2) 动画缓动函数（Easing Functions）
不同的动画效果（匀速、加速、减速、弹性）。

```typescript
type EasingFunction = (t: number) => number;

const linear: EasingFunction = (t) => t;
const easeInQuad: EasingFunction = (t) => t * t;
const easeOutQuad: EasingFunction = (t) => t * (2 - t);

function animate(element: any, easing: EasingFunction) {
  // 使用 easing 计算位置
  const progress = easing(Date.now() / 1000);
  element.style.transform = `translateX(${progress * 100}px)`;
}
```

#### (3) 排序算法
`Array.prototype.sort()` 接收的比较函数就是一个策略。
```typescript
const numbers = [3, 1, 4];
numbers.sort((a, b) => a - b); // 升序策略
numbers.sort((a, b) => b - a); // 降序策略
```

#### (4) 登录/支付方案
根据环境选择登录方式（微信、支付宝、账号密码）。
```typescript
const loginStrategy = env === 'wechat' ? wechatLogin : passwordLogin;
loginStrategy(credentials);
```

### 7. 优缺点分析

#### ✅ 优点
1.  **消除条件语句**：完美替代复杂的 `if-else` 或 `switch`。
2.  **符合开闭原则**：新增算法无需修改现有代码。
3.  **算法复用**：策略类可以在不同上下文中复用。
4.  **易于测试**：每个策略可以独立单元测试。
5.  **避免继承爆炸**：不用为每种组合创建子类。

#### ❌ 缺点
1.  **类/文件数量增加**：每个策略一个类/文件，如果策略很多，文件会变多。
2.  **客户端需了解策略**：客户端必须知道有哪些策略可选，并决定使用哪一个（可以通过工厂模式缓解）。
3.  **策略开销**：如果策略很简单（如只是返回一个常数），创建对象可能有轻微开销（可用函数式策略优化）。

---

### 8. 最佳实践与注意事项

1.  **结合工厂模式**：不要让客户端直接 `new ConcreteStrategy`，使用工厂或配置对象来创建策略，进一步解耦。
2.  **函数式优先**：在 TypeScript/JavaScript 中，如果策略无状态，**优先使用函数**而不是类，减少样板代码。
3.  **策略注册表**：对于动态策略，可以使用 Map 注册策略，通过 Key 获取。
    ```typescript
    const strategyMap = new Map<string, DiscountStrategy>();
    strategyMap.set('VIP', new VIPDiscount());
    const s = strategyMap.get(type);
    ```
4.  **默认策略**：提供一个默认策略，防止传入无效 Key 导致崩溃。
5.  **状态 vs 策略**：如果行为变化是由**内部状态**驱动的（如订单状态流转），请用**状态模式**；如果是由**外部配置/用户选择**驱动的，请用**策略模式**。

---

### 9. 总结

**策略模式的核心价值在于“封装变化”与“算法替换”。**

*   **何时使用**：
    *   系统中有大量 `if-else` 或 `switch` 分支，且分支代表不同的算法或行为。
    *   同一个接口有多种实现，需要在运行时动态切换。
    *   希望算法独立于使用它的客户端变化。
*   **核心思想**：**组合优于继承**，**多态替代条件判断**。
*   **前端地位**：它是**表单验证库**、**动画库**、**排序/过滤逻辑**的底层设计思想。
*   **与 SOLID 的关系**：是 **OCP（开闭原则）** 和 **SRP（单一职责）** 的最佳代言人。
*   **TS 最佳实践**：
    *   简单逻辑用 **函数类型**。
    *   复杂逻辑用 **类 + 接口**。
    *   创建逻辑用 **工厂模式** 封装。

结合你之前的学习路径：
*   用 **单例模式** 管理无状态策略实例（节省内存）。
*   用 **工厂模式** 创建策略。
*   用 **装饰器模式** 增强策略功能（如给策略加缓存）。
*   用 **AOP** 记录策略执行日志。
*   区分 **状态模式**（内部驱动）与 **策略模式**（外部选择）。

掌握策略模式，意味着你掌握了**消除复杂条件判断**的终极武器，能够写出清晰、易扩展的业务逻辑代码。
