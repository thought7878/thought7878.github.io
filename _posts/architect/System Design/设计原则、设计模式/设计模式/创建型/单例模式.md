`单例模式（Singleton Pattern）` 是软件工程中创建型设计模式（Creational Pattern） 中最简单但也最常被讨论的一种。

它的`核心目标`非常明确：**确保一个类在整个应用程序生命周期中只有一个实例，并提供一个全局访问点。**

结合你之前学习的 工厂模式 和 SOLID 原则，*单例模式可以看作是对象创建的一种特殊约束（限制数量为 1）*。

---

## 1. 为什么要用单例模式？（解决的问题）

*在某些场景下，创建多个实例是没有意义的，甚至是有害的：*

1.  **资源消耗大**：例如数据库连接池、线程池。*创建多个池会浪费内存和系统资源*。
2.  **状态需要共享**：例如*Redux的Store对象*、配置管理器（Configuration）、缓存（Cache）。*如果多个实例存在，可能导致配置不一致或缓存命中率低*。
3.  全局控制点：例如日志记录器（Logger）。所有模块应该写入同一个日志流，而不是分散到多个文件句柄中。
4.  硬件交互：例如打印队列、显卡驱动。硬件本身通常是唯一的，软件映射也应唯一。

**核心意图**：*控制实例数量，节省资源，避免冲突*。

---

## 2. 单例模式的核心要素

要实现单例，必须满足三个条件：

1.  **私有构造函数（Private Constructor）**：*防止外部通过 new 关键字创建实例*。
2.  **静态私有实例（Static Private Instance）**：在类内部持有唯一的实例对象。
3.  **静态公共访问方法（Static Public Method）**：提供一个全局入口（如 getInstance()）来获取该实例。

---

## 3. 常见的实现方式（以 Java 为例）

单例模式的实现难点在于**多线程环境下的安全性**和**延迟加载（Lazy Loading）** 的平衡。

### (1) 饿汉式（Eager Initialization）
类加载时就创建实例，不存在线程安全问题，但可能浪费资源。

```java
public class Singleton {
    // 1. 类加载时就初始化
    private static final Singleton instance = new Singleton();
    
    // 2. 私有构造
    private Singleton() {}
    
    // 3. 全局访问点
    public static Singleton getInstance() {
        return instance;
    }
}
```
*   **优点**：简单，线程安全。
*   **缺点**：即使不使用该实例，也会占用内存（无法延迟加载）。

### (2) 懒汉式 + 双重检查锁（Double-Checked Locking, DCL）
最常用的手动实现方式。既支持延迟加载，又保证线程安全，且性能较好。

```java
public class Singleton {
    // volatile 关键字至关重要，防止指令重排序
    private static volatile Singleton instance;
    
    private Singleton() {}
    
    public static Singleton getInstance() {
        if (instance == null) { // 第一次检查（避免不必要的同步）
            synchronized (Singleton.class) {
                if (instance == null) { // 第二次检查（确保线程安全）
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```
*   **优点**：延迟加载，线程安全，性能高（只在第一次创建时同步）。
*   **缺点**：代码复杂，容易写错（忘记 `volatile` 会导致多线程下获取到未初始化完成的对象）。

### (3) 静态内部类（Static Inner Class）
利用 Java 类加载机制保证线程安全，推荐方式之一。

```java
public class Singleton {
    private Singleton() {}
    
    // 内部类在第一次被使用时才加载
    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }
    
    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```
*   **优点**：延迟加载，线程安全，代码简洁。
*   **缺点**：只能用于 Java 等支持内部类的语言。

### (4) 枚举（Enum）
**《Effective Java》作者 Joshua Bloch 推荐的最佳实践。**

```java
public enum Singleton {
    INSTANCE;
    
    public void doSomething() {
        // 业务逻辑
    }
}
// 使用：Singleton.INSTANCE.doSomething();
```
*   **优点**：天然线程安全，防止反射和序列化攻击（其他实现方式可能被反射破解创建新实例）。
*   **缺点**：不够灵活（无法继承，无法延迟加载复杂逻辑）。

## 4. 单例模式与 SOLID 原则的关系

这是一个非常关键的思考点。虽然单例模式很常用，但它经常与 SOLID 原则**冲突**。

| 原则 | 单例模式的表现 | 分析 |
| :--- | :--- | :--- |
| **S (单一职责)** | **可能违反** | 单例类既负责业务逻辑，又负责管理自己的生命周期（创建/销毁）。 |
| **O (开闭原则)** | **支持较差** | 单例类通常很难被扩展或替换。如果需要改变行为，往往需要修改单例内部代码。 |
| **L (里氏替换)** | **不支持** | 单例通常没有子类，无法通过子类替换父类。 |
| **I (接口隔离)** | **无关** | 单例关注实例数量，与接口粒度关系不大。 |
| **D (依赖倒置)** | **严重违反** | **这是最大的问题**。单例通过全局静态方法访问，导致调用者直接依赖具体类，而不是依赖抽象。这使得**单元测试非常困难**（无法 Mock 全局状态）。 |

**结论**：在现代开发中，过度使用单例模式被视为一种**“代码异味（Code Smell）”**，因为它引入了**全局状态（Global State）**，增加了耦合度。

---

## 5. 单例模式 vs. 工厂模式

你之前学习了工厂模式，这两者经常一起出现，但侧重点不同：

| 特性 | 单例模式 (Singleton) | 工厂模式 (Factory) |
| :--- | :--- | :--- |
| **核心目标** | 控制**实例数量**（只能有一个）。 | 控制**创建过程**（解耦创建与使用）。 |
| **返回结果** | 永远返回同一个对象引用。 | 每次可能返回新对象，也可能返回单例（取决于实现）。 |
| **结合使用** | 工厂类本身通常设计为单例（因为工厂不需要多个实例）。 | 工厂方法可以返回单例对象。 |

## 6. 现代开发中的单例（Spring 容器）

在传统的 Java SE 开发中，我们需要手动写上述代码来实现单例。

但在现代框架（如 **Spring**）中，**单例是由容器管理的**：

*   **Spring Bean 默认作用域**：`Singleton`。
*   **区别**：
    *   **传统单例**：类加载器级别，整个 JVM 只有一个实例。
    *   **Spring 单例**：容器级别，每个 Spring 容器（ApplicationContext）中只有一个实例。
*   **优势**：Spring 通过**依赖注入（DI）** 解决了传统单例违反 **依赖倒置原则（DIP）** 的问题。你不需要调用 `Singleton.getInstance()`，而是通过构造函数注入，这样更容易测试和替换。

```java
// Spring 风格（推荐）
@Component // Spring 默认就是单例
public class ConfigManager {
    // 业务逻辑
}

// 使用时通过注入，而不是全局调用
public class Service {
    @Autowired
    private ConfigManager configManager; 
}
```

## 7. 单例模式的优缺点总结

### ✅ 优点
1.  **内存节约**：避免频繁创建和销毁对象。
2.  **全局访问**：方便在系统任何地方获取实例。
3.  **控制并发**：对于共享资源（如连接池），单例便于统一管理锁和状态。

### ❌ 缺点
1.  **扩展困难**：单例类很难继承，难以通过子类改变行为。
2.  **隐藏依赖**：调用者看似没有依赖，实则依赖了全局状态，导致代码耦合度高。
3.  **测试困难**：单元测试需要隔离环境，但单例的状态会保留在内存中，导致测试用例相互影响（除非每次重置单例，但这很麻烦）。
4.  **并行开发限制**：全局状态可能成为多线程开发的瓶颈。

---

## 8. 什么时候使用单例？

**✅ 推荐使用：**
*   资源管理器（数据库连接池、线程池）。
*   配置信息加载器。
*   日志记录器。
*   硬件驱动交互类。

**❌ 不推荐使用（反模式）：**
*   **业务逻辑类**：不要为了图方便把业务类做成单例。
*   **需要携带状态的对象**：如果单例内部有可变状态（如 `currentUser`），在多线程环境下极易出错。
*   **为了测试方便**：应该使用依赖注入，而不是全局单例。

## 9. 总结

单例模式是设计模式中的“经典”，但也是一把**双刃剑**。

*   **初学者**：容易滥用，把它当作全局变量使用，导致代码难以维护。
*   **进阶者**：理解其背后的资源控制意图，并意识到它对测试和耦合的负面影响。
*   **高手**：在现代框架（如 Spring）中，利用容器的单例作用域 + 依赖注入，既享受了单例的资源优势，又避免了全局访问带来的耦合问题。

**最佳实践建议**：
1.  优先使用**枚举**实现单例（防止反射/序列化破坏）。
2.  在框架环境下，优先使用**依赖注入**管理单例，而不是手动写 `getInstance()`。
3.  时刻警惕**全局状态**带来的测试困难。
