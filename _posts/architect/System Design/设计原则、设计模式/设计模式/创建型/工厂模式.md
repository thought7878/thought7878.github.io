**工厂模式（Factory Pattern）** 是软件工程中**创建型设计模式（Creational Pattern）** 中最著名、最常用的一种。

它的核心思想非常直观：**将对象的创建过程封装起来，让使用者不需要关心对象是如何创建的，只需要关心对象的使用。**

通俗地说，就是 **“不要直接 `new` 一个对象，而是找工厂要”**。

结合你之前学习的 **SOLID 原则**，工厂模式是实现 **依赖倒置原则（DIP）** 和 **开闭原则（OCP）** 的重要手段。

---

### 1. 为什么要用工厂模式？（解决的问题）

在没有工厂模式时，我们通常这样创建对象：

```java
// 客户端代码
public class Client {
    public static void main(String[] args) {
        // 问题：客户端直接依赖了具体类 "MySQLDatabase"
        Database db = new MySQLDatabase(); 
        db.connect();
    }
}
```

**这种写法的问题：**
1.  **耦合度高**：客户端代码（`Client`）强依赖于具体实现类（`MySQLDatabase`）。
2.  **难以扩展**：如果明天要换成 `OracleDatabase`，你必须修改 `Client` 的代码，重新编译。这违反了 **开闭原则（OCP）**。
3.  **创建逻辑复杂**：如果创建对象需要复杂的配置（如读取配置文件、连接池初始化），把这些逻辑散落在各个 `new` 的地方会导致代码重复（违反 **DRY 原则**）。

**工厂模式的解决方案：**
引入一个“工厂”类，由它负责 `new` 对象，客户端只向工厂索要接口。

```java
// 客户端代码
public class Client {
    public static void main(String[] args) {
        // 解决：客户端只依赖抽象 "Database"，具体由工厂决定
        DatabaseFactory factory = new DatabaseFactory();
        Database db = factory.createDatabase("mysql"); 
        db.connect();
    }
}
```

### 2. 工厂模式的三种形态

工厂模式其实是一个“家族”，通常分为三种，复杂度依次递增：

1.  **简单工厂（Simple Factory）**
2.  **工厂方法（Factory Method）**
3.  **抽象工厂（Abstract Factory）**

#### (1) 简单工厂（Simple Factory）
*严格来说，它不属于 GoF 23 种设计模式，但非常常用。*

*   **结构**：一个工厂类，根据传入的参数（如字符串、枚举），通过 `if-else` 或 `switch` 决定创建哪个具体对象。
*   **场景**：创建逻辑简单，产品类型固定。
*   **代码示例**：

```java
// 1. 抽象产品
interface Payment {
    void pay();
}

// 2. 具体产品
class Alipay implements Payment { public void pay() { System.out.println("支付宝支付"); } }
class WechatPay implements Payment { public void pay() { System.out.println("微信支付"); } }

// 3. 简单工厂
class PaymentFactory {
    public static Payment createPayment(String type) {
        if ("alipay".equals(type)) {
            return new Alipay();
        } else if ("wechat".equals(type)) {
            return new WechatPay();
        }
        return null;
    }
}

// 4. 使用
Payment p = PaymentFactory.createPayment("alipay");
p.pay();
```
*   **缺点**：违反 **开闭原则（OCP）**。如果要新增“银联支付”，必须修改 `PaymentFactory` 的代码，增加新的 `if` 分支。

#### (2) 工厂方法（Factory Method）
*这是标准的 GoF 设计模式。*

*   **结构**：定义一个创建对象的接口（工厂接口），但让**子类**决定实例化哪一个类。工厂方法让类的实例化推迟到子类。
*   **核心**：**多态**。每个具体产品对应一个具体工厂。
*   **场景**：系统需要扩展新产品，且不希望修改现有代码。
*   **代码示例**：

```java
// 1. 抽象工厂
interface PaymentFactory {
    Payment createPayment();
}

// 2. 具体工厂
class AlipayFactory implements PaymentFactory {
    public Payment createPayment() { return new Alipay(); }
}

class WechatFactory implements PaymentFactory {
    public Payment createPayment() { return new WechatPay(); }
}

// 3. 使用
PaymentFactory factory = new AlipayFactory(); // 这里可以配置化
Payment p = factory.createPayment();
p.pay();
```
*   **优点**：符合 **开闭原则（OCP）**。新增“银联支付”时，只需新增 `UnionPay` 类和 `UnionPayFactory` 类，无需修改原有代码。
*   **缺点**：每增加一个产品，就要增加一个对应的工厂类，类数量翻倍。

#### (3) 抽象工厂（Abstract Factory）
*工厂模式的终极形态。*

*   **结构**：提供一个接口，用于创建**相关或依赖对象的家族**，而不需要明确指定具体类。
*   **核心**：**产品族**。不仅创建一个对象，而是创建一组配套的对象。
*   **场景**：系统中有多个产品族（如：Mac 风格 UI、Windows 风格 UI），每个产品族有多个产品（按钮、菜单、窗口），且客户端一次只能使用一个族的产品。
*   **代码示例**：

```java
// 1. 产品接口
interface Button { void render(); }
interface Menu { void show(); }

// 2. 具体产品族 - Mac
class MacButton implements Button { public void render() { System.out.println("Mac 风格按钮"); } }
class MacMenu implements Menu { public void show() { System.out.println("Mac 风格菜单"); } }

// 3. 具体产品族 - Windows
class WinButton implements Button { public void render() { System.out.println("Win 风格按钮"); } }
class WinMenu implements Menu { public void show() { System.out.println("Win 风格菜单"); } }

// 4. 抽象工厂
interface GUIFactory {
    Button createButton();
    Menu createMenu();
}

// 5. 具体工厂
class MacFactory implements GUIFactory {
    public Button createButton() { return new MacButton(); }
    public Menu createMenu() { return new MacMenu(); }
}

class WinFactory implements GUIFactory {
    public Button createButton() { return new WinButton(); }
    public Menu createMenu() { return new WinMenu(); }
}

// 6. 使用
GUIFactory factory = new MacFactory(); // 切换这里即可切换整套皮肤
Button btn = factory.createButton();
Menu menu = factory.createMenu();
```
*   **优点**：保证客户端始终使用同一产品族的产品（不会出现 Mac 按钮配 Windows 菜单的情况）。
*   **缺点**：难以支持“新产品等级结构”。例如，如果想新增一个 `TextField` 产品，需要修改所有工厂接口和实现类，违反开闭原则。

### 3. 工厂模式与 SOLID 原则的联系

你之前学习了 SOLID，工厂模式是这些原则的最佳实践载体：

| SOLID 原则 | 工厂模式如何体现 |
| :--- | :--- |
| **S (单一职责)** | 将“创建对象”的职责从“业务逻辑”中分离出来。业务类只负责用，工厂类负责造。 |
| **O (开闭原则)** | **工厂方法**和**抽象工厂**完美支持 OCP。新增产品时，只需新增类，无需修改客户端代码。 |
| **L (里氏替换)** | 工厂返回的是抽象接口（如 `Payment`），具体实现类（如 `Alipay`）可以无缝替换。 |
| **I (接口隔离)** | 工厂接口通常很精简，只暴露创建方法，不暴露具体实现细节。 |
| **D (依赖倒置)** | **这是最核心的联系**。客户端依赖抽象（接口），工厂负责注入具体实现。实现了“高层模块不依赖低层模块”。 |

### 4. 三种工厂模式的对比总结

| 特性 | 简单工厂 (Simple Factory) | 工厂方法 (Factory Method) | 抽象工厂 (Abstract Factory) |
| :--- | :--- | :--- | :--- |
| **本质** | 一个类 + 条件判断 | 多态（子类决定） | 产品族（一套对象） |
| **扩展性** | 差（改代码） | 好（加类） | 中（加产品族好，加新产品难） |
| **复杂度** | 低 | 中 | 高 |
| **适用场景** | 逻辑简单，几乎不扩展 | 单一产品层级，需频繁扩展 | 多个产品层级，需保证配套 |
| **形象比喻** | **自动售货机**：投币选号，机器吐出商品。 | **连锁店加盟**：每个分店（子类）自己决定进什么货。 | **全屋定制**：要么全套中式家具，要么全套欧式家具。 |

### 5. 什么时候使用工厂模式？

✅ **推荐使用：**
1.  **不知道具体需要哪个类**：代码在编译时无法确定需要实例化哪个具体类（例如根据配置文件、用户输入决定）。
2.  **希望解耦**：希望客户端代码独立于具体类的创建、表示和实现。
3.  **创建过程复杂**：对象初始化需要很多步骤（如读取配置、连接资源），希望隐藏这些细节。
4.  **统一管理**：希望集中控制对象的创建（例如单例模式、对象池模式通常结合工厂使用）。

❌ **不推荐使用：**
1.  **业务简单**：如果只是一个简单的 `new`，没必要引入工厂，否则是**过度设计（违反 KISS/YAGNI）**。
2.  **产品极少变化**：如果系统永远不会新增产品类型，工厂模式只会增加类的数量，增加维护成本。

### 6. 现实世界中的工厂模式

*   **Java 的 `Calendar.getInstance()`**：你不知道返回的是 `GregorianCalendar` 还是其他子类，你只知道它是 `Calendar`。
*   **JDBC 的 `DriverManager.getConnection()`**：你传入 URL，它返回具体的数据库连接对象，你不需要 `new MySQLConnection()`。
*   **Spring 框架的 BeanFactory**：Spring 容器本质上就是一个巨大的工厂，负责创建和管理所有对象的生命周期（依赖注入的核心）。
*   **前端框架**：React 或 Vue 创建组件时，底层也是通过工厂机制来实例化组件树。

### 7. 总结

**工厂模式的核心价值在于“解耦”**。

它把 **“用什么”** 和 **“怎么造”** 分离开来。
*   对于**使用者**（客户端）：我只需要知道接口，不需要知道具体类名。
*   对于**维护者**：新增功能时，我只需要加新类，不需要改旧代码。

结合你之前学的 **SRP**（创建逻辑独立）、**OCP**（扩展无需修改）、**DIP**（依赖抽象），工厂模式是构建灵活、可维护架构的基石之一。
