**单一职责原则（Single Responsibility Principle，简称 SRP）** 是*面向对象设计五大原则（SOLID）中的第一个*，也是最基础、最重要的原则之一。

以下是对该原则的详细解释，包括定义、核心思想、代码示例、优缺点以及实践建议。

---

### 1. 定义
**“一个类应该只有一个发生变化的原因。”**
*(A class should have one, and only one, reason to change.)*

这句话由“面向对象设计之父”罗伯特·马丁（Robert C. Martin）提出。它意味着一个类应该只负责一项职责，或者说，一个类应该只承担一个领域的功能。

### 2. 核心思想解读
*   **什么是“职责”？** 职责可以理解为“变化的轴”。如果因为某个需求的变化，导致你需要修改这个类，那么这个需求就代表了该类的一个职责。
*   **什么是“单一”？** 并不是说一个类只能有一个方法，而是说类中的所有方法都应该服务于同一个业务目标或功能模块。
*   **通俗比喻：**
    *   **违反 SRP：** 就像让一个人同时兼任“公司 CEO"、“财务会计”和“保洁员”。如果公司战略变了，改 CEO 的职责；如果税法变了，改会计的职责；如果扫地机器人坏了，改保洁的职责。这个人（类）会因为太多不相关的原因被修改，容易出错且难以管理。
    *   **遵循 SRP：** CEO 只管战略，会计只管账目，保洁只管卫生。各司其职，互不干扰。

### 3. 代码示例

#### ❌ 违反 SRP 的例子
假设我们有一个 `User` 类，它既负责存储用户数据，又负责将用户数据保存到数据库，还负责给用户发送欢迎邮件。

```java
// 违反 SRP 的类
class User {
    public String name;
    public String email;

    // 职责 1: 数据验证
    public boolean validateEmail() {
        return email.contains("@");
    }

    // 职责 2: 数据库操作
    public void saveToDatabase() {
        // 连接数据库，执行 SQL...
        // 如果数据库结构变了，这个类要改
    }

    // 职责 3: 发送通知
    public void sendWelcomeEmail() {
        // 调用 SMTP 服务...
        // 如果邮件服务商变了，这个类要改
    }
}
```
**问题：** 这个类有三个变化的原因（验证逻辑变、数据库变、邮件服务变）。如果我想修改数据库连接方式，可能会不小心影响到邮件发送的代码。

#### ✅ 遵循 SRP 的例子
我们将上述职责拆分成不同的类。

```java
// 职责 1: 纯数据对象
class User {
    public String name;
    public String email;
}

// 职责 2: 数据验证
class UserValidator {
    public boolean isValid(User user) {
        return user.email.contains("@");
    }
}

// 职责 3: 数据持久化
class UserRepository {
    public void save(User user) {
        // 只负责数据库操作
    }
}

// 职责 4: 通知服务
class EmailService {
    public void sendWelcome(User user) {
        // 只负责邮件发送
    }
}
```
**好处：** 现在，如果数据库变了，只改 `UserRepository`；如果邮件规则变了，只改 `EmailService`。`User` 类保持稳定。

### 4. 遵循 SRP 的好处
1.  **降低耦合（Low Coupling）：** 类与类之间的依赖关系变弱，修改一个模块不会轻易波及另一个模块。
2.  **提高内聚（High Cohesion）：** 相关的功能聚集在一起，代码逻辑更清晰。
3.  **易于测试：** 测试一个只负责“发邮件”的类，比测试一个既管数据库又管发邮件的类要容易得多（Mock 依赖更简单）。
4.  **并行开发：** 不同的开发人员可以负责不同的职责类，减少代码冲突。
5.  **可维护性高：** 代码意图明确，新人更容易上手。

### 5. 常见误区与注意事项

#### 误区 1：一个类只能有一个方法
**纠正：** 不是的。一个类可以有多个方法，只要这些方法都是为了完成**同一个业务目标**。例如，`List` 类有 `add`, `remove`, `get` 等多个方法，但它们的职责都是“管理列表数据”，所以符合 SRP。

#### 误区 2：过度拆分（Over-engineering）
**纠正：** 不要为了追求 SRP 而把类拆得细碎无比。如果一个类目前很稳定，且没有明显的不同变化轴，强行拆分只会增加系统的复杂度（需要维护更多的文件和类关系）。
*   **原则：** 当且仅当**不同的职责表现出不同的变化趋势**时，才进行拆分。

#### 误区 3：所有方法都必须公开
**纠正：** 遵循 SRP 的类，其公开接口（Public Methods）应该只暴露其核心职责。辅助方法可以是私有的（Private）。

### 6. 如何识别是否违反了 SRP？
如果你发现一个类出现以下“代码异味（Code Smell）”，通常意味着违反了 SRP：
*   **上帝类（God Class）：** 一个类包含了数百行甚至数千行代码，几乎什么都做。
*   **方法过多：** 一个类有几十个公开方法，且这些方法看起来属于不同的业务领域。
*   **依赖过多：** 构造函数或成员变量依赖了太多不相关的服务（如同时依赖 `Database`, `Email`, `FileSystem`, `Network`）。
*   **注释分类：** 代码里出现类似 `// 用户相关方法`、`// 数据库相关方法` 这样的注释分隔，通常暗示应该拆分成不同的类。

### 7. 总结
单一职责原则的核心在于 **“关注点分离”**。

*   **目的：** 让代码更健壮、更易修改。
*   **判断标准：** 问自己，“如果需求 X 变了，我需要改这个类吗？如果需求 Y 也变了，我也需要改这个类吗？”如果是，那么它可能承担了太多职责。
*   **平衡：** 设计是权衡的艺术。不要教条地执行 SRP，而应根据项目的复杂度和未来的变化可能性来灵活应用。

遵循 SRP 是写出高质量、可扩展代码的第一步。
