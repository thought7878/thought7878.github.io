本视频讲述了前端开发中自定义事件的内存泄漏风险及正确解绑方法，重点强调组件销毁前必须使用同一函数引用调用off以避免因匿名函数导致解绑失效，并对比说明发布订阅模式与观察者模式的区别及其实际应用场景。


### **自定义事件及时解绑的重要性 [00:00](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=a191ff84cab54538af78c790dfebe79e#?seek_t=0)**

- **内存泄漏风险警示 [00:06](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=a191ff84cab54538af78c790dfebe79e#?seek_t=6)**
    - 在 Vue/React 组件中，自定义事件必须在组件销毁前（如 `beforeUnmount`）调用 `off` 解绑，否则引发隐性内存泄露。
    - 内存泄露表现为系统运行渐慢、最终崩溃，且极难定位。

### **解绑操作的核心规范 [00:53](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=a191ff84cab54538af78c790dfebe79e#?seek_t=53)**

- **函数引用一致性原则 [03:53](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=a191ff84cab54538af78c790dfebe79e#?seek_t=233)**
    - `off` 必须传入与 `on` 绑定时**完全相同的函数引用**，不可使用结构相同但独立创建的匿名函数。
- **匿名函数陷阱演示 [04:29](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=a191ff84cab54538af78c790dfebe79e#?seek_t=269)**
    - 两个语法一致的匿名函数（如 `() => console.log(1)` 与 `() => console.log(1)`）在 JavaScript 中是**不同对象**，`===` 比较结果为 `false`。
    - 因此 `on('change', () => {...})` 后执行 `off('change', () => {...})` **无法解绑**。

### **正确解绑的实现方案 [04:50](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=a191ff84cab54538af78c790dfebe79e#?seek_t=290)**

- **方法抽离法 [04:55](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=a191ff84cab54538af78c790dfebe79e#?seek_t=295)**
    - 在组件 `methods` 中定义具名处理函数：
        
        js
        
        ```js
        methods: {  
          handleChange() { console.log('changed'); }  
        }
        ```
        
    - 绑定时传入 `this.handleChange`，解绑时同样传入 `this.handleChange`。
- **引用一致性验证 [05:20](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=a191ff84cab54538af78c790dfebe79e#?seek_t=320)**
    - `this.handleChange` 在绑定与解绑中指向**同一函数对象**，确保 `off` 生效。

    - 若组件中使用 `setInterval` 或 `setTimeout`，必须在 `beforeUnmount` 中对应调用 `clearInterval` 或 `clearTimeout`。

### **发布订阅模式解析 [06:19](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=a191ff84cab54538af78c790dfebe79e#?seek_t=379)**

- **与观察者模式的本质区别 [06:27](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=a191ff84cab54538af78c790dfebe79e#?seek_t=387)**
    - 观察者模式：Subject 与 Observer **直接双向关联**。
    - 发布订阅模式：发布者（Publisher）与订阅者（Subscriber）**通过 Event Bus 解耦**，仅依赖事件通道通信。
- **模式定位说明 [06:42](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=a191ff84cab54538af78c790dfebe79e#?seek_t=402)**
    - 发布订阅模式是观察者模式的延伸，非 GoF 23 种经典设计模式之一，但实践价值明确。
- **核心应用场景 [06:53](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=a191ff84cab54538af78c790dfebe79e#?seek_t=413)**
    - 自定义事件通信（组件间松耦合）；
    - `postMessage` 跨 iframe/窗口通信；
