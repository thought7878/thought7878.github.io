    - **流的概念类比 [02:48](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=f9cedfe7f47d4c2781ece143cbd16a47#?seek_t=168)**  
        以“大水桶倒水”类比文件读取：直接加载整个文件需巨大内存（如人力抱桶），成本高昂；而流（Stream）如同连接两桶的管道，允许数据分块持续流动，仅需少量内存缓冲，显著降低硬件成本。
    - **流的适用场景 [04:54](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=f9cedfe7f47d4c2781ece143cbd16a47#?seek_t=294)**  
        处理超大文件（如百万行日志）时，传统方式需将全部内容载入内存，导致内存需求与文件大小线性正相关（如几十GB文件需几十GB内存），而流式读取按批次处理，内存占用恒定且极小。
    - **Stream代码实现与观察者模式体现 [07:20](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=f9cedfe7f47d4c2781ece143cbd16a47#?seek_t=440)**
        - 使用`fs.createReadStream()`创建读取流；
        - `.on('data', chunk => { length += chunk.toString().length })`监听数据块到达事件，每次触发回调累加当前块字符长度；
        - `.on('end', () => console.log(length))`监听流结束事件并输出总长度；
        - 执行结果为二百多万字符，验证分块读取过程：每次`data`事件触发约6万字符的chunk，多次触发后累加至总量；
        - 此机制与定时器、点击事件完全一致：事件发生（data/end）→ 自动调用注册回调 → 无需主动轮询。

    - **Readline的必要性 [11:49](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=f9cedfe7f47d4c2781ece143cbd16a47#?seek_t=709)**  
        基础Stream按固定字节块读取，无法保证行边界完整性（如长日志行可能被截断为半行或跨块），故需按逻辑行单位处理。
    - **Readline代码实现与观察者模式体现 [12:36](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=f9cedfe7f47d4c2781ece143cbd16a47#?seek_t=756)**
        - 引入`readline`模块，`const rl = readline.createInterface({ input: fs.createReadStream(...) })`；
        - `.on('line', () => lineNumber++)`监听每行到达事件，逐行累加行号；
        - `.on('close', () => console.log(lineNumber))`监听流关闭事件并输出总行数（实测50181行）；
        - 其底层仍基于Stream，仅事件名由`data`/`end`变为`line`/`close`，本质仍是观察者模式：事件驱动、回调注册、自动触发。

    - **HTTP Server基础代码 [15:46](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=f9cedfe7f47d4c2781ece143cbd16a47#?seek_t=946)**
        - 定义回调函数`serverCallback(req, res)`，其中`req.url`获取请求路径，`res.end('hello')`返回响应；
        - `http.createServer(serverCallback).listen(8081)`创建服务器并监听端口；
        - 浏览器访问`http://localhost:8081`返回"hello"，访问`/api/list`则`req.url`为`/api/list`。
    - **观察者模式体现 [17:44](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=f9cedfe7f47d4c2781ece143cbd16a47#?seek_t=1064)**  
        `createServer()`传入回调函数，该函数仅在客户端发起HTTP请求时被自动调用，请求次数决定回调执行次数；此机制与DOM点击事件绑定回调完全一致：事件（请求/点击）发生→ 系统自动调用预设回调→ 无事件则不执行。
- **总结：观察者模式的普适性 [18:18](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=f9cedfe7f47d4c2781ece143cbd16a47#?seek_t=1098)**  
    定时器、DOM事件、Promise、Stream、Readline、HTTP Server等所有异步场景，其共同本质是：主体（被观察者）维护回调函数列表，当特定事件（timer fire / click / resolve / data / line / request）发生时，自动遍历并执行对应回调。该模式贯穿前端与Node.js全栈开发，是异步编程的基石范式。
