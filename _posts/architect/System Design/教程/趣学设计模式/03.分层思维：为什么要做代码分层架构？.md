- 聚焦点声明 [02:36](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=55d5f60647984c89a08a2ca3040259df#?seek_t=156)  
    开发者更关心*应用程序内部的代码分层架构，而非DB等其他系统分层*。
- **代码分层架构定义** [03:03](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=55d5f60647984c89a08a2ca3040259df#?seek_t=183)  
    *将软件元素（代码）按“层”（即代码间关系）组织起来的结构*。
- **核心原则：单向数据流** [03:16](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=55d5f60647984c89a08a2ca3040259df#?seek_t=196)  
    请求或数据必须自上而下传递，如View → Controller → Model → 数据库。
- **反例** [03:33](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=55d5f60647984c89a08a2ca3040259df#?seek_t=213)  
    禁止跨层调用（如View直连Model），否则将导致：
    - Model层变更（如接口修改、逻辑调整）同时影响Controller层与View层；
    - 破坏层内变化封闭性。
- ***分层的本质*：** *让相似的变化在各自层内变化，而不造成层与层之间的相互影响*

### 代码分层架构解决什么问题？（两大核心价值）
[04:02](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=55d5f60647984c89a08a2ca3040259df#?seek_t=242)

- **价值一：拆解复杂功能问题** [04:15](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=55d5f60647984c89a08a2ca3040259df#?seek_t=255)
    - *功能需求常为笼统复杂问题，需逐层分解为子问题*。
    - 示例：HTTP发送字符串 → 拆解为：① 创建HTTP连接；② 发送字符串；③ 关闭HTTP连接。
    - 进一步分解：创建连接 → 打开连接 → 建立会话 → 使用TCP协议 → 网络传输。
    - 结果：*大问题转化为多个易解子层问题*；部分子问题（如HTTP协议通信）已被前人解决，实际需关注问题减少。
- **价值二：提升代码可扩展性** [06:17](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=55d5f60647984c89a08a2ca3040259df#?seek_t=377)
    - 动因：真实系统负载持续增长（如电商并发量从1万→1000万）。
    - 单体架构缺陷：一处修改波及全局；扩展困难。
    - 分层应对：
        - 将复杂逻辑切分为多层，大问题变小问题；
        - 小问题易抽象为组件，组件扩充/替换影响可控；
        - 提升组件复用性；
        - 支持横向扩展：如将Model层抽象为独立数据库服务集群部署，负载增加时快速扩容，且不影响业务层。`横向扩展`：用多台配置较低的服务器共同提供服务，也就是集群部署服务方式

### 分层架构的优势
[07:31](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=55d5f60647984c89a08a2ca3040259df#?seek_t=451)

- **优势：** [07:31](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=55d5f60647984c89a08a2ca3040259df#?seek_t=451)
    - 仅需关注整个结构中的某一层的具体实现；
    - 降低层与层之间的依赖；
    - 易于用新实现替换旧层的实现；
    - 有利于标准化的统一；
    - 各层逻辑方便复用。
- **目标：** [07:52](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=55d5f60647984c89a08a2ca3040259df#?seek_t=472)  
    *实现职责分离、解耦、组件复用、标准制定*。
- **反面例证：** [07:58](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=55d5f60647984c89a08a2ca3040259df#?seek_t=478)  
    *不分层则逻辑紧耦合，修改一处必影响多处*；如上传功能未分层，修改存储路径需同步改数据库连接逻辑，维护成本极高。
- **接口标准化机制** [08:25](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=55d5f60647984c89a08a2ca3040259df#?seek_t=505)  
    层即组件，通过统一接口交互（非功能依赖）；接口是模块间约定标准，只要符合标准，实现变更不影响调用。

    请求需经多层处理，执行路径延长，性能损耗不可避免。
- **劣势三：调用链复杂化** [09:05](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=55d5f60647984c89a08a2ca3040259df#?seek_t=545)  
    层间强耦合导致组合功能调用需大量层间跳转，代码复杂度上升。
