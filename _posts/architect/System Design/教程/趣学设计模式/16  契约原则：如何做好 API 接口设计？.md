    - **问题一：API期望的是什么？** [02:26](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=923886033544463fac18a69af7b551ec#?seek_t=146)  
        API必须要保证输入是接收者期望的输入条件。这里的重点就在于“期望”两个字。换句话说就是使用者使用API的前提条件是什么？只有满足了这个前提条件，API才能提供正常的功能。比如API需要ABC3个参数，那么使用者就需要提前准备好这三个参数。好的API接口都会提前告知使用者他需要什么，不需要什么。当条件不满足时他会拒绝，当条件满足时才会开始处理。
    - **问题二：API要保证的是什么？** [03:17](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=923886033544463fac18a69af7b551ec#?seek_t=197)  
        API必须要保证输出结果的正确性。API在处理过程中可能会遇见各种异常或错误的情况，这时API就不应该把错误或异常抛给其他系统去处理，而是在内部就得做好异常处理，并最终输出正确结果给使用者。
    - **问题三：API要保持不变的是什么？** [03:38](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=923886033544463fac18a69af7b551ec#?seek_t=218)  
        API必须要保持处理过程中的一致性。比如同一个API被部署在十个服务器上，那么只要外部输入了正确条件后，每一个API内部的处理过程就应该是相同的、不变的。也就是说当修改源代码后，你应该要同时部署这十个服务，才能让API整体的服务看起来是一个整体。不变条件通常会有绘画、信息共享的上下文、数据状态等。所谓不变你可以理解为多个相同的副本对同一个代码含义的统一解释。

    - **技巧一：接口职责分离** [05:51](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=923886033544463fac18a69af7b551ec#?seek_t=351)  
        在前面第十二讲中我们讲过接口隔离原则，它是能帮助你做好接口职责分离的一个好方法。当你在设计API的时候，应该尽量让每一个API只做一个职责的事情。这样做的好处在于能保证API功能的简单易用性和稳定性。一旦接口直接混在一起，就会造成接口间功能相互影响。假设一个API既用于修改商品的价格，又用于修改订单优惠价格，还用于修改库存数量等。那么当调用修改商品价格的接口流量增大时，势必会影响同一个接口中修改订单或修改库存的功能。在使用接口隔离原则时要注意，尽量找到变化相似的职责来进行封装。不要在一个接口中加很多方法，方法越多越有可能变成多个职责，那么就无法实现真正的接口隔离。
    - **技巧二：API命名规范** [06:50](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=923886033544463fac18a69af7b551ec#?seek_t=410)  
        好的API设计应该给使用者提供一种简单直观一致的体验，其中最关键的就是对API的命名。常用的命名技巧有以下三种：一使用英文的小写，例如使用美式英语的license color，并且尽量使用小写，因为大写稍不注意就会看错。2、使用被大家都接受的通用缩写，比如API就比application programming interface好。三通过命名含义描述接口，换句话说API的名字应该能自解释。比如list user in结合其英文字面含义就知道是查询用户信息的列表。当然现在也有很多自动生成接口说明的工具，比如swagger。但是对于开发者来说，依然不如直接阅读命名来的方便。并且像swagger这类的工具一旦暴露到公网，还会引发安全问题。
    - **技巧三：少用自定义错误码** [07:49](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=923886033544463fac18a69af7b551ec#?seek_t=469)  
        在API设计中很多人会常犯一个错误，那就是想方设法的创建自己的error code，用以表达自己对不同错误的个性化理解。但实际上在API设计中，这是一种错误应用管理原则的实践。在前面14管理原则如何提升编程中的沟通效率一文中，我们知道自定义管理的风险就在于各自理解可能有偏差，进而导致业务更大的混乱。比如当使用方在使用HTTPAPI时获得了一个404错误码。在HTTP中这是未找到资源的错误。而这时你恰好也定义了404错误码，假设代表接口的数据未找到，那结果必然出现冲突。你可能会说自定义的l code有助于传递信息，但这有一个前提条件，必须有系统且能对自定义的air code进行处理，这样才有意义。如果只是传递信息的话，error message里面的字段也可以达到同样的效果。因为当API出现错误的时候，API的使用方是否可以清晰的理解错误信息是非常重要的。比如以下几种常见的rist，API返回阶层数据形式。可以看到第三种形式就能很好的传递内部出现故障的信息。而第一种和第二种格式看上去虽然信息更多，但是很容易造成理解上的偏差。第一种格式里success和code作用有重合，这会让使用者不知道到底是应该看code还是看success，又或者两者都要看。至于第二种格式，不同的人对于叉叉叉system error可能理解完全不同。
    - **技巧四：同一接口幂等性保障** [09:36](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=923886033544463fac18a69af7b551ec#?seek_t=576)  
        什么是幂等呢？简单来说就是当一个操作多次执行所产生的影响均与一次执行的影响相同，则它是幂等的。幂等是一种通用策略。实际上HTTP规范中就明确指出，get put和delete方法必须是幂等的，但post的方法不一定保证是幂等的。如果post的方法创建新资源，通常不能保证此操作是幂等的。为什么要实现幂等呢？这是因为现代服务越来越多的往分布式服务方向发展，而分布式服务的特性就在于服务的分散性。分散性的服务会带来一个问题，多个服务在同一时间可能会并行修改同一份数据。比如你在楼下超市买完一瓶水后进行微信支付，第一次支付时网络延迟，于是你又刷了一次，那么钱应该只减1份。如果扣两次，那就是没有实现幂等。那么如何在API设计中做到接口幂等呢？通常有以下5种方法：一使用天然幂等的操作，比如数据库中的select查询，只要数据没发生改变，那么查询一次和多次的结果始终是一样的。还有delete删除操作，删除一次和多次删除都是把数据删除了，影响是一样的，典型的天然币等操作。2、使用唯一键值，比如在数据库中加唯一索引或是使用UUID做唯一ID都可以防止在新增数据时出现不必要的脏数据。因为不同的服务节点操作数据时都收到了唯一ID的约束。3、使用枷锁测验，比如悲观锁、乐观锁、分布式锁等。枷锁的目的就是让不同的服务在同一个数据变更时。不被其他服务所影响。比如订单服务API部署了500个相同实例，那么即便通过NGX网关做了负载均衡的流量分配，在修改订单的时候如果不加锁，就会导致数据被重复多次的修改，也就是无法保证幂的使用source加token及验证机制，这和使用微键值有一点类似。不过这种方法更多用在对外提供的API中去保证幂等性。这两个字段实际上既做了联合的唯一缩影，又做了使用来源的日志记录。这样既能保证接口的幂等信，也能记录不同客户端使用API的调用情况。五使用有限状态机。在一些状态变更比较频繁的业务中，会经常使用到状态机，比如订单支付、秒杀等业务。当状态机已经处于下一个状态，这时候又来了一个上一个状态的变更，那么这时就不允许再进行状态变更了。正是通过这种状态流转的约束，保证了接口服务的幂等性。
    - **技巧五：内外系统差异化安全策略** [12:29](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=923886033544463fac18a69af7b551ec#?seek_t=749)  
        在做API设计时，对于内部系统和外部系统的API所考虑的安全策略会有所不同。对于内部系统来说，更多的是考虑输入与输出数据的准确性，通常都采用了更为基础的安全测验。比如对接受的数据要做足够的验证，出现错误要能及时抛出异常并进行异常处理。当然对于一些核心镜头，比如财务订单、用户数据等，依然还是需要进行安全加固，避免外部被供货后内部数据的泄露。对于外部系统来说，API面临的主要挑战是来自外部的一些针对安全上的攻击，错误调用接口滥用的。在设计的时候，针对黑客攻击一般是购买业界一些专门做安全的防护公司的产品。针对错误的调用方式，API在寓意是条件判断时，就不应该让调用进行处理，一定要及时给出错误信息。对于接口滥用的情况，就需要做一些限流的措施。
    - **技巧六：API版本管理** [13:27](https://b.quark.cn/apps/5AZ7aRopS/routes/mofb35Rkb?debug=0&fid=923886033544463fac18a69af7b551ec#?seek_t=807)  
        从本质上来讲，API是服务提供者与服务使用者之间的一种合同协议。一旦服务提供者对API进行更改，则有破坏服务使用者使用API的风险。因此你在做API设计时，一定要考虑更改次数所带来的风险。另外API还可能因为不断升级优化而出现旧功能与新功能兼容的问题。实际上解决这两个问题的办法就是对你的API进行版本控制和管理。当需要进行重大的API更改时要提供一个完整的新版本，同时还要继续支持以前的版本。这里有两种简单的办法，一种是在同一服务API中公开两个版本，比如在接口中使用V一和V2。另一种是并行运行，两个版本的服务可以通过增加API网关来根据路由规则将请求发送到对应版本中。不过要注意的是，同时维护两个或多个版本会提升维护成本，因此在版本管理时最好提前计划一个旧版本的下线时间。对于内部API来说，要及时通知使用方进行升级和迁移。对于外部公共API则只能不断告知和减缓升级的周期，来让使用方及时跟上升级的节奏。

        不过在现实中你会发现很多API连基本三要素都没做好。要么是不做输入校验，要么就是补货异常，不处理不给出错误信息，或者没按照约定输出结果，或者扩展后不保证事务一致性的。为此我还专门总结了六大关键技巧。在我看来要想设计好API接口，除了通用的方法和技巧外，你还得时不时回到本质的三要素上去思考，看看你还有没有没想到的点，这样才能真正的找到适合你自己的API设计方法。

        原则虽然很好记忆，但是应用时其实不如模式好把握度。这时就需要你反复实践再思考这个原则，甚至需要试着去打破这个原则，看看在实际编程中会带来什么负面效果，才能找到正确应用的路径。原则更多的是给你启发和思考，而不是非要生搬硬套他们。不过随着我们继续深入学习后面的模式，你会渐渐发现模式里包含了很多原则思想是有原因的。
