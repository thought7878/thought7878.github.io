## 二、系统架构总览

### 2.1 架构模式

本项目采用 **SPA（单页应用）** 架构，前端独立运行，通过 RESTful API 与后端交互。数据层采用 Mock 数据 + API 接口定义双轨方案，支持前端独立开发和联调。

```
┌─────────────────────────────────────────────┐
│                  浏览器端                     │
├─────────────────────────────────────────────┤
│  React + TypeScript 应用                     │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │  路由层   │  │  状态层   │  │  组件层   │   │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘   │
│       └─────────────┼─────────────┘          │
│                     ▼                         │
│              API 请求层（Axios）              │
│                     │                         │
├─────────────────────┼─────────────────────────┤
│                     ▼                         │
│         ┌───────────────────────┐             │
│         │   Mock Service Worker  │ (开发环境)  │
│         └───────────────────────┘             │
│                     │                         │
└─────────────────────┼─────────────────────────┘
                      ▼
              ┌───────────────┐
              │   后端 API     │ (生产环境)
              └───────────────┘
```

### 3.2 模块详细设计

#### M0. 认证与权限模块

| 属性     | 说明                   |
| ------ | -------------------- |
| **职责** | 用户登录认证、Token 管理、权限校验 |
| **输入** | 用户凭证（账号/密码/验证码）      |
| **输出** | 认证 Token、用户角色信息、权限树  |

**核心组件：**
- `LoginForm` — 登录表单（账号密码/验证码双模式）
- `RegisterForm` — 注册表单
- `AuthRoute` — 路由级权限守卫
- `PermissionGuard` — 组件级权限守卫

**权限模型：** RBAC（基于角色的访问控制）

```
用户 ──N:M── 角色 ──N:M── 权限
  │
  └── 数据权限（店铺维度/角色维度）
```

**角色与权限映射：**

| 角色    | 菜单权限  | 数据权限 | 操作权限 |
| ----- | ----- | ---- | ---- |
| 超级管理员 | 全部    | 全量数据 | 全部   |
| 店铺管理员 | 全部    | 本店数据 | 全部   |
| 运营人员  | 运营相关  | 本店数据 | 部分操作 |
| 数据分析师 | 全部查看类 | 本店数据 | 仅查看  |

**Token 策略：**
- Access Token + Refresh Token 双 Token 机制
- Access Token 有效期 2 小时
- Refresh Token 有效期 7 天
- *静息刷新*：过期前自动刷新

#### M2. 主题与国际化模块

| 属性 | 说明 |
|-----|------|
| **职责** | 主题切换（亮/暗模式）、自定义主题色、多语言切换 |
| **被依赖** | 所有业务模块 |

**核心组件：**
- `ThemeProvider` — 主题上下文 Provider
- `ThemeSwitcher` — 主题切换 UI
- `LanguageSwitcher` — 语言切换 UI
- `ConfigProvider` — Ant Design 全局配置

**主题方案：**
- 基于 Ant Design 5.x 的 Design Token 系统
- 亮/暗模式：`theme.algorithm.defaultAlgorithm` / `theme.algorithm.darkAlgorithm`
- 自定义主题色：通过 `token.colorPrimary` 动态注入
- CSS 变量兜底：自定义样式通过 CSS 变量适配主题

**国际化方案：**
- `react-i18next` 管理翻译资源
- 语言包按模块组织：`locales/{zh-CN,en-US}/modules/*.json`
- 日期/数字格式化：`dayjs` locale 插件

**导出流程：**

```
1. 选择数据模块（流量/交易/商品/用户/营销/行业）

3. 选择数据维度和筛选条件
4. 选择导出格式（Excel/CSV）
5. 提交任务，后台异步生成
6. 完成后通知用户下载（≤100MB）
```

**实现方案：**

- 移动端侧边栏改为 Drawer 抽屉式

### 9.2 性能优化策略

| 优化项 | 方案 |
|-------|------|
| **首屏加载 ≤ 2s** | 路由懒加载、组件按需引入、Gzip 压缩、资源分包 |
| **页面切换 ≤ 500ms** | 路由预加载、数据缓存（SWR 模式） |
| **图表渲染 ≤ 1s** | ECharts 按需引入、大数据抽样渲染、Web Worker 处理 |
| **实时刷新 2s** | 增量数据更新、防抖合并、WebSocket/SSE 优先 |
| **内存控制** | 组件卸载时清理定时器/订阅、图表实例 dispose |

**Vite 分包配置：**

```
vendor-utils/    → dayjs, lodash 等
```

## 十、关键设计决策

### 10.1 状态管理选型：Zustand

| 方案 | 优势 | 劣势 |
|-----|------|------|
| Redux Toolkit | 生态完善、DevTools 强大 | 样板代码多、学习成本高 |
| **Zustand（选）** | API 简洁、无 Provider 包裹、TypeScript 友好 | 生态较小、大型团队经验少 |

选择 Zustand 的原因：
- 本项目模块独立性强，各模块状态相对隔离
- 减少样板代码，提升开发效率
- 与 React 18 并发特性兼容良好

### 10.2 Mock 方案选型：MSW

| 方案 | 优势 | 劣势 |
|-----|------|------|
| Mock.js | 配置简单、文档丰富 | 拦截 XMLHttpRequest，不支持 Fetch API |
| **MSW（选）** | Service Worker 层拦截、支持 Fetch、与生产行为一致 | 配置稍复杂 |

选择 MSW 的原因：
- 不侵入业务代码，切换 Mock/真实 API 只需开关
- 支持 Fetch API，与 Axios 底层兼容
- 浏览器 DevTools 可直接观察请求

### 10.3 实时数据方案

| 方案                      | 适用场景            | 选择      |
| ----------------------- | --------------- | ------- |
| 轮询（Polling）             | 刷新间隔 ≥ 10s，实现简单 | 默认方案    |
| SSE（Server-Sent Events） | 单向推送、轻量         | 后端支持时优先 |
| WebSocket               | 双向通信、极低延迟       | 互动场景需要时 |

**当前方案：** 默认 HTTP 轮询，后端支持 SSE 时可无缝切换。2 秒刷新需后端配合限流。

### 10.4 数据导出方案

| 方案 | 说明 |
|-----|------|
| 前端导出 | 适合小数据量（< 1万行），使用 xlsx.js |
| **后端异步导出（选）** | 大数据量场景，提交任务 → 异步生成 → 通知下载 |

选择后端异步导出的原因：
- 需求涉及 1 年时间跨度数据，数据量大
- 100MB 限制，前端导出易导致浏览器内存溢出
- 支持定时导出、模板导出等高级功能

## 十二、风险与建议

| 风险项 | 影响 | 建议 |
|-------|------|------|
| 2 秒实时刷新对后端压力 | 高 | 建议后端采用 SSE/WebSocket 推送，或增加网关限流 |
| 1 年数据跨度的查询性能 | 中 | 建议后端预聚合 + 分页，前端懒加载 |
| ECharts 大数据量渲染 | 中 | 万级以上数据点建议抽样或使用 canvas 渲染模式 |
| 多语言翻译工作量 | 中 | 建议先完成中文版本，国际化框架就绪后再补充英文 |
| 导出文件 100MB 限制 | 低 | 前端可设置分段导出提示，引导用户拆分查询 |
