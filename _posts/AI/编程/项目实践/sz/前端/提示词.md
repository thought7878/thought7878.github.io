# 京东商智前端 - Vibe Coding 提示词

> 本文档作为 Vibe Coding（AI Agent 驱动开发）的总提示词，用于指导主 Agent 和子 Agent 协同完成京东商智前端项目的开发。

---

## 一、项目概述

### 1.1 项目名称
京东商智 - 一站式运营数据开放平台前端

### 1.2 项目描述
仿京东商智的独立前端产品，为京东生态内的商家/品牌商提供全方位、全链路的数据分析解决方案。覆盖流量分析、交易分析、商品分析、用户画像、行业竞品、营销分析等核心业务场景。

### 1.3 技术栈
| 类别 | 选型 |
|------|------|
| 核心框架 | React 18+ / TypeScript 5+ |
| UI 组件库 | Ant Design 5.x（Design Token 主题定制） |
| 图表库 | ECharts 5.x |
| 路由 | React Router v6+ |
| 状态管理 | Zustand |
| 构建工具 | Vite 5+ |
| 样式 | Less + CSS Modules |
| HTTP | Axios |
| 国际化 | react-i18next + i18next |
| Mock | MSW (Mock Service Worker) |
| 日期 | dayjs |

### 1.4 已确认需求
- 支持多语言（国际化）
- 支持暗黑模式
- 实时刷新频率最低 2 秒
- 导出文件大小限制 100MB
- 支持自定义主题色
- 品牌色/Logo 暂无，使用默认
- 需要消息通知系统
- 数据时间跨度最长 1 年

## 二、架构与分工

### 2.1 Agent 协作模型

```
主 Agent（协调者）
│
├── 监控整体开发进度
├── 按模块顺序生成子 Agent 任务
├── 校验每个子 Agent 的交付成果
├── 处理模块间的依赖协调
└── 确保最终集成的一致性
    │
    ├── 子 Agent 1：M0 认证与权限
    ├── 子 Agent 2：M1 布局与导航
    ├── 子 Agent 3：M2 主题与国际化
    ├── 子 Agent 4：M3 通知与消息
    ├── 子 Agent 5：M4 首页仪表盘
    ├── 子 Agent 6：M5 实时数据看板
    ├── 子 Agent 7：M6 流量分析
    ├── 子 Agent 8：M7 交易分析
    ├── 子 Agent 9：M8 商品分析
    ├── 子 Agent 10：M9 用户画像
    ├── 子 Agent 11：M10 行业与竞品
    ├── 子 Agent 12：M11 营销分析
    └── 子 Agent 13：M12 数据导出中心
```

### 2.2 主 Agent 职责
1. **任务编排**：按模块依赖关系依次生成子 Agent 任务
2. **进度监控**：跟踪每个模块的开发进度和完成状态
3. **质量校验**：确保子 Agent 交付的代码符合项目规范
4. **依赖协调**：处理模块间的类型共享和接口依赖
5. **集成测试**：模块完成后进行整体集成验证

### 2.3 子 Agent 职责
1. **模块实现**：完成指定模块的所有代码实现
2. **单元测试**：为模块内核心逻辑编写单元测试
3. **Mock 数据**：为模块 API 接口编写 MSW Mock Handler
4. **国际化**：为模块内所有文本编写 i18n 语言包
5. **类型定义**：确保模块内 TypeScript 类型完整准确

### 2.4 开发顺序（按依赖关系）
1. **第一批（基础支撑层）**：M0 → M1 → M2 → M3
2. **第二批（核心业务层）**：M4 → M5 → M6 → M7 → M8 → M9 → M10 → M11
3. **第三批（通用功能层）**：M12

## 四、全局规范

### 4.1 编码规范
- **TypeScript 严格模式**：所有文件必须使用 TypeScript，禁止 `any` 类型
- **组件规范**：使用函数组件 + Hooks，禁止 Class 组件
- **命名规范**：
  - 文件：PascalCase（组件）、camelCase（工具/Hook）
  - 变量/函数：camelCase
  - 类型/接口：PascalCase，接口以 `I` 开头或使用 `Type` 后缀
  - 常量：UPPER_SNAKE_CASE
- **导入顺序**：第三方库 → 别名路径（`@/`） → 相对路径
- **样式方案**：Less + CSS Modules，组件样式与组件同目录

### 4.2 API 响应格式（统一）
```typescript
interface ApiResponse<T> {
  code: number;        // 0: 成功, 非 0: 错误码
  message: string;
  data: T;
}

interface PaginatedResponse<T> {
  code: number;
  message: string;
  data: {
    list: T[];
    total: number;
    page: number;
    pageSize: number;
  };
}
```

### 4.3 状态管理规范（Zustand）
- 每个模块独立一个 Store 文件
- 复杂 Store 将 Actions 拆分到 `store/{module}/actions.ts`
- 持久化状态使用 `zustand/middleware` 的 `persist`
- 派生状态使用 `get` 函数计算

### 4.4 国际化规范
- 语言包按模块组织：`locales/{lang}/modules/{module}.json`
- 组件内使用 `useTranslation()` Hook
- 所有用户可见文本必须通过 i18n，禁止硬编码中文
- 日期/数字使用 dayjs 和 Ant Design 的格式化能力

### 4.5 图表规范
- 所有图表组件封装在 `src/components/Chart/` 下
- 统一 Props 接口：`data`, `loading?`, `title?`, `height?`
- 支持响应式：监听窗口 resize 调用 `chart.resize()`
- 组件卸载时调用 `chart.dispose()` 清理

### 4.6 Mock 规范
- 使用 MSW（Mock Service Worker）在 Service Worker 层拦截
- Mock 数据符合真实业务场景的数据分布
- 时间序列数据符合真实波动规律
- 排行榜数据符合二八定律分布

## 六、模块详细需求

### M0 认证与权限模块

**职责**：用户身份认证、Token 生命周期管理、RBAC 权限校验

**核心组件**：
- `LoginForm` — 登录表单（账号密码 + 验证码）
- `RegisterForm` — 注册表单
- `AuthRoute` — 路由级权限守卫
- `PermissionGuard` — 组件级权限守卫

**关键类型**：
```typescript
interface UserInfo {
  id: string; username: string; nickname: string;
  email: string; phone: string; roles: RoleInfo[];
  permissions: string[]; dataScope: DataScope; avatar?: string;
}
type DataScope = 'all' | 'shop' | 'self';
interface TokenPair {
  accessToken: string; refreshToken: string;
  expiresIn: number; refreshExpiresIn: number;
}
```

**API 接口**：
- `POST /api/auth/login` — 登录
- `POST /api/auth/logout` — 登出
- `POST /api/auth/register` — 注册
- `POST /api/auth/refresh-token` — 刷新 Token
- `GET /api/auth/captcha` — 获取验证码
- `GET /api/auth/user` — 获取当前用户信息
- `GET /api/auth/permission-tree` — 获取权限树
- `POST /api/auth/change-password` — 修改密码
- `GET /api/auth/login-log` — 登录日志

**Store**：`authStore` — 管理认证状态、Token、用户信息、权限树

**Token 策略**：
- Access Token 有效期 2 小时，Refresh Token 有效期 7 天
- 过期前 5 分钟自动静息刷新
- 并发请求队列等待刷新完成

**权限工具函数**（`utils/permission.ts`）：
- `hasPermission()` — 检查用户权限
- `hasRole()` — 检查用户角色
- `filterMenuByPermission()` — 菜单权限过滤
- `checkDataScope()` — 数据权限范围检查

**测试要点**：
- Token 刷新流程（Mock 401 响应）
- 权限校验逻辑（不同角色/权限组合）
- 数据权限范围控制
- 登录成功/失败/网络异常场景
- AuthRoute 路由守卫行为

**职责**：全局布局框架、动态菜单生成、面包屑导航

**依赖**：M0（权限数据用于动态菜单）

**核心组件**：

- `SidebarMenu` — 动态侧边菜单
- `Breadcrumb` — 面包屑导航
- `HeaderBar` — 顶栏（用户信息、通知入口、主题/语言切换）

**关键类型**：
```typescript
interface MenuItem {
  key: string; label: string; icon?: ReactNode;
  path: string; children?: MenuItem[];
  hideInMenu?: boolean; permission?: string;
}
```

**Store**：
- `layoutStore` — 侧边栏折叠状态、面包屑、当前路由

**菜单生成逻辑**：从 M0 权限树中提取 type='menu' 的节点，递归转换为 MenuItem 树

**测试要点**：
- 不同权限树生成正确的菜单
- 面包屑根据路由路径自动生成
- 侧边栏折叠/展开布局变化
- 无权限路由渲染 403

### M2 主题与国际化模块

**职责**：亮/暗模式切换、自定义主题色、多语言切换

**核心组件**：
- `ThemeProvider` — 通过 Ant Design ConfigProvider 注入主题

```typescript
type ThemeMode = 'light' | 'dark';
interface ThemeConfig {
```

**测试要点**：
- setMode 后 algorithm 正确切换

- changeLanguage 调用和资源加载

### M3 通知与消息模块

**职责**：系统通知、数据预警、任务通知的接收、展示和管理

```typescript
type NotificationType = 'system' | 'alert' | 'task';
```

### M5 实时数据看板模块

**职责**：大促实时数据监控，支持 2s/10s/30s/60s 刷新间隔

**依赖**：M3（预警通知）

**核心组件**：
- `RealtimePage` — 页面容器
- `NumberRoll` — 数字翻牌器（requestAnimationFrame 帧动画）
- `RealtimeTrend` — 实时趋势（ECharts appendData 增量更新）
- `ConversionFunnel` — 实时转化漏斗
- `TrafficSource` — 实时流量来源
- `ProductRealtimeRank` — 实时商品排名
- `AlertList` — 预警列表

**关键类型**：
```typescript
type RealtimeInterval = 2 | 10 | 30 | 60;
interface RealtimeOverview {
  gmv: number; gmvTarget?: number; orders: number;
  visitors: number; conversionRate: number; timestamp: string;
}
```

**Store**：`realtimeStore` — 刷新间隔、自动刷新状态、各维度实时数据

**轮询逻辑**：
- 页面加载后默认开启自动刷新（默认 10s）
- 数据追加而非全量重绘（趋势数据限制最大 300 点）
- 错误时指数退避重试（最多 3 次）
- 用户可暂停/恢复/切换间隔

**性能优化**：
- ECharts `appendData` 增量更新
- `useMemo` 缓存 option 对象
- 限制最大数据点数量

**测试要点**：
- startPolling/stopPolling 定时器控制
- 间隔切换后定时器重新计算
- 暂停期间不发请求
- 趋势数据追加和最大点数限制
- 数字滚动动画触发
- API 失败指数退避重试

### M6 流量分析模块

**职责**：流量来源、搜索关键词、活动效果、流量路径分析

## 九、注意事项

1. **任何不明确的地方都必须向用户提问**，不要自行假设
2. **严格按照模块依赖顺序执行**，不可跳过依赖
3. **每个子 Agent 完成后必须汇报**，主 Agent 校验后方可继续
4. **所有文本必须通过 i18n**，不允许硬编码中文
5. **TypeScript 严格模式**，不允许使用 `any`
6. **Mock 数据要符合真实业务场景**，不是随机数
7. **组件必须支持 loading 状态**和空数据兜底
8. **图表组件必须处理响应式**和组件卸载清理
9. **Store 状态必须考虑持久化需求**（主题、语言等）
10. **错误处理必须完善**，用户友好的错误提示
