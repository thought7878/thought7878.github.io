## JS代码的执行顺序、执行机制、执行原理

JavaScript 的执行是一个从宏观调度到微观底层编译的复杂过程。要彻底搞懂 JS，我们需要从三个维度来剖析：**执行机制**（代码运行时的环境组织）、**执行顺序**（异步与同步的调度规则）、**执行原理**（JS引擎底层的编译与内存管理）。

以下是全面且详细的总结：

---

### 第一部分：JS 的执行机制（运行时基础）

执行机制决定了 JS 引擎如何“理解”和“组织”代码。核心概念是**执行上下文**和**调用栈**。

#### 1. 执行上下文 (Execution Context, EC)
每当 JS 引擎执行一段代码时，都会创建一个执行上下文。它包含了代码运行所需的所有环境信息。
*   **全局执行上下文 (GEC)**：代码开始执行时创建，只有一个。它会创建全局对象（浏览器中是 `window`），并将 `this` 指向该全局对象。
*   **函数执行上下文 (FEC)**：每次调用函数时创建。
*   **Eval 执行上下文**：执行 `eval()` 时创建（不推荐使用）。

**执行上下文的生命周期分为两个阶段：**
1.  **创建阶段（预编译）**：
    *   **创建变量对象 (VO/AO)**：扫描变量声明（`var`）和函数声明，进行**变量提升**（只提升声明，不提升赋值）。`let` 和 `const` 会被创建但处于**暂时性死区 (TDZ)**。
    *   **建立作用域链**：确定当前上下文可以访问哪些外部变量。
    *   **确定 `this` 指向**：根据函数的调用方式绑定 `this`。
2.  **执行阶段**：
    *   逐行执行代码，进行变量赋值、函数调用等操作。

#### 2. 调用栈 (Call Stack)
*   JS 是**单线程**的，这意味着它只有一个调用栈。
*   调用栈是一种 **LIFO（后进先出）** 的数据结构，用于管理执行上下文。
*   当函数被调用时，其执行上下文被**压入**栈顶；当函数执行完毕（或 return）时，其上下文从栈中**弹出**。
*   **栈溢出 (Stack Overflow)**：如果递归调用没有终止条件，调用栈会无限增长，最终超出内存限制报错。

#### 3. 作用域与闭包
*   **词法作用域**：JS 的作用域在**代码编写时**（词法分析阶段）就已经确定了，而不是在运行时确定。
*   **作用域链**：当访问一个变量时，引擎会先在当前作用域查找，找不到就沿着作用域链向上一级查找，直到全局作用域。
*   **闭包**：函数和其声明时的**词法环境**的组合。即使外部函数已经执行完毕（出栈），内部函数依然持有对外部函数变量的引用，导致这些变量不会被垃圾回收。

---

### 第二部分：**JS 的执行顺序**（Event Loop 调度）

由于 JS 是单线程的，为了防止耗时任务（如网络请求）阻塞主线程，JS 引入了**事件循环 (Event Loop)** 机制来处理异步任务。

#### 1. 同步与异步
*   **同步任务**：直接在**主线程（调用栈）** 上排队执行的任务。
*   **异步任务**：不进入主线程，而是进入**任务队列 (Task Queue)**。只有当任务队列通知主线程“某个异步任务可以执行了”，该任务才会进入主线程。

#### 2. 宏任务 (MacroTask) vs 微任务 (MicroTask)
为了精细化控制执行顺序，任务队列被分为两种：

*   **宏任务**：
    *   整体代码 `script`
    *   `setTimeout`, `setInterval`
    *   I/O 操作、UI 渲染
    *   `setImmediate` (Node.js)
*   **微任务**：
    *   `Promise.then / catch / finally`
    *   `MutationObserver` (监听 DOM 变化)
    *   `process.nextTick` (Node.js 独有，优先级最高)

#### 3. **浏览器环境下的 Event Loop 流程**（核心考点）
1.  **执行同步代码**：首先执行整体的 `script` 代码（这本身是一个宏任务）。
2.  **清空微任务队列**：当调用栈为空时，引擎会检查微任务队列，并**依次执行所有微任务**，直到微任务队列清空。（如果在执行微任务时又产生了新的微任务，会在本轮继续执行）。
3.  **UI 渲染**：浏览器根据情况决定是否进行 UI 渲染（通常每秒 60 次，即 16.6ms 一次）。
4.  **执行宏任务**：从宏任务队列中取出**一个**宏任务执行。
5.  **循环**：执行完后，再次检查并清空微任务队列，然后进入下一次循环。

**执行顺序口诀**：
**同步代码 -> 微任务 -> (UI渲染) -> 宏任务 -> 微任务 -> (UI渲染) -> 宏任务...**

#### 4. 经典代码执行顺序分析
```javascript
console.log('1. 同步'); // 同步

setTimeout(() => {
  console.log('2. 宏任务 setTimeout');
}, 0);

Promise.resolve().then(() => {
  console.log('3. 微任务 Promise 1');
  setTimeout(() => {
    console.log('4. 宏任务 setTimeout in Promise');
  }, 0);
}).then(() => {
  console.log('5. 微任务 Promise 2');
});

console.log('6. 同步'); // 同步

// 输出顺序：1 -> 6 -> 3 -> 5 -> 2 -> 4
```

### 第三部分：JS 的执行原理（V8 引擎底层视角）

JS 代码本质上是纯文本，V8 引擎（Chrome 和 Node.js 使用）需要将其翻译成机器能懂的机器码。现代 V8 采用了 **JIT (Just-In-Time) 即时编译** 技术。

#### 1. 从文本到机器码的流水线
V8 的执行流程主要分为以下几个阶段：

1.  **词法分析与语法分析 (Parsing)**：
    *   **词法分析**：将代码字符串拆解成一个个有意义的词法单元（Tokens），如关键字、变量名、运算符。
    *   **语法分析**：将 Tokens 组装成**抽象语法树 (AST)**。如果语法有误，这一步会抛出 SyntaxError。
2.  **解释执行 (Ignition 解释器)**：
    *   V8 将 AST 转换为**字节码 (Bytecode)**。
    *   **Ignition** 解释器逐行解释执行字节码。解释执行的优点是**启动快**，不需要等待编译，但运行速度相对较慢。
    *   在解释执行的同时，Ignition 会收集**类型反馈 (Type Feedback)**（例如：某个变量是不是总是数字类型）。
3.  **优化编译 (TurboFan / Maglev 编译器)**：
    *   如果某段代码被频繁执行（称为**热点代码 Hot Code**），V8 会将其交给 **TurboFan**（或较新的 **Maglev**）编译器。
    *   编译器利用 Ignition 收集的“类型反馈”，将字节码编译成**高度优化的机器码**。机器码执行速度极快。
4.  **去优化 (Deoptimization)**：
    *   JS 是动态弱类型语言，变量的类型随时可能改变。
    *   如果优化后的机器码在运行时，发现变量的类型与之前收集的“类型反馈”不一致（例如原本一直是数字，突然变成了字符串），V8 会触发**去优化**，丢弃优化后的机器码，退回到 Ignition 解释器重新执行字节码。

#### 2. 内存分配与垃圾回收 (Garbage Collection, GC)
JS 开发者不需要手动管理内存，V8 引擎自动处理。

*   **内存分配**：
    *   **栈 (Stack)**：存储基本数据类型（Number, String, Boolean 等）和对象的**引用地址**。空间小，由操作系统自动分配和释放。
    *   **堆 (Heap)**：存储引用数据类型（Object, Array, Function 等）的**实际内容**。空间大，由 GC 管理。
*   **垃圾回收算法**：
    *   V8 采用**可达性分析 (Reachability)** 来判断对象是否是垃圾。从根对象（Roots，如全局对象、当前执行栈中的变量）出发，能访问到的对象就是存活的，访问不到的就是垃圾。
*   **分代回收机制**：
    *   **新生代 (New Space)**：存储存活时间较短的对象（如局部变量）。容量小，采用 **Scavenge 算法**（复制算法），将存活对象从 From 空间复制到 To 空间，效率极高。
    *   **老生代 (Old Space)**：存储存活时间较长的对象（如闭包、全局变量）。新生代中经过多次 GC 依然存活的对象会被移入老生代。容量大，采用 **标记-清除 (Mark-Sweep)** 和 **标记-整理 (Mark-Compact)** 算法。

---


### 总结：三者的关系

1.  **执行原理** 是地基：V8 引擎通过词法分析、JIT 编译将文本变成机器码，并管理内存，让代码具备运行的物理基础。
2.  **执行机制** 是骨架：引擎为代码创建执行上下文、作用域链和调用栈，规定了代码在运行时的组织结构、变量查找规则和 `this` 指向。
3.  **执行顺序** 是血液：在单线程的限制下，通过 Event Loop 和宏/微任务队列，机制灵活地调度同步和异步代码，保证了程序的非阻塞和高效流转。

理解了这三个层面，无论是应对复杂的异步 Bug、进行极致的性能优化（如避免触发 V8 去优化、减少 GC 停顿），还是面对深度的前端面试，都能做到游刃有余。

## 详细介绍浏览器中执行JS的原理

在浏览器中执行 JavaScript 是一个涉及**浏览器宿主环境**与**JS 引擎（如 V8）** 深度协作的复杂过程。要彻底理解其原理，我们需要从宏观的架构分工，深入到微观的 JIT 编译、内存管理以及线程调度。

以下是浏览器中执行 JS 原理的详细全景剖析：

---

### 一、 宏观架构：浏览器内核与 JS 引擎的边界

首先必须明确一个核心概念：**JS 引擎本身并不包含事件循环（Event Loop）、定时器（setTimeout）或 DOM API**。

*   **JS 引擎（如 V8）**：只负责解析、编译和执行 JS 代码，管理 JS 内存。它是纯粹的“代码执行器”。
*   **浏览器宿主环境**：提供了 Web APIs（DOM、BOM、Ajax、定时器等）和**事件循环机制**。

**协作流程**：当 JS 代码调用 `setTimeout` 时，JS 引擎会将这个任务交给浏览器的**定时器线程**去计时；计时结束后，浏览器将回调函数推入**任务队列**；JS 引擎的主线程空闲时，再通过事件循环从队列中取出回调执行。

---

### 二、 微观编译：V8 引擎的 JIT 编译原理

JS 是解释型语言，但为了追求极致的性能，现代 V8 引擎采用了 **JIT（Just-In-Time，即时编译）** 技术，结合了解释器启动快和编译器执行快的优点。

当一段 JS 代码进入 V8 引擎时，会经历以下流水线：

#### 1. 解析阶段 (Parsing)
*   **词法分析**：将代码字符串拆分成 Token（如关键字、变量名、符号）。
*   **语法分析**：将 Token 流转换为**抽象语法树 (AST)**。如果语法有误，此阶段会抛出 `SyntaxError`。

#### 2. 解释执行阶段 (Ignition 解释器)
*   V8 将 AST 转换为**字节码 (Bytecode)**。字节码是一种介于源码和机器码之间的中间代码。
*   **Ignition 解释器**逐行解释执行字节码。
*   **类型反馈收集**：在解释执行过程中，Ignition 会记录变量的类型信息（例如：某个函数每次传入的参数都是整数），这被称为**类型反馈 (Type Feedback)**。

#### 3. 优化编译阶段 (TurboFan / Maglev 编译器)
*   如果某段代码被频繁执行（成为**热点代码 Hot Code**），V8 会将其交给 **TurboFan**（或较新的中层编译器 **Maglev**）。
*   编译器利用 Ignition 收集的“类型反馈”，进行激进的优化（如内联函数、消除冗余检查），将字节码直接编译成**高度优化的机器码**。
*   机器码由 CPU 直接执行，速度极快。

#### 4. 去优化 (Deoptimization)
*   JS 是动态类型语言，变量类型随时可变。
*   如果优化后的机器码在运行时，发现传入的参数类型与之前收集的“类型反馈”**不一致**（例如原本一直是数字，突然传入了字符串），V8 会触发**去优化**。
*   引擎会丢弃优化后的机器码，退回到 Ignition 解释器重新执行字节码，并重新收集类型反馈。*（频繁的去优化会严重损耗性能）*

#### 💡 核心优化机制：隐藏类 (Hidden Classes)
为了优化对象属性的访问速度，V8 引入了**隐藏类**（类似 C++ 的类结构）。
*   当创建一个对象时，V8 会为其创建一个隐藏类。
*   当给对象添加新属性时，V8 会基于原隐藏类创建一个新的隐藏类，并通过**转换链**连接。
*   **原理**：如果两个对象以**相同的顺序**添加相同的属性，它们将共享同一个隐藏类。V8 可以通过**内联缓存 (Inline Caching)** 直接通过内存偏移量访问属性，而不需要每次都去哈希表中查找。
*   **启示**：在构造函数中初始化所有属性，并始终保持相同的属性添加顺序，能大幅提升对象访问性能。

---

### 三、 运行时环境：执行上下文与内存管理

#### 1. 执行上下文与调用栈
*   当 JS 引擎执行代码时，会创建**执行上下文 (Execution Context)**，包含变量对象、作用域链和 `this` 指向。
*   上下文被压入**调用栈 (Call Stack)**。由于 JS 是单线程的，调用栈只有一个。函数执行完毕后，上下文出栈，其内部的局部变量随之销毁（除非形成闭包）。

#### 2. 内存分配：栈与堆
*   **栈 (Stack)**：由操作系统自动管理，存储**基本数据类型**（Number, String, Boolean 等）和**对象的引用地址**。空间小，访问快。
*   **堆 (Heap)**：由 V8 引擎管理，存储**引用数据类型**（Object, Array, Function）的实际内容。空间大，访问相对较慢。

#### 3. 垃圾回收 (Garbage Collection, GC)
V8 采用**可达性分析**算法，通过根对象（Roots，如全局对象、执行栈中的变量）向下遍历，无法被访问到的对象即为垃圾。V8 将堆内存分为两代：

*   **新生代 (New Space)**：
    *   存储存活时间短的对象（如局部变量）。
    *   容量小（通常 1-8MB）。
    *   采用 **Scavenge 算法**：将内存分为 From 和 To 两个空间。GC 时，将 From 中存活的对象复制到 To 空间，然后清空 From 空间。效率极高，但浪费了一半空间。
*   **老生代 (Old Space)**：
    *   存储存活时间长的对象（如闭包、全局变量），以及经历过多次新生代 GC 依然存活的对象。
    *   容量大。
    *   采用 **标记-清除 (Mark-Sweep)** 和 **标记-整理 (Mark-Compact)** 算法。先标记存活对象，清除未标记对象；为防止内存碎片，还会将存活对象向一端移动（整理）。

---

### 四、 异步调度：浏览器环境下的 Event Loop
参考：[[异步原理（Event Loop）]]

JS 引擎本身是同步单线程的，浏览器通过**事件循环**赋予了 JS 异步处理能力。

1.  **主线程执行**：*JS 引擎从上到下执行同步代码，遇到 Web APIs（如 `setTimeout`, `fetch`, DOM 事件）时，将其交给**浏览器的其他线程**（定时器线程、HTTP 请求线程等）处理*。
2.  **任务入队**：*当 Web APIs 完成任务（如定时器到期、网络请求返回），浏览器会将对应的回调函数推入任务队列 (Task Queue)*。
3.  **事件循环**：*当 JS 引擎的**调用栈为空**（同步代码执行完毕）时，Event Loop 开始工作：*
    *   首先，清空所有的**微任务**（Microtask，如 `Promise.then`, `MutationObserver`）。
    *   然后，浏览器可能会进行**UI 渲染**（更新 DOM、计算样式、绘制）。
    *   接着，从**宏任务**（Macrotask，如 `setTimeout`, UI 事件）队列中取出**一个**任务执行。
    *   循环往复。

---

### 五、 渲染协作：JS 线程与 GUI 渲染线程的互斥

在浏览器中，**JS 引擎线程**和 **GUI 渲染线程**是**互斥**的（不能同时运行）。

#### 为什么互斥？
因为 JS 可以修改 DOM 和 CSSOM。如果 JS 执行和 GUI 渲染同时进行，JS 刚刚修改了 DOM，渲染线程正在绘制旧的 DOM，就会导致页面显示混乱或数据不一致。为了保证一致性，浏览器设计为两者互斥。

#### 互斥带来的影响：
1.  **JS 阻塞渲染**：当 JS 引擎在执行耗时任务时，GUI 渲染线程会被挂起。这就是为什么长任务（Long Task）会导致页面卡顿（掉帧）。
2.  **CSS 阻塞 JS**：虽然 CSS 不直接阻塞 DOM 解析，但 JS 可能会读取样式信息（如 `getComputedStyle`）。为了保证 JS 获取到最新的样式，浏览器会**等待 CSSOM 构建完成后再执行 JS**。因此，CSS 会间接阻塞 JS 的执行，进而阻塞 DOM 的构建和渲染。

#### 优化启示：
*   将 `<script>` 标签放在 `<body>` 底部，或使用 `defer` / `async` 属性，避免 JS 阻塞 DOM 解析。
*   将 CSS 放在 `<head>` 中，尽早开始 CSSOM 的构建。
*   避免在 JS 中执行耗时计算，使用 `Web Workers` 将计算密集型任务移出主线程。

---

### 六、 总结：一段 JS 代码在浏览器中的完整一生

1.  **下载**：浏览器通过网络下载 JS 文件。
2.  **解析**：V8 引擎将源码解析为 AST。
3.  **编译**：Ignition 将 AST 转为字节码并解释执行；热点代码被 TurboFan 编译为优化的机器码。
4.  **执行**：JS 引擎在主线程（调用栈）上执行机器码，分配内存（栈/堆）。
5.  **交互**：遇到 DOM 操作或异步 API，交由浏览器宿主环境处理。
6.  **调度**：异步任务完成后，回调进入任务队列，通过 Event Loop 重新进入 JS 引擎执行。
7.  **渲染**：JS 执行完毕（或让出主线程），GUI 渲染线程接管，将 DOM/CSSOM 绘制到屏幕上。
8.  **回收**：V8 的 GC 机制在后台默默清理不再使用的内存。

理解了这套底层原理，你就能明白为什么“避免在循环中创建对象”、“保持对象属性顺序一致”、“避免长任务阻塞主线程”不仅是最佳实践，更是顺应引擎底层设计的必然选择。
