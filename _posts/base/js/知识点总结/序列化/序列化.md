
序列化（Serialization）是前端与 Node.js 开发中**数据流转的核心环节**。理解它不仅能避开大量隐蔽 Bug，还能在架构选型、性能优化、跨端通信中做出正确决策。

---

## 一、核心定义：什么是序列化？

`序列化`：**将*内存中*的数据结构或对象状态，转换为*可存储*或*可传输*的格式（通常是字符串或字节流）的过程**。  
`反序列化`：**将序列化后的数据还原为内存对象的过程**。

### 为什么需要序列化？
| 场景                             | 作用          |
| ------------------------------ | ----------- |
| 网络通信（API / WebSocket）          | 跨语言/跨进程传输数据 |
| 本地存储（localStorage / IndexedDB） | 持久化保存状态     |
| 进程/线程通信（Web Worker / Node 子进程） | 突破内存隔离边界    |
| 调试与日志                          | 将复杂对象转为可读文本 |
| 状态管理（Redux / Vuex 持久化）         | 快照保存与恢复     |

> 💡 本质：**`序列化`是“数据扁平化 + 格式标准化”的过程，不是对象克隆。**

---

## 二、JS 的标准方案：JSON 序列化

JavaScript *原生提供 `JSON` 对象作为序列化标准：*

```js
// 序列化：对象 → JSON 字符串
const str = JSON.stringify({ name: 'Alice', age: 25 });
// '{"name":"Alice","age":25}'

// 反序列化：JSON 字符串 → 对象
const obj = JSON.parse(str);
// { name: 'Alice', age: 25 }
```

### ⚠️ 重要认知：JSON ≠ JS 对象
JSON 是 JS 对象字面量的**严格子集**，但两者在类型系统、行为上存在本质差异。序列化过程会**丢弃非数据特征**。

---

## 三、JSON 序列化的“潜规则”与信息丢失（重点）

`JSON.stringify()` 遵循 ECMAScript 规范，对不同类型的处理如下：

| 原始值/结构 | 序列化结果 | 说明 |
|-------------|------------|------|
| `Object` / `Array` | 正常转换 | 仅遍历**自身可枚举字符串键** |
| `String` / `Number` / `Boolean` / `null` | 原样输出 | 基础类型安全 |
| `undefined` / `Function` / `Symbol` | **对象中：忽略该键**<br>**数组中：转为 `null`**<br>**单独调用：返回 `undefined`（非字符串）** | 非 JSON 标准类型 |
| `Date` | ISO 字符串 `"2024-01-01T00:00:00.000Z"` | **丢失 Date 原型与方法** |
| `RegExp` / `Error` | `{}` | 内部属性不可枚举 |
| `Map` / `Set` / `WeakMap` / `WeakSet` | `{}` 或 `[]` | 结构信息完全丢失 |
| `BigInt` | ❌ 抛出 `TypeError` | JSON 规范不支持 |
| 循环引用 | ❌ 抛出 `TypeError` | 无法表示图结构 |
| 原型链 / getter / setter / 不可枚举属性 | 不序列化 | 仅快照“当前自有数据” |

```js
// 示例：信息丢失演示
const obj = {
  a: 1,
  b: undefined,
  c: function(){},
  d: Symbol('s'),
  e: new Date(),
  f: /test/g,
  g: new Map([['x', 1]])
};
console.log(JSON.stringify(obj));
// '{"a":1,"e":"2024-xx-xxT...","f":{},"g":{}}'
// b,c,d 被忽略；e 变字符串；f,g 变空对象
```

> 🔍 **核心结论**：JSON 序列化产出的是**纯数据快照**。方法、原型、特殊对象类型、引用关系均会丢失或降级。

## 四、如何定制序列化行为？

### 1. `toJSON()` 方法（对象自控）
`JSON.stringify` 会优先调用对象的 `toJSON()`，以其返回值作为序列化目标。

```js
const user = {
  id: 1,
  password: 'secret',
  createdAt: new Date(),
  toJSON() {
    return { id: this.id, created: this.createdAt.toISOString() };
  }
};
JSON.stringify(user); // '{"id":1,"created":"2024-..."}'
```

### 2. `replacer` 函数（`stringify` 第二参数）
遍历每个键值对，可过滤、转换或拦截序列化过程。

```js
const data = { a: 1, b: 2n, c: 3 };
JSON.stringify(data, (key, value) => {
  if (typeof value === 'bigint') return value.toString();
  if (key === 'c') return undefined; // 过滤
  return value;
});
// '{"a":1,"b":"2"}'
```

### 3. `reviver` 函数（`parse` 第二参数）
反序列化时自底向上遍历，用于**类型还原**。

```js
const str = '{"id":1,"loginAt":"2024-05-20T10:00:00.000Z"}';
const obj = JSON.parse(str, (key, value) => {
  if (key === 'loginAt') return new Date(value);
  return value;
});
console.log(obj.loginAt instanceof Date); // true
```

## 五、常见痛点与工程级解决方案

| 痛点 | 原因 | 解决方案 |
|------|------|----------|
| **循环引用报错** | JSON 是树结构，无法表示图 | ① 手动断环 ② 使用 `flatted` / `circular-json` ③ 改用 `structuredClone`（仅深拷贝） |
| **Date/Map/Set 丢失类型** | JSON 无对应语法 | ① `toJSON` + `reviver` 手动编解码 ② 使用 `superjson` / `devalue` 等全类型序列化库 |
| **BigInt 不支持** | JSON 规范制定早于 BigInt | 转字符串传输，或自定义 `replacer` |
| **大对象阻塞主线程** | `JSON.stringify` 是同步 CPU 密集型操作 | ① 分片序列化 ② Web Worker 后台处理 ③ 改用二进制协议（MessagePack/Protobuf） |
| **反序列化安全风险** | `JSON.parse` 本身安全，但解析后直接合并可能原型污染 | ① 校验数据结构（Zod/Yup） ② 避免 `Object.assign(target, parsed)` 直接覆盖原型 |

## 六、序列化 vs 深拷贝 vs 其他数据格式

| 概念 | 目标 | 输出形式 | 典型 API/方案 | 适用场景 |
|------|------|----------|---------------|----------|
| **序列化** | 跨环境传输/存储 | 字符串 / 字节流 | `JSON.stringify`、MessagePack、Protobuf | API 通信、localStorage、日志 |
| **深拷贝** | 内存中克隆对象 | 新对象（引用独立） | `structuredClone()`、lodash.cloneDeep | 状态隔离、撤销重做、表单草稿 |
| **URL 编码** | 表单/查询参数传输 | `key=value&...` | `URLSearchParams`、`qs` | GET 参数、`application/x-www-form-urlencoded` |
| **二进制协议** | 高性能/低带宽传输 | ArrayBuffer / Uint8Array | MessagePack、Protocol Buffers、CBOR | 游戏、音视频、IoT、高频 WebSocket |

> 📌 `structuredClone()` 是浏览器/Node 原生深拷贝 API，支持循环引用、Map/Set/Date/RegExp 等，**但它不生成字符串，不属于序列化**，仅用于内存克隆。

## 七、最佳实践与选型指南

### ✅ 日常开发守则
1. **明确目的**：传输/存储 → 序列化；内存隔离 → 深拷贝。不要混用。
2. **统一日期格式**：前端发 ISO 字符串，后端返时间戳或 ISO，解析时统一用 `reviver` 转 `Date`。
3. **避免序列化复杂对象**：含 DOM、函数、闭包、循环引用的对象不应直接 `JSON.stringify`。
4. **大 payload 性能监控**：`JSON.stringify(10MB对象)` 可能阻塞 50~200ms，考虑分页、压缩或 Worker。
5. **类型安全先行**：反序列化后务必校验（TypeScript 类型守卫 / Zod / io-ts），防止脏数据击穿。

### 🧰 现代工具链推荐
| 需求 | 推荐方案 |
|------|----------|
| 标准 API 通信 | `JSON` + `fetch`/`axios` |
| 全类型序列化（含 Date/Map/循环引用） | `superjson`、`devalue`、`serialize-javascript` |
| 高性能/低带宽 | `MessagePack`、`Protocol Buffers`、`CBOR` |
| 内存深拷贝 | `structuredClone()`（首选）、`lodash.cloneDeep` |
| 表单数据 | `FormData`、`URLSearchParams` |

## 八、一句话总结

> JS 中的序列化是**将内存对象转为可传输/存储格式的过程**，`JSON` 是默认标准但会丢失类型、方法与引用关系。掌握 `toJSON`/`replacer`/`reviver` 可解决大部分定制需求；现代工程应严格区分“序列化传输”与“内存深拷贝”，按场景选用 JSON、二进制协议或 `structuredClone`，并始终对反序列化结果做类型校验。

如需深入某个方向（如：如何用 `superjson` 实现全类型安全序列化、`structuredClone` 的底层实现原理、或 Protobuf 在前端的接入实战），可提供具体场景，我将给出可落地的代码方案。
