## 二、核心知识领域详解

### 1. 架构设计

#### 1.1 模块化与组件化

**模块化设计**
- **定义**：将系统拆分为独立、可复用的模块
- **核心原则**：
  - 单一职责（每个模块只做一件事）
  - 高内聚低耦合（模块内部紧密相关，模块间依赖最小化）
  - 显式依赖（明确声明模块依赖关系）
- **实现方式**：
  - ES Modules（原生模块系统）
  - CommonJS（Node.js 环境）
  - AMD/CMD（浏览器环境，已较少使用）
- **实践要点**：
  - 合理划分模块边界
  - 建立清晰的模块命名规范
  - 维护模块依赖图表

**组件化设计**
- **定义**：将 UI 拆分为独立、可复用的组件
- **组件层次**：
  - 原子组件（Button、Input、Icon 等基础元素）
  - 分子组件（由原子组件组合，如 SearchBar）
  - 有机体组件（完整的功能模块，如 Header、Sidebar）
- **设计原则**：
  - Props 传递数据，事件传递状态变化
  - 组件无状态化优先（Stateless Component）
  - 容器组件与展示组件分离

#### 1.2 微前端架构

**核心概念**
- 将前端应用拆分为多个可独立开发、测试、部署的子应用
- 每个子应用可以使用不同的技术栈

**落地模式**

| 模式             | 描述            | 适用场景        |
| -------------- | ------------- | ----------- |
| Build-time 集成  | 子应用作为 NPM 包构建 | 团队固定、技术栈统一  |
| Run-time 集成    | 子应用独立部署，运行时组合 | 多团队协作、技术栈多样 |
| iframe 隔离      | 完全隔离的子应用      | 强隔离需求、老系统集成 |
| Web Components | 基于标准组件模型集成    | 跨框架复用       |

**关键技术点**
- **应用注册表**：管理子应用的生命周期
- **沙箱隔离**：CSS 样式隔离、JavaScript 执行隔离
- **通信机制**：父子应用间消息传递
- **统一路由**：根据路由动态加载子应用

#### 1.3 分层架构模式

**MVC 模式**
- Model（模型）：数据层，处理业务逻辑和数据
- View（视图）：表现层，负责 UI 渲染
- Controller（控制器）：协调 Model 和 View

**MVVM 模式**
- Model：数据模型
- View：视图层
- ViewModel：视图模型，双向数据绑定

**Flux 模式**
- Dispatcher：统一事件分发
- Store：数据存储和业务逻辑
- Action：用户行为描述
- View：视图层

**Clean Architecture**
```
表现层 (Presentation)
    ↓
用例层 (Use Cases)
    ↓
领域层 (Domain)
    ↓
数据层 (Data)
```

#### 1.4 架构设计原则

**SOLID 原则**
- **S**ingle Responsibility（单一职责）
- **O**pen/Closed（开闭原则）
- **L**iskov Substitution（里氏替换）
- **I**nterface Segregation（接口隔离）
- **D**ependency Inversion（依赖反转）

**DRY & KISS**
- DRY（Don't Repeat Yourself）：避免重复
- KISS（Keep It Simple, Stupid）：保持简单

**关注点分离**
- 将不同关注点分离到不同模块
- 便于独立开发和测试

### 2. 组件设计

#### 2.1 组件抽象与拆分

**何时拆分组件**
- 组件过于庞大（超过 200-300 行）
- 存在独立可复用的部分
- 多个组件承担相似功能
- 组件状态过于复杂

**如何拆分**
1. 识别组件的单一职责
2. 提取纯展示部分为独立组件
3. 提取可复用逻辑为 Hooks/工具函数
4. 通过 Props 控制组件行为

**组件拆分粒度**
- 太粗：复用性差
- 太细：管理成本高、维护困难
- 适中：独立可测试、职责单一

#### 2.2 组件 API 设计

**Props 设计原则**
```jsx
// ❌ 反例：Props 过多
<UserCard 
  name={name}
  avatar={avatar}
  email={email}
  phone={phone}
  address={address}
  bio={bio}
  onEdit={handleEdit}
  onDelete={handleDelete}
  onMessage={handleMessage}
/>

// ✅ 正例：使用配置对象
<UserCard 
  user={user}
  onAction={handleAction}
/>
```

**组件变体设计（Variants）**
```jsx
// 使用组合模式
<Button variant="primary" size="large" />
<Button variant="secondary" size="small" />
```

**插槽设计**
```jsx
// 默认插槽
<Modal>
  <Content />
</Modal>

// 命名插槽
<Layout>
  <Header slot="header" />
  <Sidebar slot="sidebar" />
</Layout>
```

#### 2.3 组件复用策略

**继承 vs 组合**
- 优先使用组合而非继承
- Mixin 模式（已较少使用）
- HOC（高阶组件）模式
- 自定义 Hooks 模式
- Render Props 模式

**设计模式**
- **Compound Components**：隐式共享状态
```jsx
<Select>
  <Select.Option value="1">选项1</Select.Option>
  <Select.Option value="2">选项2</Select.Option>
</Select>
```
- **Render Props**：逻辑复用
```jsx
<MouseTracker render={(position) => (
  <div>鼠标位置: {position.x}, {position.y}</div>
)} />
```
- **Provider Pattern**：跨层级数据传递
```jsx
<ThemeProvider theme={darkTheme}>
  <App />
</ThemeProvider>
```

#### 2.4 组件性能考量

**避免不必要的渲染**
- React.memo() 缓存组件
- useMemo() 缓存计算结果
- useCallback() 缓存回调函数
- 合理使用 Key

**按需加载**
```jsx
const HeavyComponent = lazy(() => import('./HeavyComponent'));

<Suspense fallback={<Loading />}>
  <HeavyComponent />
</Suspense>
```

**虚拟列表**
- 处理长列表渲染
- react-window / react-virtualized

### 3. 状态管理

#### 3.1 客户端状态管理

**本地状态 vs 全局状态**
- **本地状态**：仅组件内部使用，使用 useState/useReducer
- **全局状态**：跨组件共享，使用状态管理库

**Redux 生态**
- **Redux Toolkit**：现代化 Redux 开发体验
```jsx
const slice = createSlice({
  name: 'counter',
  initialState: { value: 0 },
  reducers: {
    increment: (state) => { state.value += 1; },
    decrement: (state) => { state.value -= 1; },
  },
});

export const { increment, decrement } = slice.actions;
export default slice.reducer;
```
- **Redux Thunk / Redux Saga**：异步逻辑处理
- **Reselect**：计算属性缓存

**Vue 状态管理**
- **Pinia**（Vue 3 推荐）
```jsx
const useCounterStore = defineStore('counter', {
  state: () => ({ count: 0 }),
  getters: {
    double: (state) => state.count * 2,
  },
  actions: {
    increment() { this.count++; },
  },
});
```

**轻量级方案**
- **Zustand**：简洁 API
```jsx
const useStore = create((set) => ({
  count: 0,
  inc: () => set((state) => ({ count: state.count + 1 })),
}));
```
- **Jotai**：原子化状态管理
- **Recoil**：Facebook 推出的状态管理库

#### 3.2 服务端状态管理

**SWR / React Query**
```jsx
const { data, error, isLoading } = useSWR('/api/user', fetcher);
```
- 自动缓存和去重
- 后台数据更新
- 乐观更新
- 错误重试

**Apollo Client（GraphQL）**
- 声明性数据获取
- 自动缓存管理
- 实时订阅支持

#### 3.3 路由状态管理

**路由参数**
- 路径参数：/user/:id
- 查询参数：?tab=active
- 路由状态：隐式状态

**状态同步策略**
- URL 作为状态源
- 状态变化时更新 URL
- 页面刷新时恢复状态

#### 3.4 状态管理模式

**单向数据流**
```
User Action → Dispatch → Store → View Update
```

**不可变数据**
- 便于追踪状态变化
- 支持时间旅行调试
- 优化重渲染判断
- Immer.js 简化不可变操作

**状态设计原则**
- 最小化状态
- 避免冗余状态
- 分离 UI 状态和业务状态
- 分离服务端状态和客户端状态

### 4. 性能优化

#### 4.1 加载性能优化

- **Tree Shaking**：移除未使用的代码
- **压缩**：Gzip / Brotli 压缩

- **预加载/预取**
```html

```

**缓存策略**
- 浏览器缓存（Cache-Control, ETag）
- Service Worker 缓存
- CDN 边缘缓存

**首屏渲染优化**
- SSR（服务端渲染）
- SSG（静态站点生成）
- ISR（增量静态再生）
- 流式 SSR

#### 4.2 运行时性能优化

**JavaScript 执行优化**
- 避免长任务（Long Tasks）
- 使用 Web Workers 处理计算密集型任务
- 事件委托代替多个事件监听
- 防抖/节流
```jsx
// 防抖
const debounce = (fn, delay) => {
  let timer = null;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), delay);
  };
};

// 节流
const throttle = (fn, limit) => {
  let inThrottle = false;
  return (...args) => {
    if (!inThrottle) {
      fn(...args);
      inThrottle = true;
      setTimeout(() => { inThrottle = false; }, limit);
    }
  };
};
```

**内存管理**
- 避免内存泄漏
- 及时清理定时器
- 移除事件监听
- 合理使用闭包

#### 4.3 渲染性能优化

**React 渲染优化**
- shouldComponentUpdate / PureComponent
- React.memo() 优化
- Virtual DOM 批量更新
- 避免不必要的重渲染

**CSS 渲染优化**
- 避免重排（Reflow）和重绘（Repaint）
- 使用 transform/opacity 实现动画
- will-change 提示浏览器优化
- 避免 CSS 表达式

**列表渲染优化**
- 使用 Key
- 虚拟列表（Virtual Scrolling）
```jsx
import { FixedSizeList } from 'react-window';

<FixedSizeList
  height={400}
  itemCount={10000}
  itemSize={50}
>
  {({ index, style }) => (
    <div style={style}>Row {index}</div>
  )}
</FixedSizeList>
```

#### 4.4 性能监控体系

**Core Web Vitals**
- **LCP**（Largest Contentful Paint）：最大内容绘制 < 2.5s
- **FID**（First Input Delay）：首次输入延迟 < 100ms

### 5. 安全设计

#### 5.1 前端安全威胁

**XSS（跨站脚本攻击）**
| 类型 | 描述 | 防御措施 |
|------|------|----------|
| 存储型 XSS | 恶意脚本存入数据库 | 输入校验、输出编码 |
| 反射型 XSS | 恶意脚本通过 URL 反射 | CSP、URL 编码 |
| DOM 型 XSS | 前端代码漏洞 | 安全 DOM API |

**防御策略**
```jsx
// React 自动转义
const UserInput = ({ value }) => (
  <div>{value}</div>  // 自动转义
);

// 手动转义（谨慎使用）
const safeHTML = DOMPurify.sanitize(dirtyHTML);

// CSP 配置
<meta http-equiv="Content-Security-Policy" 
  content="default-src 'self'; script-src 'self'">
```

**CSRF（跨站请求伪造）**
- 利用用户已登录状态发起恶意请求
- 防御措施：
  - CSRF Token
  - SameSite Cookie
  - 验证请求来源（Origin/Referer）

**点击劫持（Clickjacking）**
- 使用 iframe 覆盖页面诱导点击
- 防御：X-Frame-Options / CSP frame-ancestors

#### 5.2 认证与授权

**认证机制**
- **JWT（JSON Web Token）**
```jsx
// Token 存储
localStorage（需 XSS 防护）/ httpOnly Cookie

// Token 刷新
access_token（短期）+ refresh_token（长期）
```

- **OAuth 2.0**
  - Authorization Code Flow（最安全）
  - PKCE 扩展（无后端应用）
  - Implicit Flow（已废弃）

**授权模型**
- **RBAC**（基于角色的访问控制）
- **ABAC**（基于属性的访问控制）
- 前端权限控制：路由守卫、按钮权限

#### 5.3 数据安全

**敏感数据处理**
- 前端脱敏展示
- 敏感数据不存储在前端
- 请求加密（HTTPS）

**前端加密**
- 端到端加密场景下使用
- Web Crypto API
- 注意：前端加密不能替代 HTTPS

#### 5.4 安全最佳实践

**输入验证**
- 客户端验证 + 服务器端验证
- 白名单校验
- 长度限制

**依赖安全**
- 定期更新依赖
- 使用 npm audit / snyk
- 锁定依赖版本

**安全 Headers**
```html
<meta http-equiv="X-Content-Type-Options" content="nosniff">
<meta http-equiv="X-Frame-Options" content="DENY">
<meta http-equiv="Referrer-Policy" content="strict-origin-when-cross-origin">
```

### 7. 跨系统交互

#### 7.1 API 设计原则

**RESTful API 设计**

| 方法 | 用途 | 示例 |
|------|------|------|
| GET | 获取资源 | GET /users |
| POST | 创建资源 | POST /users |
| PUT | 更新资源 | PUT /users/1 |
| DELETE | 删除资源 | DELETE /users/1 |

**API 设计规范**
- 使用名词而非动词：/users 而非 /getUsers
- 层级嵌套：/users/1/posts
- 版本控制：/api/v1/users
- 统一响应格式
- 状态码规范

**GraphQL vs REST**

| 特性 | REST | GraphQL |
|------|------|---------|
| 数据获取 | 多个端点 | 单一端点 |
| 灵活性 | 固定返回 | 按需获取 |
| 缓存 | 简单（HTTPS 缓存） | 需额外配置 |
| 学习曲线 | 低 | 中 |

#### 7.2 数据流设计

**前后端数据交互**
```
前端 → HTTP Request → API Gateway → Backend Service
         ↑                              ↓
         └─────── JSON Response ←───────┘
```

**数据请求模式**
- **REST API**：资源导向
- **GraphQL**：查询导向
- **gRPC**：高性能场景（需 Protocol Buffers）

**实时数据**
- WebSocket：双向实时通信
- Server-Sent Events（SSE）：服务端推送
- Polling：轮询（低实时性需求）

#### 7.3 系统集成模式

**与后端服务集成**
- 统一错误处理
- 请求重试机制
- 限流处理
- 熔断降级

**与数据库交互**
- 前端不直接操作数据库
- 通过 API 间接访问
- 离线数据同步（IndexedDB / LocalStorage）

**第三方服务集成**
- SDK 封装
- API 适配层
- 统一配置管理
