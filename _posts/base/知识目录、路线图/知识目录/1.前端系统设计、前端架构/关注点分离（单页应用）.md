
>对于典型的React的单页应用，如何关注点分离（SoC）？
>如何将 UI、业务逻辑、数据请求、样式拆分，降低模块间耦合？


在 React 单页应用中实现`关注点分离（Separation of Concerns, SoC）`，核心不是“把文件拆得越细越好”，而是**划定清晰的逻辑边界 + 规范依赖方向**，*让每一层只关心自己的职责，且可独立测试、替换与演进*。

下面以现代 React 工程实践为例，按你提到的 4 个维度拆解，并给出可直接落地的结构与代码范式。

---
## 一、四层职责划分与依赖流向
```
UI 组件层  ←── 依赖 ── 业务逻辑层  ←── 依赖 ── 数据请求层
   ↓
样式层（独立作用于 UI，不参与逻辑流转）
```
✅ **依赖铁律**：下层不知道上层的存在。UI 调用 Hook，Hook 调用 API，API 不感知 React；样式只依附于 DOM，不触碰数据。

---
## 二、逐层拆分实战（以“用户列表”为例）

### 数据请求层（Network / API）
**职责**：发起 HTTP 请求、处理拦截/重试/超时、返回结构化数据。  
**原则**：纯函数或类，不包含 React 代码，不关心数据怎么展示。

```ts
// features/user-list/api/userApi.ts
import axios from 'axios';
import type { User } from '../types';

const api = axios.create({ baseURL: import.meta.env.VITE_API_BASE });

export const fetchUsers = async (page: number, pageSize = 20) => {
  const res = await api.get<{ data: User[]; total: number }>('/users', {
    params: { page, limit: pageSize },
  });
  return res.data; // 仅返回业务数据，不抛异常（由上层统一捕获）
};
```

### 业务逻辑层（State / Hooks）
**职责**：数据转换、分页/筛选逻辑、加载/错误状态管理、表单校验规则。  
**原则**：用 Custom Hook 封装，可脱离 UI 独立测试（用 `@testing-library/react-hooks`）。

```ts
// features/user-list/hooks/useUserList.ts
import { useQuery } from '@tanstack/react-query';
import { fetchUsers } from '../api/userApi';
import type { UseUserListOptions } from '../types';

export const useUserList = (options: UseUserListOptions) => {
  const { page, keyword } = options;

  // React Query 负责缓存、重试、并发控制
  const { data, isLoading, error, refetch } = useQuery({
    queryKey: ['users', page, keyword],
    queryFn: () => fetchUsers(page, 20),
    select: (res) => ({
      // 业务逻辑：过滤、排序、衍生状态
      users: res.data.filter(u => u.name.includes(keyword || '')),
      total: res.total,
      hasMore: page * 20 < res.total,
    }),
  });

  return { users: data?.users || [], total: data?.total, isLoading, error, refetch };
};
```

### UI 展示层（Components / View）
**职责**：布局、交互、事件绑定、无障碍属性。  
**原则**：尽量写成“展示型组件”（Presentational），通过 props 接收数据与回调，内部不写 `fetch`、不写复杂计算。

```tsx
// features/user-list/components/UserList.tsx
import { useUserList } from '../hooks/useUserList';
import { UserCard } from './UserCard';
import styles from './UserList.module.css';
import type { UserListProps } from '../types';

export const UserList: React.FC<UserListProps> = ({ keyword }) => {
  const [page, setPage] = useState(1);
  const { users, isLoading, hasMore, error } = useUserList({ page, keyword });

  if (error) return <ErrorMessage onRetry={() => setPage(p => p)} />;
  if (isLoading) return <LoadingSkeleton />;

  return (
    <section className={styles.container}>
      {users.map(u => <UserCard key={u.id} user={u} />)}
      {hasMore && (
        <button onClick={() => setPage(p => p + 1)} className={styles.loadMore}>
          加载更多
        </button>
      )}
    </section>
  );
};
```

### 样式层（Styles / Theme）
**职责**：视觉呈现、响应式、主题变量、动画。  
**原则**：与组件强绑定但逻辑解耦，优先使用 CSS Modules / Tailwind / CSS-in-JS，避免内联样式写条件逻辑。

```css
/* features/user-list/styles/UserList.module.css */
.container {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(280px, 1fr));
  gap: 1rem;
  padding: 1rem;
}

.loadMore {
  composes: btn-primary from '../../shared/styles/buttons.css'; /* 共享设计 Token */
  margin: 2rem auto;
  display: block;
}
```
> 💡 现代方案：CSS Modules（作用域隔离） + Tailwind（原子类加速开发） + `:global()`（覆盖第三方库）。

## 三、推荐目录结构（Feature-based）
按**功能模块**而非技术类型划分，天然符合 SoC 且利于团队并行开发：
```
src/
├── features/
│   └── user-list/              # 独立功能域
│       ├── api/                # 数据请求
│       ├── hooks/              # 业务逻辑
│       ├── components/         # UI 组件
│       ├── styles/             # 样式
│       ├── types/              # TS 接口
│       └── index.ts            # 统一导出（对外暴露最小 API）
├── shared/                     # 跨功能复用（按钮、图标、工具函数、全局主题）
├── config/                     # 环境变量、路由配置、全局 Provider
└── app/                        # 应用入口、Layout、权限守卫、错误边界
```

## 四、层间通信规范
| 流向       | 推荐方式                                  | 禁止做法                                                     |
| -------- | ------------------------------------- | -------------------------------------------------------- |
| UI → 逻辑  | 调用 Custom Hook 返回的函数/状态               | 直接修改全局变量或 `useState` 散落在多个组件                             |
| 逻辑 → 数据  | 调用 API 层函数，传入参数                       | 在 Hook 里写 `axios.post()` 或拼 URL                          |
| 逻辑 → UI  | 通过 Hook 返回 `data/loading/error`       | 在 API 层直接 `toast.success()` 或操作 DOM                      |
| UI ←→ 样式 | `className={styles.xxx}` 或 Tailwind 类 | `<div style={{ display: isActive ? 'block' : 'none' }}>` |

## 五、常见反模式 vs 正确做法
| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| `useEffect(() => fetch('/api'), [])` 写在 UI 组件里 | 数据请求与 UI 耦合，难测试、难复用、难处理并发/缓存 | 抽离为 `api/user.ts` + `useQuery` |
| 在 JSX 里写 `if (user.role === 'admin') ...` | 业务规则硬编码在视图，改动需找遍全项目 | 抽成 `usePermission('admin')` 或 `hasAccess(user)` |
| 全局 Redux 存所有数据 | 状态树臃肿，组件耦合度高，调试困难 | 局部状态优先，仅跨页面共享状态上提至 Context/Zustand |
| 内联样式写交互逻辑 | 样式与逻辑混在一起，覆盖困难，性能差 | 用 CSS 变量 + 类名切换，或 `clsx` 管理条件类名 |

## 六、初学者避坑指南
1. **不要过度拆分**：一个 3 人小项目，`components/` + `hooks/` + `services/` 三个目录足够，等模块超过 5 个再按 `features/` 拆分。
2. **SoC 是逻辑概念，不是物理强制**：有时把 Hook 和组件放在同一个文件里（Colocation）反而更易维护，只要边界清晰即可。
3. **测试是 SoC 的试金石**：如果某个逻辑无法脱离 UI 单独写单元测试，说明关注点没分离干净。
4. **数据流永远单向**：`API → Hook → Component → Render`，禁止反向依赖或循环引用。

## 核心口诀（便于记忆）
🔹 **视图只渲染，不碰网络**  
🔹 **逻辑管状态，不问 DOM**  
🔹 **请求纯函数，不绑框架**  
🔹 **样式靠类名，不写内联**  
🔹 **依赖单向走，下层不知上层**
