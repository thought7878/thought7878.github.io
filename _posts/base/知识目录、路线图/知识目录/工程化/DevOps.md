## 一、基础概念与文化
| 知识点 | 一句话解释 |
|--------|-----------|
| **DevOps** | 开发（Dev）与运维（Ops）的协作文化+自动化实践，目标是更快、更稳地交付价值 |
| **敏捷开发** | 小步快跑、快速迭代的开发方法，是 DevOps 的前置实践 |
| **精益思想** | 消除浪费、持续改进，指导流程优化的核心哲学 |
| **反馈闭环** | 快速收集用户/系统反馈并驱动改进，是 DevOps 的核心机制 |
| **共享责任** | 开发对运维负责，运维对开发赋能，打破"甩锅"文化 |
| **自动化优先** | 能自动化的绝不手动，减少人为错误，提升可重复性 |
| **度量驱动** | 用数据（如部署频率、故障恢复时间）衡量改进效果 |

## 二、版本控制（代码协作基石）
| 知识点 | 一句话解释 |
|--------|-----------|
| **Git** | 分布式版本控制系统，记录代码变更历史，支持多人协作 |
| **分支策略** | 如 Git Flow、Trunk-Based，规范团队协作与发布流程 |
| **Pull Request** | 代码合并前的审查机制，保障代码质量与知识共享 |
| **Code Review** | 同行评审代码，提前发现缺陷，促进团队成长 |
| **.gitignore** | 声明哪些文件不应纳入版本控制（如密钥、编译产物） |
| **Git Hook** | 在特定 Git 事件（如 commit）前后自动执行脚本 |
| **语义化版本** | 用 `主版本.次版本.补丁号` 规范版本变更影响范围 |

---

## 三、持续集成/持续交付（CI/CD）
| 知识点 | 一句话解释 |
|--------|-----------|
| **持续集成（CI）** | 代码频繁合并到主干，自动构建+测试，快速发现集成问题 |
| **持续交付（CD）** | 代码随时可安全发布到生产，发布决策由业务驱动 |
| **持续部署** | 通过所有测试的代码自动发布到生产，无需人工干预 |
| **流水线（Pipeline）** | 将构建、测试、部署等步骤编排成自动化工作流 |
| **制品库** | 存储构建产物（如 Docker 镜像、JAR 包）的仓库，如 Nexus、Harbor |
| **流水线即代码** | 用 YAML/DSL 定义 CI/CD 流程，支持版本控制与复用 |
| **并行/串行执行** | 合理编排任务顺序，平衡构建速度与依赖关系 |

---

## 四、配置管理（环境一致性保障）
| 知识点 | 一句话解释 |
|--------|-----------|
| **配置管理** | 用代码定义服务器状态，确保环境可重复、可审计 |
| **Ansible** | 基于 SSH 的无代理配置工具，YAML 编写，上手简单 |
| **Puppet** | 声明式配置管理，适合大规模基础设施统一管理 |
| **Chef** | 基于 Ruby 的配置工具，灵活强大，学习曲线较陡 |
| **SaltStack** | 高性能配置管理，支持事件驱动与远程执行 |
| **幂等性** | 多次执行同一配置操作，结果与执行一次相同，避免副作用 |
| **环境隔离** | 开发/测试/生产环境配置分离，避免"在我机器上能跑"问题 |

---

## 五、容器化与编排（应用交付新标准）
| 知识点 | 一句话解释 |
|--------|-----------|
| **Docker** | 轻量级容器引擎，实现"一次构建，到处运行" |
| **容器镜像** | 只读模板，打包应用代码、依赖、配置，保证环境一致 |
| **Kubernetes (K8s)** | 开源容器编排平台，自动化部署、扩缩容、管理容器集群 |
| **Pod** | K8s 最小调度单元，可包含一个或多个紧密耦合的容器 |
| **Deployment** | 声明式管理 Pod 副本，支持滚动更新与回滚 |
| **Service** | 为 Pod 提供稳定网络入口，实现服务发现与负载均衡 |
| **ConfigMap/Secret** | 分别管理配置项与敏感信息，实现配置与代码分离 |
| **Helm** | K8s 应用包管理工具，用 Chart 模板复用复杂部署 |

---

## 六、基础设施即代码（IaC）
| 知识点 | 一句话解释 |
|--------|-----------|
| **IaC** | 用代码定义和管理基础设施（服务器、网络、存储），实现版本化与自动化 |
| **Terraform** | 多云 IaC 工具，声明式语法，支持状态管理与资源依赖 |
| **AWS CloudFormation** | AWS 原生 IaC 服务，JSON/YAML 定义云资源栈 |
| **Pulumi** | 用真实编程语言（Python/TS 等）编写 IaC，适合开发者 |
| **不可变基础设施** | 服务器一旦部署不再修改，变更时直接替换新实例，提升可靠性 |
| **状态文件** | IaC 工具记录实际资源状态，用于差异比对与增量更新 |
| **模块/组件复用** | 将通用资源封装为模块，提升代码复用与维护效率 |

---

## 七、监控与日志（系统可观测性基础）
| 知识点 | 一句话解释 |
|--------|-----------|
| **监控（Monitoring）** | 持续采集系统指标（CPU/内存/请求量），发现异常 |
| **日志（Logging）** | 记录应用与系统事件，用于问题排查与行为分析 |
| **告警（Alerting）** | 指标超阈值时自动通知，实现故障早发现早处理 |
| **Prometheus** | 开源时序数据库+监控系统，擅长指标采集与查询 |
| **Grafana** | 可视化面板工具，对接 Prometheus 等数据源展示监控图 |
| **ELK Stack** | Elasticsearch+Logstash+Kibana，集中式日志收集分析方案 |
| **日志分级** | DEBUG/INFO/WARN/ERROR，按需输出，平衡信息量与性能 |
| **结构化日志** | 用 JSON 等格式记录日志，便于机器解析与聚合分析 |

## 八、云原生与云平台
| 知识点 | 一句话解释 |
|--------|-----------|
| **云原生** | 基于容器、微服务、DevOps 等构建弹性、可扩展的云应用方法论 |
| **公有云** | AWS/Azure/GCP 等提供按需使用的计算/存储/网络资源 |
| **私有云** | 企业自建云基础设施，满足合规与数据本地化需求 |
| **混合云** | 公有云+私有云组合，平衡灵活性与安全性 |
| **Serverless** | 无需管理服务器，按函数执行付费，极致弹性 |
| **Service Mesh** | 微服务通信基础设施（如 Istio），处理流量管理、安全、可观测性 |
| **云成本优化** | 通过预留实例、自动伸缩、资源标签等控制云支出 |

## 九、安全与合规（DevSecOps）
| 知识点 | 一句话解释 |
|--------|-----------|
| **DevSecOps** | 将安全实践左移，嵌入开发全流程，而非最后补救 |
| **静态应用安全测试（SAST）** | 扫描源代码发现安全漏洞（如 SQL 注入、硬编码密钥） |
| **动态应用安全测试（DAST）** | 运行中测试应用，模拟攻击发现运行时漏洞 |
| **依赖扫描** | 检查第三方库已知漏洞（如使用 Trivy、Snyk） |
| **密钥管理** | 用 Vault、云 KMS 等工具安全存储与轮换敏感凭证 |
| **最小权限原则** | 用户/服务仅授予完成任务所需的最小权限 |
| **合规即代码** | 用策略引擎（如 OPA）自动校验资源是否符合安全规范 |

## 十、测试策略（质量内建）
| 知识点 | 一句话解释 |
|--------|-----------|
| **测试金字塔** | 单元测试（多）→ 集成测试（中）→ E2E 测试（少），平衡速度与覆盖 |
| **单元测试** | 验证单个函数/类逻辑，快速执行，是 CI 的基石 |
| **集成测试** | 验证模块间交互（如 API+数据库），发现接口问题 |
| **端到端测试** | 模拟真实用户操作全流程，覆盖系统级行为 |
| **契约测试** | 验证服务间接口约定（如 OpenAPI），保障微服务协作 |
| **混沌工程** | 主动注入故障（如断网、杀进程），验证系统韧性 |
| **测试数据管理** | 用脱敏/生成/隔离策略，保障测试有效性与数据安全 |

## 十一、发布与部署策略
| 知识点 | 一句话解释 |
|--------|-----------|
| **蓝绿部署** | 准备两套环境，切换流量实现零停机发布，回滚快 |
| **金丝雀发布** | 先让小部分用户使用新版本，验证稳定后再全量 |
| **滚动更新** | 分批替换旧实例，平衡发布速度与系统可用性 |
| **特性开关** | 用配置控制功能可见性，实现代码部署与功能发布解耦 |
| **回滚策略** | 发布失败时快速恢复旧版本，减少故障影响时间 |
| **部署窗口** | 选择业务低峰期发布，降低变更风险 |
| **变更审批** | 关键发布需人工确认，平衡自动化与风险控制 |

---

## 十二、可观测性（超越监控）
| 知识点 | 一句话解释 |
|--------|-----------|
| **可观测性三支柱** | 指标（Metrics）、日志（Logs）、链路追踪（Traces）互补分析 |
| **分布式追踪** | 记录请求跨服务调用路径（如 Jaeger），定位性能瓶颈 |
| **指标类型** | 计数器（累计值）、仪表盘（瞬时值）、直方图（分布），按需选用 |
| **日志聚合** | 集中收集多节点日志，支持全文检索与关联分析 |
| **根因分析** | 结合多源数据快速定位故障源头，缩短平均修复时间（MTTR） |
| **SLO/SLI** | 服务等级目标/指标，用数据定义"系统是否健康" |
| **错误预算** | 允许的创新空间，平衡稳定性与迭代速度 |

---

## 十三、站点可靠性工程（SRE）
| 知识点 | 一句话解释 |
|--------|-----------|
| **SRE** | 用软件工程方法解决运维问题，是 DevOps 的一种实现 |
| **错误预算策略** | 预算耗尽时暂停发布专注稳定性，平衡创新与可靠 |
| **故障演练** | 定期模拟故障（如 Chaos Mesh），提升团队应急能力 |
| **事后复盘（Postmortem）** | 无责分析故障根因，输出改进项，避免重复犯错 |
| **自动化运维** | 用代码替代重复手工操作，释放人力聚焦高价值工作 |
| **容量规划** | 基于历史数据预测资源需求，避免过载或浪费 |
| **变更管理** | 标准化发布流程，记录、评审、回滚机制完备 |

## 十四、协作与自动化文化
| 知识点 | 一句话解释 |
|--------|-----------|
| **跨职能团队** | 开发、测试、运维融合协作，减少交接损耗 |
| **聊天机器人集成** | 在 Slack/钉钉中触发部署、查询状态，提升协作效率 |
| **文档即代码** | 用 Markdown+Git 管理文档，与代码同步更新 |
| **知识库沉淀** | 记录常见问题、解决方案，降低新人学习成本 |
| **自动化测试门禁** | 代码合并前必须通过测试，保障主干质量 |
| **价值流映射** | 可视化需求从提出到上线的全流程，识别瓶颈环节 |
| **持续学习文化** | 鼓励试错、分享、复盘，团队能力持续进化 |

## 🎯 初学者学习路径建议（6 个月计划）

```
第 1-2 月：打基础
✅ 掌握 Git 分支协作 + 命令行操作
✅ 学会写 Dockerfile + docker compose 跑多容器应用
✅ 用 GitHub Actions 实现"代码提交→自动构建"

第 3-4 月：建流程
✅ 用 Terraform 创建一台云服务器
✅ 搭建 Jenkins/GitLab CI 流水线：构建→测试→部署
✅ 配置 Prometheus+Grafana 监控一个简单服务

第 5-6 月：练实战
✅ 用 K8s 部署一个微服务应用（含 ConfigMap/Service）
✅ 实现蓝绿发布或金丝雀发布流程
✅ 编写简单混沌实验，验证服务自愈能力
```

---

✅ **DevOps 核心心法**：  
🔹 文化 > 流程 > 工具：先改变协作方式，再选合适工具  
🔹 自动化不是目的，**快速反馈**才是  
🔹 小步提交、频繁集成，问题越早发现成本越低  
🔹 监控不是为了"看"，是为了**驱动决策**  
🔹 安全与质量必须"左移"，嵌入开发早期  

> 💡 建议：每学一个知识点，**立刻动手实践**（哪怕只是本地跑通一个 demo）。DevOps 是"做"出来的，不是"看"出来的。
