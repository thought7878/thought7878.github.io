## 一、 架构基础与设计原则
> **学习重点**：建立系统思维，掌握架构选型依据与可维护性基石。

| 子领域 | 核心知识点 | 设计要点/关键技术 |
|:---|:---|:---|
| **架构模式** | SPA / MPA / SSR / SSG / ISR | 根据内容更新频率、SEO需求、首屏体验选择渲染范式 |
|  | 微前端架构 | 模块联邦(Module Federation)、qiankun、独立部署与状态隔离 |
|  | 边缘架构(Edge-First) | 边缘函数渲染、就近计算、降低中心源站压力 |
| **设计原则** | SOLID / DRY / KISS / YAGNI | 指导代码结构拆分与抽象边界划定 |
|  | 关注点分离 & 单一职责 | UI、逻辑、数据、样式解耦，便于测试与替换 |
|  | 渐进增强 & 优雅降级 | 保证基础功能可用，高级特性按环境按需加载 |
| **技术选型** | 框架评估矩阵 | 生态成熟度、渲染能力、TS支持、团队学习曲线、社区活跃度 |
|  | 依赖治理策略 | 锁定版本、语义化版本控制、定期依赖审计与升级策略 |

## 二、 组件与UI系统设计
> **学习重点**：构建可复用、可配置、高内聚的UI原子库与业务组件。

| 子领域 | 核心知识点 | 设计要点/关键技术 |
|:---|:---|:---|
| **组件模型** | 无状态(Presentational) vs 有状态(Container) | 逻辑与视图分离，提升测试性与复用度 |
|  | 受控组件 / 非受控组件 | 表单数据流向设计，避免状态不同步 |
|  | 复合组件 / 渲染道具 / 插槽模式 | 灵活组合UI结构，避免Props膨胀 |
| **设计系统** | 原子设计理论(Atomic Design) | Token → 原子 → 分子 → 组织 → 模板 → 页面 |
|  | 样式隔离与主题化 | CSS Modules / CSS-in-JS / Tailwind / 动态Theme切换 |
|  | 无障碍设计(A11y) | 语义化标签、ARIA属性、键盘导航、对比度校验 |
| **交互与动效** | 性能安全动画 | `transform`/`opacity` 优先、`will-change`、`requestAnimationFrame` |
|  | 手势与复杂交互抽象 | 事件委托、防抖/节流、交互状态机设计 |
| **最佳实践** | 组件API设计 | 显式Props、默认值、类型约束、扩展插槽、向后兼容 |
|  | 逻辑复用模式 | Custom Hooks / Composables / Mixins、关注点横向抽离 |

## 三、 状态管理与数据流设计
> **学习重点**：厘清数据流向，区分UI状态与服务端状态，设计可预测的数据流。

| 子领域 | 核心知识点 | 设计要点/关键技术 |
|:---|:---|:---|
| **状态分类** | UI状态 / 业务状态 / 全局状态 / 服务端缓存状态 | 按需划分作用域，避免全局状态污染 |
|  | 瞬时状态 vs 持久状态 | 路由/表单临时数据 vs LocalStorage/IndexedDB |
| **管理模式** | 集中式状态机 | Flux/Redux单向流、不可变数据、中间件扩展(Thunk/Saga) |
|  | 响应式/信号模式 | MobX、Vue Reactivity、Signals、自动依赖追踪 |
|  | 轻量级状态库 | Zustand、Pinia、Context+useReducer |
| **数据流模式** | 单向数据流 / 双向绑定 | 明确数据源，降低调试复杂度 |
|  | 响应式流编程 | RxJS/异步事件流、合并/过滤/重试操作符 |
| **服务端状态同步** | 缓存与请求抽象 | TanStack Query / SWR / VueUse、Stale-While-Revalidate策略 |
|  | 乐观更新 & 错误回滚 | 提升交互体验，配合后端幂等接口保障数据一致性 |
|  | 分页/虚拟列表/无限滚动 | 数据切片加载、DOM复用、滚动位置记忆 |
| **持久化与同步** | 离线存储策略 | IndexedDB封装、LocalStorage限容、跨Tab通信(BroadcastChannel) |
|  | 离线队列与重试 | Background Sync、请求队列、网络恢复自动同步 |

## 🗺️ 循序渐进学习路径建议

| 阶段 | 学习目标 | 推荐实践项目 |
|:---|:---|:---|
| **L1 基础构建** | 掌握组件拆分原则、状态分类、基础渲染模式 | 设计一个支持主题切换、表单验证、路由懒加载的后台管理系统 |
| **L2 数据与性能** | 实现服务端状态缓存、代码分割、Core Web Vitals达标 | 开发一个支持分页/搜索/乐观更新的电商商品列表，接入性能监控 |
| **L3 工程与安全** | 搭建CI/CD流水线、分层测试、CSP/错误边界防护 | 将项目接入Monorepo，配置Playwright E2E，实现权限路由与降级策略 |
| **L4 架构与协同** | 设计BFF层交互、微前端拆分、边缘渲染/流式水合 | 拆分主站与子应用，接入网关/CDN缓存策略，实现SSR+部分水合架构 |

> 💡 **系统设计核心心法**：
> 1. **先画边界，再定协议**：明确前端管什么、后端管什么，用契约约束交互。
> 2. **数据流比UI更重要**：状态混乱是系统崩溃的根源，坚持单向流与单一数据源。
> 3. **性能是设计出来的，不是优化出来的**：在架构选型阶段就考虑加载策略、缓存层级与渲染范式。
> 4. **可观测性即生产力**：没有监控的系统等于黑盒，日志、指标、链路必须贯穿全生命周期。

本框架可作为你的知识地图。建议按阶段逐项攻克，每掌握一个子领域即输出**设计文档(ADR)**或**架构图**，逐步培养系统化设计思维。
