## 一、基础概念与架构原则
| 知识点            | 一句话解释                                             |
| -------------- | ------------------------------------------------- |
| `前端系统设计`       | *规划应用结构、技术选型、数据流与协作方式的系统化方法*，**目标**是*可扩展、可维护、高性能* |
| **关注点分离（SoC）** | 将 UI、业务逻辑、数据请求、样式拆分，降低模块间耦合                       |
| **单一职责原则**     | 每个组件/模块只负责一件事，便于测试、复用与替换                          |
| **开闭原则**       | 对扩展开放，对修改关闭，通过抽象接口适应新需求                           |
| 架构权衡           | 在性能、开发效率、可维护性、团队技能间做合理取舍，无银弹方案                    |
| 渐进增强 vs 优雅降级   | 优先保障核心功能在基础环境可用，再叠加高级特性或兼容旧浏览器                    |
| 声明式 vs 命令式     | 声明式描述“期望状态”，框架自动更新 DOM；命令式手动操作 DOM，易出错难维护         |

参考：[[关注点分离（单页应用）]]

## 二、组件化与模块化
| 知识点                     | 一句话解释                                             |
| ----------------------- | ------------------------------------------------- |
| **组件**                  | 独立可复用的 UI 单元，包含视图、逻辑与样式，支持参数化配置                   |
| **模块化**                 | 将代码按功能拆分为独立文件/包，通过导入导出管理依赖与暴露接口                   |
| **组件通信**                | 父子通过 props/事件、兄弟通过状态共享/事件总线、跨级通过 Context/Provider |
| **受控 vs 非受控组件**         | 表单值由父组件状态管理（可控）vs 由 DOM 自身管理（简单但难同步）              |
| **原子设计理论**              | 按原子→分子→组织→模板→页面层级构建 UI，提升设计一致性与复用率                |
| **设计系统（Design System）** | 统一组件库、设计规范、交互标准与文档的集合，保障多端体验一致                    |
| **组件懒加载**               | 按需动态导入组件，减少首包体积，提升非关键路径加载速度                       |

## 三、状态管理
| 知识点                  | 一句话解释                                       |
| -------------------- | ------------------------------------------- |
| **本地状态**             | *组件内部临时数据*（如展开收起、输入框值），*生命周期与组件绑定*          |
| **全局状态**             | *跨组件共享数据*（如用户信息、主题、购物车），需*集中管理避免混乱*         |
| **单向数据流**            | *数据自上而下传递*，变化通过明确路径触发，便于调试与追踪来源             |
| **不可变数据（Immutable）** | 状态更新时创建新对象而非修改原对象，*简化变更检测与性能优化*             |
| **状态分层**             | 按作用域分为 UI 状态、业务状态、服务器缓存状态，分别用不同方案管理         |
| **乐观更新 vs 悲观更新**     | 先假设成功更新 UI 再请求（体验快）vs 等请求成功再更新（数据稳）         |
| **状态持久化**            | 将关键状态存入 localStorage/IndexedDB，刷新页面不丢失用户上下文 |

## 四、路由与导航
| 知识点 | 一句话解释 |
|--------|-----------|
| **客户端路由** | 不刷新页面，通过 History API 或 Hash 切换视图，实现 SPA 体验 |
| **路由懒加载** | 按路径拆分代码包，访问对应路由时才下载模块，优化首屏 |
| **路由守卫** | 导航前后拦截处理，用于权限校验、页面埋点、加载提示等 |
| **动态路由** | 路径含参数（如 `/user/:id`），匹配后提取参数渲染差异化内容 |
| **嵌套路由** | 页面内部再划分子路由，实现复杂布局与功能模块化 |
| **路由模式选择** | History 需服务端配置 fallback，Hash 兼容性好但 URL 含 `#` 影响 SEO |
| **路由状态同步** | 将查询参数（`?tab=1`）与 URL 绑定，支持分享链接与刷新保留状态 |

## 五、性能优化
| 知识点                      | 一句话解释                                        |
| ------------------------ | -------------------------------------------- |
| **关键渲染路径**               | 浏览器从解析 HTML 到绘制首屏的阻塞链，优化目标是缩短该路径             |
| **代码分割（Code Splitting）** | 将大文件拆分为小块，配合路由/组件按需加载，减少初始下载量                |
| **资源压缩与合并**              | JS/CSS/图片压缩，减少请求体积与 HTTP 连接数                 |
| **缓存策略**                 | 结合强缓存、协商缓存、CDN 边缘缓存与 Service Worker 降低重复请求   |
| **渲染优化**                 | 虚拟 DOM 差异更新、防抖节流、避免强制同步布局、合理使用 `will-change` |
| **图片优化**                 | 采用 WebP/AVIF、响应式 `srcset`、懒加载、占位图，降低带宽消耗     |
| **核心指标**                 | FCP（首次内容绘制）、LCP（最大内容绘制）、CLS（布局偏移）、INP（交互延迟）  |

## 六、网络与数据交互
| 知识点                | 一句话解释                                    |
| ------------------ | ---------------------------------------- |
| **API 设计规范**       | RESTful/GraphQL 约定，状态码语义化，版本控制，降低前后端联调成本 |
| **请求封装层**          | 统一拦截器处理 Token 注入、错误重试、加载状态、请求取消，避免散乱调用   |
| **数据缓存策略**         | 内存缓存（当前会话）、HTTP 缓存（静态资源）、本地存储（持久数据）分层使用  |
| **错误边界与降级**        | 网络失败或组件崩溃时展示友好提示，避免白屏或阻断核心流程             |
| **批量请求与合并**        | 减少并发连接数，避免浏览器限制与服务器压力，提升整体吞吐量            |
| **数据转换层（Adapter）** | 将后端原始数据映射为前端视图模型，隔离 API 变更对 UI 的影响       |
| **实时通信**           | WebSocket/SSE 替代轮询，实现低延迟消息推送与状态同步        |

## 七、安全与防护
| 知识点             | 一句话解释                                                      |
| --------------- | ---------------------------------------------------------- |
| **XSS 防护**      | 恶意脚本注入页面，防范靠输入过滤、输出编码、严格 CSP 策略                            |
| **CSRF 防护**     | 诱导用户执行非预期操作，防范靠 Anti-Token、SameSite Cookie 属性              |
| **内容安全策略（CSP）** | 声明可加载资源白名单，阻断内联脚本与恶意外链，提升纵深防御                              |
| **敏感数据保护**      | Token 存 HttpOnly Cookie，避免明文存 localStorage，全站强制 HTTPS      |
| **依赖安全**        | 定期扫描 npm 包漏洞，锁定版本，避免引入供应链攻击                                |
| **点击劫持防护**      | 通过 `X-Frame-Options` 或 CSP `frame-ancestors` 禁止恶意嵌套 iframe |
| **前端权限控制**      | 路由级、组件级、按钮级校验，前端拦截提升体验，后端兜底保障安全                            |

## 九、测试策略
| 知识点             | 一句话解释                                    |
| --------------- | ---------------------------------------- |
| **单元测试**        | 验证独立函数/组件逻辑，执行快，覆盖核心算法与边界条件              |
| **集成测试**        | 验证模块间交互（如组件+状态+路由），发现衔接与数据流问题            |
| **E2E 测试**      | 模拟真实用户操作全流程（Playwright/Cypress），保障核心业务链路 |
| **快照测试**        | 记录组件渲染结构，防意外 UI 变更，适合设计系统维护              |
| **测试覆盖率**       | 衡量代码被测试执行的比例，重核心链路质量而非盲目追求 100%          |
| **Mock 与 Stub** | 模拟网络请求/外部依赖，隔离不稳定环境，提升测试可重复性             |
| **可视化回归测试**     | 对比 UI 像素差异，自动捕获样式与布局意外变化                 |

## 十、部署与运维
| 知识点          | 一句话解释                                                 |
| ------------ | ----------------------------------------------------- |
| **静态托管平台**   | Vercel/Netlify/GitHub Pages 提供零配置部署、自动 HTTPS 与 CDN 加速 |
| **Nginx 配置** | 路由 fallback、Gzip 压缩、缓存头设置、跨域代理、静态资源优化                 |
| **环境变量管理**   | 区分 dev/test/prod 配置，敏感信息不入代码，构建时安全注入                  |
| **灰度发布**     | 逐步放量新版本，监控核心指标稳定后再全量，降低发布风险                           |
| **资源指纹**     | 文件名加 Hash（`app.[hash].js`），配合长效缓存与自动失效更新              |
| **降级与容灾**    | 核心功能失败时展示备用内容或静态降级页，保障基础可用性                           |
| **快速回滚**     | 发布异常时一键切换至上一稳定版本，缩短故障影响时间（MTTR）                       |

## 十一、可访问性与国际化
| 知识点 | 一句话解释 |
|--------|-----------|
| **可访问性（a11y）** | 让视障/听障/行动不便用户也能使用，依赖语义化 HTML 与 ARIA 属性 |
| **屏幕阅读器兼容** | 提供 `alt`、`aria-label`、合理焦点管理，避免纯图片传达关键信息 |
| **色彩对比度** | 文字与背景对比度≥4.5:1，保障弱视用户可读性，避免仅靠颜色区分状态 |
| **国际化（i18n）** | 多语言支持，文案抽离为资源文件，运行时动态切换语言包 |
| **本地化（l10n）** | 适配地区习惯（日期、货币、数字格式、RTL 布局），提升地域用户体验 |
| **无障碍测试** | 使用 axe、Lighthouse 或真实辅助工具测试，验证合规性与可用性 |

## 十二、监控与可观测性
| 知识点           | 一句话解释                                |
| ------------- | ------------------------------------ |
| **错误监控**      | 捕获 JS 运行时错误、Promise 拒绝、资源加载失败，自动上报堆栈 |
| **性能监控（RUM）** | 采集真实用户指标（FCP/LCP/CLS），分析慢页面与地域/设备差异  |
| **行为埋点**      | 记录用户点击、停留、转化路径，驱动产品迭代与运营决策           |
| **结构化日志**     | 统一日志格式发送至后端平台，支持多维度检索、聚合与告警          |
| **告警机制**      | 错误率/性能阈值超限时自动通知相关人员，缩短故障响应时间         |
| **隐私合规**      | 埋点脱敏处理，遵守 GDPR/个保法，提供用户数据收集同意与退出选项   |

## 十三、架构模式与进阶实践
| 知识点                           | 一句话解释                               |
| ----------------------------- | ----------------------------------- |
| **SPA（单页应用）**                 | 单次加载，前端路由切换，体验流畅但首屏慢、SEO 弱          |
| **MPA（多页应用）**                 | 每次跳转刷新页面，SEO 友好，适合内容型/营销型站点         |
| **SSR（服务端渲染）**                | 服务器生成 HTML 返回，提升首屏速度与 SEO，增加服务器负载   |
| **SSG（静态站点生成）**               | 构建时预渲染所有页面，极致性能与 CDN 友好，适合内容更新不频繁场景 |
| **ISR/增量渲染**                  | 结合 SSG 与 SSR，按需后台更新静态页面，平衡性能与数据实时性  |
| **微前端**                       | 将大型应用拆为独立子应用，技术栈无关、独立开发部署，集成到主容器    |
| **BFF（Backend for Frontend）** | 后端为特定前端定制聚合接口，减少端侧数据处理逻辑，提升端体验      |

## 十四、团队协作与规范
| 知识点             | 一句话解释                                       |
| --------------- | ------------------------------------------- |
| **组件库共享**       | 内部 npm 包或私有 Registry 发布，统一 UI 与交互标准，减少重复造轮子 |
| **文档即代码**       | Storybook/MDX 自动生成组件文档与交互演示，降低沟通与上手成本       |
| **Code Review** | 合并前同行评审，保障代码质量、规范一致与知识在团队内流动                |
| **技术债管理**       | 定期评估、量化影响并排期重构，避免累积导致迭代停滞与团队倦怠              |
| **知识沉淀**        | 内部 Wiki、技术分享、故障复盘、最佳实践库，提升团队整体工程能力          |
| **技术选型评估**      | 对比社区生态、团队熟悉度、长期维护成本、业务匹配度，理性决策不追新           |

✅ **前端系统设计核心心法**：  
🔹 **没有完美架构，只有合适取舍**：根据团队规模、业务阶段、性能要求做选择  
🔹 **复杂度的本质是状态与交互**：控制状态流向、减少隐式依赖，系统自然清晰  
🔹 **性能是设计出来的，不是优化出来的**：首屏、缓存、分割应在架构期规划  
🔹 **安全与可访问性不是附加项**：默认纳入设计标准，后期补救成本极高  
🔹 **可观测性决定系统寿命**：无监控的线上环境等于盲人摸象，数据驱动迭代  

> 💡 给初学者的特别提醒：  
> 前端系统设计不是“背面试题”，而是**培养权衡思维**；  
> 不要追求“大而全”的技术栈，先跑通“小闭环”再逐步扩展；  
> 每学一个知识点，问自己三个问题：`解决什么问题？带来什么成本？何时该用/不该用？`
